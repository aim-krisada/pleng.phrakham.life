## 0 · ระดับโลกเป็นค่าเริ่มต้น

ทุกงาน UX/ฟีเจอร์ ต้องได้มาตรฐานสากล (Material · Apple HIG · WCAG 2.2 AA+) **โดยไม่ต้องมีใครสั่ง**

- **P'Aim ให้ content ภาษาคน · AI ออกแบบให้ถึงระดับโลกและเสนอสิ่งที่ดีที่สุด** — ไม่ใช่ทำตามสั่งแล้วจบ
- ของอ้างอิงต่ำกว่ามาตรฐาน → **ยกทั้งคู่ขึ้น** ไม่ดึงลงมาก๊อป
- **เปิด spec จริง วัดเอง ไม่อ้างจากความจำ**
- ⛔ **ห้ามถาม P'Aim ว่า "อะไรถูกตามมาตรฐาน"** — นั่นคือการบ้านของเรา · ถามได้แค่ *อยากได้อะไร / เพราะไหม / สวยไหม*
- ⚠️ **แต่ระดับโลก ≠ ขัดจนไม่ได้ส่ง** — P'Aim: *"manual tune ทุกอันไม่ไหว · ไม่ perfect ไม่เป็นไร"* งานที่ต้องจูนมือทีละชิ้นไม่ scale = ไม่ใช่ระดับโลก (ดู evidence §E9)

## 1 · โครงทีม (เปลี่ยน 20 ก.ค. 2026)

**เดิม:** AI ตัวเดียว แตกเป็นหลาย session (PM / SA / dev / tester)
**ตอนนี้:** **1 session = ครบทุกบทบาทในตัว** (วางแผน · ออกแบบ · เขียน · ทดสอบ) · มี **4 session รวม** = 2 โปรเจกต์ × 2 ค่าย (Claude Code = **[C]** · Antigravity = **[A]**)

**P'Aim = PM ทั้ง 2 โปรเจกต์** และเป็น**คนเดินสารระหว่าง [C] กับ [A]**

### 1.1 · ⛔ ขนาดงาน — ถ้าต้องส่งไม้ต่อ แปลว่าใหญ่เกินไป

> **1 งาน = จบได้ใน session เดียว โดยไม่ต้องเขียนเอกสารส่งไม้ต่อให้ใคร**
> ถ้าเริ่มรู้สึกว่าต้องสรุปสถานะให้คนถัดไปอ่าน = **หยุด แตกงานก่อน แล้วค่อยเริ่ม**

หลักฐาน: `board.md` บวมจนเปิดทีกิน **95,000 token** เพราะงานใหญ่เกิน session เดียวมาตลอด เลยต้องสรุปส่งต่อทุกครั้ง และทุกครั้งที่สรุป ก็มีของหาย

**scope เล็กแก้ได้จริง 3 อย่าง:** context ไม่เต็มกลางทาง · blast radius เล็กเอง · แก้ทีละอย่าง = รู้ว่าต้องเปิดของจริงตรงไหน
⚠️ **แต่แก้ไม่ได้ 1 อย่าง — งานเล็กที่คนเขียนตรวจเอง ก็ยังคือคนเขียนตรวจเอง** และความพลาดที่แพงที่สุดของเรา ("คานหาย 92%" · "`resize_window` พัง") ไม่ได้เกิดจากงานใหญ่ แต่เกิดจาก**วินิจฉัยผิดแล้วไม่มีใครค้าน** ⇒ ต้องมี §1.3 คู่กันเสมอ

### 1.2 · ⏸ คู่รับรอง (พักไว้ — ต้องมี 2 ค่าย)

> **1 session ทำได้ทุกอย่าง ยกเว้นรับรองงานของตัวเอง — คู่ของมันเป็นคนรับรอง**

เหตุผลไม่ใช่ทฤษฎี วัดมาแล้ว: **ผู้ตรวจอนุมัติ 7/7 ทันทีที่ถูกบอกว่า P'Aim อยากได้อะไร · อัตราอนุมัติขยับ 17% → 59% เพราะวิธีตั้งคำถามล้วน ๆ** (evidence §E1)
ตอนแยก tester เราได้ความเป็นอิสระนี้มาฟรี · พอยุบรวมแล้ว **ต้องเขียนเป็นกฎ ไม่งั้นหายเงียบ**

**ตอนนี้ไม่มีคู่** → ใช้ §1.3 แทน · เวลาต้องการตาที่สองจริง ๆ ให้ **เปิด session ใหม่ที่ไม่รู้เรื่องงานนี้เลย แล้วส่งให้แค่ `REPRO`** — ไม่ใช่ความเป็นอิสระจากค่ายอื่น แต่เป็น**ความเป็นอิสระจากบริบท** ซึ่งเป็นตัวที่จับ "วัดผิดเรื่อง" ได้จริง

### 1.3 · ⭐ ตรวจที่ชั้นอื่นจากชั้นที่ลงมือทำ (กฎที่ทำงานแทนคู่รับรอง)

> **แก้ที่ `source` → ห้ามตรวจที่ `source`** ต้องไปดู `build-output` / `live-page` / ไฟล์ผลลัพธ์จริง

**ทั้ง 16 กรณีใน `evidence.md §E3` ตกกับดักเดียวกันหมด: ตรวจที่ชั้นเดียวกับที่ลงมือ** — grep ซอร์สแล้วสรุปเรื่องหน้าเว็บ · อ่านโค้ดที่ลบ metadata แล้วสรุปเรื่องไฟล์ mp3 · ดู exit code แล้วสรุปเรื่อง HTML
นี่คือสิ่งที่ตาที่สองเคยให้เราฟรี ๆ เพราะคนอื่นย่อมไม่ได้มองจากที่ที่เราเพิ่งพิมพ์ **และทำคนเดียวได้ ถ้าเขียนเป็นกฎ**

### กฎข้อรอง

- **ที่ตรงกัน = หลักฐานอ่อน** โดยเฉพาะถ้าทั้งคู่เป็น Claude (pl กับ pk รุ่นเดียวกัน เห็นตรงกันแทบไม่มีน้ำหนัก) · **ต่างค่ายเท่านั้นที่อิสระจริง**
- **ที่ต่างกัน = ของจริงที่ซื้อมา** ให้ P'Aim ดูตรงนั้นก่อน
- **ทั้งคู่บอก "ไม่แน่ใจ / ต้องให้คนดู" = คำตอบที่ถูก** ไม่ใช่ความล้มเหลว — หยุดแล้วส่งคน
- **ไม่ต้องเกรงใจกัน** งานของผู้ตรวจคือหักล้าง ไม่ใช่รับรอง · **เจอว่าอีกฝ่ายผิด = ทำงานสำเร็จ**

## 2 · ชั้นของหลักฐาน (layer) — ใครเซ็นได้บ้าง

ทุกเคลม **ต้องบอกในตัวประโยคเลยว่าวัดที่ชั้นไหน**
*"ไม่มีภาษากรีกในซอร์ส"* จริงและไร้ประโยชน์ · *"ไม่มีภาษากรีกบนหน้าเว็บ"* **ผิด (จริง ๆ มี 21%)** — คำเกือบเหมือนกัน แต่คนละเคลม

| ชั้น | ใครเซ็นได้ |
|---|---|
| `source` · `build-output` · `test` · `git` | [C] และ [A] ได้ทั้งคู่ |
| `live-page` | **[C] เท่านั้น** ([A] เปิดเบราว์เซอร์ไม่ได้) |
| `printed-pdf` · `audio` · `human-eye` | ⛔ **ไม่มี AI ตัวไหนเซ็นได้เลย — คนเท่านั้น** |

**ชั้นของคน = ต้องนัดคนไว้ก่อนเริ่มงาน ไม่ใช่ไปหาเอาตอนจบ**
แผนต้องระบุ **ใคร (P'Aim / พี่เปา) และเมื่อไหร่** ตั้งแต่ก่อนลงมือ · ที่ผ่านมารอดเพราะ P'Aim บังเอิญว่าง = **โชค ไม่ใช่ระบบ**
นัดไม่ได้ = **งานนี้ยังไม่พร้อมเริ่ม** ไม่ใช่ "เริ่มไปก่อนแล้วค่อยหาคน"

> วันเดียว (20 ก.ค.) มี **5 เรื่องที่ตัดสินได้ด้วยหูเท่านั้น และ 3 ใน 5 เครื่องวัดบอกว่าผ่าน** (evidence §E2)

## 3 · ⏸ ด่านออกแบบปิดตา (พักไว้ — ต้องมี 2 ค่าย)

> **ส่วนที่ยังใช้ได้แม้อยู่คนเดียว:** คำถาม 5 ข้อในขั้นที่ 2 — ตอบมันก่อนเขียนโค้ดทุกครั้ง โดยเฉพาะ **ข้อ 4 "กำลังสมมติอะไรที่ยังไม่ได้พิสูจน์"** ที่ห้ามว่าง · สิ่งที่พักคือการให้ AI 2 ตัวตอบแยกกันแล้วเทียบ

ของแพงที่สุดไม่ใช่โค้ดผิด — คือ **การวินิจฉัยผิดที่ไม่มีใครค้าน** ("คานหายตอนพิมพ์ 92%" เกือบสั่งรื้อระบบ 3-5 วัน แก้บั๊กที่ไม่มีจริง)
**ตรวจโค้ดจับไม่ได้ เพราะโค้ดยังไม่ได้เขียน** → ด่านต้องอยู่ก่อนหน้านั้น

### ⛔ ห้ามให้ AI ตัวหนึ่งอ่านคำตอบอีกตัวก่อนตอบ

ถ้าให้ตัวที่สองอ่านแผนตัวแรกแล้วรีวิว **จะได้เสียงสะท้อน ไม่ใช่ความเห็นที่สอง** · **P'Aim = ช่องทางเดียวที่ปิดตาได้จริง** ต้นทุน = paste 4 ครั้ง

**ใช้กับงานที่ต้องใช้วิจารณญาณ** (แผน · หาสาเหตุ · "แก้ถูกจุดไหม") **ไม่ใช่ทุกงาน** — เกณฑ์อยู่ §4

1. **P'Aim เขียนโจทย์ครั้งเดียว** ในบล็อก `## PROBLEM` (ภาษาคน ไม่ต้องแปลงเป็นสเปก)
2. **paste โจทย์เดียวกันให้ทั้ง 2 ตัว แยกกัน ห้ามบอกว่าอีกตัวว่าไง** → แต่ละตัวเขียน `## DESIGN-[C]` / `## DESIGN-[A]` ตอบ 5 ข้อ:
   1. **สาเหตุจริงคืออะไร** — และ**พิสูจน์ยังไง** (ไม่ใช่การให้เหตุผล)
   2. **จะทำอะไร** และ **จะไม่ทำอะไร** (กัน gold-plating)
   3. **จะพังตรงไหนได้** — ของเดิมอะไรเสี่ยงถอยหลัง
   4. 🔴 **กำลังสมมติอะไรที่ยังไม่ได้พิสูจน์** — **ช่องนี้ว่างไม่ได้**
   5. **อะไรที่ต้องให้คนดู/คนฟัง** — เตรียม URL + ขั้นตอนไว้เลย
3. **P'Aim paste ทั้ง 2 คำตอบให้ทั้ง 2 ตัว** → เขียน `## COMPARE-[C]` / `## COMPARE-[A]` **เฉพาะจุดที่ต่าง** · ตรงไหนอีกฝ่ายถูกกว่า **พูดออกมาตรง ๆ**
4. **P'Aim ตัดสิน** เขียน `## DECISION` แล้วค่อยเริ่มเขียนโค้ด

### เกณฑ์พร้อมสร้าง (ก่อนลงมือเขียนโค้ด ต้องครบ)

ทุกปุ่ม/ควบคุม (ชื่อ+id+หน้าที่) · ทุก state (ปกติ/ว่าง/error/loading/selected) · ทุก breakpoint + device matrix (Fold พับ-กาง · tablet · phone · desktop) · a11y (aria/คีย์บอร์ด/SR) · **ไฟล์ที่จะแตะ + แชร์ 2 เว็บไหม** · **ของเดิมมีอะไรแล้ว vs ใหม่จริง (เสนอเฉพาะส่วนต่าง)**
ขาดช่องใด = **ยังไม่พร้อม** · ครบรอบเดียวเร็วกว่าวนแก้

## 4 · ⭐ Blast radius — ตัดสินว่างานไหนข้ามการตรวจซ้ำได้

**ไม่ใช่ทุกงานต้องผ่าน 2 ตัว แต่ "ขนาดงาน" เป็นเกณฑ์ที่ใช้ไม่ได้** — เพราะคนประเมินขนาดคือคนเดียวกับที่อยากข้ามการตรวจ ([A] ยอมรับข้อนี้ 20 ก.ค.)

**ถามตัวเอง 4 ข้อ:**
1. **แตะของกี่ชิ้น** (CSS ค่าเดียว vs 1,679 ข้อพระคัมภีร์)
2. **ย้อนกลับได้ไหม** (แก้สี vs เขียนทับไฟล์ต้นฉบับ)
3. **ต้องใช้ชั้นไหนตรวจ** — ถ้ามี `audio` / `human-eye` = **ไม่มีวันเป็นงานเล็ก** ไม่มี AI ตรวจได้เลย
4. **ถ้าผิดแล้วมองเห็นไหม** — ⚠️ **มองไม่เห็น = radius ใหญ่ขึ้น** ตัวเลขผิดในข้อที่ไม่มีใครเปิดอยู่ได้เป็นปี · layout พังโดนแจ้งวันเดียวกัน

⇒ **งานเล็กจริง = แตะน้อย + ย้อนได้ + AI ตรวจได้ + ผิดแล้วเห็น** ครบ 4 ข้อถึงข้ามได้
ตัวอย่าง: แก้ CSS ค่าเดียว = ผ่านครบ 4 → ข้ามได้ · regex บรรทัดเดียวบนคลังพระคัมภีร์ = ตก 3 ข้อ → **ห้ามข้าม**

## 5 · รูปแบบการส่งงาน (CLAIM / VERDICT / HUMAN)

**1 งาน = 1 ไฟล์** `pair/<task-id>.md`

| บล็อก | ใครเขียนได้ |
|---|---|
| `## CONTEXT` · `## PROBLEM` | ผู้เปิดงาน / P'Aim |
| `## DESIGN-[C]` `## COMPARE-[C]` `## CLAIM-<n>` | **[C] เท่านั้น** |
| `## DESIGN-[A]` `## COMPARE-[A]` `## VERDICT-<n>` | **[A] เท่านั้น** |
| `## HUMAN-<n>` · `## DECISION` | **คนเท่านั้น** |
| `## OPEN` | ทั้งคู่ (ต่อท้าย ลงชื่อกำกับ) |

**⛔ ห้ามแก้บล็อกของอีกฝ่ายเด็ดขาด** ไม่เห็นด้วย = เขียนในบล็อกตัวเอง · **ต่อท้ายอย่างเดียว ไม่ลบของเก่า**
**⛔ ห้ามย่อความของคนอื่น** — สรุปได้เฉพาะตาของตัวเอง · **หลักฐานคือของที่ต้องเก็บ ไม่ใช่ข้อสรุป** (เกิดจริง 20 ก.ค.: ลบเหตุผล 26 บรรทัดทิ้งทั้งที่บันทึกข้อสรุปถูก)

```markdown
## CLAIM-1
**CLAIM:** <ข้อเดียว ประโยคเดียว ต้องผิดได้ถ้ามันผิด>
**LAYER:** source | build-output | live-page | printed-pdf | audio | human-eye
**REPRO:**
```sh
<คำสั่งที่ copy ไปวางรันได้เลย ตั้งแต่ cd จนถึงบรรทัดที่ให้ตัวเลข>
```
**EXPECT:** <ตัวเลข/ข้อความเป๊ะ ๆ ที่ต้องเห็น>
**ACTUAL (self-run):** <output จริง วางมาดิบ ๆ>
**STATUS:** self-verified (layer=<ชั้น>)
**ยังไม่ได้พิสูจน์:** <ห้ามว่าง — ว่าง = ยังไม่เสร็จ>
```

**⏸ บล็อก `VERDICT` + เพดาน 3 รอบ = พักไว้** (ต้องมีผู้ตรวจอิสระ) · **บล็อก `CLAIM` ใช้ต่อเต็มรูปแบบ** — มันคือกลไกที่ทำให้เคลมรันซ้ำได้ ซึ่งเป็นสิ่งที่จับความผิดพลาดได้จริง ไม่ใช่ตัวผู้ตรวจ

⛔ **`ACTUAL` ต้องก๊อปจาก terminal ทั้งก้อน ห้ามพิมพ์ซ้ำ ห้ามเรียบเรียง ห้ามตัดส่วนหัว** — ตัดตรงไหนใส่ `[...]` · อะไรที่ไม่ได้มาจาก terminal ให้ไปอยู่ `หมายเหตุ`
*(เกิดจริง 20 ก.ค.: [A] แปะตาราง `--stat` ที่ถูกเป๊ะทุกตัวเลข ใต้ส่วนหัว commit ที่มันเขียนขึ้นเอง — hash ที่อ้างไม่มีอยู่ใน repo · **"output ดิบ" ที่ครึ่งจริงครึ่งแต่ง คนอ่านแยกไม่ออก และครึ่งที่แต่งคือส่วนที่บอกว่าตรวจของชิ้นไหน**)*

```markdown
## VERDICT-1
**RAN:** <คำสั่งที่รันจริง ถ้าต่างจาก REPRO บอกว่าต่างตรงไหนและทำไม>
**ACTUAL (independent run):** <output จริง วางมาดิบ ๆ ห้ามสรุปเป็นคำพูด>
**MATCH:** yes | no | inconclusive
**STATUS:** cross-verified | refuted | blocked
```

**คำที่ห้ามใช้:** ⛔ **"ผ่าน / เสร็จ / แก้แล้ว / ใช้ได้แล้ว"** ถ้าไม่ได้เปิดของจริงดู — เขียนได้สูงสุดแค่ `self-verified (layer=…)`
**`cross-verified` สงวนไว้ให้ผู้ตรวจอิสระเท่านั้น** · **`blocked` เป็นคำตอบที่ยอมรับได้** ("รันไม่ได้เพราะต้องล็อกอิน" ดีกว่าเดาแล้วเซ็น)

**ขัดกัน = ห้ามวนเถียง** — รันซ้ำได้คนละ 1 ครั้ง ยังไม่ตรง → **หยุด ส่ง P'Aim พร้อม output ทั้ง 2 ฝั่ง** ห้ามหาข้อสรุปกันเอง ห้ามประนีประนอม
**งานจบเมื่อ:** ทุก CLAIM มี VERDICT `cross-verified` **และ** ทุกชั้นของคนมี `## HUMAN` แล้ว
**เพดาน 3 รอบต่อหัวข้อ** ตกลงกันไม่ได้ = แต่ละฝ่ายเขียน 1 ย่อหน้า P'Aim ตัดสิน · **ตกลงกันได้ก่อน = หยุดก่อนได้เลย**

## 6 · ⛔⛔ verify ที่ "ของจริง" ไม่ใช่ที่ "สัญญาณ"

> **เคลมอะไรก็ตามเกี่ยวกับสิ่งที่ผู้ใช้เห็น/ได้ยิน → ต้องเปิดของที่ผู้ใช้เห็นดู**
> exit code · timestamp · ค่าใน config · diff · ผลที่ cache ไว้ — **ทั้งหมดคือ "รายงานเกี่ยวกับ" ของจริง และถูกจับได้แล้วว่าโกหกทุกตัว**

**16 กรณีจริงในวันเดียว ไม่มีสักอันที่จับได้ด้วยการอ่านโค้ด/config/diff** → ตารางเต็มอยู่ [`evidence.md §E3`](evidence.md)

**กฎลูกที่ต้องจำ (แต่ละข้อมาจากกรณีจริงคนละแบบ):**
1. **เครื่องมือที่พังเงียบ อันตรายกว่าพังดัง** · **gate ที่ผ่านทั้งที่ของหาย แย่กว่าไม่มี gate** เพราะคนเชื่อมัน
2. **ลบฟีเจอร์ยังไม่จบตอนลบโค้ด** — ต้องเปิดไฟล์ผลลัพธ์ดู ของที่ลบแล้วรอดมาบ่อยกว่าที่คิด
3. **migration ที่สำเร็จแต่ย้าย 0 รายการ = ล้มเหลว ไม่ใช่ no-op** — ทุก migration/backfill ต้องยืนยันว่าย้ายได้จำนวนที่คาดไว้และไม่ใช่ศูนย์
4. **ผ่านเทสต์ 1 รอบ ไม่ใช่ข้อพิสูจน์ของ concurrency** — ต้องทดสอบการชนมากกว่า 1 ระดับ และ **พิสูจน์ตัวกันด้วยการจงใจทำพังดู**
5. **ค่าที่ cache ไว้ ดีเท่ากับสมมติฐานที่ผลิตมันเท่านั้น** — คำนวณใหม่จากของจริงถ้าถูก · ถ้าแพงให้เขียนสมมติฐานกำกับไว้
6. **เทสต์ว่า "ตัวสร้าง" ผลิต X ไม่ใช่หลักฐานว่า "ไฟล์ที่ส่งมอบ" มี X** — แก้ตัวสร้างแล้วต้องรันใหม่ แล้วถามคำถามน่าเบื่อ: *"แล้วไฟล์จริงเปลี่ยนจริงไหม"*
7. **คำบ่นที่พูดถึง "ส่วนหนึ่ง" ห้ามตอบด้วยค่าเฉลี่ยของทั้งก้อน** — ×1.03 "เฉย ๆ" ถูกทางเลข แต่ผิดทางหู เพราะเฉลี่ยข้ามรอยต่อ · **บ่นถึงส่วนไหน วัดส่วนนั้น**
8. **พิสูจน์ว่าไม่ได้ทำพัง ≠ พิสูจน์ว่ามันทำงาน** — เทสต์ถูกและครบในเรื่องที่มันคุม แต่เรื่องที่มันคุมไม่ใช่เรื่องที่สำคัญ (ข้อนี้ P'Aim จับได้เอง คมที่สุด)
9. **⭐ เคลมที่เข้าข้างเรา คือเคลมที่ต้องตรวจหนักที่สุด** — คำอธิบายที่บอกว่าความล้มเหลวของเรา "เป็นเรื่องโครงสร้าง ไม่ใช่วินัยเรา" มาในรูปน้ำใจจากเพื่อน · ของจริง: อธิบายได้ 1 ใน 3 อีก 2 คือวินัยล้วน ๆ (evidence §E4)
10. **ห้าม relay คำว่า "ทำไม่ได้" ของคนอื่นโดยไม่ลองเอง** — agent บอก "เครื่องมือพัง" ลองเอง = ใช้ได้ปกติ (แค่ลืมพารามิเตอร์)
11. **บ่นเรื่องเดิมซ้ำครั้งที่ 2 = แก้ผิดจุด ไม่ใช่บั๊กใหม่** — หา root cause เดียว อย่าปะทีละใบ

## 7 · ไฟล์ · branch · การไม่ชนกัน

- **จองไฟล์ก่อนแก้** — ทุก session เขียนใน `board.md` ว่าจะแตะไฟล์ไหน · **ซ้ำกับสายอื่น = หยุด ถาม P'Aim** (เคยเกิดจริง 2 สายแก้ไฟล์เดียวกัน ต้อง rebase แก้)
- **`main` มีเจ้าของคนเดียวต่อโปรเจกต์** เขียนชื่อไว้ใน `board.md` · ⛔ **[A] ห้าม commit ลง `main` ของโปรเจกต์ไหนก็ตาม** — ทำงานบน branch ของตัวเองเสมอ (เกิดจริง 20 ก.ค.: [A] commit ลง phrakham `main` ตอนที่อีกฝ่ายถือ lock อยู่ · ตั้งใจดี แต่กฎต้องชัด)
- **pleng `main` = auto-deploy** merge ผิด = ขึ้นเว็บทันที
- **1 งาน = 1 branch = 1 worktree** · ไฟล์คนละไฟล์ = ทำขนานได้ · ไฟล์ที่แชร์ (`_quarto.yml` · theme · `pk-*.js` · `webdata`) = **ต้องเข้าคิว**
- **⛔ ขั้นตอนที่ 0 ก่อนคำสั่ง git แรก — ยืนยันว่า worktree มีอยู่จริง** (เกิด 3 ครั้งในวันเดียว)
  ```sh
  git rev-parse --show-toplevel    # ต้องเท่ากับ cwd ของตัวเอง
  git rev-parse --git-common-dir   # worktree จริงต้องต่างจาก .git
  ```
  `git branch --show-current` **จับข้อนี้ไม่ได้** — repo ผิดตัวก็ตอบได้อย่างมั่นใจ · **ต้องเช็กใหม่ทุกครั้งหลัง `cd`** (cwd ของ Bash ค้างข้ามคำสั่ง)
- **ของใช้ร่วม 2 เว็บ มีเจ้าของคนละฝั่ง** — ⛔ **ต้องได้รับการ*ยืนยัน*จากทั้ง 2 โปรเจกต์ก่อน deploy ไม่ใช่แค่*แจ้งให้ทราบ*** (ข้อเสนอของ [A] 20 ก.ค. · ของเดิมเขียนแค่ "แจ้ง board" ซึ่งอ่อนเกินไป — แจ้งแล้วไม่มีใครตอบก็ deploy ได้)
  - **`DockKey.vue` — เพลงเป็นเจ้าของ** พระคำยืมใช้
  - **`assets/pk-audio.js` — พระคำเป็นเจ้าของ** ⛔ ห้ามแก้ตรง ขอผ่านฝั่งพระคำเท่านั้น
- **commit เจาะจงไฟล์** (`git add <file>`) · เช็ก `git branch --show-current` ก่อน commit ทุกครั้ง
- ⛔ **ห้าม deploy จนกว่า P'Aim สั่ง "go"** · **go มีอายุแค่รอบนั้น** ไม่ใช่ใบอนุญาตถาวร

## 8 · รายงาน — 3 ช่อง ขาดไม่ได้

ทุกงาน ทุก session ทุกใบ:
1. **แก้อะไร** — diff/commit จริง
2. **พิสูจน์ยังไง** — คำสั่งที่ copy ไปรันซ้ำได้ + **output ดิบ** (ไม่ใช่ "ตรวจแล้วถูก")
3. **อะไรที่ยังไม่ได้พิสูจน์** — **ว่างไม่ได้ · ว่าง = งานยังไม่เสร็จ**

**รายงานเอง ไม่ต้องให้ใครตาม** — เสร็จหรือติดปัญหาเมื่อไหร่ เขียนรายงาน + แจ้ง board ทันที · **session ที่เสร็จแล้วเงียบ = ถือว่ายังไม่เสร็จ**
**แก้ซอฟต์แวร์ที่ผู้ใช้เห็น = ต้องอัปคู่มือผู้ใช้ในงานเดียวกัน** (เพลง: `src/views/Guide.vue`) ไม่อัป = ยังไม่เสร็จ

## 9 · ไม่แน่ใจ = ห้ามเดา (แยกให้ถูก 2 แบบ)

- **ไม่แน่ใจว่า P'Aim อยากได้อะไร / ขอบเขตแค่ไหน / จะเปลี่ยนพฤติกรรมเดิมไหม** → **หยุด ถามก่อนลงมือ**
- **ไม่แน่ใจเรื่องเทคนิค/มาตรฐาน** → **หาคำตอบเอง** (อ่านโค้ด/spec/มาตรฐาน) ⛔ ห้ามกวน P'Aim

**สเปกบอกว่า "ใช้ของเดิม/ต่อยอด" = ใช้ตามเดิมเป๊ะ** ห้ามเปลี่ยนหน้าตา ห้ามเพิ่มปุ่มใหม่ ห้าม redesign · ทำเท่าโจทย์ ไม่ทำเกิน
**การเปลี่ยน UI ที่มีคนใช้อยู่แล้ว = ต้อง confirm ก่อน** ไม่ใช่จัดระเบียบบนสมมติฐาน

## 10 · มาตรฐานที่ผูกไว้

- **ISO/IEC 29110-5-4** — ตามรอยได้ `โจทย์ → ออกแบบ → โค้ด → รายงานตรวจ → deploy` · **หลักฐานทุกด่าน commit ไว้**
- **UI:** เพลง `docs/ui-standards.md` · พระคำ `memory/web-quality-standards.md` (WCAG 2.2 AA · ARIA APG · HIG/Material 3)
- **เนื้อหาพระคำ:** `memory/content-style.md` + `memory/articles-section.md` · ความถูกต้องผ่าน N/G
- **บังคับใช้ แข็ง→อ่อน:** automate ใน test/CI → ผู้ตรวจอิสระเซ็น → brief อ้างมาตรฐาน → ด่านสุดท้าย

## 11 · การปรับปรุงไฟล์นี้

เจอปัญหา → ถามว่า **"process ตรงไหนปล่อยหลุด"** → เพิ่มกฎ/automate · **ไม่แก้จุดเดียว ไม่โทษคน**
⛔ **อย่าเสียเวลาขอโทษซ้ำ ๆ — เอาบทเรียนใส่ไฟล์นี้แทน**
**กฎใหม่ทุกข้อต้องมีหลักฐานคู่กันใน `evidence.md`** — กฎที่ไม่มีที่มา คนรุ่นถัดไปจะลบทิ้งเพราะไม่รู้ว่าซื้อมาด้วยอะไร
**อ่านแล้วไม่เปลี่ยนสิ่งที่ลงมือทำ = ไม่ใช่กฎ ไม่ต้องอยู่ในไฟล์นี้**
