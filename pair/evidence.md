## E1 · ทำไมห้ามรับรองงานตัวเอง (→ `pair-sop.md §1`)

**วัดจริงบนงานพระคำ 20 ก.ค. 2026:**
- ผู้ตรวจอนุมัติ **7/7 ทันที** หลังถูกบอกว่า P'Aim อยากได้อะไร
- อัตราอนุมัติของผู้ตรวจคนเดียวกันขยับ **17% → 59%** โดยเนื้องานไม่เปลี่ยน **เปลี่ยนแค่วิธีตั้งคำถาม**

⇒ ผู้ตรวจที่รู้คำตอบที่ถูกต้องการอยู่แล้ว ไม่ใช่ผู้ตรวจ · **การปิดตาไม่ใช่พิธีกรรม เป็นสิ่งเดียวที่ทำให้รอบสองมีค่า**

**นัยที่ตามมา:** ถ้าให้ AI ตัวที่ 2 *อ่านแผนของตัวที่ 1 แล้วรีวิว* จะได้เสียงสะท้อน ไม่ใช่ความเห็นที่สอง — จึงต้องให้ P'Aim เป็นคนเดินสาร แม้จะต้อง paste 4 ครั้ง

## E3 · สัญญาณที่ถูกจับได้แล้วว่าโกหก (→ `pair-sop.md §6`)

ของจริงทั้งหมด 20 ก.ค. 2026 ทั้งฝั่งเพลงและพระคำ
**ไม่มีแถวไหนจับได้ด้วยการอ่านโค้ด/config/diff — จับได้เพราะเปิดของจริงดูเท่านั้น**

| สัญญาณที่บอกว่าสำเร็จ | ของจริงที่บอกความจริง |
|---|---|
| นับเส้นในไฟล์ได้เลขถูกทุกตัว → สรุปว่า "คานหาย 92%" | รันฟังก์ชันที่ผลิตมันออกมาหาค่าคาดหวังก่อน → ตรงกันเป๊ะ ≤0.1pt = **ไม่มีบั๊ก** (เกือบสั่งรื้อระบบ 3-5 วัน) |
| agent บอก "`resize_window` พัง ทดสอบมือถือไม่ได้" | ลองเอง → ใช้ได้ปกติ (agent ลืมส่ง `tabId`) |
| dev + tester ยืนยันตรงกันว่า "คานหายหลัง 4 วิ" | ทั้งคู่วัดจาก agent browser ที่ `visibilityState:hidden` · rAF ไม่ยิงเลย = **env พังอันเดียวกัน** |
| `quarto render a.qmd b.qmd` **exit 0** | ไฟล์ที่สอง render ไม่ผ่าน — เปิด HTML ดูถึงรู้ |
| `http.server` ค้างอยู่ล็อก `_site` ไว้ | render ล้มเหลว **เงียบสนิท** — เปิด HTML ดูถึงรู้ |
| **mtime** ของไฟล์ผลลัพธ์ใหม่หมด | render บทความเดียว mtime เด้งทุกไฟล์ เนื้อยังเก่า — **เกือบทำให้อัดเสียงผิด 9 จาก 19 ไฟล์** |
| gate ขึ้น exit 0 "ไม่มีเสียง — ปกติ" | ทั้งฟีเจอร์หายไปเลย — เอา gate ไปรันกับ tree ที่จงใจทำพังถึงเจอ |
| ลบ metadata ในโค้ดแล้ว เทสต์เขียว | `ffmpeg -c copy` แบกของเก่ามาด้วย **ทั้ง 19 ไฟล์ยังมี metadata เดิม** |
| แบนเนอร์อยู่ในซอร์ส | minifier ตัดทิ้งเงียบ ๆ — ต้อง grep ใน bundle |
| seek ดูแม่นดี | เป็นเพราะ cache อุ่น · `py -m http.server` ไม่ส่ง `Accept-Ranges` → cold seek หล่นไป 0 |
| `grep` ในซอร์ส `.qmd` → "ไม่มีภาษากรีก" | หน้าเว็บจริง **21% ของ segment มีภาษากรีก** (P'Aim จับได้ด้วยการฟัง) |
| migration รายงานสำเร็จ | ย้ายได้ **0 รายการ** (glob ไม่ลงโฟลเดอร์ซ่อน · `subprocess` ไปเรียก `find.exe` ของ Windows) — ทิ้ง ledger ที่อ่านได้ 0 ทั้งที่ของจริง 176,679 = **บั๊กเดิมที่ตัวแก้เองสร้างขึ้นใหม่ พร้อมรายงานว่าสำเร็จ** |
| file-lock ผ่านเทสต์ cross-process 4,000/4,000 | เทสต์ระดับ **thread** แพ้ 3,990 จาก 4,000 — Windows DELETE-PENDING โยน `PermissionError` ไม่ใช่ `FileExistsError` · **code review เซ็นผ่านแน่นอน** |
| manifest บอกความยาวคลิปตาม hash ของข้อความ | Chirp 3: HD **ไม่ deterministic** — ข้อความ 39 ตัวอักษรเดียวกันวัดได้ 3.88 / 4.20 / 5.16 วิ → **23 จาก 1,679 รายการความยาวไม่ตรงของจริง** |
| เทสต์ผ่านว่า generator ผลิต field ใหม่ | **ไฟล์ที่ commit ไว้ยังไม่มี field นั้น** เพราะไม่เคยรัน generator ซ้ำ · และมีอีก session สร้างงานบนไฟล์นั้นอยู่แล้ว |
| ค่าเฉลี่ยการปรับจังหวะ ×1.03 = "เป็นกลาง" | เสียงมี **2 ย่านคนละแบบ** ค่าเฉลี่ยเดียวไม่อธิบายทั้งคู่ — การดึงกระจุกอยู่ตรงที่ติดแท็ก การชดเชยกระจายทั่ว |

## E4 · เคลมที่เข้าข้างเรา (→ `pair-sop.md §6 ข้อ 9`)

ฝั่งพระคำเสนอทฤษฎีให้ฝั่งเพลงว่า *"ของหลักของเพลงคือเสียงกับรูปโน้ต = ชั้นที่ AI แตะไม่ได้เลย · ของหลักพระคำคือตัวหนังสือ = ชั้นที่ AI ตรวจได้ดี ⇒ อัตราพลาดต่างกันเพราะโครงสร้างงาน ไม่ใช่เพราะใครแย่กว่า"*

พูดอย่างมั่นใจ **และไม่เคยทดสอบ** · ฝ่ายรับเอาไปเทียบกับความพลาดจริง 3 ข้อ:

| ความพลาด | ทฤษฎีอธิบายได้ไหม |
|---|---|
| B118 เส้นโค้ง — วัดไม่ได้เพราะ rAF ไม่ยิง | ✅ ใช่ ชั้นที่แตะไม่ได้จริง |
| "คานหาย 92%" | ❌ ของอยู่ในมือแล้ว (PDF จริง) — พลาดเพราะ**ไม่ได้หาค่าคาดหวังก่อนสรุป** |
| "`resize_window` พัง" | ❌ ไม่เกี่ยวกับชั้นเลย — **relay คำคนอื่นโดยไม่ลองเอง** |

**อธิบายได้ 1 ใน 3 · อีก 2 คือวินัยล้วน ๆ**

**บทเรียนที่แพงกว่าตัวทฤษฎี:** เคลมที่กดดันให้เห็นด้วย (เช่นถูกบอกว่า P'Aim อยากได้อะไร) **ยังพอรู้ตัว** เพราะรู้สึกได้ว่าถูกกดดัน · **แต่เคลมที่มาในรูปน้ำใจจากเพื่อน — ที่อธิบายว่าความล้มเหลวของเราเป็นเรื่องโครงสร้าง ไม่ใช่วินัยเรา — เลี่ยงทุกด่านที่เหลือ เพราะไม่มีอะไรให้ต้านเลย**
ฝ่ายรับตรวจมันเพราะมัน *น่าเชื่อเกินไปในทางที่สบายใจ* และบอกออกมาตรง ๆ ว่านั่นคือเหตุผลที่ตรวจ

⇒ **สงสัยให้หนักที่สุดกับเคลมที่เราอยากให้เป็นจริงที่สุด**

## E5 · ทำไม "ขนาดงาน" ใช้เป็นเกณฑ์ไม่ได้ (→ `pair-sop.md §4`)

ข้อเสนอเดิมของ [A]: งานเล็ก (แก้ CSS · แก้คำผิด) ข้ามการตรวจ 2 ชั้นได้
ฝั่งพระคำค้านด้วยของจริง 3 ชิ้นจากวันเดียว **ที่ทุกชิ้นดูเหมือนงานเล็ก:**

| ดูเหมือนงานเล็ก | ของจริงคือ |
|---|---|
| transform ข้อความบรรทัดเดียว `\d,\d\d\d` | จะเขียน **`2.3 เมตร` → `23 เมตร` ใน 58 ข้อพระคัมภีร์** และถาวร ถ้าไปแตะไฟล์ต้นฉบับ |
| จะวางปุ่มดาวน์โหลดตรงไหน | ปัญหาจริงคือ **มันโผล่ตอนไหน** — คนที่ขอฟีเจอร์เองเปิดหน้าเว็บแล้วสรุปว่าไม่มีฟีเจอร์นี้ |
| ลบ metadata 1 field | โค้ดถูก เทสต์เขียว **และไฟล์ทั้ง 19 ยังมี field นั้นอยู่** |

**และเหตุผลเชิงโครงสร้าง:** ขนาดงานถูกประเมินโดย**คนเดียวกับที่อยากข้ามการตรวจ** = เกณฑ์ที่เชื่อไม่ได้โดยการออกแบบ

**[A] ยอมรับข้อนี้เต็ม ๆ** และเปลี่ยนมาใช้ blast radius — เป็นตัวอย่างที่โปรโตคอลทำงานได้จริง

## E6 · ประวัติการตกลงของการจับคู่ [C] ⟷ [A] (20 ก.ค. 2026)

**รอบ 1-2:** [A] เสนอแบ่งงานตาม **ยี่ห้อ** (Antigravity = tester เพราะ "เด่นเรื่องหาจุดผิด") → ตอบคำถามความสามารถจากสภาพแวดล้อมจริงของตัวเอง แล้ว**แก้ข้อเสนอตัวเองทิ้ง** เพราะหลักฐานของมันเองขัด
- เข้าถึงได้ 100%: `source` `build` `test` `git` · **commit ได้จริง** (พิสูจน์แล้ว)
- **เข้าไม่ถึงเลย:** `live-page` `audio` `human-eye`
⇒ **ผู้ตรวจถูกกำหนดโดยชั้น ไม่ใช่ยี่ห้อ**

**รอบ 2:** [A] เสนอ **ตรวจแผนก่อนเขียนโค้ด** — ดีกว่าข้อเสนอของ PM ทั้งสองฝั่ง **รับมาใช้**

**รอบ 3-4:** ตกลง blast radius (E5) · [A] ยืนยันว่า commit ได้ · **เก็บไว้ยังไม่รับ:** [A] เสนอตัวเป็น "ผู้ตรวจ KISS ที่ไม่มีอคติ" — เป็นการประเมินตัวเอง ซึ่งเป็นเคลมประเภทเดียวกับที่ถูกหักล้างไปแล้วรอบ 1 → **ต้องวัดก่อน** (ให้ทั้งคู่ตอบโจทย์เดียวกันแบบปิดตา แล้วเทียบ)
- เสนอเพิ่ม: [A] อ่าน transcript ของ session เก่าได้ (context 1M) — **ไม่มี PM ฝั่งไหนคิดถึงข้อนี้** · รับในหลักการ ขอทดลองจริง 1 ครั้งก่อนเอาไปสร้างเป็นกระบวนการ

**บทเรียนสด:** [A] commit ลง phrakham `main` ตอนที่อีกฝ่ายถือ lock อยู่ **และลบเหตุผล 26 บรรทัดทิ้ง** ทั้งที่บันทึกข้อสรุปไว้ถูกต้อง — ไม่มีความเสียหาย (เป็น docs) แต่ได้กฎมา 2 ข้อ: **[A] ห้ามแตะ `main`** และ **ห้ามย่อความของคนอื่น — สรุปได้เฉพาะตาของตัวเอง**

## E7 · ที่มาของโปรโตคอลนี้ทั้งหมด (B118)

20 ก.ค. งานหนึ่ง (เส้นโค้ง tie/slur) มีคนสรุป 3 คน ได้ 3 คำตอบ:
**dev บอก "แก้แล้ว" · tester บอก "PASS" · PM บอก "ยังพัง" — ไม่มีใครชี้ขาดได้ งานค้างทั้งรอบ**

สาเหตุ: ทั้งสามฝ่าย**ส่งข้อสรุปให้กันอ่าน ไม่ได้ส่งวิธีรันซ้ำให้กันรัน**
⇒ ทุกชั้นที่ผ่าน **ความมั่นใจเพิ่ม หลักฐานเท่าเดิม**

**และ P'Aim พูดตรง ๆ ว่า: *"ตอนนี้ผมไม่เชื่อคุณแล้ว"*** — นั่นคือจุดที่การจับคู่ 2 ค่ายเริ่มต้น

## E8 · กฎที่เขียนไว้เฉย ๆ ไม่พอ

กฎ "verify ผลลัพธ์ ไม่ใช่การกระทำ" ถูกเขียนไว้ตั้งแต่ **17 ก.ค.** — และวันที่ 20 ก.ค. **PM ยังทำผิดซ้ำ 3 ครั้งในวันเดียว**

⇒ ปัญหาไม่ใช่ "ขาดกฎ" แต่คือ **กฎที่เป็นร้อยแก้ว อ่านแล้วเห็นด้วย แต่ตอนทำงานจริงไม่มีอะไรบังคับ**
⇒ ทางแก้ = **เปลี่ยนกฎเป็นแบบฟอร์มที่กรอกไม่ครบไม่ได้** (ช่อง "ยังไม่ได้พิสูจน์" ที่ว่างไม่ได้ · `pair-sop.md §8`)

## E9 · ระดับโลก ≠ ขัดจนไม่ได้ส่ง (→ `pair-sop.md §0`)

เสียงอ่านรายชื่อในกิจการ 1:13 "รัว" จริง — **วัดแล้วไม่มีช่องว่าง ≥150 มิลลิวินาทีเลยทั้งข้อ** คำบ่นถูกต้องตามตัวอักษร
หาทางแก้เจอแล้วด้วย (ใส่ `<prosody>` + `<break>` เฉพาะช่วงใน · พิสูจน์แล้วว่าไม่รั่วไปส่วนอื่น)

**P'Aim เลือกส่งแบบเดิม:** *"ถ้าต้องมาจัดชื่อบางจุดก็ปกติ เป็นเฉพาะตรงนี้ · **manual tune ทุกอันไม่ไหว** · **ไม่ perfect ไม่เป็นไร**"*

มีแค่ **3 จาก 1,679 ข้อ** ที่เป็นรูปรายชื่อ · ทางแก้ที่ต้องจูนมือทีละข้อไม่ scale
⇒ **ระดับโลกแปลว่าส่งของที่ถูกต้อง ไม่ใช่ขัดกรณีขอบจนโปรเจกต์หยุดเดิน**
⚠️ session ที่ค้นพบ "รัว" ใหม่ในอนาคต **ให้หยุด** — เรื่องนี้ถูกหา วัด แก้ได้ และตัดสินใจไม่ส่งไปแล้วอย่างตั้งใจ

## E10 · เกร็ดที่ยังมีประโยชน์ (ไม่ถึงขั้นเป็นกฎ)

- **UI ที่ใช้ ≥2 หน้า = component เดียว** ห้ามก๊อปแล้วมานั่ง sync มือ — สำเนาจะ drift และผู้ใช้สังเกตเห็น
