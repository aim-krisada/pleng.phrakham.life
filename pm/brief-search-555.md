**สั่งโดย:** pm4 · **ฐาน:** `studio-shell-redesign` (HEAD ล่าสุด · git-verify ก่อนเริ่ม) · **branch ใหม่:** `git switch -c search-short-notes studio-shell-redesign`

## อาการ (P'Aim)
พิมพ์ค้น **`555`** → **0 เพลง** (ควรเจอเพลง 1 พระเจ้าเป็นความรัก ที่ขึ้นต้น 5 5 5 6…) · P'Aim ถาม: ค้นโน้ต (เลขล้วน) ไม่บังคับใส่/ไม่ใส่ space ดีไหม

## ต้นเหตุ (อ่านก่อน)
`src/lib/songSearch.js` `isNoteQuery(q)` บรรทัด ~84-89:
```
return hasSpace || hasDecoration || q.replace(/[^0-7]/g, '').length >= 4
```
เลขล้วน **≤3 หลักไม่เว้นวรรค = ถือเป็นเลขเพลง** (กัน "42"/"100"/"117" ให้ค้นเลขเพลง) → "555" ไม่เข้า note-path เลย → หาเป็นเลขเพลง 555 (ไม่มี) = 0

## เป้าหมายการแก้
**เลขล้วนยาว ≥3 หลัก = ค้นทั้ง "เลขเพลง" และ "ลำดับโน้ต" (union)** — ไม่ใช่สลับทางเดียว:
- `555` → ไม่มีเพลงเลข 555 · แต่เจอเพลงที่ทำนองมี `5 5 5` (เพลง 1) ✓
- `100` → เพลงเลข **100 มาก่อน** (เจตนาแรง) + เพลงที่ทำนองมี `1 0 0` ตามหลัง
- `117` → เพลงเลข 117 ก่อน + ทำนอง `1 1 7`
- **≤2 หลัก (`1`,`42`) = เลขเพลงอย่างเดียว** (ค้นโน้ต 1-2 ตัว = รกมาก ทุกเพลงมี "1") — คงไว้
- มี/ไม่มีเว้นวรรค ผลตรงกัน (ต่อยอด note-search-verify ที่เพิ่ง merge — exact-sequence ไม่ fuzzy)

## แนวทาง (dev ปรับได้ถ้ามีทางสะอาดกว่า)
- อย่าเปลี่ยน `isNoteQuery` เป็นสลับ (จะทำเลขเพลง 3 หลัก 100/117 หาย) → ทำเป็น **union**: ถ้า query เป็นเลขล้วน ≥3 หลัก ให้ผลลัพธ์ = (เพลงที่เลขตรง · rank สูงสุด) ∪ (เพลงที่ notesCompact มี substring นั้นต่อเนื่อง)
- คง exact-sequence (ไม่ fuzzy) สำหรับ note-path · ค้นเนื้อ/ชื่อไม่กระทบ
- ❓ ถ้าเจอ edge เช่น "100" ควรจัดอันดับเลขเพลงเหนือทำนองยังไง — ใส่คำถามท้ายรายงาน (ไม่บล็อก · default = เลขเพลงก่อน)
