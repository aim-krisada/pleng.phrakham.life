## ที่มา — P'Aim เสนอเอง (17 ก.ค.) แทนกฎเดิม

พี่เปา (issues10): *"จะลบเพลงทั้งเพลง หาปุ่มไม่เจอ"* → **PM เจอว่า Editor ลบไม่ได้เลย และติด 3 ชั้น**
**P'Aim เคาะทางใหม่:**
> **ทุกคน (Editor/Admin) ลบได้ทุกเพลง → เข้าถังขยะ → ทุกคน restore ได้ → เฉพาะ Admin เทถังได้**
> **+ Editor กู้เพลงที่ Admin ลบได้ ("เพื่อความง่าย")** · **+ ของในถังอยู่ตลอดไป** (ไม่หายเอง · Admin เทเท่านั้น)

**ทำไมทางนี้ชนะกฎเดิม ("Editor ลบเพลงตัวเอง") — เหตุผลที่ต้องเข้าใจก่อนออกแบบ:**
1. **"เพลงของตัวเอง" ไม่มีอยู่จริงในข้อมูล** — **121/124 เพลง (97.6%) `author_id = NULL`** (ยิง REST นับจริง) · และ **121 เพลงนั้นคือชุด import ที่พี่เปาอยากลบมากที่สุดพอดี** → กฎเดิมไม่มีวันช่วยเขา
2. **ถังขยะทำให้ไม่ต้องรู้เจ้าของอีกเลย** → อุปสรรค 3 ชั้นหายหมด
3. **⭐ ปลอดภัยกว่ากฎเดิมมาก** — ทุกวันนี้ "ลบ" = **หายถาวร** · พี่เปามี 121 เพลงต้องไล่ลบของซ้ำ/ผิด **ถ้าลบแล้วกู้ไม่ได้ เขาจะไม่กล้าลบ = ยังติดขัดอยู่ดี**
4. **เป็นมาตรฐานสากล** (Gmail/Drive/ทุกเจ้า) = ตรงหลัก world-class by default

**⚠️ PM บอก P'Aim ตรง ๆ แล้วว่า "ไม่ได้ง่ายกว่า — งานเยอะกว่า แต่ถูกกว่า"** → **ห้ามตัดของเพราะยาก · ถ้าแพงจริง เสนอ *ลำดับเฟส* ไม่ใช่เสนอทำครึ่งเดียว** (memory `feedback_never_ask_user_what_is_correct`)

## 🔴 ความเสี่ยงที่แท้จริง = **ของรั่ว** (นี่คือหัวใจของงานออกแบบนี้)

**ทุกที่ที่ดึงเพลงจากฐานข้อมูล ต้องกรองของที่ลบแล้วออก** — รายการเพลง · ค้นหา · หน้าเพลง · แผ่นพิมพ์ · PDF · MP3 · เล่นเพลง · Studio · JSON export · sitemap/PWA cache · **ที่ไหนอีก?**

> **ลืมกรองจุดเดียว = เพลงที่ "ลบแล้ว" ยังโผล่ให้คนทั้งประเทศเห็น และเราจะไม่รู้เลย**

**= โรคเดียวกับที่เจอ 9 ครั้งวันนี้ ("โค้ดบอกว่าทำ แต่ไม่ทำ") แต่รอบนี้เดิมพันคือของที่ผู้ใช้เห็น**

**→ งานแรกของ SA: ไล่หา *ทุกจุด* ที่อ่านตาราง `songs` แล้วนับให้ครบ** (`grep from('songs')` + ไล่ทุก caller) · **ห้ามเดา ห้ามใช้เลขจากเอกสารเก่า**
**→ เสนอกลไกที่ *กันรั่วโดยโครงสร้าง* ดีกว่าไล่แก้ทีละจุด** — เช่น **กรองที่ RLS/view ชั้นเดียว** แทนการหวังให้ทุก caller จำได้
_(บทเรียนสด: สาย MP3 เจอว่า 2 จุดที่ "ต่างคนต่างต้องจำเอง" **ลืมพร้อมกันทั้งคู่** → เขารวมเป็น seam เดียว · **ทำแบบเดียวกัน**)_

## สิ่งที่ต้องออกแบบ

1. **ไล่จุดอ่าน `songs` ให้ครบ + นับจริง** (ข้างบน) → **ตารางว่าจุดไหนต้องกรอง จุดไหนไม่ต้อง (เช่น หน้าถังขยะเอง)**
2. **กลไกลบ:** soft delete (`deleted_at`?) หรืออื่น · **⚠️ soft delete = UPDATE ไม่ใช่ DELETE → RLS policy เปลี่ยนความหมายทั้งหมด** (`db/002:75` ปัจจุบัน = `"Approvers can delete songs" using (app_role() = 'approver')`)
3. **สิทธิ์ตามที่ P'Aim เคาะ:** ทุกคนลบ · **ทุกคนกู้ (รวมกู้ของที่ Admin ลบ)** · **Admin เท่านั้นเทถัง** · **อยู่ตลอดไป**
4. **ปฏิสัมพันธ์กับของที่มีอยู่ — ไล่ให้ครบ:** **GATE `verified`** (เพลงที่ตรวจแล้วแต่ถูกลบ = ต้องหายจาก public ทันที) · `song_drafts` (ร่างลบถาวรอยู่แล้ว — **ควรเข้าถังด้วยไหม? เสนอมา**) · audit log (`classifyChange`) · `number` **ไม่ unique ข้ามเล่ม (#32 มี 2 เพลง)** → **อ้าง `id` เท่านั้น**
5. **UI:** ปุ่มลบ · **หน้าถังขยะ** · ปุ่มกู้ · ปุ่มเทถัง (Admin) · **confirm ที่บอกชื่อเพลง** · **เทถัง = ทำลายถาวร → confirm ต้องหนักกว่า** · **WCAG 2.2 AA · target ≥44px · 360/412px**
6. **เสนอลำดับเฟส** + **ทางถอย** (ถ้าปล่อยแล้วมีของรั่ว จะกลับยังไง)
