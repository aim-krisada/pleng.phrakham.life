## สิ่งที่ต้องส่ง (วิเคราะห์ · ไม่ build)

### 1. ⭐ เทียบ pattern ระดับโลกจริง — **เปิดของจริง อย่าเล่าจากความจำ**
โปรแกรมโน้ต/แต่งเพลงที่แก้ปัญหานี้มาแล้ว: **MuseScore** (คลิกโน้ต→properties panel) · **Flat.io / Noteflight** (เว็บ · มือถือ) · **Ultimate Guitar / Songsterr** (chord editor) · **Hooktheory** · + **pattern มือถือทั่วไป:** contextual toolbar · iOS text-selection callout · Figma/Canva (เลือก object→inspector โผล่)
**ตอบ:** เขาโชว์เครื่องมือตอนไหน · เก็บตอนไหน · มือถือทำยังไง · **อันไหนเข้ากับ pleng (เลข+เนื้อ+คอร์ด ไม่ใช่ staff)** · อันไหนไม่เข้า เพราะอะไร

### 2. feasibility กับของที่เรามีจริง — **เปิดโค้ดดู อย่าเดา**
- **editor เรามี selection อยู่แล้วแค่ไหน?** (`EditorMode.vue` มี active note/bar/line ไหม · หรือทุกปุ่มลอยอิสระ) → **เลือกก่อน-โผล่ทีหลัง ทำได้บนโครงเดิม หรือต้องรื้อ**
- **มือถือ (พี่เปาใช้จริง):** จิ้มแล้วเครื่องมือโผล่ที่ไหนไม่บังโน้ต · แป้นพิมพ์กับ selection ชนกันไหม
- **โน้ต 1 ตัวมี ~13 ปุ่มรอบตัว** (baseline) → selection-driven ยุบเหลือเท่าไหร่ได้จริง

### 3. 🚧 ต้องเข้ากับของที่กำลังทำ — **ห้ามออกแบบให้รื้อทับ**
- **orientation (`91b05cf3`) วางแถบตัวตนบน shell = เหนือพื้นที่ dock → รอด selection-redesign อยู่แล้ว** (จุดตั้งต้นที่ดี · อย่าออกแบบให้แถบนั้นพัง)
- **เฟส 1 dev (`996eafda`) ถือ dock/`EditorMode.vue` อยู่** → บทวิเคราะห์ต้องบอกว่า **ถ้าทำจริง จะกระทบ dock ตรงไหน · ต้องรอสายไหนก่อน**
- **selection-redesign นี้เอง SA orientation ขีดเส้นแล้วว่า "รื้อ dock ทั้งก้อน คนละขนาดกับ orientation → เป็นสายแยก"** = ถูก · **นี่คือ "รื้อ" ตัวจริง (`refine_not_redesign`) → ต้องเสนอชัด ไม่แอบทำ**

### 4. เสนอ + phasing + **ความเสี่ยง/ต้นทุนซื่อ ๆ**
- แนวทางที่แนะนำ + **ทำไมดีกว่าทางอื่น** (ฟันธง อ้างมาตรฐาน ไม่ list เฉย ๆ)
- **แบ่งเฟสให้พี่เปาได้ของเป็นรอบ ๆ** (พี่เปาคือคอขวด · เพลง 30% · อย่าเสนอ big-bang ที่ 3 เดือนไม่มีอะไรถึงมือเขา)
- **ต้นทุนจริง:** รื้อขนาดนี้เสี่ยงอะไร · พี่เปาต้องเรียนรู้ใหม่แค่ไหน (เขาชินของเดิมแล้ว การเปลี่ยนใหญ่ = สะดุดชั่วคราว) · **บอกตรง ๆ ถ้าไม่คุ้ม**
