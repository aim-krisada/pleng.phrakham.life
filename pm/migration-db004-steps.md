# วิธีรัน migration `db/004` (audit log) บน Supabase — สำหรับ P'Aim

**ใครรัน:** P'Aim (approver) เท่านั้น · dev/PM ไม่แตะ prod DB
**รันเมื่อไหร่:** ตอน deploy รอบที่รวม audit log (หลัง tester ผ่าน + PM merge เข้า base) · **ปลอดภัยรันก่อน deploy โค้ดหน้าเว็บได้** (migration นี้ additive + backward-compatible — เว็บเวอร์ชันเก่ายังทำงานปกติ · ประวัติแค่ยังไม่โชว์จนโค้ดใหม่ขึ้น)

## ขั้นตอน
1. เปิด Supabase → โปรเจกต์ pleng → เมนูซ้าย **SQL Editor** → **New query**
2. เปิดไฟล์ `db/004-audit-log-events.sql` (อยู่ใน repo หลัง PM merge) → **copy ทั้งไฟล์**
3. วางในช่อง SQL Editor → กด **Run**
4. ควรขึ้นสำเร็จ (ไม่มี error สีแดง) · migration ใช้ `add column if not exists` / `create or replace` → **รันซ้ำได้ ไม่พัง** ถ้าเผลอรัน 2 รอบ
5. บอก PM ว่ารันแล้ว → PM/พี่เปา ทดสอบ end-to-end บนเว็บจริง (แก้เพลง → ดู "ประวัติการแก้ไข" ว่าบันทึกใครแก้อะไร)

## migration นี้ทำอะไร (ภาษาคน)
- เพิ่มคอลัมน์ใน `song_revisions` ให้เก็บ: เหตุการณ์ที่มีความหมาย (สร้าง/แก้/ส่งตรวจ/อนุมัติ/ตีกลับ/ถอน) · **สำเนาชื่อคนทำ (ไม่หายแม้ลบ user)** · ก่อน-หลัง · เวลา
- เปลี่ยน trigger ให้จับทั้งฝั่งร่าง (`song_drafts`) และเพลงจริง (`songs`)
- เพิ่มคำสั่ง `approve_and_publish` ให้ "อนุมัติ+เผยแพร่" เป็น 1 เหตุการณ์
- ล็อกให้ **แก้/ลบ log ไม่ได้** (เขียนได้เฉพาะระบบ) = audit เชื่อถือได้

## ข้อควรรู้
- ก่อนรัน: หน้า "ประวัติการแก้ไข" จะว่าง (โค้ดจับ error คืนค่าว่างอย่างนุ่มนวล ไม่พัง)
- ไม่กระทบเพลง/ข้อมูลเดิม (data เก่าไม่หาย · วิวัฒน์ตารางเดิม)
