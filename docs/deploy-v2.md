# รันสองรุ่นพร้อมกัน — `/` (ของเดิม) กับ `/v2/` (รุ่นใหม่)

เอกสารนี้เขียนให้พี่เอมอ่านแล้วรู้ว่า "ต้องกดอะไร" ไม่ต้องรู้เรื่องเทคนิค
ส่วนล่างมีรายละเอียดสำหรับคนที่ต้องแก้ระบบ

| ที่อยู่ | รุ่น | มาจากสาขา | ใครใช้ |
|---|---|---|---|
| <https://pleng.phrakham.life/> | ของเดิม (v1) | `main` | พี่เปาใช้ทำเพลงทุกวัน — **ห้ามสะดุด** |
| <https://pleng.phrakham.life/v2/> | ของใหม่ (v2) | `studio-shell-redesign` | ไว้ลองของใหม่ |

**ฐานข้อมูลเดียวกัน** (Supabase ตัวเดิม) ตั้งใจให้เป็นแบบนั้น — เพลงที่แก้ในรุ่นไหนก็เห็นอีกรุ่นทันที
ไม่มีการย้าย ไม่มีการก๊อปฐานข้อมูล

ในรุ่น v2 จะมีป้าย **`v2 รุ่นทดลอง ⇄`** อยู่ข้างชื่อเว็บบนหัวจอทุกหน้า **แตะครั้งเดียวกลับไปรุ่นเดิม**
(ถ้าเปิดเพลงอยู่ ก็จะกลับไปเพลงเดิมในรุ่นเดิม) ในรุ่นเดิมไม่มีป้ายนี้ — ยังเหมือนที่พี่เปาใช้อยู่ทุกอย่าง

---

## 1. ต้องกดอะไรตอนจะ "ปล่อยของ" (ครั้งแรก)

ตอนนี้โค้ดพร้อมแล้ว แต่ **ยังไม่ได้ปล่อย** — รอพี่เอมสั่ง

สิ่งที่ต้องเกิดขึ้นคือ **เอาไฟล์ `.github/workflows/deploy.yml` ตัวใหม่ขึ้นทั้งสองสาขา** (`main` และ
`studio-shell-redesign`) เพราะไฟล์นี้คือตัวที่สั่งว่า "สร้างทั้งสองรุ่นแล้วเอาขึ้นด้วยกัน"
ถ้าขึ้นแค่สาขาเดียว อีกสาขาจะยังใช้ไฟล์เก่าที่ลบอีกรุ่นทิ้ง

ลำดับที่ปลอดภัยที่สุด (PM เป็นคนกด · พี่เอมแค่สั่ง "go"):

1. เอาการแก้ทั้งหมดเข้า `studio-shell-redesign` ก่อน — ยังไม่มีอะไรเปลี่ยนบนเว็บจริง
2. เอา **เฉพาะไฟล์ `.github/workflows/deploy.yml`** เข้า `main`
3. พอไฟล์เข้า `main` งาน deploy จะเริ่มเอง — ใช้เวลาราวสองสามนาที
4. เปิด <https://pleng.phrakham.life/> เช็คว่าของเดิมยังปกติ **ก่อน** แล้วค่อยเปิด `/v2/`

หลังจากนี้ **ทุกครั้งที่มีคนแก้สาขาไหนก็ตาม ระบบจะสร้างใหม่ทั้งสองรุ่นเสมอ** ไม่มีใครลบใครทิ้ง

## 2. ถ้า `/v2/` มีปัญหา — เอาออกยังไง

**ทางที่เร็วที่สุด (ไม่แตะของเดิมเลย):** ทำให้สาขา `studio-shell-redesign` สร้างไม่ผ่าน หรือ
เอาการแก้ล่าสุดของ v2 ออก แล้วสั่ง deploy ใหม่ — ระบบจะปล่อยเฉพาะของเดิม และ `/v2/` จะหายไป
ของเดิมที่หน้าแรกไม่กระทบ (ดูข้อ 5 ว่าทำไม)

**ถอนถาวร:** เอาไฟล์ `deploy.yml` เวอร์ชันเก่า (ที่สร้างแค่ `main`) กลับเข้า `main`
แล้ว deploy หนึ่งรอบ — เว็บกลับไปเป็นรุ่นเดียวเหมือนก่อนหน้านี้ทุกอย่าง

**คนที่เคยเข้า `/v2/` แล้วเครื่องจำไฟล์ไว้** จะไม่ค้าง เพราะไฟล์ที่จำไว้ของ `/v2/` แยกกล่องกับของหน้าแรก
คนละกล่อง — เมื่อ `/v2/` หายไป เขาจะเห็นหน้าไม่พบไฟล์ แล้วพิมพ์ที่อยู่หน้าแรกเข้าได้ตามปกติ

## 3. วันที่จะให้ v2 ขึ้นเป็นตัวหลัก (แทนของเดิม)

1. รวมงานของ `studio-shell-redesign` เข้า `main` (นี่คือ "ปล่อยจริง" — พี่เอมสั่งเท่านั้น)
2. เอา `deploy.yml` กลับไปเป็นแบบสร้างสาขาเดียว (ไม่ต้องมี `/v2/` แล้ว) — ป้าย `v2 รุ่นทดลอง`
   จะหายไปเอง เพราะมันขึ้นเฉพาะตอนสร้างแบบ `/v2/` ไม่ต้องไปไล่ลบโค้ด
3. `/v2/` เดิมจะหายจากเว็บในรอบ deploy นั้น
4. **คนที่เคยติดตั้งเป็นแอป (PWA) จาก `/v2/`** ต้องถอนแล้วติดตั้งใหม่จากหน้าแรก — ของสองที่นี้
   ระบบมองเป็นคนละแอป ตั้งใจให้เป็นแบบนั้นเพื่อไม่ให้ปนกันระหว่างทดลอง
5. คนทั่วไปที่ใช้หน้าแรกอยู่แล้ว **ไม่ต้องทำอะไร** — เปิดครั้งถัดไปได้ของใหม่เอง

### ถ้าปล่อยเป็นตัวหลักแล้วพัง — ถอยยังไง

`git revert` การรวมเข้า `main` แล้ว push → deploy รอบใหม่จะเอาของเดิมกลับมาทั้งหมด
(หน้าเว็บที่ยังอยู่ตอนนี้ไม่ถูกลบระหว่างรอ — GitHub Pages เก็บของเดิมไว้จนกว่ารอบใหม่จะสำเร็จ)

## 4. รายละเอียดทางเทคนิค (สำหรับคนแก้ระบบ)

**การสร้าง (`vite.config.js`)** — `PLENG_V2=1 npm run build` เปลี่ยน `base` จาก `'./'` เป็น `'/v2/'`
และตั้ง `__APP_VARIANT__ = 'v2'` (ตัวที่ทำให้ `VersionSwitch.vue` แสดงป้าย) ทุกอย่างที่เคยเขียน
พาธตรงๆ ถูกแก้ให้อ่านจาก base แล้ว:

| จุด | เดิม | ตอนนี้ |
|---|---|---|
| `dist/asset-manifest.json` | `/assets/…` | `<base>assets/…` |
| `sampler.js` `SAMPLE_HOSTS` | `/samples/` | `import.meta.env.BASE_URL + 'samples/'` |
| ลงทะเบียน service worker | `register('sw.js')` | `register(BASE_URL + 'sw.js')` |
| `site.webmanifest` | ชื่อเดียว | บิลด์ v2 เติม ` (v2 ทดลอง)` + `id: '/v2/'` ตอน build |

**Service worker (`public/sw.js`)** — ไฟล์เดียวกันถูกเสิร์ฟทั้ง `/sw.js` และ `/v2/sw.js`
โฟลเดอร์ของสคริปต์ = ขอบเขตของมัน (`scope`) ดังนั้นทุกอย่างอ่านจาก `registration.scope`:

- `ROOT` = `/` หรือ `/v2/` — ใช้ตั้งพาธ app shell, samples, asset-manifest ทั้งหมด
- ชื่อกล่องเก็บไฟล์ (cache) ต่อท้าย `--v2` เฉพาะรุ่นย่อย → **หน้าแรกยังใช้ชื่อเดิมเป๊ะ** (`pleng-app-v1`)
  คนที่ใช้อยู่จึงไม่โดนล้างของ
- ตอน `activate` ลบเฉพาะกล่องที่เป็น "ของ scope ตัวเอง" → รุ่นหนึ่งลบของอีกรุ่นไม่ได้
- `fetch` ไม่ยุ่งกับพาธนอก scope ตัวเอง
- รายชื่อ sample ในไฟล์ `manifest.json` เขียนเป็น `/samples/…` → SW ของ v2 แปลงเป็น `/v2/samples/…`
  ก่อนโหลด (ไม่งั้นจะไปดูดไฟล์ 205 ชิ้นของหน้าแรกมาเก็บผิดกล่อง — เจอตอนวัดจริง)

**Workflow (`.github/workflows/deploy.yml`)** — GitHub Pages ไม่มี "ปล่อยทีละส่วน"
ทุกครั้งที่ปล่อย มันแทนที่ทั้งเว็บด้วยของที่อัปโหลดรอบนั้น ฉะนั้น workflow เดียวจึงต้อง
**สร้างทั้งสองสาขาเสมอ** ไม่ว่าใครเป็นคนจุดชนวน:

1. checkout `main` → `npm ci` → `npm run build` → ได้ root
2. checkout `studio-shell-redesign` → build ด้วย `PLENG_V2=1` → ก๊อปเป็น `dist/v2`
3. อัปโหลด `dist` ทั้งก้อน

ขั้นตอนของ v2 ทั้งสองขั้นตั้ง `continue-on-error: true` **จงใจ** — v2 คืองานที่ยังทำอยู่
ถ้ามันสร้างไม่ผ่าน ต้องไม่ลากให้ของเดิมปล่อยไม่ได้ ผลคือรอบนั้นปล่อยเฉพาะของเดิม
แล้ว `/v2/` หายไปชั่วคราว (มีบรรทัด log บอกว่า "NOT included")

## 5. ทำไมของเดิมถึงปลอดภัย

1. ไฟล์ที่ปล่อยขึ้น root มาจากสาขา `main` เท่านั้น — workflow `checkout ref: main` เสมอ
   ไม่ว่าสาขาไหนจุดชนวน โค้ดของ v2 จึงเข้าไปที่ root ไม่ได้เลย
2. โค้ดฝั่งแอปที่แตะไปทั้งหมด (ป้ายเวอร์ชัน, sampler, การลงทะเบียน SW) **ยังไม่อยู่บน `main`**
   และเมื่อถึงวันที่ขึ้น ป้ายเวอร์ชันก็ยังไม่แสดง เพราะ `__APP_VARIANT__` เป็นค่าว่างในบิลด์ root
3. ชื่อกล่อง cache ของหน้าแรกไม่เปลี่ยน และ SW ของ `/v2/` ลบของหน้าแรกไม่ได้ตามกฎ scope
4. v2 สร้างไม่ผ่าน = ปล่อยของเดิมตามปกติ (`continue-on-error`)

## 6. ทดสอบเองในเครื่อง

```sh
npm run build && cp -r dist /tmp/site
PLENG_V2=1 npm run build && cp -r dist /tmp/site/v2
py -3 -m http.server 5396 --bind 127.0.0.1 --directory /tmp/site
```
เปิด <http://127.0.0.1:5396/> แล้ว <http://127.0.0.1:5396/v2/> แล้วกลับมาหน้าแรก —
ใน DevTools → Application ต้องเห็น service worker สอง scope และ cache สี่กล่อง
(`pleng-app-v1`, `pleng-samples-v1`, และคู่ `--v2` ของมัน) โดยกล่อง `--v2` มีแต่พาธ `/v2/…`
