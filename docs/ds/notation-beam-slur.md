# บั๊ก 1 — issue8: เส้นใต้ (beam) ตามพยางค์

## กฎ (SSOT เดียว)
**เส้นใต้เชื่อม (beam) = การรวมพยางค์.** โน้ตเขบ็ต (มีขีดใต้ `_`) ที่ติดกันจะ **เชื่อมเส้น** ก็ต่อเมื่อโน้ตตัวหลังเป็น **เอื้อน (ไม่มีคำร้องของตัวเอง)** ของโน้ตตัวหน้า. โน้ตที่ **ขึ้นคำใหม่** จะ **ตัดเส้น** (เริ่ม beam ใหม่/ยืนเดี่ยว).

สัญญาณจากข้อมูล (zero migration):
- ช่อง syllable **ว่าง** (`''` / เว้นวรรค) = เอื้อน → **ต่อเส้น**
- ช่อง syllable **มีคำ** = พยางค์ใหม่ (attack) → **ตัดเส้นก่อนหน้า**

> กฎนี้ **ครอบ issues2 เดิมโดยอัตโนมัติ:** `(6_ 5_)` บน "ดี" → `6_`=ดี(มีคำ), `5_`=ว่าง(เอื้อน) → เชื่อม 2 ตัวเป็น beam เดียว + arc ถูก suppress (beamOnly) เหมือนเดิม.

## renderer ต้องตัด beam กลางจังหวะได้ (partial beam)
เดิมตัด beam ที่ **ขอบจังหวะ** เท่านั้น. ต้องเพิ่มเงื่อนไขตัด **ก่อนโน้ตที่ขึ้นคำใหม่** ด้วย — จึงเกิด "beam ครึ่งจังหวะ" ได้ (เช่นในจังหวะเดียวมีเอื้อนแล้วขึ้นคำใหม่). เก็บเงื่อนไขตัดเดิมทั้งหมด (ขอบจังหวะ · โน้ตไม่มีขีดใต้ · rest `0` · triplet) แล้ว **เพิ่ม** เงื่อนไข "โน้ตนี้ขึ้นคำใหม่".

## อัลกอริทึม `beamGroups(noteString, syllables)` → `[{start, end, u2}]`
คืน "รันของ beam" อ้างด้วย `t.idx` (running index ต่อ token — เลขเดียวกับที่ NoteRow ใช้ยิง `data-idx`).

```
gs = groupNotes(parseNotes(noteString))
stamp t.idx = 0,1,2,... ให้ทุก token (note / ext / raw) — ตรงกับ NoteRow

// map พยางค์: เดินทุก token ที่ "ไม่ใช่ struct bracket" ใช้ syllables ทีละช่อง
// isAttack(t) = ช่องพยางค์ของ t มีข้อความ (trim แล้วไม่ว่าง)
//   - ถ้า syllables == null (v1 / ไม่ส่งมา) → isAttack = false เสมอ (fallback = พฤติกรรมเดิม beat-only)

beat = 0 ; run = [] ; runBeat = -1 ; beams = []
flush(): ถ้า run.length >= 2 → mark ทุกตัว beamed=true, push {start:run[0].idx, end:last.idx,
         u2: run.some(underlines>=2)} ; run = []

for g in gs:
  isTrip = g.group === 'triplet'
  for t in g.tokens:
    if t.type === 'note':
      dur = (1/2**underlines) * DOT_FACTOR[dots]; if isTrip: dur = dur*2/3
      startBeat = floor(beat + 1e-9)
      beamable = !isTrip && underlines > 0 && pitch !== '0'
      if beamable && run.length>0 && startBeat===runBeat && !isAttack(t):
          run.push(t)                       // เอื้อน จังหวะเดียวกัน → ต่อ beam
      else:
          flush()
          if beamable: run=[t]; runBeat=startBeat   // ขึ้น beam ใหม่ (มีคำ / ข้ามจังหวะ / ตัวแรก)
      beat += dur
    elif t.type === 'ext': flush(); beat += 1
    else: flush()
flush()
return { beams, ... }   // (คืน gs พร้อม flag beamed ให้ NoteRow ใช้ต่อ)
```

**ตรวจกับ 3 รูป (ผ่านครบ):**

| รูป | โน้ต + (คำ) | ปัจจุบัน (beat-only) | ต้องได้ (syllable) |
|---|---|---|---|
| 1 | `1.`(ต้อง) `.6_`(ไขว่) `1_`(คว้า) `2_`(ว่าง) `3`(เสาะ) | `.6`เดี่ยว · `1–2`beam | **เท่ากัน** ✓ (บังเอิญถูก) |
| 2 | `1`(ใส่) `1_`(อา) `.6_`(ภรณ์) `1`(งาม) `2`(นัก) | `1_–.6_`**beam(ผิด)** | `1_`เดี่ยว · `.6_`เดี่ยว ✓ |
| 3 | `5`(น-) `3`(ภา) `–` `3_`(ไม่) `2_`(เคย) | `3_–2_`**beam(ผิด)** | `3_`เดี่ยว · `2_`เดี่ยว ✓ |

> รูป 1 วันนี้ถูก **เพราะขอบจังหวะบังเอิญตรงกับขอบคำ**. รูป 2/3 คำ 2 คำตกในจังหวะเดียว → กฎ beat จึงเชื่อมผิด. กฎ syllable แก้ 2/3 โดยไม่ทำ 1 พัง.

## มาตรฐาน + เหตุผลที่เลือก (บันทึกกัน future "fix" ผิดทาง)
- ใช้ **ธรรมเนียมโน้ตร้องดั้งเดิม (traditional vocal beaming) = beam ตามพยางค์** · ตรงกับ **jianpu/โน้ตตัวเลขสายร้อง** และ **ตรงหนังสืออ้างอิงไทย** (เป้าหมายหลักของ pleng).
- **ไม่ใช้** ธรรมเนียมฝรั่งสมัยใหม่ (Elaine Gould, *Behind Bars*: instrumental beaming ตามจังหวะ ไม่สนพยางค์) — **จงใจ**. ใครมาเห็นภายหลังแล้วคิดจะ "แก้ให้ beam ตามจังหวะ" = ผิดทาง อย่าทำ. (Gould เองก็ระบุว่าโน้ตร้องแบบดั้งเดิม beam ตามพยางค์ — เราเลือกสายนั้น.)

## AC — issue8
1. รูป 2 (`1 1_ .6_ 1 2`, คำ ใส่/อา/ภรณ์/งาม/นัก): `1_` กับ `.6_` **แยกเส้น** (2 beam element = 0 เพราะ len<2 → ไม่มี beam เชื่อม, ต่างตัวมีขีดใต้ของตัวเอง).
2. รูป 3 (`5 3 - 3_ 2_`, คำ น-/ภา/-/ไม่/เคย): `3_` กับ `2_` **แยกเส้น**.
3. รูป 1 (`1. .6_ 1_ 2_ 3`, คำ ต้อง/ไขว่/คว้า/ว่าง/เสาะ): `1_`+`2_` **เชื่อม 1 beam**, `.6_` เดี่ยว — **ไม่เปลี่ยนจากปัจจุบัน**.
4. issues2: `(6_ 5_)` บน "ดี" ยังได้ **1 beam + ไม่มี slur arc**.
5. v1 (ไม่มี `syllables`): พฤติกรรมเดิม (beat-only) — ไม่ crash, ไม่เปลี่ยน.

# บั๊ก 2 — issues5: slur ข้ามห้อง / ข้ามบรรทัด

## วิเคราะห์สาเหตุ (ปัจจุบันจับคู่ผิดยังไง)
`( )` ถูกจับกลุ่มใน `groupNotes` **ภายในสตริงโน้ตของกล่องเดียว**. เมื่อผู้ใช้พิมพ์ `(1'` ท้ายห้องหนึ่ง แล้ว `6)` ต้นห้องถัดไป → `(` กับ `)` อยู่คน string ของคนละ segment:
- กล่องเปิด (`… (1'`): `groupNotes` เปิด slur ค้าง ไม่มี `)` ปิด → กลุ่มมีโน้ตตัวเดียว → NoteRow วาด **arc ค้างครอบโน้ตตัวเดียว (ผิด)**.
- กล่องปิด (`6) …`): `)` ไม่มี `(` คู่ → ตัด group เฉยๆ → `6` เป็นโน้ตเดี่ยว **ไม่มี arc**.

= อาการที่พี่เปาเห็น: เครื่องหมายโค้งค้าง/หายเมื่อข้ามเส้นห้อง. **โครงสร้างเดียวกับ tie ข้ามห้องเป๊ะ** (B069/B099 แก้ไปแล้วด้วย line-level overlay).

## แนวทาง — ใช้กลไก overlay เดิม (KISS, refine ไม่ redesign)
slur ข้ามห้อง = **arc จากโน้ต anchor เปิด → โน้ต anchor ปิด** พาดข้ามเส้นห้อง เหมือน tie จาก source→receiver ทุกประการ. เงื่อนไขต่างจาก tie แค่: anchor คือโน้ตที่ **ติดหลัง `(`** และโน้ตที่ **ติดหน้า `)`** (คนละ pitch ได้), และ slur พาดได้หลายห้อง.

**แบ่งเคส:**
- **slur ในกล่องเดียว (within-segment):** เดิม NoteRow วาด `.slur-arc` ได้ถูกอยู่แล้ว → **ไม่แตะ** (ไม่ regress B062/B076).
- **slur ข้ามกล่อง (cross-segment):** ให้ SongSheet overlay วาด + ซ่อน arc ค้างของ NoteRow.

## `slurSpans(noteStrings[])` → คู่ slur ระดับบรรทัด (pure, unit-test ได้)
input = อาเรย์สตริงโน้ตของทุก segment ในบรรทัด (เรียงซ้าย→ขวา) · output = คู่ slur:
```
{ open:{si, idx}, close:{si, idx}, sameSegment }
   si  = ลำดับ segment ในบรรทัด
   idx = t.idx ของโน้ต anchor ใน segment นั้น (โน้ตแรกหลัง '(' / โน้ตสุดท้ายก่อน ')')
```
วิธี: เดิน token ทุก segment ตามลำดับ, ใช้ **stack** เก็บ `(` ที่เปิดค้าง (พร้อม si+idx ของโน้ตถัดไป); เมื่อเจอ `)` จับคู่กับ top ของ stack แล้วบันทึก. `sameSegment = open.si === close.si`.
- **เฉพาะคู่ `sameSegment===false`** ที่ต้องใช้ overlay (คู่ในกล่องเดียว NoteRow ทำเองแล้ว).
- เคสปกติ depth 1; รองรับซ้อน/ต่อเนื่องด้วย stack. (nested slur = edge หายาก → ดู open questions.)

## การวาด (ขยาย `measureTies` เดิม — เปลี่ยนชื่อเป็น `measureArcs` ได้)
สำหรับแต่ละคู่ cross-segment ในบรรทัด:
1. หา DOM anchor: `.segment[data-seg="{li}-{open.si}"] .nt[data-idx="{open.idx}"]` และของ close เช่นเดียวกัน. (ใช้ `data-seg`+`data-idx` ที่มีอยู่แล้ว — **ไม่ต้องเพิ่ม attribute**.)
2. วัด rect ทั้งสอง เทียบกับ rect ของ `.song-line` (เหมือน tie).
3. **ตัดสินการวาด:**
   - **แถวเดียวกัน (`|a.top − b.top| ≤ h*0.6`) — ข้ามห้องแต่ไม่ wrap:** วาด **arc เดียว** พาดจาก open→close ข้ามเส้นห้อง ด้วย `buildArc(x_open, x_close, yTop, h)` (มาตรฐานสากล: slur ในระบบเดียว = โค้งต่อเนื่องข้ามเส้นห้อง).
   - **คนละแถว (wrap) — slur ถูกตัดด้วยการขึ้นบรรทัดใหม่:** วาด **2 ส่วน**
     - ส่วนหน้า: จาก open → **ขอบขวาสุดของแถวที่ open อยู่** (ปลายเปิดลอยท้ายแถว)
     - ส่วนหลัง: จาก **ขอบซ้ายสุดของแถวที่ close อยู่** → close (ปลายเข้าต้นแถวถัดไป)
     - ("ขอบแถว" = ใช้ขวาสุดของ `.nt` ตัวสุดท้ายในแถวนั้น / ซ้ายสุดของ `.nt` ตัวแรกในแถวถัดไป เพื่อไม่ให้ arc ล้นออกนอกโน้ต).
4. เพิ่ม path ที่ได้ลง overlay SVG ต่อบรรทัด (โครงเดิม `lineArcs[li]`).
5. **ซ่อน arc ค้างของ NoteRow:** `.segment[data-seg="{li}-{open.si}"] .slur-arc` → `display:none` (ใส่ใน restore-list เดียวกับ `hiddenHalves` เพื่อให้ hide/draw sync — ถ้าเคสไหน overlay ไม่วาด (เช่น wrap ที่วัดไม่ได้) ให้คืน arc เดิมเป็น fallback).

> arc ของ slur ควรพาดสูงกว่า/คนละแถบกับ tie เล็กน้อยได้ แต่ **ใช้ `buildArc` ตัวเดิม** (รูปทรง lens เดียวกับ tie/slur ทั้งระบบ) — สีเดียวกับโน้ต (`--note-blue`).

## AC — issues5
1. `… (1'` ท้ายห้อง + `6)` ต้นห้องถัดไป **ในแถวเดียวกัน**: มี arc **1 เส้น** พาดจาก `1'`→`6` ข้ามเส้นห้อง; **ไม่มี** arc ค้างครอบ `1'` เดี่ยว.
2. เคสเดียวกันแต่ `6)` **ตกลงบรรทัดถัดไป (wrap)**: มี arc **2 ส่วน** (ท้ายแถวบน + ต้นแถวล่าง); ไม่มี arc ค้าง.
3. slur ในกล่องเดียว (`(1 2 3 4)`): ยังเป็น **1 arc** ของ NoteRow เหมือนเดิม (ไม่ถูก overlay ยึด).
4. tie ข้ามห้องเดิม (B069/B099) ยังวาดถูก ไม่ถูก slur overlay รบกวน.

## มาตรฐานที่อ้าง
- slur ที่ถูก **system break (ขึ้นบรรทัดใหม่) ตัด** → วาด **2 ส่วน** ปลายละแถว: หลักการ engraving สากล (Gould *Behind Bars* — slurs across system breaks; Read/Ross *Music Notation*). **ไม่ขัด** กับการที่เราไม่ใช้ Gould เรื่อง beam — คนละหัวข้อ (นี่คือเรื่องการตัด slur ตามระบบ ซึ่งเป็นมาตรฐานเดียวทั้งวงการ).
- slur ข้ามเส้นห้องใน **ระบบเดียว** = โค้งต่อเนื่อง (ไม่ตัด) — มาตรฐานเดียวกัน.

# DoD — ระบบตรวจถาวร 2 ชั้น (P'Aim สั่ง · เป็นข้อบังคับ)

### ชั้น 2ข — สคริปต์ตรวจข้อมูลทุกเพลง (re-run ได้ · QA tool ถาวร)
`tools/audit-notation.mjs` — ดึงทุกเพลงจาก Supabase (public read key ที่มีใน `src/supabase.js` อยู่แล้ว — read-only) แล้วพ่นรายงาน markdown ไป `docs/reports/notation-audit.md`. รันซ้ำ: `node tools/audit-notation.mjs`.

**รูปแบบรายงาน (หน้าเดียวต่อเพลง · ให้พี่เปากวาดตาเทียบหนังสือเร็ว):**
```
## #<number> <title_th>   (key <k> · <timeSig>)
บรรทัด L, ห้อง B [จังหวะ b0–b1]:  <โน้ต>(คำ)  →  NEW=ตัดเส้น / TIE=ต่อเส้น(เอื้อน)
  ตัวอย่าง:
  L1 ห้อง2 [beat1]:  1_(อา)=NEW  .6_(ภรณ์)=NEW      → แยกเส้น
  L1 ห้อง3 [beat2]:  1_(คว้า)=NEW 2_( )=เอื้อน        → เชื่อมเส้น
⚑ slur ข้ามห้อง: L1 ห้อง4 (1' → L1 ห้อง5 6)   [ข้ามห้อง | ข้ามบรรทัด: วัดตอน render]
```
- ต่อจังหวะ ระบุทุกโน้ตเขบ็ตว่า **NEW (มีคำ→ตัด)** หรือ **เอื้อน (ว่าง→ต่อ)** — ใช้ `beamGroups` ตัวเดียวกับ render (SSOT เดียว ไม่ใช่ตรรกะซ้ำ).
- ท้ายเพลงลิสต์ **จุด slur ข้ามห้อง** ทุกจุด (จาก `slurSpans`).
- สรุปหัวรายงาน: จำนวนเพลง · จำนวนจุด beam-cut · จำนวน slur ข้ามห้อง · เพลงที่มีช่อง attack **ว่างผิดปกติ** (อาจกรอกคำไม่ครบ) เป็น ⚠️ ให้พี่เปาสอบ.
