# Command palette (Ctrl+K) — สเปกหน้าจอ

**สโคป:** หน้าแก้เพลง **ตัวใหม่** (โหมดแก้บนแผ่นเพลง — `SongViewer.vue` + `NoteInputBar.vue`) เท่านั้น ·
⛔ ไม่แตะตัวแก้ตัวเก่า (`EditorMode.vue`) — มติ PM 24 ก.ค. คือ *ยกของเข้าตัวใหม่ทีละชิ้นแล้วค่อยตัดแท็บเก่า*
การใส่ palette ลงตัวเก่าด้วย = สร้างทางเข้าที่ 3 ให้ของที่กำลังจะถูกทิ้ง

**ต่อจาก:** `docs/ds/note-symbol-set.md` §3.1 (คำสั่ง `[…]`) · §4.1 (แป้นสอนคีย์) · §4.1.5 (มติ palette=ทางหลัก) ·
§4.2 (ป๊อปอัปกระจกสะท้อน) · §7.3 (ทางค้นพบ) · `docs/ds/repeat-flow-override.md` (คำสั่งวนรอบที่ palette ต้องใส่ได้)

> ## 🔒 หลักที่ล็อก ห้ามแก้
> **palette ไม่ใช่ภาษาที่สอง มันคือประตูที่สองของภาษาเดียวกัน**
> เลือกจาก palette แล้วสิ่งที่ลงในบรรทัด = **ข้อความเดียวกับที่พิมพ์เองได้เป๊ะ**
> ⇒ ผู้ใช้เห็นทุกครั้งว่ามันเขียนเป็นข้อความว่าอะไร แล้วค่อย ๆ เลิกเปิด palette ไปเอง

## 1. ทำไมต้องมี — หลักฐาน 3 ชิ้น (ไม่ใช่ความชอบ)

| หลักฐาน | ตัวเลข/ที่มา | แปลว่า |
|---|---|---|
| ฟีเจอร์ที่ "มีแต่มองไม่เห็น" ตายสนิท | volta ใช้ **0 ครั้งจาก 155 เพลง** ทั้งที่ dropdown "ห้องจบ" มีอยู่ใน ⋯ มาตลอด (`note-symbol-set.md` §7.3) | ทางเข้าที่ต้องกดเปิดเมนูก่อน = ไม่มีอยู่จริงสำหรับผู้ใช้ |
| ของที่ต้องหาทางเข้ามีเยอะเกินแถบปุ่ม | นับ `title`/`aria-label` ที่เป็นคำสั่งในตัวแก้ตัวเก่า **≈60 คำสั่ง** (ฟังท่อนนี้ · ย้ายบรรทัดขึ้น/ลง · คัดลอกห้อง · ห้องต่อกัน · ร้องรับทุกข้อ · เปลี่ยนคีย์ท่อนนี้ …) กระจายอยู่หลังเมนู ⋯ หลายชั้น | แถบปุ่มรับไม่ไหวแน่นอน ต้องมีทางที่ค่าใช้จ่ายไม่โตตามจำนวนคำสั่ง |
| ของจะเพิ่มอีกมาก | P'Aim ยืนยันเอง (§4.1.5) | ต้องเลือกโครงที่ **โตแล้วไม่เพิ่มจำนวนการกด** ตั้งแต่วันนี้ |

**ค่าใช้จ่ายต่อการใช้ 1 ครั้ง** — แถบปุ่ม = กวาดสายตา O(จำนวนปุ่ม) + ยกมือไปเมาส์ ·
palette = 1 คีย์ + พิมพ์ 2-3 ตัว **คงที่ไม่ว่าจะมี 20 หรือ 200 คำสั่ง** ⇒ นี่คือเหตุผลเชิงโครงสร้าง
ที่ palette เป็นทางหลัก ไม่ใช่เพราะ "แอปสมัยใหม่เขาทำกัน"

## 2. เปิด/ปิดยังไง — 3 ประตู (ทั้งหมดเข้าห้องเดียวกัน)

| # | ประตู | ใครใช้ | เปิดแล้วช่องค้นมีอะไร |
|---|---|---|---|
| **P-1** | **`Ctrl+K`** (mac: `⌘K`) | คนพิมพ์รัว | ว่าง · ระบบพิมพ์ `[` ให้ในบรรทัดก็ต่อเมื่อผู้ใช้เลือกคำสั่งที่เป็น `[…]` (ดู §8) |
| **P-2** | **พิมพ์ `[` ในบรรทัดโน้ต** → รายการแนะนำเปิดเอง (**ไม่แย่งโฟกัส** — ดู §2.2) | คนที่ไม่รู้ว่ามี Ctrl+K | ตัวที่พิมพ์ต่อจาก `[` = คำค้น · บรรทัดจริงเห็น `[` อยู่แล้ว |
| **P-3** | **ปุ่ม `⌘ คำสั่ง · Ctrl+K` ในแถบเครื่องมือ** (เห็นตลอดเวลา) | ทัช/แท็บเล็ต + คนที่ไม่เคยรู้จัก palette | ว่าง |

**ทำไม `[` ถึงเป็นประตูที่ถูก ไม่ใช่ระบบที่สอง:** คำสั่งทุกตัวใน §3.1 ขึ้นต้นด้วย `[` อยู่แล้ว และ
`[ ]` เป็นพื้นที่สงวนของคำสั่งระบบ (ห้ามใช้ในเนื้อร้อง) ⇒ การพิมพ์ `[` **คือการเริ่มเขียนคำสั่ง**อยู่แล้วโดยนิยาม ·
palette แค่ยื่นรายการมาให้ตอนที่ผู้ใช้เริ่มเขียนพอดี = autocomplete ไม่ใช่ภาษาใหม่
(เทียบ: VS Code/GitHub ใช้ตัวนำหน้า `>` `#` `@` เป็นตัวเปลี่ยนโหมดในช่องเดียวกัน — เราไม่ต้องประดิษฐ์ตัวนำหน้าเลย
เพราะภาษาของเรามี `[` เป็นตัวนำหน้าอยู่แล้ว)

### 2.1 กติกาคีย์ (dev ต้องทำตามนี้เป๊ะ)

1. **จับด้วย `event.code === 'KeyK'` ⛔ ไม่ใช่ `event.key`** — `code` ผูกกับ**ปุ่มกายภาพ** จึงไม่เปลี่ยนตามภาษาแป้น
   (W3C UI Events: `code` = "the physical key"; `key` = อักขระที่ได้ ซึ่งเปลี่ยนตาม layout)
   ⚠️ ของเดิมในแอปทำผิดข้อนี้อยู่: `EditorMode.vue:2060` ใช้ `e.key.toLowerCase()` จับ Ctrl+Z/Ctrl+Y
   → **น่าจะ**พังเมื่อแป้นอยู่โหมดไทย (ยังไม่ได้ทดสอบจริง — อยู่ใน §13)
2. รับทั้ง `ctrlKey` และ `metaKey` · **ต้องไม่ทำงานเมื่อ `altKey`** (สงวน `Ctrl+Alt+K` ไว้เป็นทางสำรอง ดูข้อ 4)
3. `preventDefault()` เสมอเมื่อจับได้
4. **ทางสำรองที่ต้องมี: `Ctrl+Alt+K`** เปิด palette ได้เหมือนกัน — เพราะ `Ctrl+K` เป็นคีย์ที่เบราว์เซอร์ก็ใช้
   (โฟกัสช่องค้นหาของเบราว์เซอร์) และในบางบริบทของ**ตัวแก้ข้อความ** เจ้าอื่นก็ชนกัน ·
   **แหล่งอ้างที่เปิดตรวจได้:** GitHub ประกาศคีย์ของ command palette ไว้ว่า `Ctrl+K` (Win/Linux) / `⌘K` (mac)
   **และให้ `Ctrl+Alt+K` เป็นคีย์สำรองสำหรับตอนที่กำลังแก้ข้อความ Markdown** — <https://docs.github.com/en/get-started/accessibility/github-command-palette>
   ⇒ ทำตามรูปแบบที่พิสูจน์แล้วในแอปที่คนใช้ทุกวัน ไม่ใช่รูปแบบที่เราคิดเอง
5. **Esc** ปิด และ **คืนเคอร์เซอร์กลับตำแหน่งเดิมเป๊ะ** (โน้ตตัวเดิม · ตำแหน่ง caret เดิม) — ห้ามทำ selection หาย
6. คลิกนอกกล่อง = ปิดเหมือน Esc · ⛔ ห้ามปิดเมื่อผู้ใช้แค่เลื่อนผลลัพธ์
7. 🔴 **ถ้าโฟกัสไม่ได้อยู่ในแผ่นเพลงตอนกด `Ctrl+K`** (ผู้ใช้เพิ่งคลิกที่ว่าง) → **ดึงโฟกัสกลับไปที่เคอร์เซอร์ล่าสุดก่อน แล้วค่อยเปิด palette** · ถ้าไม่เคยมีเคอร์เซอร์เลย = วางที่โน้ต/ช่องแรก · ⛔ ห้ามให้ `Ctrl+K` ตายเงียบเพราะไม่มีที่แทรก (จุดที่ Pro จับ — "ประตูบานนี้จะเปิดไม่ออกในหลายสถานการณ์")

> ⚠️ **ที่ยังไม่พิสูจน์:** ผมยิงคีย์ Ctrl+K ผ่าน CDP บน Chromium ของ session นี้แล้ว หน้าเว็บได้รับ
> `keydown {code:'KeyK', ctrlKey:true}` จริง **แต่คีย์สังเคราะห์ไม่ปลุกคีย์ลัดระดับเบราว์เซอร์เลย
> (แม้ในรอบที่ไม่เรียก preventDefault)** ⇒ การทดลองนี้ **ไม่ได้พิสูจน์** ว่า preventDefault ชนะเบราว์เซอร์ ·
> สิ่งที่ยืนยันได้คือหลักฐานทางอ้อมจาก GitHub/VS Code ที่ใช้คีย์นี้ในเบราว์เซอร์ได้จริง (ข้อ 4) ·
> **ต้องมีคนกดด้วยนิ้วจริง 1 ครั้งก่อนประกาศใช้** (§13 ข้อ 1)

### 2.2 🔴 P-2 (`[`) = **รายการแนะนำที่ห้ามแย่งการพิมพ์** (แก้ตามที่ G ค้าน — รับเต็ม)

ประตูนี้ต่างจาก P-1/P-3 ตรงที่ **มือผู้ใช้ยังพิมพ์อยู่บนแผ่นเพลง** ⇒ ถ้ามันยึดคีย์เมื่อไหร่ คนพิมพ์รัวพัง
(เคสจริงที่ G ยกมา: ระบบที่ preselect แถวแรกแบบ VS Code ทำให้คนพิมพ์เร็วกด Enter แล้วได้ของที่ไม่ได้ตั้งใจ
หรือเคอร์เซอร์ไม่ยอมไปตัวถัดไป — **นี่เป็นความเห็นจากประสบการณ์ของ G ไม่ใช่มาตรฐาน** แต่ตรงกับข้อเท็จจริง
ในตัวแก้ของเราที่ **Enter = ไปโน้ตตัวถัดไป** อยู่แล้ว `SongViewer.vue:389`)

| กฎ | รายละเอียด |
|---|---|
| **โฟกัสไม่ขยับ** | โฟกัสยังอยู่ที่ช่องพิมพ์ของแผ่นเพลง · รายการลอยใต้เคอร์เซอร์ · ⛔ ห้าม `focus()` ไปที่ลิสต์ |
| **ยังไม่มีแถวไหนถูกเลือก (unselected by default)** | ⇒ `Enter` `space` และอักขระทุกตัว **ไหลลงบรรทัดตามปกติ 100%** |
| **`↓` = ติดอาวุธ** | กด `↓` (หรือแตะแถว) ครั้งแรกจึงมีแถวถูกเลือก · ตั้งแต่นั้น `Enter` = "เอาแถวนี้" · `↑` เลื่อนกลับจนหลุดแถวแรก = กลับสู่สถานะไม่มีแถวถูกเลือก |
| **ยกเว้น: ตรงเป๊ะตัวเดียว = ติดอาวุธให้เลย** | ถ้าสิ่งที่พิมพ์ต่อจาก `[` **แมตช์คำสั่งเดียวแบบตรงเป๊ะ** (เช่นพิมพ์ `[ซ้ำ` เหลือ `[ซ้ำ]` อันเดียว) → preselect อันนั้นให้ `Enter` ยืนยันได้เลย · **ปลอดภัยเพราะการยืนยัน = ได้ข้อความเดียวกับที่พิมพ์เองจนจบเป๊ะ** ไม่ใช่ของเซอร์ไพรส์ (แก้ข้อกังวลรอบ Pro โดยไม่ทิ้งการปกป้องคนพิมพ์รัว — ดู §13) |
| **`Esc`** | ปิดรายการเฉย ๆ · `[` และตัวอักษรที่พิมพ์ไปแล้ว **ยังอยู่ในบรรทัด** · เคอร์เซอร์ไม่ขยับ |
| **ไม่มีดีเลย์** | รายการต้องไม่หน่วงการพิมพ์ (กรองแบบ sync บนทะเบียนไม่กี่สิบรายการ · ไม่ต้อง debounce จนสะดุด) |

> P-1 (`Ctrl+K`) และ P-3 (ปุ่ม) ตรงกันข้าม: **โฟกัสย้ายเข้าช่องค้นของ palette** และไฮไลต์แถวแรกได้
> เพราะผู้ใช้ตั้งใจ "หยุดพิมพ์เพื่อมาหา" แล้ว

### 2.3 ⛔ ห้ามแตะประวัติย้อน/ทำซ้ำ (จุดที่ G จับได้ · ผมมองข้าม)

การ **เปิด/ปิด/ยกเลิก** palette ต้องไม่สร้างรายการในประวัติ Undo เลย ·
มีเพียง **การแทรกจริง** เท่านั้นที่เป็น 1 ก้าวของ `Ctrl+Z`
(อาการที่จะเกิดถ้าพลาด: เปิด palette แล้ว Esc → กด `Ctrl+Z` แล้วได้ "ย้อนการเปิด palette"
แทนที่จะย้อนโน้ตที่เพิ่งพิมพ์ = ผู้ใช้เสียงานเงียบ ๆ)

## 3. พิมพ์แล้วค้นอะไรได้

### 3.1 ทะเบียนคำสั่ง = ไฟล์เดียว (นี่คือกลไกกัน "2 ระบบ")

```js
// src/lib/editorCommands.js — ทะเบียนเดียวของทั้งแถบปุ่มและ palette
{
  id: 'repeat-end',
  group: 'วนซ้ำ',              // หมวด — ชุดเดียวกับที่แถบปุ่มใช้ (§4)
  kind: 'insert',             // 'insert' (ลงบรรทัด) | 'action' (สั่งงาน ไม่ลงบรรทัด) — คุมไอคอน+ตำแหน่ง §4.1
  barLabel: 'ซ้ำ',            // ชื่อสั้นบนปุ่มแถบ (พื้นที่จำกัด)
  paletteLabel: 'ซ้ำท่อนนี้',  // ชื่อเต็มใน palette (อธิบายได้ยาว) — แยกจาก barLabel (จุดที่ Pro ชี้ว่าทะเบียนไฟล์เดียวมักพัง)
  alias: ['ซ้ำ', 'วน', 'ร้องอีก', 'repeat', 'ย้อน'],   // คำพ้อง ไทย+อังกฤษ (คีย์ค้นซ่อน)
  insert: '[ซ้ำ]',            // ⭐ ข้อความที่ลงบรรทัด = สิ่งที่พิมพ์เองได้เป๊ะ (kind:'insert' เท่านั้น)
  run: null,                  // ฟังก์ชันสำหรับ kind:'action' (เช่น ฟังบรรทัดนี้) — payload แยกจาก insert
  arg: null,                  // หรือ { placeholder:'2', pattern:'[ซ้ำ N รอบ]' }
  onBar: false,               // โผล่บนแถบปุ่มด้วยไหม
  available: (ctx) => true,   // สิทธิ์แสดงตามบริบท — ปุ่มแถบอาจกดได้เสมอ แต่ palette แสดงเฉพาะบางสถานะ (Pro §5)
  scope: 'note',              // 'note' | 'word' | 'song' — ชั้นหยาบที่ `available` อ่านต่อ
  needs: ['G16'],             // ปลดล็อกเมื่องานนี้เสร็จ (ก่อนหน้านั้นไม่ลงทะเบียน)
}
```

> **ทำไม 4 ฟิลด์นี้ (`kind` · `barLabel`/`paletteLabel` แยก · `run` แยกจาก `insert` · `available`) ต้องมีตั้งแต่วันแรก:**
> รอบ Pro ชี้ 3 จุดที่ทะเบียนไฟล์เดียว "มักพังตอนโต" — ชื่อบนปุ่มต้องสั้นแต่ใน palette ต้องอธิบายยาว ·
> คำสั่งบางตัวกดจากแถบได้เสมอแต่ palette ต้องเช็คบริบท · action ในอนาคตต้องการ payload ไม่ใช่ string ธรรมดา ·
> ทั้งหมดเป็น**การเผื่อโครง ไม่ใช่งานเพิ่มในรุ่นนี้** (ค่าปริยายง่าย ๆ ก็เดินได้)

**กฎเหล็ก:** แถบปุ่ม (`NoteInputBar.vue`) ต้อง render จากทะเบียนนี้ **โดยกรอง `onBar:true`** ·
⛔ ห้ามมีลิสต์สัญลักษณ์ 2 ชุดในโค้ด (วันนี้ `SYMBOL_GROUPS` ถูก hard-code อยู่ใน `NoteInputBar.vue:38-64`
และอักขระที่คีย์บอร์ดรับก็ hard-code แยกอีกที่ที่ `SongViewer.vue:295 SYMBOL_CHARS` — **สองที่นี้ต้องยุบเป็นทะเบียนเดียว
พร้อมกับงานนี้** ไม่งั้น palette จะกลายเป็นชุดที่ 3)

### 3.2 ค้นด้วยอะไรได้บ้าง

| ค้นด้วย | ตัวอย่างพิมพ์ | ได้ |
|---|---|---|
| ชื่อไทย | `ซ้ำ` | ซ้ำท่อนนี้ · เริ่มซ้ำ · ข้อนี้ไม่ซ้ำ |
| คำพ้องไทย | `วน` `ร้องอีก` | เหมือนบรรทัดบน |
| ชื่อสากล/โรมัน | `repeat` `volta` `fine` `dc` `segno` `coda` | รายการเดียวกัน |
| **ตัวอักขระเอง** | `^` `~` `_` | ปุ่มสัญลักษณ์ตัวนั้น (พร้อมบรรทัด "อยู่ตรงไหนบนแป้น" §4.1) |
| ตัวเลข/ค่า | `3/4` `120` | เปลี่ยนจังหวะ · ตั้งความเร็ว |

**กติกาการแมตช์** (reuse `normalize()` จาก `src/lib/songSearch.js` — NFC + lower + ยุบช่องว่าง · ห้ามเขียนใหม่):
1. **ขึ้นต้นตรง** (prefix) ชนะ **มีอยู่ข้างใน** (substring) ชนะ **fuzzy** (ระยะแก้ ≤1 ต่อ 5 ตัวอักษร)
2. แมตช์ที่ **ชื่อหลัก** ชนะแมตช์ที่ **คำพ้อง**
3. **ไฮไลต์ตัวอักษรที่แมตช์** ในผลลัพธ์ (ผู้ใช้เห็นว่าทำไมแถวนี้ขึ้นมา)
4. ไม่เจอ = **ไม่เดาแทน** — ขึ้นบรรทัดเดียว *"ไม่พบ 'xxx' — ลองคำใกล้เคียง: …"* (P5 · ห้ามแทรกอะไรเอง)
5. ⛔ **ไม่ทำ** การเดาว่าพิมพ์ผิดภาษา (ไทย↔อังกฤษ) ในรุ่นนี้ — ดู §10

หมวดชุดเดียวที่ใช้ทั้งแถบปุ่มและ palette · **ลำดับนี้คือลำดับที่แสดงเสมอ**
(3 หมวดแรก = ของที่แถบปุ่มแสดงอยู่แล้ววันนี้ ชื่อเดิมเป๊ะ ไม่เปลี่ยนคำ)

| # | หมวด | สมาชิก | อยู่บนแถบปุ่มด้วยไหม |
|---|---|---|---|
| 1 | **ความยาว** | `_` เขบ็ต · `.` จุดเพิ่ม · `-` ลากเสียง · `~` โยงเสียง · `^` ยืดเสียง · (`!` staccato เมื่อ G3 เสร็จ) | ✅ ทั้งหมด |
| 2 | **เสียง** | สูงหนึ่งช่วง `'` · ต่ำหนึ่งช่วง `,` · `#` · `b` · `n` | ✅ (บางตัวเฉพาะมือถือ) |
| 3 | **กลุ่ม/ห้อง** | `(` `)` `{` `}` `\|` | ✅ |
| 4 | **คอร์ด** | ใส่/เปลี่ยน/ลบคอร์ด | ✅ (ปุ่ม "คอร์ด ▾") |
| 5 | **วนซ้ำ** | `[ซ้ำ]` `[เริ่มซ้ำ]` `[ซ้ำ N รอบ]` `[รอบ 1]` `[รอบ 2,3]` `[รับ]` `[ข้อนี้ไม่ซ้ำ]` | ❌ palette เท่านั้น |

### 4.1 🔴 หมวด 8 "คำสั่ง (ไม่ลงบรรทัด)" — 3 กฎกันกดผิด (ตกลงกับ G แล้ว)

G ค้านว่าคำสั่ง "แทรกข้อความ" กับ "สั่งงานระบบ" คนละ mental model — แทรกข้อความ ผลอยู่ที่เคอร์เซอร์ ·
สั่งงานระบบ ผลอยู่ที่ทั้งหน้าจอ · **ถ้ากดผิดไปโดนคำสั่งสั่งงาน (เช่นเล่นเสียง) = สมาธิคนพิมพ์ขาดทันที**
G เสนอให้แยกเป็นคนละ palette · **ผมไม่แยก** เพราะ palette 2 ตัว = ผู้ใช้ต้องจำว่าของอยู่ประตูไหน = ระบบ 2 ชุด
(สิ่งที่ห้ามไว้ตรง ๆ) → แยกด้วย**โครงในตัวเดียว** แทน · G ตรวจแล้วตอบว่า *"ปิดความเสี่ยงได้ครบ ไม่มีจุดตายเพิ่ม"*:

1. **อยู่ท้ายสุดเสมอ** ใต้หัวข้อ `คำสั่ง (ไม่ลงบรรทัด)` — ไม่ว่าคะแนนการแมตช์จะสูงแค่ไหน · **ห้ามเข้าโซนใช้ล่าสุด** (§4 การเรียงผล)
2. **ไม่มีทางถูกไฮไลต์เอง** — ต้องกด `↓` ลงไปถึงหรือแตะเท่านั้น (Enter รัว ๆ ไม่มีทางไปโดน)
3. **หน้าตาแถวต่างชัดด้วยไอคอน** — แถว `insert` มีช่องข้อความ mono ท้ายแถว (สิ่งที่จะลงบรรทัด · §8 ข้อ 1) · แถว `action` **ไม่มี mono + มีไอคอนต่างกลุ่ม** (เช่นเฟือง/ไอคอนคำสั่ง) เพื่อบอกด้วยตาว่า "อันนี้สั่งงาน ไม่ได้พิมพ์ลงเพลง" — รอบ Pro ย้ำว่าต้องแยกด้วย **ไอคอน**ไม่ใช่แค่ตำแหน่ง (kind ในทะเบียน §3.1 คุมข้อนี้)

**การเรียงผล — 🔄 แก้จากร่างแรกหลังรอบ Pro (ดู §13 "รอบ Pro ต่างจากรอบ Flash ตรงไหน"):**
- **ยังไม่พิมพ์อะไร → มีโซน `ใช้ล่าสุด` (สูงสุด 3 แถว) บนสุด → เส้นคั่น → หมวดตายตัวตามตารางนี้**
  - ร่างแรกผมเขียนว่า "ตายตัวล้วน ห้ามสลับตามความถี่" ด้วยเหตุผลกล้ามเนื้อมือจำตำแหน่ง · **รอบ Pro ค้านและผมรับ**:
    เหตุผลกล้ามเนื้อจำตำแหน่งเป็นของ **แถบปุ่ม 2 มิติ (spatial · Fitts's Law)** — ปุ่มอยู่พิกัดเดิมตาเล็งได้ ·
    แต่ **palette เป็นลิสต์ 1 มิติที่เดินด้วยลูกศร/พิมพ์ (temporal)** ไม่ใช่พิกัด → ถ้าคำสั่งที่ใช้บ่อยที่สุด
    ตกไปอยู่แถวที่ 28 ต้องกด `↓` 28 ครั้งหรือพิมพ์ค้นทุกครั้ง = ภาระจริง (Hick's Law)
  - **ขอบเขตที่กันของเดิมพัง:** โซนใช้ล่าสุด **สูงสุด 3 แถว · มีป้าย `ใช้ล่าสุด` ชัด · หมวดตายตัวข้างล่างไม่ขยับเลย**
    → ยังคงหลัก"แถบปุ่มห้ามขยับ" (แถบปุ่มคนละเรื่อง ไม่มี MRU) และหลัก "สอนภาษา" (ลิสต์เต็มยังอยู่ครบข้างล่าง)
  - ⚠️ **ที่ Pro อ้างว่า "VS Code/Slack โชว์ recently-used บนสุด" ผมเปิด docs ทางการของ VS Code แล้ว
    ไม่พบระบุพฤติกรรมนี้** → ถือเป็น**ข้อสังเกต ไม่ใช่มาตรฐานที่อ้างได้** · การตัดสินใจนี้ยืนบนเหตุผล
    Hick's Law + เคสแถวที่ 28 เอง ไม่ได้ยืนบน "เพราะเจ้าอื่นทำ"
- **พิมพ์แล้ว → เรียงตามคะแนนการแมตช์ (§3.2)** · คะแนนเท่ากันใช้ความถี่ของผู้ใช้คนนี้เป็นตัวตัด · **โซนใช้ล่าสุดหายไป** (พิมพ์แล้ว = กำลังหา ไม่ใช่หยิบซ้ำ)
- แสดงหัวข้อหมวดคั่นเสมอ (ทั้งตอนว่างและตอนกรอง) — ผู้ใช้เรียนโครงของภาษาไปในตัว
- โซนใช้ล่าสุดเก็บใน localStorage (ไม่มีบัญชี ตรง mission) · คำสั่งหมวด 8 "ไม่ลงบรรทัด" **ห้ามเข้าโซนใช้ล่าสุด** (กันกดพลาดไปโดนคำสั่งสั่งงานตั้งแต่แถวบน — §4.1)

## 5. เลือกด้วยคีย์บอร์ดล้วน

| คีย์ | ทำอะไร |
|---|---|
| `↓` `↑` | เลื่อนแถบเลือก (วนรอบที่หัว/ท้ายลิสต์) |
| `Enter` | แทรกคำสั่งที่เลือก แล้วปิด · เคอร์เซอร์อยู่ต่อจากข้อความที่แทรก (หรือคลุมค่าที่ต้องกรอก §7) |
| `Esc` | ปิด · คืนเคอร์เซอร์ที่เดิมเป๊ะ · ถ้าเปิดด้วย `[` (P-2) ให้ `[` ที่พิมพ์ไว้**ยังอยู่** (ผู้ใช้พิมพ์ต่อเองได้) |
| `Home` / `End` | แถวแรก / แถวสุดท้าย |
| ⛔ `Tab` | **ห้ามผูก** — ในโหมดแก้ Tab = ข้ามโน้ตอยู่แล้ว (`EditorMode.vue:2154`) |

**โครงสร้าง a11y = combobox + listbox ตาม W3C ARIA APG**
(<https://www.w3.org/WAI/ARIA/apg/patterns/combobox/>) — โฟกัส **อยู่ที่ช่องพิมพ์ตลอด**:
- ช่องค้น: `role="combobox"` · `aria-expanded` · `aria-controls="<id ของ listbox>"` · `aria-activedescendant="<id แถวที่เลือก>"`
- ลิสต์: `role="listbox"` · แต่ละแถว `role="option"` + `aria-selected` บนแถวที่เลือก
- จำนวนผลลัพธ์ประกาศผ่าน live region (เช่น *"3 รายการ"*) เพื่อให้คนใช้โปรแกรมอ่านหน้าจอรู้ว่าลิสต์เปลี่ยน
- `Esc` ปิดลิสต์แล้วโฟกัสกลับช่องค้น → ปิดอีกครั้ง = ปิด palette (ตาม APG)
- 🔴 **ทุกครั้งที่เลื่อนด้วย `↑↓` ต้องสั่ง `el.scrollIntoView({block:'nearest'})` ที่แถวที่เลือกด้วย**
  — `aria-activedescendant` **ไม่ทำให้ลิสต์เลื่อนตามให้เอง** แถวที่ถูกเลือกจึงหลุดออกนอกจอได้ทั้งที่ระบบบอกว่าเลือกอยู่
  (จุดที่ G จับได้ · ตรงกับที่ APG อธิบายว่าผู้เขียนต้องดูแลการเลื่อนเอง)
- ⚠️ **`aria-haspopup` ไม่ได้บังคับ** สำหรับ popup ที่เป็น listbox — รายการที่ APG ระบุคือ `role=combobox` ·
  `aria-expanded` · `aria-controls` · `aria-activedescendant` เท่านั้น · ใส่เพิ่มก็ไม่ผิด แต่ **ห้ามเขียนว่าเป็นข้อบังคับ**
  (G ยืนยันรอบแรกว่า "ต้องมี" แล้ว **ถอนคำเองรอบ 2 ว่า "ไม่มี ผมพูดเกิน"** เมื่อถูกขอประโยคในสเปก — transcript `04-reply-from-G-round2.md`)

## 7. คำสั่งที่ต้องกรอกค่า

`[ซ้ำ N รอบ]` · `[รอบ 1,2]` · `[3/4]` · `[Key:C]` · `[T:120]`

**ทำแบบ snippet placeholder ไม่ใช่หน้าต่างถามค่า:** กด Enter แล้ว **แทรกข้อความทั้งก้อนลงบรรทัดเลย
โดยคลุม (select) เฉพาะส่วนค่าไว้** → พิมพ์ทับได้ทันที · กด Esc/ลูกศร = รับค่าตั้งต้นตามที่แทรก

```
เลือก "ซ้ำ N รอบ" →  … | [ซ้ำ 2 รอบ]        ← "2" ถูกคลุมไว้ พิมพ์ 3 ทับได้เลย
```

**⛔ 2 ข้อบังคับที่ต้องทำ ไม่งั้นพังเงียบ:**
1. 🔴 **หยุดการ re-render พรีวิว/autosave จนกว่าการคลุมจะถูกใช้หรือโฟกัสหลุด** — ถ้ามีอะไรไป
   render ใหม่หลังแทรก **การคลุมจะหลุดโดยไม่มีสัญญาณ** แล้วตัวเลขที่ผู้ใช้พิมพ์จะ**ต่อท้ายแทนที่จะทับ**
   (`[ซ้ำ 2 รอบ]` → `[ซ้ำ 23 รอบ]`) = ข้อมูลผิดแบบที่ผู้ใช้ไม่ทันสังเกต · **จุดนี้ G จับได้ ผมมองข้าม**
2. ค่าที่กรอกได้ **จำกัดเป็นตัวเลข/ASCII เท่านั้น** (`2` `3/4` `120` `C`) — ไม่มีค่าไหนเป็นภาษาไทย
   จึงไม่มีเรื่อง composition ของ IME ไทยมาเกี่ยว (นี่คือเหตุผลที่ผมไม่ถอยไปใช้ inline input step ตามที่ G เสนอ)
3. **`Esc` ตอนตัวเลขถูกคลุม = ยกเลิกการคลุมเฉย ๆ ข้อความยังอยู่** (palette ปิดไปแล้ว ไม่มีสถานะซ้อน → ไม่มี Esc 2 ชั้น)
4. **ถ้าทดสอบจริงแล้วพบว่าการคลุมไม่นิ่ง** (IME/re-render) → ถอยไปใช้ **inline input step ในแถว palette**
   (กด Enter → แถวกลายเป็นช่องกรอกตัวเลข → Enter อีกครั้งจึงแทรกทั้งก้อนครั้งเดียว) ตามที่ G เสนอ

**ทำไมไม่ถามค่าในกล่อง palette ตั้งแต่แรก:** (ก) ผลลัพธ์ที่ลงบรรทัดต้องเป็นข้อความที่พิมพ์เองได้เป๊ะอยู่แล้ว —
การให้ผู้ใช้เห็นข้อความจริงแล้วแก้ตัวเลขในนั้น **สอนภาษาไปด้วยในตัว** ตรงกับหลักที่ล็อก ·
(ข) ไม่ต้องออกแบบ UI ขั้นที่ 2 ที่จะโตเป็น wizard (ซึ่ง §3.1 ตัดทิ้งไปแล้วเพราะขัด keyboard-first)

## 8. สิ่งที่เกิดขึ้นตอนแทรก (หัวใจของหลักที่ล็อก)

1. แต่ละแถวใน palette **แสดงข้อความจริงที่จะลง** ด้วยฟอนต์ mono ท้ายแถว เช่น `[ซ้ำ 2 รอบ]` / `^`
   → ผู้ใช้เห็น "คำตอบ" ทุกครั้งที่ใช้ ไม่ใช่เห็นแค่ชื่อปุ่ม
2. แทรกที่ตำแหน่งเคอร์เซอร์ **ผ่านทางเดินเดียวกับที่ปุ่มบนแถบใช้** (`emit('symbol')` / `markSel()` / `insertBoxSel()` …)

## 10. ⛔ สิ่งที่ palette **ไม่** ทำในรุ่นนี้ (แต่เผื่อโครงไว้)

| ไม่ทำตอนนี้ | เผื่อไว้ยังไง |
|---|---|
| ค้นข้ามเพลง / เปิดเพลงอื่น / ไปหน้าอื่น | ฟิลด์ `scope` ในทะเบียน (`'song'`) รองรับอยู่แล้ว แค่ยังไม่ลงทะเบียนรายการ |
| คำสั่งระดับแอป (พิมพ์ A4 · แชร์ · ดาวน์โหลด · ตั้งค่าเสียง) | หมวด 8 ในตาราง §4 — ใส่เมื่อของถูกยกเข้าตัวแก้ใหม่แล้ว (ตามมติ PM) |
| palette บน**หน้าอ่าน** (ไม่ได้อยู่โหมดแก้) | ไม่ผูกอะไรตายตัว · ตัวเปิด (`Ctrl+K`) ลงทะเบียนเฉพาะตอน `editMode` |
| เดาว่าพิมพ์ผิดภาษา (ไทย↔อังกฤษ) | แยกเป็นฟังก์ชันแมตช์ต่างหาก เติมทีหลังได้โดยไม่แตะ UI · **เหตุผลที่ยังไม่ทำ:** สมมติฐานว่ากล่องโน้ตบังคับโหมดละตินอยู่แล้วยังไม่ได้พิสูจน์ (§13) — ทำก่อนพิสูจน์ = เดา |
| ประวัติคำสั่งล่าสุด / คำสั่งโปรด | เก็บสถิติการใช้ตั้งแต่ P1 (localStorage) แต่ใช้แค่เป็นตัวตัดสินเมื่อคะแนนเท่ากัน (§4) |
| ทำงานหลายคำสั่งรวด (macro) | ไม่เผื่อ — ยังไม่มีหลักฐานว่าต้องการ |
| **ตัวนำหน้าเปลี่ยนโหมดค้น** (`@` = สัญลักษณ์/คอร์ด · พิมพ์เปล่า = คำสั่ง) | 🆕 **เผื่อจากรอบ Pro** — เมื่อลิสต์แบนราบเริ่มพัง ให้เพิ่ม prefix routing แบบ VS Code (`>` `@`) เพื่อหดขอบเขตการค้นโดยไม่ต้องมี dropdown ตัวกรอง (ต้องใช้เมาส์) · **ยังไม่ทำในรุ่นนี้** เพราะรุ่นนี้ ~30 รายการ ยังไม่ถึงจุดพัง |

### 10.1 🆕 เกณฑ์ตัดสินว่า "ถึงเวลาต้องเพิ่ม prefix routing" (จากรอบ Pro)

ลิสต์แบนราบไม่มีตัวนำหน้า **เริ่มพังเมื่อเข้าเงื่อนไขใดเงื่อนไขหนึ่ง** (Pro ให้ตัวเลข · ผมเก็บเป็น**เกณฑ์เฝ้าดู** ไม่ใช่คำสั่งทำตอนนี้):

## 12. เกณฑ์ตรวจรับ (ให้ tester ใช้ตรงนี้ได้เลย)

1. เปิดโหมดแก้ → กด `Ctrl+K` → palette เปิด · ช่องค้นได้โฟกัส · ⛔ ช่องค้นหาของเบราว์เซอร์ต้องไม่เปิด **(กดด้วยนิ้วจริง)**
2. พิมพ์ `ซ้ำ` → มีผลลัพธ์ · `↓↓` + `Enter` → ข้อความที่ลงบรรทัด **ตรงกับที่แถวนั้นแสดงไว้ทุกตัวอักษร**
3. พิมพ์คำสั่งเดียวกันด้วยมือล้วน (ไม่เปิด palette) → ผลบนแผ่นเพลง **เหมือนกันเป๊ะ** (นี่คือการทดสอบหลักที่ล็อก)
4. กด `[` ในบรรทัดโน้ต → palette เปิดเอง · กด Esc → `[` ยังอยู่ · เคอร์เซอร์อยู่ที่เดิม
5. ปุ่ม `คำสั่ง · Ctrl+K` มองเห็นได้ที่ 360/768/1280 **โดยไม่ต้อง hover** และตอนบังคับ `hover:none`
6. แถวลิสต์สูง ≥24px (ผ่าน AA) · ที่ตั้งไว้จริง 44px · โฟกัสมี `focus-visible` เห็นชัด
7. โปรแกรมอ่านหน้าจอ: ช่องค้นประกาศเป็น combobox · เลื่อน `↓` แล้วประกาศชื่อแถว (aria-activedescendant)
8. มือถือ 360/412: แผ่นขึ้นจากล่างเต็มความกว้าง · ช่องค้นไม่ถูกคีย์บอร์ดบัง · ไม่มีเลื่อนซ้าย-ขวา
9. 🔴 **ประวัติไม่เพี้ยน:** พิมพ์โน้ต 1 ตัว → เปิด palette → `Esc` → `Ctrl+Z` **ต้องย้อนโน้ตตัวนั้น**
   ⛔ ไม่ใช่ "ย้อนการเปิด palette" (§2.3)
10. 🔴 **Enter ไหลผ่านตอนยังไม่ติดอาวุธ:** พิมพ์ `[` → พิมพ์ต่อ → กด `Enter` โดยไม่กด `↓` เลย →
   ต้องได้พฤติกรรมเดิมของตัวแก้ (ไปโน้ตตัวถัดไป) ⛔ ห้ามมีคำสั่งไหนถูกแทรก (§2.2)
11. 🔴 **การคลุมค่าไม่หลุด:** เลือก `[ซ้ำ N รอบ]` → พิมพ์ `3` ทันที → ต้องได้ `[ซ้ำ 3 รอบ]`
   ⛔ ไม่ใช่ `[ซ้ำ 23 รอบ]` (§7)
12. เลื่อน `↓` ลงไปจนพ้นจอ → แถวที่เลือก **ต้องเลื่อนเข้ามาให้เห็นเอง** (§5 `scrollIntoView`)
13. กด `↓` ยาว ๆ ลงไปถึงหมวด "คำสั่ง (ไม่ลงบรรทัด)" ได้ · แต่ **ตอนเปิดมาใหม่ ๆ ต้องไม่มีแถวในหมวดนี้ถูกไฮไลต์เอง** (§4.1)
14. **ตรงเป๊ะตัวเดียว = ยืนยันได้เลย:** พิมพ์ `[` แล้วพิมพ์จนเหลือแมตช์เดียว → กด `Enter` → ได้คำสั่งนั้น (ต้องเท่ากับพิมพ์เองจนจบเป๊ะ · §2.2)
15. **โซนใช้ล่าสุด:** ใช้คำสั่ง A → ปิด → เปิด palette ใหม่ (ยังไม่พิมพ์) → A อยู่ในโซน `ใช้ล่าสุด` บนสุด (สูงสุด 3) · หมวดตายตัวข้างล่างไม่ขยับ · คำสั่งหมวด "ไม่ลงบรรทัด" **ไม่โผล่ในโซนนี้** (§4)
16. 🔴 **KPI ความเร็ว (ความเสี่ยงอันดับ 1 จากรอบ Pro):** จับเวลาเปิด palette → เห็นผลลัพธ์แรก · **ต้องไม่มี layout shift และไม่หน่วงจนคนพิมพ์รัวพิมพ์อักขระถัดไปแซง** · วิธีวัดระยะยาว = **Cancellation Rate** (เปิดแล้ว Esc/ลบทิ้งทันทีโดยไม่เลือกอะไร) — ถ้าทะลุ ~20% แปลว่าช้าเกินไปสำหรับผู้ใช้ (ดู §14 — ยังไม่มีเครื่องมือวัด · ต้องเพิ่ม)

### 13.2 ที่ทั้ง Flash และ Pro ตรงกัน (รับเข้าสเปกแล้วรอบก่อน)

`[` ห้ามแย่งโฟกัส/กิน Enter (§2.2) · เปิด-ปิดห้ามโผล่ใน Undo (§2.3 · §12/9) · `scrollIntoView` (§5) · การคลุมค่าหลุดได้ (§7 · §12/11) · แยก insert/action (§4.1)

> **สรุปคุณภาพแหล่ง:** รอบ Flash อ้างมาตรฐานผิด 1 ครั้งแล้วถอน (`aria-haspopup` · ครั้งที่ 3 ของโปรเจกต์ · 2 ครั้งก่อนใน `note-symbol-set.md` §8.2) · **รอบ Pro แยก "มาตรฐาน" กับ "ความเห็น" เองชัดเจน ไม่พลาดซ้ำ** แต่ยังมีข้ออ้างที่ผมต้องตัด (VS Code/Slack เรื่อง MRU · Notion/Slack เรื่อง ghost text) เพราะเปิด docs แล้วยืนยันไม่ได้ ⇒ กฎเดิมยังจริง: **เปิดของจริงก่อน เอาคำตอบ G มาถ่วง ไม่ใช่กลับกัน**

## 14. ยังไม่พิสูจน์ (ห้ามนับว่าผ่าน)

1. 🔴 **`preventDefault()` ชนะ `Ctrl+K` ของเบราว์เซอร์จริงไหม — ยังไม่พิสูจน์ด้วยการกดจริง**
   คีย์สังเคราะห์ผ่าน CDP ไม่ปลุกคีย์ลัดระดับเบราว์เซอร์เลย (ทดลองแล้วทั้งรอบที่ preventDefault และไม่ preventDefault
   ผลเท่ากัน = การทดลองไม่ตัดสินอะไร) · ที่มีคือหลักฐานทางอ้อม (GitHub ใช้คีย์นี้ในเบราว์เซอร์) →
   **ต้องมีคนกดด้วยนิ้ว 1 ครั้งบน Chrome/Edge/Safari ก่อนประกาศใช้** · ทางสำรอง `Ctrl+Alt+K` ต้องมีตั้งแต่ต้นเพราะเหตุนี้
2. 🔴 **แป้นภาษาไทยกับคีย์ลัด** — ผมเขียนกฎให้ใช้ `event.code` (ถูกตามสเปก) แต่ **ไม่ได้ทดสอบบนเลย์เอาต์ไทยจริง**
   (สลับเลย์เอาต์ระบบ = รบกวนเครื่องที่ P'Aim ใช้อยู่) · และข้อสังเกตว่า Ctrl+Z ของเดิมพังบนแป้นไทยเป็น **การอ่านโค้ด ไม่ใช่การวัด**
