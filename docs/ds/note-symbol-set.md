## 1. ปัญหาแกน: จุด `.` มี 4 ความหมาย

### 1.1 ของจริงวันนี้

| ความหมาย | เขียนยังไง | พบในคลังจริง |
|---|---|---|
| อ็อกเทฟ**ล่าง** | `.` **นำหน้า**เลข (`.5`) ซ้ำได้ | 1,641 |
| อ็อกเทฟ**บน** | `'` **ตามหลัง**เลข (`5'`) ซ้ำได้ | 868 |
| **จุดเพิ่มค่า** (×1.5 · `..` ×1.75) | `.` **ตามหลัง**เลข (`5.`) | 1,085 |
| **staccato** (จุดสั้น) | ⛔ **ยังไม่มีในระบบ** | 0 |

**3 ปัญหาซ้อนกัน:**
1. **`.` แบกความหมายเดียวกัน 2 อย่าง** แยกด้วยตำแหน่งเท่านั้น (ผิด P1)
2. **แนวคิดเดียวกัน (อ็อกเทฟ) อยู่คนละฝั่งคนละอักขระ** — ล่างใช้ `.` ข้างหน้า บนใช้ `'` ข้างหลัง
   → ผู้ใช้ต้องจำกฎ 2 ชุดสำหรับเรื่องเดียว
3. **staccato ยังไม่มี และเป็นระเบิดเวลา** — Wikipedia *Numbered musical notation* เตือนไว้เองว่า
   *"Special attention has to be paid on the staccato dot since it looks like the octave changer"*
   → ถ้าใส่ทีหลังโดยไม่วางระบบตอนนี้ = จุดที่ 4 มาชนของเดิมแน่นอน

### 1.2 ข้อพิสูจน์ว่านี่เป็นปัญหาจริง ไม่ใช่ปัญหาสมมติ

ของพังในคลังจริง **7 จุด / 5 เพลง** — **5 ใน 7 เป็นเรื่องลำดับของ modifier ล้วน**
(ใบแจ้ง `broken-notation-for-content-owner.md`):

| อาการ | สตริงที่พัง | ผู้ใช้ตั้งใจว่า | ทำไมพัง | order-free แก้ได้ไหม |
|---|---|---|---|---|
| 1 | `5^.` `2^.` | โน้ตยาว + ลากค้าง | จุดเพิ่มค่าต้อง**ก่อน** `^` | ✅ |
| 1 | `#4^_` `1^_` `2^_` | เขบ็ต + ลากค้าง | เขบ็ตต้อง**ก่อน** `^` | ✅ |
| 2 | `.#4` | ฟาชาร์ปเสียงต่ำ | จุดล่างต้อง**หลัง** `#` | ✅ |
| **3** | `~6)` · `…1_})` | ? | **`)` ไม่มี `(` คู่** | ❌ **คนละชั้น** |

**อาการ 3 (2 จุด) ไม่ใช่ปัญหาลำดับ** — เป็นวงเล็บไม่ครบคู่ ซึ่ง order-free แก้ไม่ได้และไม่ควรแก้
(ระบบไม่รู้ว่าคนตั้งใจให้เส้นโค้งเริ่มตรงไหน) → คงไว้เป็น 🔴 ชั้น "ผิดแน่" ใน validation §6

ผู้ใช้ไม่มีทางเดาลำดับที่ parser ต้องการได้ และไม่มีอะไรบอกเขาว่าผิด —
บนแผ่นเพลงมันพิมพ์ตัวอักษรดิบออกมาเฉยๆ (`NoteRow.vue:288`) และตอนกดฟังก็ข้ามโน้ตนั้นเงียบๆ

### 1.2.1 ⚠️ กับดักของ order-free ที่ต้องปิดก่อนสร้าง

**order-free จะเปลี่ยน "error ที่ดัง" ให้กลายเป็น "ยอมรับเงียบๆ"** — และนั่นอันตราย

ตัวอย่างจริงจากใบแจ้ง: `#4^_` `1^_` `2^_` ในเพลง #760 ท่อนรับ ·
คนเขียนใบแจ้งตั้งข้อสังเกตว่า **`^` ติดกัน 3 ตัวบนโน้ตเขบ็ต เป็นเรื่องผิดปกติทางดนตรี** →
อาจไม่ได้ตั้งใจใส่ค้างเสียงเลย แต่ใช้ `^` แทนสัญลักษณ์อื่นในหนังสือ
ถ้า order-free ไป normalize เป็น `#4_^` เงียบๆ **คำถามนี้จะไม่มีวันถูกถาม** และของผิดจะถูกตราว่าถูก

**กฎที่ต้องบังคับ (แยก 2 บริบท):**

| บริบท | ทำอะไร |
|---|---|
| **กำลังพิมพ์** (คนอยู่ตรงนั้น) | normalize ทันทีแบบ**เห็นได้** — กล่องแสดงรูปแบบที่เก็บจริง + โน้ตวาดใหม่เดี๋ยวนั้น · ผู้ใช้เห็นว่าเกิดอะไรขึ้น = เรียนรู้ไปในตัว |
| **ข้อมูลเก่าในคลัง** (ไม่มีใครอยู่ตรงนั้น) | ⛔ **ห้ามเขียนทับเงียบๆ** — ให้ขึ้นเป็น 🟡 *"ตรงนี้น่าจะหมายถึง `#4_^` — ใช่ไหม? [ใช่ แก้ให้เลย] [ไม่ใช่ ขอดูก่อน]"* · คนตัดสิน ไม่ใช่ระบบ |

เหตุผล: ของพัง 7 จุดนี้ **4 จุดที่ใบแจ้งเองยังบอกว่าต้องเปิดหนังสือต้นฉบับก่อน** —
ระบบไม่มีข้อมูลพอจะตัดสินแทน (สอดคล้องหลักเดิมของโปรเจกต์: *ไม่ชัวร์ = flag ห้ามเดา*
— `migrateToV2()` ทำแบบนี้อยู่แล้ว)

### 1.3 ทางแก้ที่เสนอ (ฟันธง)

> **ก. แยกอ็อกเทฟออกจาก `.` โดยใช้ `,` / `'` ตามมาตรฐานสากลของโน้ตแบบพิมพ์**
> **ข. ทำให้ลำดับ modifier ไม่มีความหมาย (order-free) แล้ว normalize ให้เอง**

**(ก) `,` = ลง 1 อ็อกเทฟ · `'` = ขึ้น 1 อ็อกเทฟ** (ซ้ำได้ `,,` `''`)

ไม่ใช่ของที่ผมคิดเอง — เป็น convention ของ **ระบบโน้ตแบบพิมพ์ด้วยคีย์บอร์ดระดับสากลทั้ง 2 ตัว**:

| ระบบ | ขึ้นอ็อกเทฟ | ลงอ็อกเทฟ |
|---|---|---|
| **LilyPond** (เอนจินพิมพ์โน้ตโอเพนซอร์สที่ใช้ทั่วโลก) | `c'` | `c,` |
| **ABC notation** (มาตรฐานโน้ตแบบข้อความ) | `c'` | `C,` |
| **ที่เสนอสำหรับ pleng** | `5'` (เหมือนเดิมเป๊ะ) | `5,` (ใหม่) |

**ทำไมดีกว่า:**
- `'` **ไม่เปลี่ยนเลย** — 868 จุดในคลัง + นิ้วพี่เปา ไม่กระทบ
- `,` กับ `'` อยู่ **ติดกันบนแป้น US** (แถวเดียวกัน `,` ล่างซ้ายของ `'`) · ทั้งคู่ไม่ต้องกด Shift
- ขึ้น/ลง = คู่ตรงข้ามที่หน้าตาเป็นคู่ตรงข้าม · อยู่**ฝั่งเดียวกัน** (หลังเลข) ทั้งคู่ → เหลือกฎเดียว
- ปลดปล่อย `.` ให้แปลว่า **"จุดเพิ่มค่า" อย่างเดียว** = ตรงกับ Wikipedia เป๊ะ
  (*"A dot after the plain or underlined note increases its length by half"*)

**ห้ามพังของเดิม (P4):** parser **ยังรับ `.` นำหน้าเลขเป็นอ็อกเทฟล่างตลอดไป** —
ข้อมูลเก่า 1,641 จุดไม่ต้องแตะ (และ ⛔ ห้าม re-import อยู่แล้ว) · นิ้วพี่เปาพิมพ์ `.5` ได้เหมือนเดิม
**แต่ตัวแก้จะ normalize ให้เป็น `5,` ทันทีที่พิมพ์จบกล่อง** (แบบเดียวกับที่ `fixAccidental()`
ย้าย `4#` → `#4` ให้อยู่แล้ววันนี้ — `NoteBoxes.vue:51-63`) → เป็น **superset ล้วน ไม่มี migration**

**(ข) order-free:** เมื่ออ็อกเทฟมีอักขระของตัวเองแล้ว **ทุก modifier ถูกระบุด้วยอักขระ ไม่ใช่ตำแหน่ง**
→ รับได้ทุกลำดับ แล้วเรียงกลับเป็นรูปแบบมาตรฐานให้เอง

```
พิมพ์อะไรก็ได้:   5^.   5.^   ^5.   .^5      →  เก็บเป็น  5.^
                  #4^_  4#_^  _^#4            →  เก็บเป็น  #4_^
                  .#4   #.4   #4,             →  เก็บเป็น  #4,
```
**ผลตรงๆ: ของพังในคลัง 6/6 จุด หายทันที และเกิดใหม่ไม่ได้อีก**

**(ค) staccato = `!`** — อักขระว่าง ไม่ชนกับอะไร · แสดงผลตาม Wikipedia ที่แนะให้เลี่ยงจุดกลมเพื่อกัน
สับสนกับจุดอ็อกเทฟ โดยใช้ **สามเหลี่ยมคว่ำ (staccatissimo)** หรือจุดหนา-ห่างกว่า
→ **เลือกสามเหลี่ยมคว่ำ** เพราะแยกออกด้วยตาในระยะพิมพ์งานจริง (จุดหนา/ห่าง ต่างกันน้อยเกินไป)

## 3. สัญลักษณ์ที่เป็น "คำ" (D.C. / D.S. / Fine / Coda) — ใส่ยังไงไม่ให้ปนกับเนื้อร้อง

> 🔴 **อยู่ในสโคป v2 ที่จะ release รอบนี้ — ไม่ใช่ของทีหลัง** (P'Aim 23 ก.ค.)

> ไม่ต้องเสียแรงกับ v1 ที่จะ obsolete — ดังนั้น Fine / D.C. และวนต่างๆ **ยังต้องมีใน v2 ที่จะ
> release นี้**"*
> → **ที่ห้ามทำคือไปปะของเดิมบน v1** · ที่ต้องทำคือ **สร้างของจริงใน v2** ให้ครบมาตรฐาน

**ปัญหา:** ถ้าให้พิมพ์ `D.C. al Fine` ลงไปตรงๆ มันจะไปนั่งในสายเดียวกับเนื้อร้อง/โน้ต และ

**ทางแก้: แยก "คำสั่งกระโดด" ออกเป็น item ชนิดของตัวเอง ไม่ใช่ข้อความ**

```jsonc
{ "type": "jump", "kind": "dc" | "ds" | "fine" | "coda" | "tocoda" | "segno" }
```
- **ผูกกับเอนจินจริง** — ต่อที่ `resolvePlayOrder()` ใน `songModel.js:189` ซึ่ง
  **คอมเมนต์ในโค้ดระบุไว้เองแล้วว่าเป็นจุดเสียบของเฟส 2** (*"This is the dispatch seam for
  Phase 2 jump symbols (D.C./D.S./Coda): they add more cases here; the display pass never changes"*)
  → **reuse ตรงตาม P6 ไม่ต้องเขียนใหม่**
- **ไม่ปนกับเนื้อร้อง** เพราะไม่ใช่ string ในสายเนื้อ · render เป็นตัวเอียงเหนือ/ท้ายห้องตามธรรมเนียมสากล
- **วิธีใส่:** ป๊อปอัป (เป็น "คำ" ไม่ใช่อักขระ = เข้าเงื่อนไข P3 ที่ต้องมีปุ่ม) ·

  (แบ็กสแลชนำ = ไม่มีทางชนกับโน้ต · เป็น pattern เดียวกับ LaTeX/LilyPond ที่นักพิมพ์โน้ตคุ้น)
  → **คนพิมพ์รัวไม่ต้องยกมือ · คนที่ไม่รู้ว่ามีก็เจอในป๊อปอัป** (ตอบข้อ discoverability §7.3)

## 4. หน้าตาป๊อปอัปที่เหลือจริง

### 4.1 แป้นสัญลักษณ์บนจอ — ✅ **เคาะแล้ว: เก็บไว้ · แสดงเป็นค่าเริ่มต้น · ทำให้มัน "สอนคีย์"**

**คำตอบจากพี่เปา (ผู้ใช้จริง) พลิกสมมติฐานเดิม:**
> ใช้ทั้ง 2 อย่าง · *"ถ้าคีย์บอร์ดอย่างเดียวก็ชอบ **แต่ขี้เกียจหา** เช่น `^` ไม่รู้อยู่ตรงไหนใน keyboard"*

→ **เขาไม่ได้ชอบเมาส์มากกว่าคีย์บอร์ด — เขาชอบคีย์บอร์ด แต่หาอักขระไม่เจอ**
→ **ปัญหาที่แท้จริงคือ discoverability ของ "ตัวอักขระอยู่ตรงไหนบนแป้น" ไม่ใช่ "ปุ่มบนจอเยอะไป"**
→ ⛔ **ห้ามตัดแถบทิ้ง · ค่าเริ่มต้น = แสดง** · (ทาง ก "พับได้" ยังคงไว้เป็นทางเลือกของผู้ใช้ ไม่ใช่ค่าเริ่มต้น)

#### รูปแบบที่ถูกต้องเรียกว่าอะไร — และมาตรฐานว่าไง

รูปแบบ "ปุ่มบนจอที่บอกคีย์ลัดของตัวเองไปในตัว" = **key-equivalent annotation**
(บางที่เรียก *shortcut hint* / *accelerator label*) · มีอยู่ในแนวปฏิบัติหลักทุกเจ้า:

| มาตรฐาน | ทำยังไง |
|---|---|
| **Apple HIG** (Menus / Keyboard shortcuts) | เมนูทุกอันแสดง **key equivalent** ต่อท้ายรายการเสมอ เพื่อให้ผู้ใช้เห็นทางลัดระหว่างใช้ทางยาว |
| **Microsoft Fluent — KeyTips** | กด Alt แล้ว **ตัวอักษรคีย์ลัดลอยทับปุ่มจริงบนริบบิ้น** = ปุ่มสอนคีย์ของตัวเองตรงๆ |
| **Material Design** | tooltip ของปุ่มแสดงคีย์ลัดได้ (แต่ tooltip = ต้อง hover ก่อน → **ไม่พอสำหรับเคสนี้**) |
| **Nielsen heuristic #7** *Flexibility and efficiency of use* | ระบบต้องมี **accelerator** ให้ผู้ใช้ชำนาญ · แนวปฏิบัติสมัยใหม่ต่อยอดว่า accelerator ต้อง **ค้นพบได้** ไม่ใช่ซ่อนไว้ให้เดา |

**ฟันธง: ใช้ Apple-HIG-style key-equivalent แบบ "ติดบนปุ่มตลอดเวลา" ไม่ใช่ tooltip**
— เพราะ tooltip ต้อง hover ก่อน และ **บนจอสัมผัส (Surface ของ P'Aim) hover ไม่มีจริง**
(memory `feedback_hover_media_hides_on_touchscreen_laptop` · เคยเผาความเชื่อใจมาแล้ว 3 ครั้ง)

#### ✨ จุดที่เคสเราง่ายกว่าทุกมาตรฐานข้างบน

มาตรฐานเหล่านั้นออกแบบสำหรับคีย์ลัดแบบ `Ctrl+Shift+K` ที่**ไม่เกี่ยว**กับผลลัพธ์ —
**แต่ของเรา "คีย์ลัด" คือตัวอักขระนั้นเอง** → ป้ายบนปุ่มไม่ใช่คำใบ้ **มันคือคำตอบ**

```
┌──────────┐   บรรทัด 1  = อักขระที่จะได้ (ตัวใหญ่ = ป้ายของปุ่ม)
│    ^     │   บรรทัด 2  = ชื่อภาษาคน
│ ค้างเสียง │   บรรทัด 3  = ★ ตำแหน่งจริงบนแป้นพิมพ์ ★  ← อันนี้แหละที่พี่เปาขาด
│  ⇧ + 6   │
└──────────┘
```

**บรรทัดที่ 3 คือของใหม่ที่แก้ปัญหาเขาตรงๆ** — เขารู้อยู่แล้วว่าอยากได้ `^` แต่ไม่รู้ว่ามันคือ Shift+6
ป้าย "`^` ค้างเสียง" เฉยๆ ยังไม่พอ · ต้องบอก **ปุ่มจริงที่ต้องกด**

| อักขระ | ตำแหน่งบนแป้น US | หมายเหตุ |
|---|---|---|
| `^` | ⇧ + 6 | ที่พี่เปาหาไม่เจอ |
| `_` | ⇧ + - | ใช้ 4,881 ครั้ง — คนใช้เยอะสุดแต่ยังต้อง Shift |
| `~` | ⇧ + ` | มุมซ้ายบน |
| `(` `)` | ⇧ + 9 / ⇧ + 0 | |
| `{` `}` | ⇧ + [ / ⇧ + ] | |
| `#` | ⇧ + 3 | |
| `'` `,` `.` `-` `b` `n` | กดตรงๆ | ไม่ต้อง Shift |

#### 🔴 กฎบังคับของบรรทัดที่ 3 — **ไม่ชัวร์ว่าตรง ห้ามแสดง**

ตำแหน่งในตารางข้างบนเป็นของ **แป้น US** · ถ้าเครื่องผู้ใช้ตั้งแป้นอื่นจะไม่ตรง —
และ **บอกตำแหน่งผิด แย่กว่าไม่บอกเลย** (= หลัก P5 ข้อเดียวกับที่ใช้กับ validation §6:
ผู้ใช้กด `⇧+6` ตามที่จอบอกแล้วไม่ได้ `^` = เลิกเชื่อบรรทัดนี้ทั้งบรรทัดตลอดไป)

| กรณี | แสดงอะไร |
|---|---|
| detect เลย์เอาต์ได้ และยืนยันว่าปุ่มนั้นให้อักขระนั้นจริง | `^` · ค้างเสียง · **`⇧ + 6`** (ครบ 3 บรรทัด) |
| detect ไม่ได้ / ไม่มั่นใจ / เลย์เอาต์ไม่รู้จัก | `^` · ค้างเสียง — **ตัดบรรทัดที่ 3 ทิ้ง ไม่เดา** |

- **ทางที่ถูกคือ derive ไม่ใช่ hardcode** — Chromium มี Keyboard Map API
  (`navigator.keyboard.getLayoutMap()`) คืนอักขระจริงต่อปุ่มกายภาพ
- ⚠️ **ผมยังไม่ได้ verify ว่า API นี้ใช้ได้จริงบนเบราว์เซอร์เป้าหมาย และครอบคลุม "ชั้น Shift" ไหม**
  (ถ้าคืนแต่อักขระ unshifted จะบอก `⇧+6` ไม่ได้เลย) → **dev ต้องพิสูจน์ก่อนสร้าง ห้ามเดา**
- **⛔ fallback ไม่ใช่ "ใช้ตาราง US ไปก่อน"** — ตาราง US ใช้ได้ต่อเมื่อ **ยืนยันแล้วว่าเครื่องนั้น
  เป็นแป้น US จริง** · detect ไม่ได้ = แสดง 2 บรรทัด จบ
- ผลที่ยอมรับได้: ผู้ใช้ที่ระบบ detect ไม่ได้ ยังเห็นอักขระ (ดีกว่าวันนี้ที่ปุ่มไม่บอกอะไรเลย)
  แค่ไม่ได้ตัวช่วยพิเศษ — **ไม่มีใครถูกหลอก**

#### เป้าหมายของดีไซน์นี้ (เกณฑ์วัดที่ PM ตั้ง)
> **คนที่ไม่รู้ว่า `^` อยู่ตรงไหนบนแป้น ต้องรู้ได้จากหน้าจอ โดยไม่ต้องเปิดคู่มือ**

และผลข้างเคียงที่ตั้งใจ: **ยิ่งใช้ปุ่ม ยิ่งจำคีย์ได้ → ค่อยๆ เลิกพึ่งปุ่มเอง**
(ทางเดิน มือใหม่ → มือโปร อยู่บนหน้าจอเดียว ไม่ต้องมีโหมด "สำหรับผู้เชี่ยวชาญ")
— **ไม่บังคับให้เลิก** ใครอยากกดปุ่มตลอดไปก็ได้ (P4)

#### เข้ากันกับ P3 ("พิมพ์ได้ = ไม่ต้องมีปุ่ม") ยังไง
P3 **ไม่ได้ถูกทิ้ง แต่ถูกอ่านให้ถูก** — เจตนาของ P3 คือ *"อย่าให้ต้องยกมือไปเมาส์"*
ซึ่งดีไซน์นี้ตอบได้เต็ม (พิมพ์ได้ทุกตัว ไม่ต้องแตะปุ่มเลยถ้าไม่อยาก) ·
ปุ่มที่เหลืออยู่ทำหน้าที่ **สอน** ไม่ใช่ **บังคับให้ใช้** — จึงไม่ขัดกัน

### 4.2 ป๊อปอัปที่เหลือ — 3 กลุ่ม 11 ปุ่ม

## 5. เปลี่ยนอัตราจังหวะกลางเพลง (P'Aim สั่งเพิ่ม 23 ก.ค.)

**ปัญหาวันนี้:** 1 เพลง = 1 `timeSignature` (`songModel.js:66,112`) →
เพลงที่เปลี่ยนจังหวะกลางเพลง (เพลง 515, 433) **ขึ้น ❌ "จังหวะไม่ครบ" ทุกห้องในช่วงที่เปลี่ยน**
= สอนผู้ใช้ให้เพิกเฉยต่อเครื่องหมายเตือน (ละเมิด P5 โดยตรง)

**ทางแก้ — เพิ่ม item ชนิดเดียว ไม่รื้อโมเดล:**
```jsonc
{ "type": "time", "value": "3/4" }    // วางในสายเดียวกับ bar / repeat-start
```
- **ทำไมทางนี้:** สาย item ในบรรทัดรองรับ item แปลกๆ อยู่แล้ว (`bar` `repeat-start` `volta`
  `end` `section` `marker` `label`) และ `resolveContent()` **ส่งผ่าน item ที่ไม่ใช่ segment
  แบบไม่แตะอยู่แล้ว** → ผู้บริโภคเดิมทุกตัวไม่พัง (P6)
- **lint อ่านค่าที่ "กำลังมีผล"** แทนค่าเดียวของทั้งเพลง — `expectedBeats()` รับ time ปัจจุบัน
  ที่เดินสะสมมา ไม่ใช่ `content.timeSignature`
- **แสดงผล:** เลขเศษส่วนตัวเล็กเหนือหัวห้องที่เปลี่ยน (ตาม Wikipedia: *"Change of time signature
  within the piece may be marked in-line or above the line of music"*)

## 6. ระบบเตือนที่ไม่สร้าง alert fatigue (P5)

**หลัก: เตือนได้เฉพาะเมื่อ "รู้แน่ว่าผิด" · ไม่รู้แน่ = เงียบ**

| ชั้น | เมื่อไหร่ | หน้าตา |
|---|---|---|
| 🔴 **ผิดแน่** | สตริงที่ parse ไม่ออก (จะพิมพ์ตัวประหลาดลงแผ่นเพลงจริง) · `( )` ไม่ครบคู่ | ขีดแดงใต้อักขระที่มีปัญหา + **ปุ่มแก้ให้ 1 คลิก** |
| 🟡 **น่าสงสัย** | จังหวะในห้องไม่ตรง time signature **ที่กำลังมีผล** | ขีดจางที่หัวห้อง · ไม่ใช่ ❌ ตัวโต |
| ⚪ **เงียบ** | ห้องยก (`pickup`) · ห้องต่อ (`continue`) · ช่วงที่ยังไม่ประกาศ time · โน้ตที่ยังพิมพ์ไม่จบ | ไม่แสดงอะไร |

**กฎ 4 ข้อที่ต้องบังคับ:**
1. **สาเหตุที่อธิบายได้แล้ว = ห้ามเตือน** — `pickup` 538 จุดในคลังคือเครื่องพิสูจน์ว่ากลไก
   "บอกระบบว่าตั้งใจ" ใช้ได้จริง · ต้องขยายให้ครอบคลุมการเปลี่ยนจังหวะ (§5)
2. **ห้ามเตือนระหว่างพิมพ์** — เตือนตอนออกจากกล่อง/ห้อง ไม่ใช่ทุก keystroke
3. **1 สาเหตุ = 1 คำเตือน** — เปลี่ยนจังหวะกลางเพลงต้องขึ้น **1 บรรทัด** ว่า "ช่วงนี้จังหวะไม่ตรง
   ประกาศ 3/4 ตรงนี้ไหม?" ไม่ใช่ ❌ 12 ห้องเรียงกัน
4. **ทุกคำเตือนต้องมีทางออก 2 ทาง: แก้ให้ · หรือ "ตั้งใจแบบนี้"** (แล้วจำไว้ ไม่ถามซ้ำ)

### 7.2 property ของเพลงแบบ centralized — แนวคิด (ตามที่ขอ ยังไม่ใช่สเปกละเอียด)

**ปัญหาที่แท้จริง:** พอเพลงหนึ่งมีได้**หลายคีย์ หลายจังหวะ** "แก้ที่เดียว" กลายเป็นคำที่กำกวม —
แก้ที่เดียวของ *ทั้งเพลง* หรือของ *ช่วงนี้*?

**แนวคิดที่เสนอ: "ค่าตั้งต้น + ไทม์ไลน์การเปลี่ยน"**
```
┌ ข้อมูลเพลง ────────────────────────────┐
│ ค่าตั้งต้น:  คีย์ 1=C   จังหวะ 4/4   ♩=92 │
│                                          │
│ การเปลี่ยนระหว่างเพลง:                    │
│   ห้อง 17  จังหวะ → 3/4        [ไปดู][ลบ] │
│   ข้อ 4    คีย์ → 1=A          [ไปดู][ลบ] │
│                          [+ เพิ่มการเปลี่ยน] │
└──────────────────────────────────────────┘
```
- **ที่เดียวที่เห็นทุกอย่าง** — ไม่ต้องไล่หาว่าเปลี่ยนตรงไหนบ้าง
- **แก้ได้ 2 ทาง เห็นผลเป็นอันเดียวกัน** — แก้จากหน้านี้ หรือแก้ที่ห้องนั้นตรงๆ (item `time`)
  = หน้านี้เป็น "หน้าต่างมองไทม์ไลน์" ไม่ใช่ที่เก็บซ้อน (กัน 2 ที่ไม่ตรงกัน)
- **ค่าตั้งต้นห้ามหาย** — เพลงส่วนใหญ่ (คลังจริงเกือบทั้งหมด) ไม่มีการเปลี่ยนเลย → ตารางว่าง
  = ผู้ใช้ปัจจุบันไม่เห็นความซับซ้อนที่ไม่เกี่ยวกับเขา
- **`key` ราย arrangement entry ที่มีอยู่แล้วใน v2 = แถวหนึ่งในตารางนี้** (reuse ไม่ใช่ของใหม่)

### 7.3 discoverability — ผู้ใช้จะรู้ได้ยังไงว่ามีอะไรให้ใช้

**หลักฐานว่าเป็นปัญหาจริง:** พี่เปาเข้าใจว่าระบบไม่มีเครื่องหมายซ้ำเลย · ตัวเลขยืนยัน —
**volta ใช้ 0 ครั้ง · repeat 3 คู่ · `n`(♮) 0 ตัว · `b` 1 ตัว จาก 155 เพลง** ทั้งที่โค้ดวาดได้หมด
→ **ไม่ใช่ฟีเจอร์ขาด แต่คือฟีเจอร์ที่มองไม่เห็น**

**4 ทางที่เสนอ (เรียงตามน้ำหนัก):**
1. **เห็นผลทันทีที่พิมพ์ (WYSIWYG)** — พิมพ์ `,` แล้วจุดโผล่ใต้เลขเดี๋ยวนั้น · พิมพ์ `\dc` แล้ว
   กลายเป็นป้าย D.C. ทันที → **การเรียนรู้เกิดจากการลอง ไม่ใช่จากคู่มือ** (แก้ pain point
   "มองไม่เห็นภาพความจริงขณะกรอก" ที่ G สรุปจากพี่เปาโดยตรง)
2. **ป๊อปอัป `?` = สารบัญที่มีชีวิต** — เปิดแล้วเห็นทุกอย่างที่ใส่ได้ ณ จุดนั้น **พร้อมคีย์ลัดข้างปุ่ม**
   → คนใหม่กดปุ่ม · คนชำนาญอ่านคีย์ลัดแล้วเลิกเปิดป๊อปอัป (เส้นทางจากมือใหม่→มือโปรอยู่ในที่เดียว
   ตามแนว Material/HIG)
3. **เตือนเชิงบวก** — ระบบเห็นว่าท่อนนี้ซ้ำกับท่อนก่อนเป๊ะ → เสนอ *"ท่อนนี้เหมือนท่อนบน —
   ใช้เครื่องหมายย้อนซ้ำแทนไหม?"* (คลังมีข้อมูลพอจะตรวจได้แล้ว — `melodyLineSignature()`
   ใน `songModel.js` เทียบทำนองซ้ำเป็นอยู่แล้ว = reuse)
4. **Guide ต้องอัปเดตพร้อมงาน** (SOP §4.1 · memory `pleng-guide-always-updated`)

## 8.1 ภาคผนวก — G1+G2+G3 แบบพร้อมให้ dev หยิบทำทันที

ทั้ง 3 ตัวเป็นชุดเดียวกัน (แตะ `parseNotes()` จุดเดียว) · **ไม่ขึ้นกับคำตอบพี่เปาเรื่องแป้น 21 ปุ่ม**

### รูปแบบมาตรฐานที่จะเก็บลง DB (canonical form)
```
[~] [# | b | n] <เลข 0-7> [, | ']* [_]{0,2} [.]{0,2} [~] [^] [!]
ปลายไท  accidental   ขั้นเสียง  อ็อกเทฟ    เขบ็ต    เพิ่มค่า  ต้นไท  เฟอร์มาตา  staccato
```

### กติกาการรับเข้า (order-free)
1. **ต้องมีเลข 0-7 พอดี 1 ตัวต่อ 1 โน้ต** — ไม่มีเลข = ยังเป็น `{type:'raw'}` เหมือนเดิม
2. **modifier ทุกตัวรับได้ทุกตำแหน่ง** ก่อน/หลังเลข ในลำดับใดก็ได้ → เก็บตาม canonical form
3. **ยกเว้นเดียว — `.` ยังผูกกับตำแหน่งเพื่อความเข้ากันได้ย้อนหลัง:**
   - `.` **ก่อน**เลข = อ็อกเทฟล่าง (legacy · 1,641 จุดในคลัง) → normalize เป็น `,` **หลัง**เลข
   - `.` **หลัง**เลข = จุดเพิ่มค่า (1,085 จุด) → คงไว้
   - `,` ไม่มีความกำกวม รับได้ทุกตำแหน่ง
   - ⚠️ กฎ `5..5` เดิม (`notation.js:53-61`) **ต้องคงพฤติกรรมเป๊ะ** — มีข้อมูลจริงพึ่งอยู่
4. **ซ้ำได้:** `,,` `''` `__` `..` — แต่ `,` กับ `'` ปนกันในโน้ตเดียว = 🔴 ผิดแน่ (ขึ้นล่างพร้อมกันไม่ได้)
5. **`_` เกิน 2** → วันนี้ตัดทิ้งแล้วตัวที่ 3 กลายเป็น raw (สายสำรวจ §4-B) · ให้เป็น 🟡 เตือนแทน ไม่ใช่พังเงียบ

### ต้องไม่พัง (regression ที่ต้องมีเทสต์)
| กรณี | ต้องได้ผลเท่าเดิมเป๊ะ |
|---|---|
| `.5` `..5` (อ็อกเทฟล่าง legacy) | อ่านออกเหมือนเดิม · render จุดใต้เลขเหมือนเดิม |
| `5.` `5..` (จุดเพิ่มค่า) | ×1.5 / ×1.75 เหมือนเดิม |
| `5..5` (กฎกำกวมเดิม) | = โน้ตประจุด แล้วตามด้วยอ็อกเทฟล่าง — **ห้ามเปลี่ยน** |
| `'` `‘` `’` `′` (iOS smart punctuation) | ยังรับทั้ง 4 ตัว (`notation.js:44`) |
| `4#` → `#4` (`fixAccidental` เดิม) | ยังทำงาน · ตอนนี้เป็นเคสย่อยของ order-free |
| ทุกสตริงใน 155 เพลง | parse ผลลัพธ์เหมือนเดิมทุกตัว **ยกเว้น 5 จุดของอาการ 1-2 ที่ควรกลายเป็นอ่านออก** |

**วิธี verify ที่ถูก (จาก CLAUDE.md — preview tool ไม่เห็น worktree):**
รัน node import `src/lib/notation.js` เทียบ `parseNotes()` ก่อน/หลัง บนสตริงโน้ตทั้งคลัง
(ดึงจาก DB แบบ **อ่านอย่างเดียว** — ⛔ ห้ามเขียนกลับ) → diff ต้องว่าง ยกเว้น 5 จุดที่ตั้งใจ
