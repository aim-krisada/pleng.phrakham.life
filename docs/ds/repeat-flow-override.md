## 2. ส่วนต่าง 2 อย่าง

### 2.1 เครื่องหมายโครงสร้างต้องมี **id ถาวร**

วันนี้ `{type:'repeat-end'}` ไม่มีตัวระบุ → อ้างถึง "ตัวไหน" ไม่ได้

```jsonc
{ "type": "repeat-start", "id": "r1" }
{ "type": "repeat-end",   "id": "r1", "times": 2 }   // times = ค่าเริ่มต้นของทำนอง
{ "type": "volta", "id": "v1", "num": [1] }          // num เป็นลิสต์ (รอบ 3 จบเหมือนรอบ 2 = [2,3])
```

**ทำไมต้องมี id — เคสที่ฟิลด์เดียวเอาไม่อยู่ (G ยืนยัน · ตรงกับที่ผมสงสัยไว้ก่อน):**
> ทำนองมีเครื่องหมายซ้ำ **2 จุด** · ข้อ 2 ต้อง **ข้ามจุดแรก แต่เก็บจุดที่สองไว้**
> → ถ้าไม่มี id ก็สั่งได้แค่ "ข้ามทั้งหมด/ไม่ข้ามเลย"

**id ต้องถาวร ไม่ใช่ตำแหน่ง** — นี่คือการปิดกับดัก **broken reference** ที่ G เตือน
(แก้ทำนองแล้วห้องเลื่อน/ถูกลบ → override ที่อ้างเลขห้องจะชี้ผิดเงียบๆ)
- id ออกตอน**สร้างเครื่องหมาย** และติดไปกับเครื่องหมายนั้นตลอดชีวิต
- ลบเครื่องหมาย → override ที่อ้าง id นั้นกลายเป็น **กำพร้า** → 🟡 lint เตือน **ไม่ลบให้เงียบ ไม่เล่นมั่ว**

### 2.2 คำสั่งนำทางระดับ **แถวของ `arrangement`** (แถว = 1 ข้อ)

```jsonc
{
  "stanza": "A", "label": "ข้อ 2", "syllables": [ … ],
  "flow": {                       // ไม่มี = ร้องตามทำนองทุกประการ (เคสปกติ ~80%)
    "skip":     ["r1"],           // ข้ามเครื่องหมายซ้ำตัวนี้ (อ้างเป็นตัวๆ · "*" = ข้ามทุกตัว)
    "times":    { "r2": 3 },      // ตัวนี้เล่น 3 รอบเฉพาะข้อนี้
    "ending":   3,                // บังคับเข้าห้องจบชุดที่ 3
    "jump":     "segno",          // "segno" | "capo" | "coda" | "none" — เฉพาะข้อนี้
    "skipSections": ["B"],        // ข้ามท่อนนี้เฉพาะข้อนี้
    "path":     [ … ]             // ⚠️ ทางหนีทีไล่: ระบุลำดับห้องเอง (ดู §4)
  }
}
```

**กฎการทับค่า (precedence) — ประโยคเดียว:**
> **มีค่าใน `flow` = ใช้ค่านั้น · ไม่มี = ใช้ของทำนอง** · ไม่มีการผสมครึ่งๆ
> ⇒ **ยังคงหลัก "เจ้าของเดียวต่อข้อเท็จจริง"** จาก `song-structure.md` — เจ้าของถูกกำหนดตาม
> **ขอบเขต** (ทำนอง = ทุกข้อ · แถว = ข้อนั้น) **ไม่ใช่เก็บซ้ำ 2 ที่**

**ชื่อ pattern (ไม่ต้องประดิษฐ์เอง):** **Inheritance & Override** — ในเครื่องมือทำเพลง/ออกแบบเรียก
**Master / Instance** หรือ *Template with Local Overrides*

⛔ **ห้ามใช้ชื่อ `endingOverride`** — *Ending* ในทางดนตรีแปลว่า **volta (ห้องจบรอบ)** เท่านั้น ·
เคสเราคือ **การเดินเพลง (navigation flow)** คนละเรื่อง · ชื่อผิดทำให้คนรุ่นหลังใช้ผิดที่

## 3. ครอบ 8 รูปแบบครบ — ⛔ ไม่มีข้อไหนตอบว่า "ทำไม่ได้"

| # | รูปแบบที่มีจริงในเพลง | เก็บยังไง | มี UI รุ่นนี้ |
|---|---|---|---|
| 1 | ซ้ำทุกข้อเหมือนกัน (~80%) | **ไม่มี `flow`** | — (ไม่ต้องทำอะไร) |
| 2 | ข้อสุดท้ายไม่วน | `flow.skip: ["r1"]` หรือ `["*"]` | ✅ |
| 3 | ห้องจบมากกว่า 2 แบบ | `volta.num: [2,3]` + `flow.ending: 3` | ✅ |
| 4 | จำนวนรอบไม่เท่ากันแต่ละข้อ | `flow.times: {"r1": 3}` | ✅ |
| 5 | ท่อนรับเนื้อเปลี่ยนตามเลขข้อ | **arrangement ปกติ — มีอยู่แล้ว** | ✅ (live) |
| 6 | ข้ามท่อนเฉพาะข้อ (ข้อคี่มีประสาน ข้อคู่ข้าม) | `flow.skipSections: ["B"]` | ✅ |
| 7 | D.S. / Coda เฉพาะข้อ | `flow.jump: "segno"` · `"none"` เพื่อไม่กระโดด | ✅ |
| 8 | **กระโดดซ้อนกัน (วนในวน) — เคสพิลึกสุด** | `flow.path: [ลำดับห้องจริง]` | ⛔ **ไม่มี UI รุ่นนี้ (§4)** |

**เคสทดสอบที่ต้องผ่าน (เพลง 105):** ข้อ 1 วนปกติ · ข้อ 2-3 ไม่วนเนื้อ
→ `arrangement[1].flow = { skip: ["*"] }` · `arrangement[2].flow = { skip: ["*"] }` — **จบ ไม่ต้องก๊อปทำนอง**

## 4. 🔴 แยกให้ชัด: โครงเก็บ ≠ UI

| | รุ่นนี้ |
|---|---|
| **โครงเก็บ (schema + เอนจิน)** | **ครบทั้ง 8 รูปแบบ** รวม `path` · เพราะเติมทีหลัง = ต้องรื้อ |
| **UI** | ทำ **แบบ 2-7** (สิ่งที่พี่เปาต้องใช้จริง) · **แบบ 8 (`path`) ไม่มีหน้าจอ** — เข้าได้ทาง JSON เท่านั้น |

**เหตุผลที่ยังเก็บ `path` ไว้ทั้งที่ไม่มี UI:** ถ้า schema ไม่มีที่ให้มันตั้งแต่แรก วันที่เจอเพลงพิลึก
จริงจะต้อง **แก้ schema + migrate ข้อมูลที่มีอยู่** ซึ่งแพงกว่าการเผื่อฟิลด์ไว้เฉยๆ หลายเท่า

### ช่องขั้นสูงให้ใครเห็น — **ตัดสินแล้ว ไม่ถาม P'Aim**
**เปิดให้ทุกคนที่แก้เพลงได้ เห็นเท่ากัน** (เหมือนทุกอย่างในตัวแก้)
- ตรงกับหลักเดิมของโปรเจกต์: **แก้ได้ทุก tier · เกทที่ขั้นตอน "บันทึกขึ้นเซิร์ฟเวอร์" เท่านั้น**
  (memory `pleng-edit-open-all-tiers`) — การซ่อนความสามารถตามตัวตนผู้ใช้ขัดหลักนี้ตรงๆ
- **แต่ซ่อนตาม "ความซับซ้อน" ได้** — แบบ 2-7 อยู่ในคำสั่งภาษาคนปกติ · `path` อยู่หลัง
  "ตั้งค่าขั้นสูง" ที่ต้องกดเปิด (progressive disclosure — Material/HIG ใช้เหมือนกัน)
- **ต่างกันตรงไหน:** ซ่อนตาม *ตัวตน* = กีดกัน · ซ่อนตาม *ความซับซ้อน* = ไม่รกสำหรับคน 99%
  **และคนที่ต้องใช้ยังเข้าถึงได้** — อย่างหลังคือสิ่งที่มาตรฐานแนะ

## 7. ยังไม่พิสูจน์

1. 🔴 **ไม่ได้เปิดสเปก MusicXML ตัวจริง** — ข้อ §1 อ้างจากคำตอบ G (2 วง) ที่ตรงกัน **แต่เป็นคำบอกเล่า
   ไม่ใช่การอ้างมาตรา** · ก่อน dev ลงมือ R6 ควรเปิด `<repeat>`/`<ending>`/`<sound>` จริงยืนยัน
2. **ไม่รู้ว่าคลังมีแต่ละรูปแบบใน 8 อย่างกี่เพลง** — และ **วัดจากคลังตอนนี้จะได้ 0 เกือบทุกช่อง**
   เพราะระบบเพิ่งใส่ไม่ได้ (กับดักที่พิสูจน์แล้ว) → **ต้องวัดจากหนังสือ/พี่เปา ควบกับตัวเลขคลัง**
   · พี่เปาประเมินเบื้องต้น **"น่าจะ 2 เพลง แต่มีในเล่มใหญ่ด้วย"** (ยังไม่ได้ไล่นับจริง)
3. **ชื่อฟิลด์ทั้งหมดเป็นข้อเสนอของผม ไม่ใช่ชื่อมาตรฐาน** (มาตรฐานไม่มีเรื่องนี้ให้ยึด) — dev เปลี่ยนได้
   ถ้ามีเหตุผลดีกว่า **แต่ความสามารถ 8 อย่างต้องครบ**
4. **ยังไม่ได้ออกแบบหน้าตาช่องขั้นสูง** — §4 ตัดสินแค่ว่า *ใครเห็น* และ *ซ่อนด้วยเกณฑ์อะไร*
5. **ยังไม่ได้ตรวจว่า `skipSections` ชนกับ `afterEachVerse` (ร้องรับทุกข้อ) ไหม** — ทั้งคู่ยุ่งกับ
   "ท่อนไหนถูกร้องบ้าง" · **อาจต้องนิยามลำดับก่อนหลัง** ยังไม่ได้ไล่
