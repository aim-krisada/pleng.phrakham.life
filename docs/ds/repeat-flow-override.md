## 1. สิ่งที่ **ไม่ต้อง** เปลี่ยน (สำคัญกว่าสิ่งที่ต้องเปลี่ยน)

> **ตำแหน่งการเก็บเครื่องหมายซ้ำของเรา ถูกมาตรฐานแล้ว**

✅ **ยืนยันจากสเปกตัวจริงเองแล้ว ไม่ใช่คำบอกเล่าของ G** (W3C MusicXML 4.0 reference · เปิดอ่านเอง):

| อ้าง | สิ่งที่สเปกเขียน |
|---|---|
| `<repeat>` อยู่ที่ไหน | **parent = `<barline>`** |
| `<ending>` อยู่ที่ไหน | **parent = `<barline>`** |
| `<barline>` อยู่ที่ไหน | **parent = `<measure>` (partwise)** / `<part>` (timewise) |

⇒ **สายโซ่ครบ: `measure → barline → repeat/ending`** = **ผูกกับห้องของทำนองแบบเดียวกับเราเป๊ะ**
(G ฟันธงตรงข้ามในรอบแรกแล้วถอนคำในรอบ 2 — **ตอนนี้ไม่ต้องเชื่อ G แล้ว เพราะตรวจเองได้**)

### 🎁 ของแถมจากการเปิดสเปกจริง — 2 อย่างที่ดีกว่าที่ G บอก

**(1) `<repeat times="…">` มีในมาตรฐานอยู่แล้ว** — จำนวนรอบซ้ำเป็นแอตทริบิวต์มาตรฐาน
⇒ ฟิลด์ `times` ของเรา **ไม่ใช่ของประดิษฐ์ มีคู่เทียบในมาตรฐาน**

**(2) 🔴 `<repeat after-jump="yes">` — มาตรฐานตอบเคสที่ผมเคย "ตัดสินเอง" ไว้แล้ว**
สเปกระบุว่าแอตทริบิวต์นี้ใช้กับ *backward repeats following da capo or dal segno markings*
⇒ **การมีแอตทริบิวต์นี้แปลว่า ค่าปกติคือ "หลังกระโดดแล้วไม่เล่นเครื่องหมายซ้ำซ้ำอีก"**
และ `after-jump="yes"` คือการทำเครื่องหมาย *ข้อยกเว้น*
⇒ **กฎ "หลังกระโดดแล้ว = รอบสุดท้ายเสมอ" ที่ผมประกาศไว้ใน `note-symbol-set.md` §7.1.2
ตอนนี้มีมาตรฐานรองรับตรงๆ แล้ว ไม่ใช่แค่ "ธรรมเนียม"** — และเราควรเผื่อ flag แบบเดียวกันไว้ด้วย

**(3) `<ending number="1,2">` รับหลายเลขคั่นด้วยจุลภาคได้ในมาตรฐาน**
⇒ การที่เราให้ `volta.num` เป็นลิสต์ **ตรงมาตรฐาน ไม่ใช่ส่วนขยาย**

⇒ **ไม่รื้อ 3 ชั้น · ไม่ย้ายที่เก็บ · ไม่สร้างชั้นที่สาม**
⇒ ส่วนที่เป็น **"ทำนองอันเดียวแชร์หลายข้อ แล้วแต่ละข้อเดินคนละทาง"** — **มาตรฐานมีกลไกให้แล้ว 3 ตัว**
(`time-only` · MEI `<volta>` · MEI `<expansion>`) ดูตารางแมปด้านล่าง · **สิ่งที่เราต้องทำคือแมปให้ตรง
ไม่ใช่ประดิษฐ์ใหม่**

**⚠️ ข้อจำกัดของการอ้างมาตรฐาน (ต้องเขียนกำกับ ห้ามให้ใครเข้าใจผิด):**
- ✅ อ้างได้จริง: **MusicXML `<repeat>` · `<ending>` · `<sound>`** (ฝั่งเล่นเสียงสำหรับ D.C./al Fine/segno)
- ⛔ **ห้ามอ้าง MEI `<choice>/<app>/<rdg>`** — เป็นกลไกสำหรับ *ต้นฉบับต่างฉบับในงานวิชาการ*
  ไม่ใช่การแตกทางตอนเล่น (G อ้างผิดฝาแล้วยอมรับเอง)
### 🔴 แก้ข้ออ้างที่ผมเขียนผิดเอง — **มาตรฐานรองรับ "ข้อนี้วนต่างจากข้ออื่น" อยู่แล้ว**

รอบก่อนผมเขียนว่า *"การ override รายข้อ = ส่วนขยายระดับแอปเรา ไม่ใช่ฟีเจอร์มาตรฐาน"*
โดยอ้างคำตอบ G 2 วงที่ตรงกัน — **ผิด** · เปิดเอกสารตัวจริงแล้ว **มี 3 กลไก และตรงกับของเราแทบทุกช่อง**
(ยืนยันเองทั้ง 3 ลิงก์ ไม่ได้เชื่อคำบอกเล่าซ้ำสอง)

| กลไกมาตรฐาน | สเปกเขียนว่า | ตรงกับของเรา |
|---|---|---|
| **MusicXML `time-only`** [🔗](https://www.w3.org/2021/06/musicxml40/musicxml-reference/data-types/time-only/) | *"used to indicate that a particular playback- or listening-related element only applies particular times through a repeated section"* · รับ **ลิสต์เลขรอบคั่นจุลภาค** · ใช้ได้กับ `<sound>` `<note>` `<lyric>` `<tie>` ฯลฯ | **`flow.jump` · `flow.skip` · `flow.times`** — สั่งเป็น "รอบไหน" ได้โดย **ไม่ต้องกางเป็น through-composed** |
| **MEI `<volta>`** [🔗](https://music-encoding.org/guidelines/v5/elements/volta.html) | *"Sung text for a specific iteration of a repeated section of music"* · **parent = `<verse>` / `<refrain>`** · มีไว้สำหรับกรณี *"the musical notation is repeated, but the accompanying lyrics are not"* | **เคสเพลง 105 ตรงตัว** — โน้ตซ้ำแต่เนื้อไม่ซ้ำ |
| **MEI `<expansion @plist>`** [🔗](https://music-encoding.org/guidelines/v5/elements/expansion.html) | *"Indicates how a section may be programmatically expanded into its through-composed form"* · `@plist` = **ลำดับ id ของ section/ending ตามลำดับเล่นจริง** (ตัวอย่างในสเปก: `#A #End1 #A #End2`) | **`flow.path` มีชื่อมาตรฐานอยู่แล้ว** — จัดโครงให้เหมือนกันจะ export ตรงๆ ได้ |

**ฝั่งคำสั่งกระโดด — `<sound>`** [🔗](https://www.w3.org/2021/06/musicxml40/musicxml-reference/elements/sound/)
(parent = `<direction>` · `<measure>` partwise · `<part>` timewise) **ตรวจเองแล้ว มีครบทุกตัวที่เราต้องใช้:**

| แอตทริบิวต์ | สเปกเขียนว่า | ตรงกับของเรา |
|---|---|---|
| `dacapo` | *"go back to the beginning of the movement"* · ค่าเป็น `yes` เสมอ | `flow.jump: "capo"` |
| `dalsegno` / `segno` | จุดเริ่ม / จุดหมายของการย้อนไป segno · **"multiple jumps can be distinguished using this parameter's value"** | `flow.jump: "segno"` + **ยืนยันว่าการใส่ `id` บน segno/coda ของเราตรงมาตรฐาน** |
| `tocoda` / `coda` | จุดเริ่ม / จุดหมายของการข้ามไป coda · **"By default, this jump occurs on the second pass through a repeated section"** | `flow.jump: "coda"` · **มาตรฐานระบุค่าปกติไว้ให้แล้ว = รอบที่ 2** |
| `fine` | *"Follows the final note or rest in a movement with a da capo or dal segno direction"* | `[จบ]` / `kind:'fine'` |
| `forward-repeat` | *"a forward repeat sign is implied but not displayed"* | เผื่อไว้: เพลงที่ย้อนกลับต้นโดยไม่มี `‖:` เขียนไว้ |
| **`time-only`** | *"which times to apply the sound element if the sound element applies only particular times through a repeat"* | 🔴 **นี่คือตัวที่ทำให้ `flow.jump` รายข้อ export ได้โดยไม่ต้องกางเป็น through-composed** |

⚠️ **ข้อควรระวังที่ผมต้องระบุเอง ไม่ให้แกว่งไปอีกทาง:** `time-only` นับเป็น **"รอบที่เท่าไหร่ของท่อนที่ซ้ำ"**
· แมปกับ "ข้อที่เท่าไหร่" ของเราได้ **เมื่อข้อนั้นคือรอบของการซ้ำจริงๆ** (เพลงสโตรฟิกทั่วไป = ใช่)
· แต่โมเดลเราให้ `arrangement` เรียงอิสระได้ **ถ้าข้อไหนไม่ได้เป็นรอบของ repeat การแมปจะไม่ตรง 1:1**
→ **ยังไม่ได้ไล่ว่ามีเคสแบบนั้นในคลังไหม** (ดู §7)

## 2. ส่วนต่าง 2 อย่าง

### 2.1 เครื่องหมายโครงสร้างต้องมี **id ถาวร**

วันนี้ `{type:'repeat-end'}` ไม่มีตัวระบุ → อ้างถึง "ตัวไหน" ไม่ได้

```jsonc
{ "type": "repeat-start", "id": "r1" }
{ "type": "repeat-end",   "id": "r1", "times": 2 }   // times = ค่าเริ่มต้นของทำนอง
{ "type": "volta", "id": "v1", "num": [1] }          // num เป็นลิสต์ (รอบ 3 จบเหมือนรอบ 2 = [2,3])
```

**ทำไมต้องมี id — เคสที่ฟิลด์เดียวเอาไม่อยู่ (G ยืนยัน · ตรงกับที่ผมสงสัยไว้ก่อน):**
> ทำนองมีเครื่องหมายซ้ำ **2 จุด** · ข้อ 2 ต้อง **ข้ามจุดแรก แต่เก็บจุดที่สองไว้**
> → ถ้าไม่มี id ก็สั่งได้แค่ "ข้ามทั้งหมด/ไม่ข้ามเลย"

**id ต้องถาวร ไม่ใช่ตำแหน่ง** — นี่คือการปิดกับดัก **broken reference** ที่ G เตือน
(แก้ทำนองแล้วห้องเลื่อน/ถูกลบ → override ที่อ้างเลขห้องจะชี้ผิดเงียบๆ)

> ✅ **และมาตรฐานทำแบบเดียวกัน** — MEI `<expansion @plist>` นิยามไว้ว่าเป็น
> *"an ordered list of **identifiers** of descendant section, ending, lem, or rdg elements"*
> [🔗](https://music-encoding.org/guidelines/v5/elements/expansion.html)
> ⇒ **มาตรฐานอ้างลำดับการเล่นด้วย id ไม่ใช่ตำแหน่ง เหมือนกับที่เราตัดสินจากเหตุผลวิศวกรรม**
> — เขียนอ้างไว้ตรงนี้เพื่อไม่ให้มีใครมาเถียงทีหลังว่า "ทำไมไม่ใช้เลขห้องให้ง่ายกว่า"
### 2.1.1 🔴 อ้างอิงพัง (broken reference) — ออกแบบให้จบ ไม่ใช่หมายเหตุท้ายไฟล์

**นี่คือบั๊กตระกูล "ข้อมูลหายเงียบ"** (กลุ่มเดียวกับ B108) — ผู้ใช้ไม่รู้ตัวและกู้ไม่ได้
จึงต้องนิยามพฤติกรรมให้ครบ **ทุกกรณี** ตั้งแต่ตอนออกแบบ

**กฎที่ 1 — อ้างด้วย id ไม่ใช่ตำแหน่ง**
id ออกตอน**สร้างเครื่องหมาย** ติดไปกับเครื่องหมายนั้นตลอดชีวิต ·
ย้ายห้อง แทรกห้อง ลบห้องอื่น แก้โน้ต → **id ไม่ขยับ ⇒ override ไม่พังโดยอัตโนมัติ**
⛔ ห้ามอ้างเลขห้อง/ลำดับที่ เพราะทุกการแก้ทำนองจะเลื่อนมันหมด

**กฎที่ 2 — id ต้องไม่ซ้ำ แม้แต่ตอนคัดลอก**
คัดลอกห้อง/บรรทัดที่มีเครื่องหมายไปวาง → **ต้องออก id ใหม่ให้ตัวที่วาง**
ถ้าไม่ทำ จะมี 2 เครื่องหมายถือ id เดียวกัน → override ชี้กำกวม เล่นผิดโดยไม่มีใครรู้
⚠️ **ผูกกับงาน "ย้าย/คัดลอก/วาง" ที่ PM แยกไว้อีกใบโดยตรง — ต้องบอกสายนั้น**

**กฎที่ 3 — พฤติกรรมเมื่อจะลบเครื่องหมายที่ถูกใช้อยู่**
| สถานการณ์ | ทำอะไร |
|---|---|
| กำลังจะลบเครื่องหมายที่มีข้ออื่นอ้างถึง | **เตือนก่อนลบ + บอกว่าใครใช้อยู่** *"เครื่องหมายนี้ถูกใช้โดย ข้อ 2, ข้อ 3 — ลบแล้วข้อเหล่านั้นจะกลับไปร้องตามทำนอง"* แล้วให้ยืนยัน |
| ผู้ใช้ยืนยันลบ | **เก็บ override ไว้ในข้อมูล ไม่ลบทิ้งเงียบ** · ทำเครื่องหมายว่า **กำพร้า** |
| เล่นเสียง/แสดงผล เมื่อเจอ override กำพร้า | **เพิกเฉยคำสั่งนั้น กลับไปใช้ค่าของทำนอง** ⛔ **ห้ามเดา ห้ามกระโดดมั่ว** |
| lint | 🔴 **แสดงเป็นข้อผิดพลาด** *"ข้อ 2 สั่งข้ามเครื่องหมายซ้ำที่ถูกลบไปแล้ว"* + ปุ่มลบคำสั่งทิ้ง |
| undo | คืนทั้งเครื่องหมาย**และ**สถานะ override พร้อมกัน |

**ทำไมเลือก "เตือนแล้วให้ลบได้" ไม่ใช่ "ห้ามลบ":**
ปฏิเสธการลบ = ผู้ใช้ติดกับดักแก้เพลงตัวเองไม่ได้ · **แต่ลบเงียบ = ข้อมูลหาย** ·
ทางกลางคือ **บอกผลกระทบก่อน แล้วให้เขาตัดสิน** — ซึ่งเป็น**รูปแบบที่โค้ดนี้ใช้อยู่แล้ว**
ตอนลบทำนองที่ถูกใช้ในลำดับเพลง (`EditorMode.vue:607-611`) ⇒ **ผู้ใช้เคยเห็นแบบนี้แล้ว ไม่ต้องเรียนรู้ใหม่**

### 2.2 คำสั่งนำทางระดับ **แถวของ `arrangement`** (แถว = 1 ข้อ)

```jsonc
{
  "stanza": "A", "label": "ข้อ 2", "syllables": [ … ],
  "flow": {                       // ไม่มี = ร้องตามทำนองทุกประการ (เคสปกติ ~80%)
    "skip":     ["r1"],           // ข้ามเครื่องหมายซ้ำตัวนี้ (อ้างเป็นตัวๆ · "*" = ข้ามทุกตัว)
    "times":    { "r2": 3 },      // ตัวนี้เล่น 3 รอบเฉพาะข้อนี้
    "ending":   3,                // บังคับเข้าห้องจบชุดที่ 3
    "jump":     "segno",          // "segno" | "capo" | "coda" | "none" — เฉพาะข้อนี้
    "skipSections": ["B"],        // ข้ามท่อนนี้เฉพาะข้อนี้
    "path":     [ "…id…" ]        // ลำดับเล่นจริงเป็น **ลิสต์ id** (ดู §4)
  }
}
```

**กฎการทับค่า (precedence) — ประโยคเดียว:**
> **มีค่าใน `flow` = ใช้ค่านั้น · ไม่มี = ใช้ของทำนอง** · ไม่มีการผสมครึ่งๆ
> ⇒ **ยังคงหลัก "เจ้าของเดียวต่อข้อเท็จจริง"** จาก `song-structure.md` — เจ้าของถูกกำหนดตาม
> **ขอบเขต** (ทำนอง = ทุกข้อ · แถว = ข้อนั้น) **ไม่ใช่เก็บซ้ำ 2 ที่**

**ชื่อ pattern:** **Inheritance & Override** · ในวงการออกแบบ/เครื่องมือเรียก **Master / Instance**
หรือ *Template with Local Overrides*

> ⚠️ **ต้องเขียนกำกับ: ชื่อพวกนี้ไม่ใช่ศัพท์มาตรฐานดนตรี** — ยืมมาจากวงการซอฟต์แวร์/ออกแบบ ·
> **มาตรฐานดนตรีไม่มีชื่อเรียกสำหรับ inherit + override** · ที่ MEI มีคือ `@copyof` / `@sameas`
> ซึ่งเป็น **"สำเนา" คนละเรื่องกับ "สืบทอดแล้วทับบางส่วน"**
> ⇒ ใช้ชื่อนี้ต่อได้เพื่อสื่อสารในทีม **แต่ห้ามให้ใครเข้าใจว่าเป็นศัพท์ทางการของมาตรฐานดนตรี**

## 4. 🔴 แยกให้ชัด: โครงเก็บ ≠ UI

| | รุ่นนี้ |
|---|---|
| **โครงเก็บ (schema + เอนจิน)** | **ครบทั้ง 8 รูปแบบ** รวม `path` · เพราะเติมทีหลัง = ต้องรื้อ |
| **UI** | ทำ **แบบ 2-7** (สิ่งที่พี่เปาต้องใช้จริง) · **แบบ 8 (`path`) ไม่มีหน้าจอ** — เข้าได้ทาง JSON เท่านั้น |

**เหตุผลที่ยังเก็บ `path` ไว้ทั้งที่ไม่มี UI:** ถ้า schema ไม่มีที่ให้มันตั้งแต่แรก วันที่เจอเพลงพิลึก
จริงจะต้อง **แก้ schema + migrate ข้อมูลที่มีอยู่** ซึ่งแพงกว่าการเผื่อฟิลด์ไว้เฉยๆ หลายเท่า

**รูปทรงของ `path` — ให้เหมือน MEI `<expansion @plist>` ตั้งแต่แรก**
`@plist` ของ MEI คือ **ลำดับ id ของ section/ending ตามลำดับเล่นจริง** (ตัวอย่างในสเปก: `#A #End1 #A #End2`)
→ **`flow.path` ต้องเป็นลิสต์ id เรียงตามลำดับเล่น ไม่ใช่เลขห้องดิบ** ·
ได้ 2 อย่างพร้อมกัน: **export ตรงๆ ทีหลังโดยไม่ต้องแปลง** และ **ทนการแก้ทำนอง** (กฎ §2.1.1 ข้อ 1)

### 4.1 🔴 `skipSections` ปะทะ `afterEachVerse` — ลำดับก่อนหลัง (ปิดข้อค้าง)

ทั้งคู่ยุ่งกับ *"ท่อนไหนถูกร้องบ้าง"* → **ถ้าไม่นิยาม dev จะเดาเอง แล้วผลจะต่างกันในแต่ละที่**

| | ทำอะไร | ระดับ |
|---|---|---|
| `afterEachVerse` | **เพิ่ม** ท่อนรับเข้าไปหลังทุกข้อ | ทั้งเพลง (บนแถวของท่อนรับ) |
| `flow.skipSections` | **ตัด** ท่อนออกเฉพาะข้อนี้ | ข้อเดียว (บนแถวของข้อนั้น) |

> ### 📌 กฎที่ประกาศ: **"กางก่อน แล้วค่อยตัด"**
> **1. กาง** — `afterEachVerse` ทำงานก่อน สร้างลำดับเต็มของทั้งเพลง (ข้อ 1 · รับ · ข้อ 2 · รับ …)
> **2. ตัด** — `flow.skipSections` ของแต่ละข้อ ทำงานทีหลัง บนลำดับที่กางแล้ว

**ผลลัพธ์ของเคสที่ทั้งสองมีค่าพร้อมกัน:**

| ตั้งค่า | ผลที่ต้องได้ |
|---|---|
| ท่อนรับ `afterEachVerse: true` · ข้อ 3 `skipSections: ["B"]` (B = ท่อนรับ) | `ข้อ1 · รับ · ข้อ2 · รับ · ข้อ3` — **ข้อ 3 ไม่มีรับต่อท้าย ข้ออื่นมีครบ** |
| ท่อนรับ `afterEachVerse: true` · ข้อ 2 `skipSections: ["C"]` (C = ท่อนประสาน ไม่ใช่รับ) | รับยังตามหลังทุกข้อ · ตัดเฉพาะ C ของข้อ 2 |
| ข้อ 3 `skipSections` ระบุท่อนที่ไม่มีในลำดับอยู่แล้ว | **ไม่ทำอะไร ไม่ใช่ error** (สั่งตัดของที่ไม่มี = ผลลัพธ์ตรงกับที่สั่ง) |

**ทำไม "กางก่อน" ไม่ใช่ "ตัดก่อน":**
`afterEachVerse` เป็น**นโยบายของทั้งเพลง** ส่วน `skipSections` เป็น**ข้อยกเว้นของข้อเดียว** —
ข้อยกเว้นต้องทำงานบนผลของนโยบาย ไม่ใช่กลับกัน · ถ้าตัดก่อนแล้วค่อยกาง **ท่อนที่เพิ่งตัดไปจะถูก
`afterEachVerse` ใส่กลับเข้ามา** = คำสั่งของผู้ใช้ถูกลบล้างเงียบๆ
⇒ **หลักเดียวกับกฎทับค่าใน §2.2: ของที่เจาะจงกว่าชนะเสมอ**

### ช่องขั้นสูงให้ใครเห็น — **ตัดสินแล้ว ไม่ถาม P'Aim**
**เปิดให้ทุกคนที่แก้เพลงได้ เห็นเท่ากัน** (เหมือนทุกอย่างในตัวแก้)
- ตรงกับหลักเดิมของโปรเจกต์: **แก้ได้ทุก tier · เกทที่ขั้นตอน "บันทึกขึ้นเซิร์ฟเวอร์" เท่านั้น**
  (memory `pleng-edit-open-all-tiers`) — การซ่อนความสามารถตามตัวตนผู้ใช้ขัดหลักนี้ตรงๆ
- **แต่ซ่อนตาม "ความซับซ้อน" ได้** — แบบ 2-7 อยู่ในคำสั่งภาษาคนปกติ · `path` อยู่หลัง
  "ตั้งค่าขั้นสูง" ที่ต้องกดเปิด (progressive disclosure — Material/HIG ใช้เหมือนกัน)
- **ต่างกันตรงไหน:** ซ่อนตาม *ตัวตน* = กีดกัน · ซ่อนตาม *ความซับซ้อน* = ไม่รกสำหรับคน 99%
  **และคนที่ต้องใช้ยังเข้าถึงได้** — อย่างหลังคือสิ่งที่มาตรฐานแนะ

## 7. ยังไม่พิสูจน์

1. ✅ ~~ไม่ได้เปิดสเปก MusicXML ตัวจริง~~ → **เปิดแล้ว ยืนยันแล้ว** (`measure → barline → repeat/ending`
   · `times` · `after-jump` · `ending number="1,2"`)
   ✅ **`<sound>` ตรวจเองแล้วเช่นกัน** (`dacapo` · `dalsegno`/`segno` · `tocoda`/`coda` · `fine` ·
   `forward-repeat` · `time-only`) — **ทุกข้ออ้างมาตรฐานในเอกสารนี้มี URL + ชื่อ element/attribute แล้ว**
2. **ไม่รู้ว่าคลังมีแต่ละรูปแบบใน 8 อย่างกี่เพลง** — และ **วัดจากคลังตอนนี้จะได้ 0 เกือบทุกช่อง**
   เพราะระบบเพิ่งใส่ไม่ได้ (กับดักที่พิสูจน์แล้ว) → **ต้องวัดจากหนังสือ/พี่เปา ควบกับตัวเลขคลัง**
   · พี่เปาประเมินเบื้องต้น **"น่าจะ 2 เพลง แต่มีในเล่มใหญ่ด้วย"** (ยังไม่ได้ไล่นับจริง)
3. **ชื่อฟิลด์ที่ผมตั้ง (`flow.skip` ฯลฯ) เป็นชื่อภายในของเรา** — แต่ **แนวคิดเบื้องหลังทุกตัวมีคู่เทียบ
   ในมาตรฐานแล้ว** (ตาราง §1) · dev เปลี่ยนชื่อได้ถ้ามีเหตุผลดีกว่า **แต่ (ก) ความสามารถ 8 อย่างต้องครบ
   (ข) `flow.path` ต้องคง isomorphic กับ `<expansion @plist>`**
4. **ยังไม่ได้ออกแบบหน้าตาช่องขั้นสูง** — §4 ตัดสินแค่ว่า *ใครเห็น* และ *ซ่อนด้วยเกณฑ์อะไร*
5. ✅ ~~`skipSections` ชนกับ `afterEachVerse` ไหม~~ → **ปิดแล้วใน §4.1** (กฎ "กางก่อน แล้วค่อยตัด"
   + ตารางผลลัพธ์ของเคสที่ทั้งสองมีค่าพร้อมกัน)
6. 🔴 **ข้ออ้างมาตรฐานของผมเคยผิด 1 ข้อ — และมันผ่านมาได้เพราะ "G 2 วงตรงกัน"**
   ⇒ **G หลายวงตรงกัน ≠ ถูก** (แหล่งเดียวกันย่อมผิดเหมือนกัน) · ดู §8 ของ `note-symbol-set.md`
7. **`time-only` แมปกับ "ข้อที่เท่าไหร่" ได้เมื่อข้อนั้นคือรอบของ repeat จริงๆ** — โมเดลเราให้
   `arrangement` เรียงอิสระได้ · **ยังไม่ได้ไล่คลังว่ามีข้อที่ไม่ได้เป็นรอบของ repeat ไหม**
   ถ้ามี การ export ด้วย `time-only` จะไม่ตรง 1:1 สำหรับเคสนั้น
8. **ยังไม่ได้ตรวจ `<sound>` ด้วยตาเอง** (D.C./al Fine/segno ฝั่งเล่นเสียง) — ควรตรวจก่อน dev ทำ R6
