# Pair protocol — Claude Code ↔ Antigravity

ไฟล์นี้อ่านเองจบ ไม่ต้องรู้ประวัติโปรเจกต์ก่อน · ใครเปิดอ่านครั้งแรกก็ทำงานตามได้ทันที

## 1 · ปัญหาที่โปรโตคอลนี้แก้

20 ก.ค. 2026 งานหนึ่ง (B118 เส้นโค้ง tie/slur) มีคนสรุป 3 คน ได้ 3 คำตอบ:
dev บอก "แก้แล้ว" · tester บอก "PASS" · PM บอก "ยังพัง" — **ไม่มีใครชี้ขาดได้ งานค้างทั้งรอบ**
สาเหตุ: ทั้งสามคน**ส่งข้อสรุปให้กันอ่าน** ไม่ได้ส่ง**วิธีรันซ้ำ**ให้กันรัน
→ ทุกชั้นที่ผ่าน **ความมั่นใจเพิ่ม หลักฐานเท่าเดิม**

**กติกาข้อเดียวที่เป็นแกนของทั้งไฟล์นี้:**
> เราสองคนไม่ส่ง "ความเห็น" ให้กัน — เราส่ง **คำสั่งที่อีกฝ่ายรันซ้ำแล้วได้ผลเดียวกัน**
> ถ้าเคลมไหนรันซ้ำไม่ได้ เคลมนั้นไม่นับ ไม่ว่าจะฟังดูมั่นใจแค่ไหน

## 2 · ไฟล์และเจ้าของบล็อก

1 งาน = 1 ไฟล์ `docs/pair/<task-id>.md` (เช่น `B118-slur-continuous.md`)

| บล็อก | ใครเขียนได้ | ใครห้ามแตะ |
|---|---|---|
| `## CONTEXT` | ผู้เปิดงาน (ปกติ = Claude Code) | — |
| `## CLAIM-<n>` | **Claude Code เท่านั้น** | Antigravity |
| `## VERDICT-<n>` | **Antigravity เท่านั้น** | Claude Code |

**ห้ามแก้บล็อกของอีกฝ่ายเด็ดขาด** — ถ้าไม่เห็นด้วย เขียนในบล็อกของตัวเอง
(ต่อท้ายอย่างเดียว ไม่ลบของเก่า — ประวัติการเถียงคือหลักฐาน)

## 2.5 · 🔴 ด่านออกแบบ — อุดรูตั้งแต่ยังไม่เขียนโค้ด (P'Aim 20 ก.ค. · สำคัญที่สุด)

**P'Aim = PM ทั้ง 2 โปรเจกต์ · AI 2 ตัวช่วยกันอุดรูรั่วตั้งแต่ขั้นออกแบบ ไม่ใช่ตรวจตอนของเสร็จแล้ว**

ที่มา: ของแพงที่สุดวันที่ 20 ก.ค. ไม่ใช่โค้ดผิด — คือ **การวินิจฉัยผิดที่ไม่มีใครค้าน**
("คานหายตอนพิมพ์ 92%" → เกือบสั่งรื้อระบบ 3-5 วัน แก้บั๊กที่ไม่มีจริง)
**ตรวจโค้ดจับข้อนี้ไม่ได้เลย เพราะโค้ดยังไม่ได้เขียน** → ด่านต้องอยู่ก่อนหน้านั้น

### ⛔ กฎเหล็ก: ห้ามให้ AI ตัวหนึ่งอ่านคำตอบอีกตัวก่อนตอบ

เหตุผลไม่ใช่ทฤษฎี — วัดมาแล้วบนงานพระคำ 20 ก.ค.:
**ผู้ตรวจอนุมัติ 7/7 ทันทีหลังถูกบอกว่า P'Aim อยากได้อะไร · อัตราอนุมัติขยับ 17% → 59% เพราะวิธีตั้งคำถามล้วนๆ**
→ ถ้าให้ตัวที่สองอ่านแผนของตัวแรกแล้วรีวิว **จะได้เสียงสะท้อน ไม่ใช่ความเห็นที่สอง**
→ **P'Aim คือคนเดินสาร = ช่องทางเดียวที่ปิดตาได้จริง** ต้นทุน = paste 4 ครั้ง

### ขั้นตอน (ใช้กับงานที่ต้องใช้วิจารณญาณ — แผน · สาเหตุ · "แก้ถูกจุดไหม")

1. **P'Aim เขียนโจทย์ 1 ครั้ง** ในบล็อก `## PROBLEM` (ภาษาคน ไม่ต้องแปลงเป็นสเปก)
2. **paste โจทย์เดียวกันให้ทั้ง 2 ตัว แยกกัน ห้ามบอกว่าอีกตัวว่าไง**
   แต่ละตัวเขียนบล็อกของตัวเอง (`## DESIGN-[C]` / `## DESIGN-[A]`) ตอบ 5 ข้อ:
   1. **สาเหตุจริงคืออะไร** — และรู้ได้ไงว่าใช่ (ต้องมีวิธีพิสูจน์ ไม่ใช่การให้เหตุผล)
   2. **จะทำอะไร** — และ **จะไม่ทำอะไร** (กัน gold-plating)
   3. **จะพังตรงไหนได้บ้าง** — ของเดิมอะไรเสี่ยงถอยหลัง
   4. **🔴 ผมกำลังสมมติอะไรที่ยังไม่ได้พิสูจน์** — ช่องนี้ว่างไม่ได้
   5. **อะไรที่ต้องให้คนดู** (`human-eye`) — เตรียม URL/ขั้นตอนไว้เลย
3. **P'Aim paste ทั้ง 2 คำตอบให้ทั้ง 2 ตัว** → แต่ละตัวเขียน `## COMPARE-[C]` / `## COMPARE-[A]`
   **เขียนเฉพาะจุดที่ต่าง** ห้ามสรุปซ้ำของตัวเอง · ตรงไหนอีกฝ่ายถูกกว่า **พูดออกมาตรงๆ**
4. **P'Aim ตัดสิน** เขียนใน `## DECISION` แล้วค่อยเริ่มเขียนโค้ด

### วิธีอ่านผล (สำคัญกว่าตัวคำตอบ)

- **ที่ต่างกัน = ของจริงที่ซื้อมา** — ตรงนั้นคือรูรั่ว ให้ P'Aim ดูตรงนั้นก่อน
- **ที่ตรงกัน = หลักฐานอ่อน ไม่ใช่ข้อพิสูจน์** — โดยเฉพาะถ้าทั้งคู่เป็น Claude
  (pl กับ pk เป็นรุ่นเดียวกัน เห็นตรงกันแทบไม่มีน้ำหนัก · ต่างค่ายเท่านั้นที่อิสระจริง)
- **ทั้งคู่บอก "ไม่แน่ใจ/ต้องให้คนดู" = คำตอบที่ถูก** ไม่ใช่ความล้มเหลว — หยุดแล้วส่งคน

### ⚠️ ของที่หายไปเมื่อไม่มี PM session (P'Aim ต้องรับเอง)

เดิม PM คุมเรื่องนี้ · ตอนนี้ไม่มีแล้ว = **ต้องมีกลไกถูกๆ แทน ไม่ใช่ความจำ**
- **จองไฟล์ก่อนแก้** — ก่อนเริ่มงาน ทุก session เขียนบรรทัดใน `board.md` ว่าจะแตะไฟล์ไหน
  **ซ้ำกับของอีกสาย = หยุด ถาม P'Aim** (เคยเกิดจริง: 2 สายแก้ `EditorMode.vue` พร้อมกัน ต้อง rebase แก้)
- **`main` มีเจ้าของคนเดียวต่อโปรเจกต์** เขียนชื่อไว้ใน `board.md` · **pleng `main` = auto-deploy** merge ผิด = ขึ้นเว็บทันที

## 3 · รูปแบบ CLAIM (Claude Code เขียน)

```markdown
## CLAIM-1
**CLAIM:** <ข้อเดียว ประโยคเดียว ต้องผิดได้ถ้ามันผิด>
**LAYER:** source | build-output | live-page | printed-pdf | **audio** | **human-eye**
**REPRO:**
```sh
<คำสั่งที่ copy ไปวางรันได้เลย ตั้งแต่ cd จนถึงบรรทัดที่ให้ตัวเลข>
```
**EXPECT:** <ตัวเลข/ข้อความเป๊ะๆ ที่ต้องเห็น>
**ACTUAL (self-run):** <output จริงที่ผมรันได้ วางมาดิบๆ>
**STATUS:** self-verified
**ยังไม่ได้พิสูจน์:** <ช่องนี้ห้ามว่าง — ถ้าว่างแปลว่ายังไม่เสร็จ>
```

**ข้อห้ามของ Claude Code:**
- ⛔ ห้ามเขียนคำว่า "ผ่าน / เสร็จ / แก้แล้ว / ใช้ได้แล้ว" — เขียนได้สูงสุดแค่ `self-verified`
- ⛔ `cross-verified` เป็นคำของ Antigravity เท่านั้น
- ⛔ REPRO ที่รันไม่ได้จริง (พิมพ์เอาเท่ ๆ) = โกง · REPRO ทุกอันต้องเคยรันเองมาแล้ว
- ⛔ ห้ามอ้าง exit code, ชื่อไฟล์, timestamp, diff, หรือคอมเมนต์ในโค้ด เป็นหลักฐานว่าผลลัพธ์ถูก (ดู §6)

## 4 · รูปแบบ VERDICT (Antigravity เขียน)

```markdown
## VERDICT-1
**RAN:** <คำสั่งที่รันจริง ถ้าต่างจาก REPRO บอกว่าต่างตรงไหนและทำไม>
**ACTUAL (independent run):** <output จริง วางมาดิบๆ ห้ามสรุปเป็นคำพูด>
**MATCH:** yes | no | inconclusive
**STATUS:** cross-verified | refuted | blocked
**หมายเหตุ:** <ถ้า refuted บอกว่าที่จริงเห็นอะไร · ถ้า blocked บอกว่าติดอะไร>
```

**สิ่งที่ขอจาก Antigravity โดยเฉพาะ:**
1. **แปะ output ดิบมาเสมอ** ไม่ใช่ "ตรวจแล้วถูกต้อง" — คนอ่านต้องเห็นตัวเลขเอง
2. **ถ้า REPRO ของผมพารันไปเจอผลที่ผมบอก แต่คุณคิดว่าคำถามมันผิดตั้งแต่ต้น → พูดออกมา**
   ("วัดถูก แต่วัดผิดเรื่อง" คือความผิดพลาดที่แพงที่สุดของผมรอบที่แล้ว — ผมนับเส้นในไฟล์ PDF
   ได้เลขถูกทุกตัว แล้วสรุปเองว่าเป็นบั๊ก ทั้งที่ผลนั้นถูกตามกฎที่ตั้งใจออกแบบไว้
   เกือบสั่งรื้อระบบ 3-5 วัน แก้บั๊กที่ไม่มีจริง)
3. **ไม่ต้องเกรงใจ** — งานของคุณคือหักล้าง ไม่ใช่รับรอง เจอผมผิดคือคุณทำงานสำเร็จ

## 5 · กฎการชี้ขาด

1. **ชั้นที่ไม่มี AI ตัวไหนเซ็นได้เลย — `human-eye` · `audio` · `printed-pdf`**
   ต้องคนดู/คนฟังจริง แล้วเขียนใน `## HUMAN-<n>` เท่านั้น · AI เขียนได้แค่ *"เตรียม URL + ขั้นตอนให้แล้ว"*
   - `human-eye` — สวยไหม / โค้งขาดไหม / อ่านง่ายไหม / กดถูกไหม
   - **`audio` — เพราะไหม / อ่านตัวเลขถูกไหม / เร็วไปไหม** (pk pm9 20 ก.ค.: วันเดียวมี **5 เรื่องที่ตัดสินได้ด้วยหูเท่านั้น** และ **3 ใน 5 นั้น เครื่องวัดบอกว่าผ่าน** — ×1.03 ถูกทางเลขแต่ผิดทางหู · ความยาวเท่ากันเป๊ะ "พิสูจน์" การอ่านที่ไม่มีใครเคยฟัง)
   - `printed-pdf` — ต้องสั่งพิมพ์จริง ไม่ใช่ดู CSS
2. **🔴 ชั้นพวกนี้ต้องนัดคนไว้ก่อนเริ่มงาน ไม่ใช่ไปหาเอาตอนจบ**
   แผนงานต้องระบุ **ใคร (P'Aim / พี่เปา) และ เมื่อไหร่** ตั้งแต่ก่อนลงมือ
   ที่ผ่านมารอดเพราะ P'Aim บังเอิญว่างมาฟัง/มาดู — **นั่นคือโชค ไม่ใช่ระบบ**
   ถ้ายังนัดไม่ได้ = **งานนี้ยังไม่พร้อมเริ่ม** ไม่ใช่ "เริ่มไปก่อนแล้วค่อยหาคน"
3. **ขัดกัน = ห้ามวนเถียง** — รันซ้ำได้ **คนละ 1 ครั้ง** ยังไม่ตรงกัน →
   **หยุด ส่ง P'Aim พร้อม output ทั้งสองฝั่ง** ห้ามหาข้อสรุปกันเอง ห้ามประนีประนอม
4. **`blocked` เป็นคำตอบที่ยอมรับได้** — "รันไม่ได้เพราะต้องล็อกอิน" ดีกว่าเดาแล้วเซ็น
5. **งานจบเมื่อ:** ทุก CLAIM มี VERDICT `cross-verified` **และ** ทุกข้อชั้นคน (`human-eye`/`audio`/`printed-pdf`) มี `## HUMAN` แล้ว

### ⚠️ ข้อสังเกตเรื่อง "ทำไม pleng พลาดบ่อยกว่าพระคำ" — จริงบางส่วน ห้ามใช้เป็นข้อแก้ตัว

**เอาไปเทียบกับความพลาดจริง 3 ข้อของ 20 ก.ค. แล้วเข้าแค่ 1 ใน 3:**

| ความพลาด | ทฤษฎีอธิบายได้ไหม |
|---|---|
| B118 เส้นโค้ง — วัดไม่ได้เพราะ rAF ไม่ยิง | ✅ ใช่ ชั้นที่แตะไม่ได้จริง |
| "คานหาย 92%" | ❌ **ไม่ใช่** — ของอยู่ในมือแล้ว (PDF จริง) พลาดเพราะ**ไม่ได้รันโมเดลหาค่าคาดหวังก่อนสรุป** · dev รันแล้วตรงเป๊ะ ≤0.1pt |
| "resize_window พัง" | ❌ **ไม่ใช่** — ไม่เกี่ยวกับชั้นเลย · relay คำคนอื่นโดยไม่ลองเอง ทั้งที่เพิ่งทำสำเร็จเมื่อชั่วโมงก่อน |

→ **ทฤษฎีนี้ใช้ออกแบบระบบได้ (ตั้งด่านคนที่ชั้นซึ่งไม่มีใครตรวจได้) แต่ใช้อธิบายความพลาดที่ผ่านมาไม่ได้ 2 ใน 3 ข้อ**
→ **ห้ามหยิบไปพูดว่า "pleng ยากกว่าเลยพลาดมากกว่า"** — 2 ใน 3 เป็นวินัยล้วนๆ ป้องกันได้ด้วยกฎที่มีอยู่แล้ว
→ และข้อสังเกตนี้มาจาก Claude ด้วยกัน + เป็นคำอธิบายที่**เข้าข้างเรา** = ยิ่งต้องตรวจก่อนเชื่อ (ดู §2.5 "ที่ตรงกัน = หลักฐานอ่อน")

## 6 · ⛔ สัญญาณที่เคยโกหก (เจอจริงทั้งหมด 20 ก.ค. 2026)

ทุกแถวคือของจริงที่เกิดขึ้นแล้ว ทั้งฝั่ง pleng และเว็บพี่น้อง (phrakham) วันเดียวกัน
**ไม่มีแถวไหนจับได้ด้วยการอ่านโค้ด/config/diff — จับได้เพราะเปิดของจริงดูเท่านั้น**

| สัญญาณที่บอกว่าสำเร็จ | ของจริงที่บอกความจริง |
|---|---|
| นับเส้นในไฟล์ได้เลขถูกทุกตัว → สรุปเองว่า "คานหาย 92%" | รันฟังก์ชันที่ผลิตมันออกมา แล้วเทียบค่าคาดหวัง (ตรงกันเป๊ะ ≤0.1pt = ไม่มีบั๊ก) |
| agent รายงาน "`resize_window` พัง ทดสอบมือถือไม่ได้" | ลองเอง → ใช้ได้ปกติ (agent แค่ลืมส่ง `tabId`) |
| dev + tester ยืนยันตรงกันว่า "คานหายหลัง 4 วิ" | ทั้งคู่วัดจากเบราว์เซอร์ agent ที่ `visibilityState:hidden` · rAF ไม่ยิงเลย = env พังอันเดียวกัน |
| `quarto render a b` **exit 0** ทั้งที่ไฟล์ที่สองพัง | เปิด HTML ที่ render ออกมา |
| mtime ของไฟล์ผลลัพธ์ใหม่ | grep เนื้อหาใหม่ในไฟล์ที่ build (mtime เด้งหมดแม้เนื้อไม่เปลี่ยน) |
| gate ขึ้น exit 0 "ไม่มีเสียง — ปกติ" | เอา gate ไปรันกับ tree ที่จงใจทำพัง (gate ที่ผ่านทั้งที่ของหาย = แย่กว่าไม่มี gate) |
| ลบ metadata ในโค้ดแล้ว | probe ไฟล์จริงหลังทำ (`-c copy` แบกของเก่ามาด้วยทั้ง 19 ไฟล์) |
| แบนเนอร์ในซอร์ส | grep ใน bundle ที่ minify แล้ว (ถูกตัดเงียบๆ) |
| seek ดูแม่นดี | ทดสอบ cold cache กับ server ที่รองรับ Range (`py -m http.server` ไม่รองรับ) |

**กฎที่สรุปได้:** เคลมอะไรก็ตามเกี่ยวกับ *สิ่งที่ผู้ใช้เห็น* → **ต้องเปิดของที่ผู้ใช้เห็นดู**
exit code · timestamp · ค่าใน config · diff · ผลที่ cache ไว้ — **ทั้งหมดนี้คือ "รายงานเกี่ยวกับ" ของจริง ไม่ใช่ของจริง และถูกจับได้แล้วว่าโกหกทุกตัว**

**ผลพลอยได้ที่ต้องจำ:**
- **บอกเสมอว่าวัดที่ชั้นไหน ในตัวข้อความนั้นเลย** — "ไม่มีภาษากรีกในซอร์ส" จริงและไร้ประโยชน์ · "ไม่มีภาษากรีกบนหน้าเว็บ" ผิด (จริง ๆ มี 21%) · คำเกือบเหมือนกัน แต่คนละเคลม
- **เครื่องมือที่พังเงียบ อันตรายกว่าเครื่องมือที่พังดังๆ**
- **ลบฟีเจอร์ยังไม่จบตอนลบโค้ด** — ต้องเช็กไฟล์ผลลัพธ์ ของที่ลบแล้วรอดมาบ่อยกว่าที่คิด
