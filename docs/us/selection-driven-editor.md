## 1 · ปัญหา — วัดสดใหม่บน live วันนี้ (ไม่ยกเลขจากเอกสาร)

รัน `docs/reports/editor-ux-baseline.md` (สคริปต์เดียวกัน) บน `https://pleng.phrakham.life/#/studio` เพลงเปล่า 1 ห้อง · **desktop 1280×800**:

| Metric | วัดสดวันนี้ | เอกสารฐาน | target |
|---|---|---|---|
| ปุ่ม/ช่องกดได้พร้อมกัน (M1) | **100** (79 ปุ่ม + 10 ช่องพิมพ์ + ลิงก์) | 100 | < 25 |
| ปุ่ม "ฟัง" (M3) | **5** | 5 | 1 |
| ปุ่มพรีวิว (M4) | **3** | 3 | 1 |
| ปุ่มลบ (M7) | **3** | 3 | ที่เดียว/scope |
| ปุ่ม edhead แถวเดียว (M8) | **15** | 15 | ≤ 7 |

**✅ เลขไม่เน่า — วัดสดตรงกับ SSOT เป๊ะ.** และการวัดสดได้ของแถม = **ป้ายจริงของปุ่มฟังทั้ง 5:**

```
ฟังท่อนนี้ · ฟังบรรทัดนี้ · ฟังห้องนี้ · ฟังทั้งเพลง · ฟังท่อน A
   ↑ท่อน      ↑บรรทัด       ↑ห้อง       ↑ทั้งเพลง    ↑ท่อนที่เลือก
```

**นี่คือหลักฐานต้นเหตุ (root cause) แบบชี้ตัวได้:** ฟัง 5 ปุ่ม = **ระดับ scope 5 ระดับ ต่างคนต่างวางปุ่มฟังของตัวเอง** · ลบ 3 = `ลบบรรทัด · ลบโน้ตนี้ · ลบห้องนี้` (scope 3 ระดับ) · พรีวิว 3 = `ตัวอย่างสด · ดูผลทั้งเพลง · ดูผลห้องนี้`.

> **สรุปต้นเหตุ:** editor **ไม่ได้ขาดระบบเลือก** — มันขาด **การซ่อน**. ทุกระดับ scope วาดชุดเครื่องมือเต็มของตัวเองค้างไว้ตลอด → 100 ปุ่ม. นี่ผิด **NN/g #8 (minimalist)** · **Miller's Law (~7±2)** · **Hick's Law** (ปุ่มยิ่งเยอะ ตัดสินใจยิ่งช้า) ตามที่ `ui-standards §2` อ้างไว้อยู่แล้ว.

**หมายเหตุ:** จอเปล่า = **ค่าต่ำสุด** (floor). เพลงจริงหลายท่อน/หลายห้อง จำนวนปุ่ม**คูณตามจำนวนห้อง/โน้ต** → ตัวเลขบานกว่านี้มาก. (mobile 375px = 95, ล้นแนวนอน 285px ในภาวะ headless — ต้องวัดซ้ำที่ 360/412 จริงตอน build)

## 2 · เทียบ pattern โลกจริง (เปิดของจริง ไม่เล่าจากความจำ)

เปิดเอกสาร/ของจริงมาแล้ว 6 ตัว. **แบ่งได้ 3 ตระกูล** — สำคัญคือ **ตัวไหน "เข้ากับ pleng"** (เลข+เนื้อ+คอร์ด บนช่องพิมพ์ **ไม่ใช่ staff 5 เส้น**):

### ก. แผงข้าง (persistent side panel) ที่ "เปลี่ยนเนื้อตามสิ่งที่เลือก"
| ตัว | พฤติกรรม (จากของจริง) | เข้ากับ pleng? |
|---|---|---|
| **MuseScore 4 Properties (F8)** | *"ไม่เลือกอะไร → โชว์ตัวเลือกทั้งสกอร์ · เลือกโน้ต → โชว์เฉพาะ property ของโน้ต"* | ✅ **โมเดลตรงกับ P'Aim ที่สุด** — "ไม่เลือก=จอสะอาด, เลือก=เครื่องมือโผล่" คือโมเดลนี้เป๊ะ |
| **Hookpad (Hooktheory)** | เครื่องมืออยู่ **ขวาของ staff** · *"เปลี่ยนเป็นคอนโทรลคอร์ด หรือ โน้ต ตามสิ่งที่เลือก"* | ✅ แนวคิดใช้ได้ · ⚠️ แต่ pleng จอแคบ (มือถือ) วางแผงขวาถาวรไม่ไหว |
| **Figma inspector** | เลือก object → แผงขวาโชว์ property ของ object นั้น | ✅ หลักการเดียวกัน (desktop) |

### ข. ป้ายลอยติดตัวสิ่งที่เลือก (floating contextual near the object)
| ตัว | พฤติกรรม | เข้ากับ pleng? |
|---|---|---|
| **Flat.io Note toolbar** | แตะโน้ต → แถบเครื่องมือของโน้ตโผล่ · แตะซ้ำ = เปลี่ยนความยาว/ลบ · มือถือ: เครื่องมืออยู่**ล่างจอ** | ✅ ใกล้ pleng — pleng มี `slot-tools` (◀▶) โผล่เหนือช่องโน้ตตอน focus อยู่แล้ว |
| **Songsterr tab editor** | *"คลิกโน้ต → เปิดเมนูแก้แบบ contextual · พิมพ์เลข 0–9 เปลี่ยนโน้ต · Backspace ลบ"* | ✅✅ **ใกล้ pleng สุดในเชิงโดเมน** — Songsterr เป็น tab (ตัวเลข+ช่อง ไม่ใช่ staff) และ**พิมพ์เลขแก้โน้ตเหมือน pleng เป๊ะ** |
| **iOS edit menu (Apple HIG)** | แตะค้าง/ดับเบิลแตะ → ป้ายลอย (Copy/Paste/Delete) · *"ถ้าไม่มีอะไรถูกเลือก เมนูต้องไม่โชว์ตัวเลือกที่ต้องมี selection"* | ✅ เป็น **มาตรฐาน Apple ที่หนุนแนวคิด P'Aim ตรง ๆ** |

### ค. แถบบนแปลงร่างตอนเลือก (Material 3 Contextual Action Bar)
| ตัว | พฤติกรรม | เข้ากับ pleng? |
|---|---|---|
| **Material 3 CAB** | เลือกของ → **top app bar แปลงเป็นแถบ action ของสิ่งที่เลือก** · เลิกเลือก → กลับเป็นแถบปกติ | ✅ **เข้ากับ edhead ของ pleng ที่สุด** — edhead เป็นแถบบนอยู่แล้ว → ให้มัน "แปลงร่าง" ตาม scope ที่เลือก |

### สรุปว่าตัวไหนเข้า/ไม่เข้า pleng

- ✅ **เข้า:** **Songsterr + Flat** (โดเมน tab/ตัวเลข-พิมพ์ ตรง pleng) · **MuseScore Properties + Apple HIG** (โมเดล "ไม่เลือก=สะอาด, เลือก=โผล่" + กติกาซ่อนปุ่มที่ต้อง selection) · **Material CAB** (แถบบนแปลงร่าง = ตรงกับ edhead)
- ❌ **ไม่เข้า:** **แผงข้างถาวรของ MuseScore/Hookpad/Figma แบบตายตัว** — มันกิน**พื้นที่กว้างถาวร** เหมาะจอ desktop กว้าง แต่พี่เปาใช้**มือถือ** (จอ 360–412px) → แผงขวาถาวรจะเบียดโน้ต. เอา**หลักการ**มา (เปลี่ยนเนื้อตามการเลือก) แต่**ไม่เอารูปแบบแผงข้างถาวร**.
- ⚠️ **ข้อควรระวังจาก Apple HIG:** *"อย่าใส่ทั้ง context menu และ edit menu ให้ของชิ้นเดียว จะสับสน"* → pleng ต้องเลือก **ทางเดียว** (แถบ contextual เดียว) ไม่ใช่ทั้งป้ายลอย + แถบบน + เมนูขวาคลิก ปนกัน.

> **บทสรุปมาตรฐาน:** แนวคิด P'Aim **ไม่ใช่ของแปลกใหม่ที่ต้องเสี่ยงทดลอง** — มันคือแพตเทิร์นที่ **โปรแกรมโน้ตระดับโลกทำเหมือนกันหมด** และมี **มาตรฐาน Apple/Material หนุนเป็นข้อ ๆ**. เราไม่ได้เดิมพัน เราเดินตามทางที่พิสูจน์แล้ว.

## 3 · Feasibility กับโค้ดจริง (เปิด `EditorMode.vue` อ่านเอง — 4,575 บรรทัด)

> ⚠️ ไฟล์มี null byte บรรทัด 1569 → อ่านผ่าน `tr -d '\000' < file` (ripgrep ตรง ๆ หยุดอ่านกลางไฟล์)

### 3.1 ⭐ ค้นพบสำคัญ — editor **มี "การเลือก" ครบ 5 ระดับอยู่แล้ว**

| ระดับ scope | ตัวแปรสถานะที่มีอยู่แล้ว | บรรทัด | วันนี้ทำอะไร |
|---|---|---|---|
| **โน้ต/พยางค์** | `focusedSlot` | 324 | โฟกัสช่อง → โชว์ `slot-tools` (◀▶) เหนือช่องนั้น |
| **คอร์ดบนโน้ต** | `editingChord {li,bi,si,p}` | 423 | จิ้มคอร์ด → เปิดตัวแก้คอร์ดตรงโน้ตนั้น |
| **ห้อง** | `barMenuOpen "li-bi"` | 1805 | จิ้มห้อง → เปิด popover เครื่องมือห้อง (ทีละห้อง) |
| **บรรทัด** | `activeLine` | 877 | บรรทัดที่แตะล่าสุด = เป้าของปุ่ม ▲▼/ฟัง/ลบ ใน edhead |
| **ท่อน** | `activeStanza` / `lensChoice` | 189/? | ท่อนที่เลือก → โชว์ `cshead` (แก้ชื่อ/ทำนอง/คีย์/ย้าย/ลบท่อน) |

**นี่คือหัวใจของ feasibility:** โครง "เลือกแล้วรู้ว่าเลือกอะไร" **มีครบแล้วทั้ง 5 ระดับ**. `slot-tools`, `barMenuOpen` popover, `cshead` (โผล่เฉพาะ `lensActive`) **เป็น selection-driven อยู่แล้วบางส่วน**.

### 3.2 แล้วทำไมยังมี 100 ปุ่ม?

เพราะ **"การเลือก" วันนี้แค่ "ชี้เป้า" ไม่ได้ "ซ่อน"**:
- `activeLine` เปลี่ยน**เป้า**ของปุ่ม ▲▼ — แต่ปุ่ม ▲▼/ฟังบรรทัด/ลบบรรทัด (กลุ่ม `ed-quick` 6 ปุ่ม) **โชว์ค้างตลอด** ไม่ว่าเลือกบรรทัดหรือไม่
- ปุ่ม `ฟังท่อนนี้/ฟังบรรทัดนี้/ฟังห้องนี้/ฟังทั้งเพลง` **วางค้างทุกปุ่มพร้อมกัน** ไม่ได้ยุบเป็น "ฟังสิ่งที่เลือก"
- ห้องทุกห้องมีปุ่ม `+`/เครื่องมือของตัวเอง · โน้ตทุกตัวมีเครื่องมือรอบตัว → **คูณตามจำนวน element**

> **แปลว่า:** งานนี้ **ไม่ต้องสร้างระบบเลือกใหม่** (มีแล้ว) · แค่เพิ่ม **(1) ตัวบอก "ตอนนี้เลือก scope อะไร"** ตัวเดียว (รวม 5 ตัวข้างบนให้รู้ว่าอันไหนคือ "ที่กำลังโฟกัส") + **(2) เงื่อนไขซ่อน/โชว์กลุ่มปุ่มตาม scope นั้น**.

### 3.3 ของดีอีกอย่าง — action model เป็น **declarative อยู่แล้ว**

`editItems` (บรรทัด 1744) เป็น **array ของ object** ที่มี `place`, `anchor`, `row`, `hidden`, `pinnable`, `kind` — แถบเครื่องมือ render จาก data ไม่ใช่ปุ่ม hardcode กระจาย. **การเติม field `scope:` ให้แต่ละ item แล้วกรองตาม scope ที่เลือก = แก้ที่เดียว ครอบทุกปุ่ม** (ตรงหลัก `ui-standards §3` library กลาง). มี `hidden` เป็นเงื่อนไขซ่อนอยู่แล้ว → เพิ่มเงื่อนไข scope เข้าไปในกลไกเดิมได้.

### 3.4 สรุป feasibility

| | คำตอบ |
|---|---|
| ต้องรื้อสถาปัตยกรรมไหม | **ไม่** — state การเลือกครบแล้ว · action model เป็น data อยู่แล้ว |
| งานหลักคืออะไร | เพิ่ม `activeScope` 1 ตัว (รวม 5 selection ref) + field `scope` ต่อ item + `v-if` กลุ่มปุ่มตาม scope |
| จัดอยู่ในหมวด | **refine** (ต่อยอดของเดิม) — ตรง `feedback_refine_not_redesign` · **ไม่ใช่ redesign** |
| ~13 ปุ่มรอบโน้ต 1 ตัว | ยุบได้ (ดูเฟส 1) — เป้าเหลือ **แถบ contextual เดียว** ของ scope โน้ต แทนปุ่มลอยกระจาย |

## 4 · มือถือ (พี่เปาใช้จริง) — ข้อจริงที่ต้องแก้ให้ได้

พี่เปาแก้เพลงบนมือถือ 360–412px. ประเด็นมือถือที่ selection-driven **ต้องตอบให้ได้**:

1. **🔴 แป้นพิมพ์บังแถบล่าง (คอขวดจริง):** pleng แก้โน้ต/เนื้อ = **พิมพ์ในช่อง** → คีย์บอร์ดจอเด้งขึ้นกินครึ่งล่าง. ถ้าวางแถบ contextual **ล่างจอแบบ Flat** ตอนพิมพ์อยู่ → **คีย์บอร์ดบังแถบ**. → **ข้อเสนอ:** แถบ contextual ของ "โน้ต/พยางค์" ควรเกาะ**เหนือช่องที่เลือก** (แบบ `slot-tools` วันนี้ที่โผล่เหนือช่อง — พิสูจน์แล้วว่าไม่โดนคีย์บอร์ดบัง) ไม่ใช่ตรึงล่างจอ. ส่วนแถบของ scope ที่**ไม่ต้องพิมพ์** (ห้อง/บรรทัด/ท่อน = กดปุ่มล้วน) วางล่างจอแบบ CAB ได้.
2. **ไม่บังโน้ต:** ป้าย contextual ต้องเกาะติดสิ่งที่เลือกและ **clamp ไม่หลุดจอ** (`ui-standards §2` popup ห้าม scroll/หลุดขอบ) · จอง scroll ให้ช่องที่เลือกไม่โดนแถบทับ (`scroll-margin`).
3. **แป้นพิมพ์ชนการเลือก:** วันนี้ในช่องพยางค์ Space/Enter/Backspace/Delete ถูกจับเป็น "แยก/รวมพยางค์" แล้ว (`onSylKey` บรรทัด 371) → selection-driven ต้อง**ไม่ไปแย่ง** คีย์เหล่านี้ตอนโฟกัสอยู่ในช่อง (ใช้คีย์ลัดเฉพาะตอน "เลือก scope ที่ไม่ใช่ช่องพิมพ์").
4. **target size:** วัดสดวันนี้ under-44px = 56 ปุ่ม (desktop). ยุบปุ่มซ้ำ = **โอกาสทำให้ปุ่มที่เหลือใหญ่ขึ้นถึง 44px** (M9). ต้อง verify 360/412 จริง + `right ≤ viewport` ทุกปุ่ม (`feedback_verify_mobile_real_width`).
5. **regression guard:** ห้ามทำ M9(<44)/M10(<24)/M11(h-scroll=0) แย่ลง — วัดสดหลังทุกเฟส.

## 5 · 🚧 เข้ากับงานที่กำลังทำ — อันไหนรอด อันไหนชน (ต้องบอกตรง)

| งานในมือ | ชนไหม | เหตุผล |
|---|---|---|
| **orientation — แถบระบุตัวตนบน shell** (`docs/ds/editor-orientation.md`, `91b05cf3`) | ✅ **ไม่ชน — เสริมกันด้วยซ้ำ** | แถบตัวตนตอบ *"ฉันเปิดเพลงอะไร/ฉบับใคร"* (ชั้น **สถานะ**) · selection-driven ตอบ *"ฉันเลือกอะไรจะแก้"* (ชั้น **เครื่องมือ**) → **คนละชั้น คนละแถว** · แถบตัวตนอยู่**บน shell เหนือ dock** → รอดพ้นการรื้อ dock (SA orientation ขีดเส้นไว้ถูกแล้ว) |
| **dev เฟส1 "ห้ามเลือดไหล"** (`996eafda` — D3/D4/D2) | 🔴 **ชนไฟล์ตรง ๆ** | ทั้งคู่แก้ **`EditorMode.vue` ไฟล์เดียวกัน** · เฟส1 กำลังแก้ปุ่มหลัก/เตือนทับร่าง · selection-driven จะ**ปรับโครง dock ทั้งก้อน** → ถ้าทำพร้อมกัน = merge ชนหนัก → **ต้องรอเฟส1 merge เข้าฐานก่อน** |
| **DS orientation ขีดเส้น** *"รื้อ dock ทั้งก้อน = คนละขนาดกับ orientation → เป็นสายแยก"* | ✅ **ถูกต้อง** | **นี่คือ "รื้อ" ตัวจริง** ในความหมาย `feedback_refine_not_redesign` — ต้อง**เสนอชัด ไม่แอบทำ** · เฟส 1–2 ของงานนี้ = แตะ dock/edhead หนัก → **ต้อง P'Aim เคาะ** ก่อนลงมือ (เฟส 0 ไม่แตะโครง = ทำได้ก่อน) |

> **สรุป §5:** แถบตัวตน (orientation) = คนละชั้น ปลอดภัย. แต่ **เฟส 1–2 คือการรื้อ dock จริง** และ **ชนไฟล์กับ dev เฟส1** → **ลำดับบังคับ: เฟส1 merge ก่อน → แล้วค่อยเฟส 1 ของงานนี้** · และเพราะเป็น "รื้อ" ต้องรอ P'Aim สั่ง.

## 6 · ข้อเสนอ (ฟันธง 1 ทาง + ทำไมดีกว่าทางอื่น)

### 6.1 ทางที่แนะนำ — **"แถบเครื่องมือเดียวที่เปลี่ยนเนื้อตามสิ่งที่เลือก" (one contextual toolbar)**

รวมโมเดล 3 มาตรฐานที่เข้ากับ pleng:
- **ไม่เลือกอะไร → จอสะอาด:** โชว์เฉพาะ action ระดับ**เอกสาร** (ฟังทั้งเพลง · ดาวน์โหลด · บันทึกร่าง/เผยแพร่ · ย้อน/ทำซ้ำ · ตั้งค่า) — ตรง **MuseScore Properties** ("ไม่เลือก=ตัวเลือกทั้งสกอร์") + **Apple HIG** ("ไม่มี selection อย่าโชว์ปุ่มที่ต้อง selection")
- **เลือกโน้ต → เครื่องมือโน้ต** (จุดบน/ล่าง=ทบเสียง · เขบ็ต · เอื้อน · ลบโน้ต · คอร์ดที่โน้ต) เกาะเหนือช่อง (แบบ `slot-tools` เดิม แต่ครบเครื่อง)
- **เลือกห้อง/บรรทัด/ท่อน → เครื่องมือของ scope นั้น** ในแถบเดียว (แบบ **Material CAB** — edhead แปลงร่างตาม scope)
- **ทุก verb ยุบเป็นปุ่มเดียวชี้ scope ที่เลือก:** `ฟังสิ่งที่เลือก` · `ลบสิ่งที่เลือก` · `ดูผล` (5→1, 3→1, 3→1)

### 6.2 ทำไมดีกว่าทางอื่น (ไม่ list เฉย ๆ — เทียบให้เห็น)

| ทางเลือก | ผล | ตัดสิน |
|---|---|---|
| **คงเดิม (โชว์ทุกปุ่มทุก scope)** | 100 ปุ่ม · ผิด Miller/Hick/NN#8 · พี่เปาบ่น "งง" ทั้งวัน | ❌ นี่คือปัญหา |
| **รื้อทำใหม่ทั้ง editor** | เสี่ยงสูง · พี่เปาเรียนใหม่หมด · 3 เดือนไม่มีของถึงมือ · ทิ้ง 5 selection ref ที่ดีอยู่แล้ว | ❌ ผิด `refine_not_redesign` + big-bang |
| **แผงข้างถาวรแบบ MuseScore/Figma** | กินพื้นที่กว้างถาวร · จอมือถือ 360px รับไม่ไหว | ❌ ไม่เข้า pleng (มือถือ) |
| **✅ contextual toolbar เดียว (ต่อยอดของเดิม)** | ใช้ 5 selection ref + declarative editItems ที่มีอยู่ · ยุบปุ่มซ้ำอัตโนมัติ · แก้ที่เดียว · เดินเป็นเฟสได้ | ✅ **เล็กสุดที่แตะครบทุก metric** |

## 7 · เฟส — มีของถึงมือพี่เปาทุกรอบ (ไม่ big-bang)

| เฟส | ทำอะไร | แตะอะไร | พี่เปาได้อะไร | บล็อกไหม |
|---|---|---|---|---|
| **0 · ยุบปุ่มซ้ำ** ⭐ (80/20) | 5 ฟัง→1 `ฟังสิ่งที่เลือก` · 3 ลบ→1 · 3 พรีวิว→1 · ย้าย/สำเนา ชี้ scope ที่เลือก · **ไม่แตะ layout** | เพิ่ม `activeScope` + แก้ target ปุ่ม (data ใน `editItems`) | **ปุ่มลดฮวบทันที (M3–M7) · แทบไม่ต้องเรียนใหม่** (ปุ่มหายไป = ของซ้ำ ไม่ใช่ของที่เคยใช้) | 🟡 อยากทำหลังเฟส1 dev merge (เลี่ยงชนไฟล์) · **แต่เสี่ยงต่ำ อาจทำคู่ขนานได้ถ้า PM คิว** |
| **1 · ซ่อนตามการเลือก** | ไม่เลือก=จอสะอาด · เลือก scope ไหน โชว์เครื่องมือ scope นั้น (edhead แปลงร่างแบบ CAB) | **รื้อ dock/edhead** (v-if กลุ่มปุ่มตาม scope) | จอเปล่า < 25 ปุ่ม (M1/M2/M8) · เห็นเฉพาะที่เกี่ยว | 🔴 **รอเฟส1 dev merge + P'Aim เคาะ** (นี่คือ "รื้อ" จริง) |
| **2 · ขัดมือถือ** | แถบ contextual เกาะถูกที่ (เหนือช่อง=พิมพ์ · ล่างจอ=กดล้วน) · เลี่ยงคีย์บอร์ดบัง · target 44px | dock mobile + scroll-margin | แก้บนมือถือลื่น ไม่โดนแป้นบัง | 🔴 ต่อจากเฟส 1 |

**ทำไมเรียงงี้:** เฟส 0 = **คุ้มสุด เสี่ยงต่ำสุด** (ยุบของซ้ำ = win ล้วน ไม่ต้องเรียนใหม่) → จ่ายก่อน · เฟส 1–2 = ของที่ทำให้ "สะอาดจริง" แต่แตะโครง → รอคิว + P'Aim.

## 8 · ต้นทุน/ความเสี่ยง ซื่อ ๆ (คุ้มไหม)

| ด้าน | ประเมินตรง ๆ |
|---|---|
| **พี่เปาเรียนใหม่แค่ไหน** | **เฟส 0 ≈ 0** (ปุ่มที่หายคือของซ้ำ) · **เฟส 1 = ปานกลาง** (ต้องชินกับ "จิ้มก่อน") → ลดแรงกระแทกด้วย: คงปุ่มเอกสารที่ใช้บ่อยไว้เสมอ + hint ครั้งแรก + ยังจิ้มตรงจุดที่แก้เหมือนเดิม |
| **ความเสี่ยงหลัก** | (1) มือถือ: คีย์บอร์ดบังแถบล่าง (§4.1) · (2) ชนไฟล์ dev เฟส1 (§5) · (3) refactor 5 selection ref พังของเดิม · (4) regression M9–M11 |

## 9 · สิ่งที่ผม (SA) ไม่ตัดสินแทน — รอ P'Aim

1. **จะเดินเฟส 1–2 (รื้อ dock) ไหม** — เฟส 0 คุ้มชัด แต่ 1–2 = รื้อจริง + พี่เปาเรียนใหม่ → **P'Aim เคาะว่าเอาความสะอาดสุด (เดินครบ 3 เฟส) หรือพอแค่ยุบปุ่มซ้ำ (เฟส 0)**
2. **ลำดับกับ dev เฟส1** — ให้ผมจ่อคิวหลังเฟส1 merge หรือทำเฟส 0 คู่ขนาน (PM คิวไฟล์)
3. **ยังไม่ build** — ใบนี้เป็นบทวิเคราะห์. เคาะแล้วค่อยทำ DS + จ่าย dev.
