# US — audit-log: บันทึกประวัติ "ใครทำอะไรกับเพลงในคลัง"

Backlog: B028 · DS: `docs/ds/audit-log.md` (ยังไม่เขียน — รอ US ผ่าน)
สถานะ: spec (รอ P'Aim approve)
ธีมใหม่ = **"บันทึกประวัติ"** — ไม่ทับ ps3sa (หน้าจอ/เมนู) และ sa-jianpu-rules (กฎโน้ต)

## ที่มา (ปัญหา)
เพลงในคลังกลางถูกทีม คจ. หลายคนช่วยกันทำ — เพลง 1 เพลงเดินผ่าน **คนทำเพลง** (สร้าง/แก้/ส่งตรวจ) แล้วส่งต่อ **คนอนุมัติ** (อนุมัติแล้วเผยแพร่ให้คนทั่วไปเห็น). ทุกวันนี้ถ้าเพลงเปลี่ยนไปหรือมีอะไรผิด **ไม่มีทางรู้ว่าใครทำ ทำเมื่อไหร่ แก้ตรงไหน หรือใครเป็นคนตัดสินใจให้ออกสู่สาธารณะ** → อยากได้ระบบที่จดประวัติเองอัตโนมัติ เชื่อถือได้ ย้อนดูได้

## โยงกลับ mission
- Tier 1 (คนทำเพลง) "ร่าง / ส่งตรวจ" · Tier 2 (คนอนุมัติ) "อนุมัติเข้าคลัง / ลบ / ย้อน" → ระบบต้อง **รู้ตัวตน** ว่าใครทำอะไรในแต่ละขั้น
- เส้นแบ่งสิทธิ์ = "เก็บเข้าระบบได้ไหม" → log จับเฉพาะสิ่งที่ **เข้าคลังกลาง** เท่านั้น (ของส่วนตัวไม่เกี่ยว)
- "jianpu มาตรฐานที่ส่งต่อได้" → ประวัติที่ตรวจสอบได้ทำให้คลังน่าเชื่อถือ

---

## US-1 — จดทุกการกระทำอัตโนมัติ
**ในฐานะ** ทีม คจ. ที่ดูแลคลังเพลง
**ฉันต้องการ** ให้ระบบจดเองทุกครั้งที่มีคนแตะเพลงในคลัง ว่า *ใคร · ทำอะไร · เพลงไหน · เมื่อไหร่*
**เพื่อ** ตรวจสอบย้อนหลังได้เสมอ โดยไม่ต้องอาศัยความจำหรือให้ใครมาจดมือ

**AC:**
- [ ] ทุกการ **สร้าง · แก้ · ส่งตรวจ · อนุมัติ+เผยแพร่ · ตีกลับ · ถอน/ลบ** เพลงในคลัง → มีบันทึก 1 รายการเกิดขึ้นเองอัตโนมัติ
- [ ] แต่ละรายการบอกครบ: **ชื่อผู้ทำ · ประเภทการกระทำ · เพลง (ชื่อ/รหัส) · วันเวลา**
- [ ] จดครบไม่ว่าจะทำผ่านทางไหน (คนทำงานไม่มีทาง "ลืมจด" หรือข้ามได้)

## US-2 — เห็นว่าแต่ละครั้ง "แก้ตรงไหน"
**ในฐานะ** คนอนุมัติ / คนดูแลคลัง
**ฉันต้องการ** เปิดดูรายการแก้แต่ละครั้งแล้วเห็นว่า *เนื้อ/โน้ตเปลี่ยนจากอะไรเป็นอะไร*
**เพื่อ** รู้ว่าการแก้ครั้งนั้นทำอะไรกับเพลง ก่อนจะอนุมัติหรือหาสาเหตุปัญหา

**AC:**
- [ ] เปิดรายการ "แก้" แต่ละครั้ง → เห็นการเทียบ **ก่อน ↔ หลัง** (จุดที่เปลี่ยนถูกไฮไลต์ เหมือนดูเอกสารที่ถูกแก้)
- [ ] รองรับทั้งการแก้ **เนื้อร้อง** และ **โน้ต/โครงเพลง**
- [ ] *(ยังไม่ทำในรอบนี้)* ปุ่ม "กดย้อนกลับไปเวอร์ชันนั้น" — แต่ข้อมูลที่เก็บต้องพร้อมต่อยอดได้ทันทีในอนาคต

## US-3 — จับครบ "2 มือ": คนทำ + คนอนุมัติ
**ในฐานะ** คนดูแลคลัง
**ฉันต้องการ** ให้ประวัติแสดงเส้นทางของเพลงตลอดสาย ตั้งแต่คนทำเพลงจนถึงคนอนุมัติเผยแพร่
**เพื่อ** รู้ชัดว่าใครส่งตรวจ และ **ใครเป็นคนตัดสินใจให้เพลงออกสู่สาธารณะ**

**AC:**
- [ ] จับ **ฝั่งคนทำเพลง:** สร้างร่าง · แก้ · **ส่งตรวจ** (ใครกด/เมื่อไหร่)
- [ ] จับ **ฝั่งคนอนุมัติ:** **อนุมัติ+เผยแพร่** · ตีกลับ (ส่งคืนให้แก้) · ถอน/ลบ (ใครกด/เมื่อไหร่)
- [ ] เหตุการณ์ **"อนุมัติ+เผยแพร่"** ต้องบันทึกแน่นที่สุด (ตอบให้ได้เสมอว่าใครทำให้เพลงนี้เป็นสาธารณะ)
- [ ] แต่ละรายการดูออกว่ามาจากขั้น "ทำเพลง" หรือ "อนุมัติ"

## US-4 — เปิดดูประวัติของเพลงได้
**ในฐานะ** ทีม คจ. ที่ล็อกอิน
**ฉันต้องการ** เปิดดู "ประวัติการแก้ไข" ของเพลงหนึ่งได้ เรียงตามเวลา
**เพื่อ** ไล่ดูได้ว่าเพลงนี้ผ่านมือใครมาบ้าง และเกิดอะไรขึ้นเมื่อไหร่

**AC:**
- [ ] ในหน้าเพลง (เฉพาะคนที่ล็อกอิน) มีทางเข้าดู **"ประวัติการแก้ไข"** ของเพลงนั้น
- [ ] แสดงเป็นรายการเรียงจากใหม่ → เก่า: *เวลา · ผู้ทำ · การกระทำ*
- [ ] กดที่รายการ "แก้" เพื่อดูรายละเอียด ก่อน↔หลัง (ตาม US-2)
- [ ] คนทั่วไป (ไม่ล็อกอิน) **มองไม่เห็น** เมนูนี้

## US-5 — บันทึกแก้ไม่ได้ (เชื่อถือได้จริง)
**ในฐานะ** คนดูแลคลัง
**ฉันต้องการ** ให้ประวัติเป็นของ "อ่านอย่างเดียว" ใครก็ลบ/แก้ร่องรอยของตัวเองไม่ได้
**เพื่อ** ให้บันทึกเชื่อถือได้จริงเวลาต้องใช้ตรวจสอบ

**AC:**
- [ ] ผู้ใช้ทั่วไป/ทีม **แก้หรือลบ** รายการในประวัติไม่ได้ (เพิ่มได้อย่างเดียวโดยระบบ)
- [ ] แม้แต่คนที่แก้เพลง ก็ลบร่องรอยการแก้ของตัวเองไม่ได้

## ขอบเขต / นอกขอบเขต (กันเข้าใจผิด)
- **จับเฉพาะเพลงในคลังกลาง** — คนที่ไม่ล็อกอินแล้วแก้ไฟล์ JSON ของตัวเอง = อยู่นอกคลัง = ไม่ถูกจด (เป็นของส่วนตัว)
- **รอบนี้ยังไม่ทำ:** ปุ่มกดย้อนกลับเวอร์ชัน (แค่ "ดู" ก่อน · ออกแบบข้อมูลให้ต่อยอดง่าย) · การแจ้งเตือน/สรุปรายงาน · ประวัติของสิ่งที่ไม่ใช่เพลง (บัญชีผู้ใช้ ฯลฯ)
- **ไม่ตัดสิน** ว่าการแก้นั้น "ดี/ถูกทำนองไหม" — แค่บันทึกว่าเกิดอะไรขึ้น (การตัดสินคุณภาพเป็นของคนอนุมัติ)
