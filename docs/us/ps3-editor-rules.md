# US — ps3: กฎตรวจโน้ตในโหมดแก้ไข (lint) — "เตือนสิ่งที่ไม่ควรทำ"

Backlog: B026 · DS: `docs/ds/ps3-editor-rules.md`
โฟกัสรอบนี้ = **กฎ (algorithm) ล้วน** ก่อน · UX โต๊ะแก้ไขที่เหลือแยกไว้รอบหลัง
**แหล่งอ้างอิงที่ตรวจแล้ว:** Wikipedia "Numbered musical notation" (มาตรฐาน jianpu) + โค้ด plugin จริง MuseScore (`EasyJianpu`, `Jianpu_Numbered_Notation_v410/v46`) — เก็บเฉพาะกฎที่ **แม่นตามมาตรฐาน + ช่วยได้จริง + implement ได้จริง** บนโมเดลของแอป (movable-do · ทำนองเดี่ยว + เนื้อร้อง)

## ที่มา (ปัญหา)
ทีมคีย์เพลง **ไม่ชัวร์ทฤษฎีโน้ตตัวเลข** ว่าอะไรถูก/ผิด (เคสจริง: ♮ เพลง 31 คีย์ A · เอื้อนเว้นวรรค) → อยากให้ **โปรแกรมเตือนกฎที่ตายตัว**ตรงจุดที่แก้ · **หลักการ:** เตือน = ชี้ "อะไรไม่ควรทำ + ควรทำอะไรแทน" · **ไม่บล็อกการบันทึก** · เตือนเฉพาะกฎที่ **ผิดแน่ตามระบบโน้ต** (ไม่เดา ไม่ตีความดนตรี)

---

## กฎที่จะเตือน (7 ข้อ — มั่นใจว่าแม่น + implement ได้)

### R1 — ♮ (natural) ใช้ผิด
ในฐานะคนคีย์เพลง อยากให้เตือนเมื่อใส่ ♮ ที่ "ไม่มีผล" เพื่อรู้ว่าควรใช้ b/# แทน
**AC:** มี **♮N โดยไม่มี #N/bN มาก่อนในห้องเดียวกัน** → เตือน **เหลือง** + ข้อความ "♮ ใช้ยกเลิก #/b เท่านั้น ไม่ใช่โน้ตขาว — อยากได้เสียงต่างจากคีย์ ใช้ bN (ต่ำลง)/#N (สูงขึ้น)" · ถ้า ♮ ตามหลัง #/b ตัวเดียวกันในห้อง (ยกเลิกจริง) → ไม่เตือน · แยกตามช่วงเสียง (octave)
> ยืนยัน: Wikipedia — accidental "relative to the diatonic scale (1-7), not note names" (เช่น B natural ในคีย์ Cm เขียน ♯5)

### R2 — จังหวะในห้องไม่ครบ
**AC:** ห้องที่ผลรวมจังหวะ ≠ อัตราจังหวะ (4/4 ควรได้ 4) → เตือน **เหลือง** บอก "ได้กี่/ควรกี่" · (มีบางส่วนแล้ว — รวมเข้าชุดกฎเดียว) · ห้องแรกของท่อนไม่เต็มได้ (ห้ามเตือนแรงเป็น error)

### R3 — ตัวอักษรอ่านไม่ได้
**AC:** มีตัวที่ระบบอ่านไม่ออก (พิมพ์ผิด) → เตือน **แดง** (ผิดแน่) บอกตัวที่มีปัญหา · ข้ามการนับจังหวะห้องนั้น (กันเตือนซ้อน)

### R4 — เอื้อนโดนเว้นวรรคแยกช่อง
**AC:** เอื้อน (วงเล็บ `( )` = ร้องคำเดียวหลายโน้ต) ต้องอยู่ **ช่องเดียว** · ถ้าวงเล็บถูกแยกคนละช่อง (เว้นวรรคในวงเล็บ) → เตือน **เหลือง** "เอื้อนต้องพิมพ์ติดกันในช่องเดียว ไม่งั้นนับเป็นหลายคำ" · (ตรงกับที่เพิ่งแก้คู่มือ: `(65)` ไม่ใช่ `(6 5)`)

### R5 — ไท (~) ต้องเชื่อมโน้ตเสียงเดียวกัน ⭐ใหม่
**AC:** เครื่องหมายไท `~` เชื่อมโน้ต **2 ตัวเสียงเดียวกัน** ให้ลากต่อ · ถ้า `~` เชื่อมโน้ต **คนละเสียง** → เตือน **เหลือง** "ไทใช้กับเสียงเดียวกันเท่านั้น ถ้าจะเอื้อน (คนละเสียง) ใช้วงเล็บ ( )"
> ยืนยัน: มาตรฐาน + plugin (โน้ตตัวรับของไท "ไม่พิมพ์เลขซ้ำ/ไม่รับคำใหม่")

### R6 — ตัวหยุด (0) ใส่ #/b หรือจุดอ็อกเทฟ ⭐ใหม่
**AC:** ตัวหยุด `0` ไม่มีเสียง → ใส่ **#/b หรือจุดอ็อกเทฟ (บน/ล่าง)** ไม่มีความหมาย → เตือน **เหลือง** "ตัวหยุด (0) ใส่ชาร์ป/แฟลต/จุดอ็อกเทฟไม่ได้ — น่าจะพิมพ์ผิด" · (ขีดลาก `-` และจุดเพิ่มค่ายังใส่ได้ = ความยาว)
> ยืนยัน: plugin ทุกตัว render 0 เป็นตัวเปล่า ไม่มีจุด/ชาร์ป

### R7 — ชาร์ป/แฟลตที่ "ไม่มีจริง" (#3 · b4 · #7 · b1) ⭐ใหม่
**AC:** มี-ฟา (3–4) และ ที-โด (7–1) ห่างกันครึ่งเสียงอยู่แล้ว → `#3`, `b4`, `#7`, `b1` ตกบนโน้ตที่ไม่มีครึ่งเสียง = เป็นเสียงของเลขข้าง ๆ → เตือน **เหลือง** "#3 คือเสียงเดียวกับ 4 (ควรเขียน 4)" (และ b4=3 · #7=1 · b1=7)
> ยืนยัน: plugin ใช้ตาราง 12 ช่อง — accidental มีได้เฉพาะ 5 ขั้นโครมาติก (#1/b2 · #2/b3 · #4/b5 · #5/b6 · #6/b7) เท่านั้น

## ระดับการเตือน + ที่แสดง
**AC:** 3 ระดับ: **แดง=ผิดแน่** (R3) · **เหลือง=น่าจะพลาด** (R1/R2/R4/R5/R6/R7) · **ฟ้า=คำแนะนำ** (เผื่ออนาคต) · แสดง **ใต้/ข้างแต่ละห้อง** ตรงจุดที่แก้ · หลายเตือน/ห้องได้ · **ไม่บล็อกการบันทึก/ส่งตรวจ**

## ขอบเขต — สิ่งที่ระบบ "ไม่" ตรวจ (สำคัญ กันเข้าใจผิด)
เช็กได้แค่ **"สัญลักษณ์ผิดกฎ + โครงสร้างจังหวะ"** · **ตรวจไม่ได้**ว่าโน้ตนั้น "ถูกทำนองไหม" · คอร์ดเหมาะไหม · คีย์ถูกไหม — เป็นดุลพินิจดนตรี ต้องใช้หูคน

## จงใจ "ยังไม่ทำ" (บันทึกเหตุผล)
- **จุดคู่ (double-dot `..`) / ตัวเชื่อม 32 (ขีดใต้ 3 ชั้น):** แอปยังไม่รองรับ = ข้อจำกัดแอป ไม่ใช่กฎ (ถ้าพิมพ์จะโดน R3 อยู่แล้ว)
- **ลำดับสัญลักษณ์ในตัวโน้ต:** editor auto-fix ให้แล้ว (`fixAccidental`) — ไม่ต้อง lint
- **ตัวหยุดยาวควรเขียน 0 0 0 แทน 0 - -:** เป็น "สไตล์" ไม่ใช่ผิด → เก็บเป็น **ฟ้า/คำแนะนำ** อนาคต
- **คอร์ดซ้อนแนวตั้ง / grace note / glissando:** แอปเป็นทำนองเดี่ยว+เนื้อ ไม่มีในโมเดล → ไม่เกี่ยว
- **#2 ↔ b3 (เสียงเท่ากัน เลือกตามทิศทางทำนอง):** เป็นคำแนะนำ (ฟ้า) อนาคต ไม่ใช่ผิด
