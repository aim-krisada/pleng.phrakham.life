## 4. ทำไมความเสียหายถึงน้อยกว่าที่กลัว

บั๊กจะทำลายข้อมูลได้ก็ต่อเมื่อมีคน **เปิดร่างของเพลงที่เผยแพร่แล้ว → กดเผยแพร่ทับ** เท่านั้น
เพลงส่วนใหญ่ในห้องสมุดถูกแก้แบบ **เปิดเพลงตรงๆ แล้วบันทึก** (ไม่ผ่านร่าง) — เส้นทางนั้นอ่านหมวด/ธีมจริงจากแถวเพลง จึงไม่ทับ
สถิติที่หนุนข้อนี้: จาก 81 เพลงอนุชนที่เทียบได้ มี **เพลงเดียว** ที่ธีมหาย ทั้งที่ทุกเพลงถูกแก้ไปแล้วหลายรอบ (updated_at กระจายทุกวันตั้งแต่ 13–23 ก.ค.)

## 5. บั๊กยังอยู่บน production ไหม — **ส่วนใหญ่แก้แล้ว**

### 5.1 ที่แก้แล้วและขึ้นเว็บจริงแล้ว ✅

| หลักฐาน | ผล |
|---|---|
| ไฟล์โค้ดจริงบนเว็บ `pleng.phrakham.life/assets/index-CSAfPjE3.js` | มีตราคอมมิต **`5661068`** = `origin/main` ล่าสุด **และ** พบตัวแปร `categoryKnown` / `themeKnown` ในบันเดิล |
| คอมมิตที่แก้บั๊กบน `main` | **`7284b34` (2026-07-20)** — "B108 per-field knownness + drafts carry their own หมวด/ธีม" |
| คอลัมน์จริงบนฐานข้อมูล | probe `song_drafts?select=category` และ `theme` → **ตอบ 200 (คอลัมน์มีจริง)** ส่วนคอลัมน์มั่ว ตอบ error 42703 → `db/010` **รันไปแล้วจริง** |
| เส้นทางผู้อนุมัติ | โค้ดที่ deploy อยู่ **ปฏิเสธการเผยแพร่** ถ้าอ่านธีมเดิมไม่สำเร็จ ("❌ อ่านหมวด/ธีมเดิมของเพลงไม่สำเร็จ") = กันธีมหายแล้ว |

⇒ **ข้อความในใบสั่งงานที่ว่า "ยังไม่ได้แก้บน production" ไม่ตรงกับของจริง** — แก้และ deploy ไปแล้ว 20 ก.ค.

### 5.2 ที่ **ยังไม่แก้** และยังเสียหายได้อยู่ 🟡

**ป้ายตรวจทาน (`review_flags`) หายทุกครั้งที่เผยแพร่ผ่านร่าง** — ยังจริงอยู่:
`song_drafts` ไม่มีคอลัมน์ `review_flags` → เปิดร่างขึ้นมา ระบบตั้งป้ายเป็นลิสต์ว่าง (`EditorMode.vue:1316`)
→ กดเผยแพร่ เขียนลิสต์ว่างทับของจริง (ทั้ง `saveDirect()` และ RPC `approve_and_publish` ที่ `coalesce` ไม่ช่วย เพราะ `[]` ไม่ใช่ null)

**ความเร่งด่วน: ต่ำ** — ป้ายตรวจทานเป็นตัวช่วยงาน ไม่ใช่เนื้อเพลง สร้างใหม่ได้ด้วยการ lint ซ้ำ · และวัดแล้ว **ยังไม่มีเพลงไหนโดนจริง** (22/22 ครบ)
