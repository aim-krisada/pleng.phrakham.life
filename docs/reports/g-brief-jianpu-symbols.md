## 1) ข้อความเปิด — ก๊อปทั้งบล็อกนี้วางให้ G ได้เลย

```text
สวัสดีครับ ต่อจากที่เราคุยกันเรื่องโครงสร้างเพลง (สมองสากล + หน้ากาก 2 แบบ)
รอบนี้ผมอยากโฟกัสเรื่องเดียวคือ "ชุดสัญลักษณ์เจียนพู่ที่ระบบจะรองรับ"
ขอความเห็นเชิงแนวคิด ไม่ต้องเขียนโค้ด

ย้ำสิ่งที่เราตกลงกันไปแล้ว (ไม่ต้องรื้อ ไม่ต้องเสนอใหม่)
- core เก็บแบบมาตรฐานสากล แล้วค่อยสวมหน้ากากตอนแสดงผล (สากล / แบบหนังสือเพลงโบสถ์ไทย)
- โครงข้อมูลเป็น "ทำนอง (stanza) เขียนครั้งเดียว + ลำดับการร้อง (arrangement) ที่ใส่เนื้อทีละพยางค์"
  และมีตัวกางลำดับการร้องเป็นเส้นตรงแล้ว (รองรับ :|| และกล่อง 1./2. เล่นกระโดดถูกแล้ว)
- 1 พยางค์ = 1 โน้ตที่ตี · เอื้อน = slur (โน้ตที่ถูกโยงไม่กินพยางค์)
- ที่ยังไม่มี: D.C./D.S./Coda/Segno · โหมดแสดงผลแบบย่อ (compact) สากล · export MusicXML

บริบทผู้ใช้
- เว็บคลังเพลงนมัสการไทยฟรีสำหรับคริสตจักร ผู้ใช้คืออาสาสมัครในโบสถ์ ไม่ใช่นักดนตรีอาชีพ
- คนแก้เพลงหลักคือ "พี่เปา" ซึ่งคีย์เพลงจากหนังสือเพลงเล่มจริงวันละหลายเพลง
- ใช้ทั้งมือถือและคอมพิวเตอร์
- ⚠️ ทิศทางที่ผมเคาะแล้ว: การป้อนสัญลักษณ์จะใช้ "คีย์บอร์ด + ป๊อปอัปเล็กๆ ที่เด้งใกล้จุดที่กำลังแก้"
  ไม่ใช่แถบเครื่องมือยาวๆ ด้านล่างจอแบบ Domisol หรือ MuseScore เพราะผมมองว่ามือใช้ง่ายกว่า
  → กรุณาตอบภายในกรอบนี้ (ช่วยทำให้ป๊อปอัปดี ไม่ใช่เสนอให้เปลี่ยนไปใช้ toolbar)

syntax ที่ผู้ใช้พิมพ์ได้จริงวันนี้
  1-7 ระดับเสียง (movable do) · 0 ตัวหยุด
  .5 = ต่ำ 1 อ็อกเทฟ (จุดนำหน้า) · 5' = สูง 1 อ็อกเทฟ (apostrophe ตามหลัง)
  5_ = เขบ็ต 1 ชั้น · 5__ = เขบ็ต 2 ชั้น
  5. = จุดประ (x1.5) · 5.. = จุดประคู่ (x1.75)
  - = ยืดอีก 1 จังหวะ · #5 b5 n5 = ชาร์ป/แฟลต/เนเชอรัล
  ( ) = slur · ~ = tie · { } = สามพยางค์ · ^ = fermata
  เส้นกั้นห้อง/ซ้ำ/กล่อง 1.2. = เก็บเป็นโครงสร้างข้อมูล ไม่ใช่ตัวอักษรในสตริง
  ยังไม่มี: D.C./D.S./Fine/Coda/Segno · tuplet ที่ไม่ใช่ 3 · โน้ตประดับ · ความดัง (p/f/hairpin)
  · ตัวหยุดยาวหลายห้อง · glissando · หลายแนวเสียงในห้องเดียว · เปลี่ยนเลขจังหวะกลางเพลง

ปัญหาที่รู้ตัวแล้ว
- ตัวอักษร "." ถูกใช้ 3 ความหมาย: อ็อกเทฟบน / อ็อกเทฟล่าง / จุดประ
  (และมาตรฐานยังมี "จุด staccato" อีกอันที่หน้าตาชนกับจุดอ็อกเทฟ)
- ยังไม่ได้ตัดสินว่าจะเก็บข้อมูลเป็น "ตัวอักษรที่ผู้ใช้พิมพ์" หรือ "ความหมายที่แปลแล้ว"

สิ่งที่อยากได้จากคุณ
ตอบคำถามด้านล่าง และที่สำคัญคือ **ช่วยหาช่องโหว่ในสิ่งที่ผมยังไม่ได้คิด** ไม่ต้องเห็นด้วยกับผม
ถ้าอ้างระบบที่มีอยู่จริงได้ (Domisol, MuseScore plugin, jianpu-ly, LilyPond, ABC, MusicXML,
Digital Common Notation, not angka อินโดนีเซีย, ฟอนต์ Doulos SIL Cipher) ช่วยระบุชื่อด้วยครับ
```

## 2) คำถามถึง G (8 ข้อ — ก๊อปต่อท้ายได้เลย)

```text
1. "." ทับซ้อน 4 ความหมาย (อ็อกเทฟบน / อ็อกเทฟล่าง / จุดประ / staccato)
   ผมพบว่ามีระบบที่เลี่ยงด้วยวิธีต่างกัน เช่น jianpu-ly ใช้ 1' กับ 1,
   Digital Common Notation ใช้ขีดตั้ง | ซ้าย/ขวาแทนจุดบน/ล่าง
   Domisol ตัดปัญหาด้วยการให้อ็อกเทฟเป็น "ปุ่มคนละแถว" ไม่ให้พิมพ์จุดเลย
   ในกรอบ "คีย์บอร์ด + ป๊อปอัปติดเคอร์เซอร์" ของผม ควรแก้ยังไง
   และถ้ายังให้พิมพ์ ASCII ได้ควรเลือกอักขระชุดไหน เพราะอะไร

2. ป๊อปอัปติดเคอร์เซอร์มีที่ว่างจำกัดมาก (โดยเฉพาะบนมือถือ)
   ควรจัดกลุ่มสัญลักษณ์ยังไงให้กดถูกตั้งแต่ครั้งแรก?
   ควรจัดตาม "ชนิดสัญลักษณ์" (ระดับเสียง/ค่าโน้ต/โครงสร้าง) หรือตาม "ความถี่ที่ใช้จริง"
   หรือตาม "สิ่งที่ใส่ได้กับสิ่งที่เลือกอยู่" (context-sensitive)?
   ถ้า context-sensitive จะเสี่ยงอะไรบ้างเมื่อผู้ใช้จำไม่ได้ว่าปุ่มไปอยู่ไหน

3. สัญลักษณ์บางตัวเกาะกับ "โน้ต 1 ตัว" (จุดประ, ชาร์ป, staccato)
   บางตัวเกาะกับ "ช่วง" (slur, hairpin, tuplet — ต้องเลือก 2 จุด)
   บางตัวเกาะกับ "ห้อง" (ซ้ำ, กล่อง 1./2., เปลี่ยนเลขจังหวะ)
   และบางตัวเป็น "คำสั่งกระโดด" ที่เปลี่ยนลำดับการเล่นทั้งเพลง (D.C./D.S./Fine/Coda/Segno)
   4 ระดับนี้ควรใช้วิธีป้อนแบบเดียวกันหรือคนละแบบ?
   Domisol แยก 2 กลุ่มชัดเจนว่าเป็น "jump target" กับ "jump command" — คุณเห็นด้วยไหม
   และมีระดับที่ 5 ที่ผมมองข้ามหรือเปล่า

4. ควรเก็บข้อมูลเป็น "ตัวอักษรที่ผู้ใช้พิมพ์" (surface string)
   หรือ "ความหมายที่แปลแล้ว" (semantic: pitch/duration/tie/…) หรือทั้งคู่?
   ชั่งน้ำหนักกับสิ่งที่เราจะต้องทำแน่ๆ คือ export MusicXML, เล่นเสียง, ย้ายคีย์,
   และผู้ใช้กลับมาแก้เพลงเดิมซ้ำหลายรอบ
   ระบบจริงระบบไหนเลือกทางไหน แล้วเจ็บตรงไหนเมื่อผ่านไปหลายปี

5. use case ที่เราระบุไว้แล้วแต่ยังไม่ทำ 4 ข้อ (วนลูปเก็บข้อแล้วเข้ารับร่วม /
   เปลี่ยนเลขจังหวะกลางเพลง / ท่อนรับสลับแถวเนื้อตามเลขข้อ / ลูกคู่ร้องเหลื่อมเวลา)
   — 4 ข้อนี้ต้องใช้ "สัญลักษณ์" อะไรเพิ่มบ้างที่มาตรฐานสากลมีอยู่แล้ว
   และข้อไหนที่มาตรฐานสากล **ไม่มี** สัญลักษณ์ให้ ต้องประดิษฐ์เอง?
   (ผมสงสัยเรื่อง "ท่อนรับสลับแถวตามเลขข้อ" มากที่สุด เพราะดูเหมือนเป็นธรรมเนียมการจัดหน้า
   ของหนังสือเพลงไทย/จีน มากกว่าจะเป็นสัญลักษณ์ที่มีในมาตรฐาน)

6. ผมค้นไม่พบว่ามีมาตรฐานทางการฉบับไหนกำหนดกฎ "การวางเนื้อร้องใต้โน้ตเจียนพู่"
   (แหล่งที่มีอธิบายแต่ตัวโน้ต ไม่อธิบายเนื้อร้อง)
   ถ้าไม่มีมาตรฐานจริง เราควรยึดอะไรแทน — กฎ lyric ของ MusicXML (syllabic
   single/begin/middle/end + extend) หรือธรรมเนียมหนังสือเพลงไทย?
   และกฎ "1 พยางค์ = 1 attack note" ของเราจะพังในเคสไหนที่ผมนึกไม่ถึง
   (เช่น โน้ตซ้ำเสียงเดียวกันใต้ slur, ตัวหยุดกลางคำ, ห้องนำ (pickup), เนื้อ 2 ภาษาซ้อน)

7. ถ้าเราจะรองรับ "ครบมาตรฐาน" จริงๆ รายการสัญลักษณ์จะยาวมาก
   (โน้ตประดับ, glissando, tremolo, นิ้ว, เทคนิคเครื่องดนตรีจีน, ตัวหยุดหลายห้อง,
   คอร์ดซ้อนแนวตั้ง, หลายแนวเสียง, เลขจังหวะหลายค่าพร้อมกัน)
   ช่วยจัดลำดับให้หน่อยว่าอะไรคือ "ต้องมีตั้งแต่วันแรก" / "ค่อยมาทีหลังได้" / "ไม่ต้องมีเลย"
   สำหรับ **หนังสือเพลงนมัสการโปรเตสแตนต์ไทย** โดยเฉพาะ ไม่ใช่ดนตรีจีนดั้งเดิม
   ใช้เกณฑ์อะไรตัดสิน

8. อะไรคือสิ่งที่เรา "ยังไม่รู้ว่าเราไม่รู้"
   — สัญลักษณ์ แนวคิด หรือกับดักในการออกแบบวิธีป้อนโน้ต
   ที่ไม่ได้ถูกพูดถึงเลยในบริบทข้างบน แต่จะกลับมาเป็นปัญหาแน่ๆ
   เมื่อคลังโตถึงหลักพันเพลงและมีคนแก้หลายคนพร้อมกัน
```

## 3) สิ่งที่อยากให้ G ค้าน (ก๊อปต่อท้าย)

```text
ขอให้ทำตัวเป็นผู้ค้าน ไม่ใช่ผู้เห็นด้วย:
- ชี้ว่า syntax ปัจจุบันข้อไหน "จะพังแน่" เมื่อเจอเพลงจริงที่ซับซ้อนขึ้น
- ถ้าคุณคิดว่า "คีย์บอร์ด + ป๊อปอัป" มีจุดตายที่ผมมองไม่เห็น บอกมาตรงๆ
  (ผมยังยืนยันทิศทางนี้ แต่ต้องรู้ว่าจะเจ็บตรงไหนเพื่อกันไว้ก่อน)

- ถ้ามีทางที่ดีกว่าแต่แพงกว่า บอกว่าแพงตรงไหนและคุ้มเมื่อไหร่
- ห้ามเกลี่ยให้ดูลงตัว ถ้าแหล่งอ้างอิงขัดกันให้บอกว่าขัดตรงไหน
- ตอบเป็นข้อ สั้น ตรง
```

3. **ธรรมเนียมหนังสือเพลงคริสเตียนไทย** — ไม่มีเอกสารอ้างอิง (มีแต่ตัวหนังสือเพลงจริง)
4. **ระบบพิมพ์เจียนพู่เป็นข้อความ (ASCII)** — คลังไม่มี · ผมค้นเพิ่มได้ = `jianpu-ly` (ดู A4) ซึ่งเทียบกับ syntax ของเราได้ตรงที่สุด
5. **แผนที่ MusicXML ↔ เจียนพู่** — ไม่มีทั้งคลังและในเว็บ (MusicXML ไม่มีที่ทางสำหรับเจียนพู่โดยตรง ต้องเก็บเป็นโน้ตสากลแล้วให้ปลายทาง render เอง)

## ภาคผนวก A2 — ตารางสัญลักษณ์มาตรฐาน (จากคลัง + ค้นเพิ่ม)

**ตัวหนา = ข้อที่ระบบเรายังไม่มี**

| กลุ่ม | สัญลักษณ์ | ความหมาย |
|---|---|---|
| ระดับเสียง | `1`–`7` | ขั้นคู่เสียงบันไดเสียงเมเจอร์ (movable do) |
| ระดับเสียง | `0` | ตัวหยุด · **ธรรมเนียมคือเขียน `0` ซ้ำ ไม่ใช่ลากขีด** (`0 0 0 0` = 1 ห้องใน 4/4) |
| ระดับเสียง | จุดบน/ล่าง | ขึ้น/ลงอ็อกเทฟ · หลายจุดซ้อนแนวตั้ง · **ถ้ามีขีดค่าโน้ตใต้เลข จุดล่างต้องเลื่อนลงไปใต้ขีด** |
| ระดับเสียง | `#` `b` `♮` | ผูกกับ**ขั้นคู่เสียง ไม่ใช่ชื่อโน้ต** (เช่น leading note ของ C minor เขียน `#5` แม้จริงๆ คือ B natural) |
| ระดับเสียง | `/` `\` (not angka อินโดฯ) | ขึ้น/ลงครึ่งเสียง — พิมพ์ทับบนตัวเลข |
| ระดับเสียง | `X` / `x` | จังหวะไม่มีระดับเสียง (เครื่องกระทบ) |
| ค่าโน้ต | ตัวเลขเปล่า | ตัวดำ 1 จังหวะ |
| ค่าโน้ต | ขีดใต้ (减时线) | 1 ขีด = ครึ่ง · 2 ขีด = 1/4 · **ขีดที่ต่อกันเชื่อมเป็นคานเดียวเหมือน beam** |
| ค่าโน้ต | `-` (增时线) | ยืดอีก 1 จังหวะต่อขีด |
| ค่าโน้ต | จุดหลังตัวเลข | จุดประ ×1.5 · 2 จุด ×1.75 |
| ค่าโน้ต | **tuplet** | เส้นโค้งคร่อม + เลขเล็ก (ไม่จำกัดแค่ 3) |
| โครงสร้าง | `|` `‖` `‖:` `:‖` `1.` `2.` | เหมือนสากล · **เลขในกล่องเขียนเล็กกว่าเลขโน้ตไม่ถึงครึ่ง** |
| โครงสร้าง | **`)0(`** | ตัวหยุดทั้งห้อง (สัญลักษณ์ทั่วไป) · ตัวหยุดหลายห้องใช้แบบสากลได้ |
| โครงสร้าง | **เส้นกั้นห้องแบบประ / ไม่มีเส้นเลย** | ท่อน cadenza |
| โครงสร้าง | **fermata เหนือเส้นกั้นห้อง** | พักสั้นๆ ระหว่างห้อง (คนละอันกับ fermata บนโน้ต) |
| โครงสร้าง | **D.C./D.S./Fine/Coda/Segno** | ใช้ชุดสากล (เจียนพู่ไม่มีของตัวเอง) |
| การเชื่อม | tie (延音线) / slur (圆滑线) | **เขียนเหนือแถวโน้ตเสมอ** (บางสำนักเขียนใต้ได้) |
| หัวเพลง | `1=C` · ไมเนอร์ `6=A` | movable do · Dorian ในทางปฏิบัติก็เขียน `1=C` |
| หัวเพลง | `4/4` | เศษส่วนแนวนอน/แนวตั้ง · **เปลี่ยนกลางเพลงได้ (เขียนในบรรทัดหรือเหนือบรรทัด)** · **บางเพลงระบุหลายค่าพร้อมกัน `4/4 2/4 3/4 5/4` = ห้องยาวไม่เท่ากัน** |
| หัวเพลง | metronome mark | **ที่เดียวในเจียนพู่ที่ใช้รูปตัวโน้ตแบบสากล** |
| อื่นๆ | **โน้ตประดับ (grace)** | ตัวเลขครึ่งขนาด บรรทัดเหนือโน้ตหลัก + เส้นโยงลงมา · ใส่ก่อนหรือหลังโน้ตหลักก็ได้ |
| อื่นๆ | expression / fermata | เขียน**เหนือ**แถวโน้ต |
| อื่นๆ | **dynamics (p f mf) + hairpin** | เขียน**ใต้**แถวโน้ต · ฟอนต์บางกว่าตัวเลข |
| อื่นๆ | ⚠️ **จุด staccato** | **หน้าตาชนกับจุดอ็อกเทฟ** → มาตรฐานแก้ด้วยการทำจุดหนาขึ้น+ห่างจากบรรทัดมากขึ้น หรือใช้สามเหลี่ยมคว่ำ (staccatissimo) แทน |
| อื่นๆ | **glissando / tremolo / portamento / นิ้ว** | มีสัญลักษณ์เฉพาะ (เส้นหยักทแยง, 3 ขีดทแยง, ฯลฯ) |
| อื่นๆ | คอร์ด | ซ้อนแนวตั้ง ต่ำสุดอยู่ล่าง · **แต่ละตัวมีจุดอ็อกเทฟของตัวเอง ขีดค่าโน้ตมีเฉพาะตัวล่างสุด** · ใช้ชื่อคอร์ด (Cm) แทนได้ |
| อื่นๆ | **หลายแนว/2 มือ** | เส้นกั้นห้องลากผ่านทุกแนว แต่**ไม่ตัดผ่านเนื้อร้อง** · เครื่อง 2 มือใช้ bracket ผูก |
| อื่นๆ | ลำดับการเรียงเมื่อชนกัน | นิ้ว > slur/tie > tuplet/roll (มีกฎว่าอะไรอยู่บนอะไร) |

**ของแถมจากคลังที่มีค่ากับเรามาก:** Wikipedia บันทึกว่ามีธรรมเนียม **พิมพ์เจียนพู่ด้วยแป้น ASCII** อยู่แล้วในโลกอินเทอร์เน็ต (ตัวอย่างในบทความใช้ `/` เป็นเส้นกั้นห้อง, `'` เป็นอ็อกเทฟบน, `( )` เป็นกลุ่ม, `!` ท้ายเพลง) → syntax ของเราไม่ได้แปลกแยก · และมีฟอนต์ OFL เฉพาะทาง **Doulos SIL Cipher** (ขาด slur ใน OpenType) กับ **x-Vacle** (ไม่ต้องเพิ่มความสูงบรรทัด) ถ้าจะทำหน้าพิมพ์ให้สวย

## ภาคผนวก A3 — Domisol ตัดสินใจอะไรไว้ (จากภาพหน้าจอในคลัง)

ใช้เป็น "ของเทียบ" ไม่ใช่ของลอก — เพราะ P'Aim เคาะว่าเราไปทาง popup ติดเคอร์เซอร์

- **แบ่งแป้นเป็น 3 แท็บ:** `123` (โน้ต) · `tr` (เครื่องหมายบนโน้ต) · `f` (ความดัง + การกระโดด)
- **แท็บ `123`:** 4 แถว = อ็อกเทฟบน (1̇–7̇) / กลาง (1–7) / ล่าง (1̣–7̣) / ค่าโน้ต (กลม ขาว ดำ เขบ็ต1 เขบ็ต2) + `0` + `—` + `.` + `# b ♮`
  → **อ็อกเทฟไม่ใช่การพิมพ์จุด แต่เป็นปุ่มคนละแถว** = ตัดปัญหา "." ทับซ้อนทิ้งทั้งหมด
- **แท็บ `tr`:** NOTE (tie, staccato, ฯลฯ) · GRACE (โน้ตประดับ 1-7) · PHRASE (slur / tuplet / Chord / gliss) · FINGER (大食中名小) · TECHNIQUE (颤上下刮揉摇) · BAR (`‖:` `:‖` `1.` `2.` +Bar −Bar) · TIME SIGNATURE (2/4 3/4 4/4 6/8)
- **แท็บ `f`:** DYNAMICS (pp→ff) · HAIRPIN ("แตะโน้ตตัวที่ 2 เพื่อจบ") · **NAV SYMBOLS (jump targets): Segno, Coda** · **JUMPS (reroute playback): D.C. Fine, D.S. Coda, To Coda, Fine**
  → **แยกชัดระหว่าง "จุดหมาย" กับ "คำสั่งกระโดด"** — ตรงกับคำถามข้อ 3 ของเรา
- **หลักการเลือกเป้าหมาย:** "แตะโน้ตในโน้ตเพลงก่อน แล้วค่อยกดสัญลักษณ์" (สัญลักษณ์ผูกกับสิ่งที่เลือก)
- **การพิมพ์:** คลิกโน้ตแล้วพิมพ์ = แทนที่ · Enter = แทรกตัวใหม่ · Esc = ต่อท้าย
- **Song settings แยกจากแป้น:** ชื่อ/ผู้แต่ง/โหมด (Solfa หรือ Jianpu)/คีย์/เลขจังหวะ/tempo/**จำนวนแนวเสียง 1-4**/ห้องต่อบรรทัด
- สิ่งที่ Domisol **ไม่มี** ให้เห็นในภาพ: คอร์ดกีตาร์เหนือโน้ต (ของเรามี), เนื้อร้องหลายข้อซ้อนกัน, `(รับ)` แบบไทย

## ภาคผนวก A4 — jianpu-ly: ระบบ "พิมพ์เจียนพู่เป็นข้อความ" ที่ใกล้เราที่สุด

| เรื่อง | jianpu-ly | ของเรา |
|---|---|---|
| อ็อกเทฟ | `1,,` `1,` `1` `1'` `1''` + ทางลัด `8` `9` | `.5` / `5'` |
| ค่าโน้ต | `s1` (เขบ็ต2) `q1` (เขบ็ต1) `1` | `5__` `5_` `5` |
| จุดประ | `q1.` `1.` | `5.` (ชนกับจุดอ็อกเทฟ) |
| tuplet | `3[ q1 q1 q1 ]` (ระบุจำนวนได้) | `{ }` (สามพยางค์เท่านั้น) |
| tie / slur | `1 ~ 1` / `1 ( 2 )` | `~` / `( )` — ตรงกัน |
| ซ้ำ + ห้องจบ | `R{ … } A{ 2 | 3 }` | flag ของห้อง |
| นำทาง | `Fine` `DC` `DS` `ToCoda` เป็นคำในสตริง | ยังไม่มี |
| คีย์/จังหวะ/tempo | `1=Bb` `4/4` `4/4,8` (ห้องนำ) `4=85` | field แยก (ไม่มีห้องนำ) |
| เนื้อร้อง | `L: …` · `H:` สำหรับอักษรจีน | array พยางค์ผูกกับ attack note |
| **จุดที่ควรขโมยมา** | **ลำดับอักขระในโน้ตสลับได้** (`#1` = `1#`, `'1` = `1'`) = ผู้ใช้ไม่ต้องจำลำดับ | ของเราลำดับตายตัว |

## ภาคผนวก A5 — จุดที่แหล่งขัดกัน / ยังไม่ยืนยัน (ไม่เกลี่ย)

1. **ไม่มีมาตรฐานทางการ** — ไม่พบ GB/ISO ของเจียนพู่ · แหล่งทั้งหมดเป็น Wikipedia / คู่มือซอฟต์แวร์ / บล็อกสอนดนตรี → ถือเป็นธรรมเนียม ไม่ใช่สเปก
2. **หน้า TablEdit ที่ดึงมาได้ เนื้อหาเกือบเหมือน Wikipedia ทุกประโยค** → น่าจะเป็น mirror นับเป็น 1 แหล่ง ไม่ใช่ 2
3. **ตำแหน่ง tie/slur** — Wikipedia บอก "เหนือแถวโน้ตเสมอ" แต่ก็ระบุเองว่าบางสำนักเขียนใต้ได้ และบางสำนักเอาขีดค่าโน้ตไปไว้เหนือเลข → มี "สำเนียงท้องถิ่น" จริง
4. **การเขียนอ็อกเทฟ** — นอกจากจุด ยังมีแบบเส้นนอน (เขียนบน/บน-เส้น/ใต้เส้น) และแบบขีดตั้ง `|` ซ้าย/ขวา (Digital Common Notation) → ไม่มีทางเดียวที่ "ถูก"
5. **การวางเนื้อร้อง** — ไม่มีแหล่งใดกำหนดกฎ (Domisol ก็ไม่มีเอกสาร) = เราต้องเลือกเอง
6. **MusicXML** — ไม่พบว่ามีที่ทางสำหรับเจียนพู่โดยตรง · MuseScore รองรับผ่านปลั๊กอินชุมชนที่ยังขาด slur → export ต้องเป็นโน้ตสากล แล้วให้ปลายทาง render
7. **โน้ตไทยเดิม (ดนตรีไทย)** ใช้จุดบน/ล่างเหมือนกันแต่ใช้**ตัวอักษร**ไม่ใช่ตัวเลข → ธรรมเนียมไทยกับเจียนพู่ไม่ใช่อันเดียวกัน
