## 2. สิ่งที่แก้

### `src/lib/slurArcs.js` (ใหม่) — เรขาคณิตเส้นโค้งที่ใช้ร่วมกัน 2 หน้า
ย้ายมาจาก `SongSheet.vue` เพื่อไม่ให้โค้ดวาดโค้ง 2 ชุด drift ออกจากกัน ([[pleng-shared-core-shell-deep]])
- `buildArc(x1,x2,yTop,h)` — filled lens เหมือนเดิมเป๊ะ
- `planArcs(openRect, closeRect, originRect, rowRects)` — ตัดสิน **โค้งเดียว** (แถวเดียวกัน)
  vs **แบ่งครึ่งตามหลัก engraving** (ตกคนละแถว: open→ขอบขวาแถว, ขอบซ้ายแถวถัดไป→close)
- `makeHalfHider()` — bookkeeping ซ่อน/คืน NoteRow half-arc

### `src/components/SongSheet.vue` — แก้น้อยที่สุด (mechanical)
- `:5` import จาก `slurArcs.js` แทนนิยามในไฟล์
- `:185-189` ลบ `buildArc` + `hiddenHalves` ท้องถิ่น → ใช้ `makeHalfHider()`
- `:261-269` บล็อก slur ข้าม segment เรียก `planArcs()` แทน logic inline 18 บรรทัด
- **บล็อก tie (`:220-242`) ไม่แตะเลย**

### `src/components/EditorMode.vue` — overlay ระดับบรรทัด (ของใหม่)
- `:6-7` import `slurSpans` + `planArcs`/`makeHalfHider`
- `:2515-2620` `measureEdArcs()` — ไล่ `.ed-strip[data-li]` ทุกอัน:
  - **tie ข้ามห้อง:** เดินย้อนข้าม `.nt-ext` หา digit ต้นทาง (เหมือนแผ่นเพลง) → ถ้า `src.bi !== rec.bi` เท่านั้นจึงวาด
  - **slur ข้ามห้อง:** `slurSpans()` บนโน้ตทั้งบรรทัด → กรองเฉพาะคู่ที่ `o.bi !== c.bi`
  - ซ่อน stub เดิม **เฉพาะเมื่อวาดแทนได้จริง** (วัดไม่ได้ = คง fallback)
  - ซ่อน tie-start ที่ **โน้ตต้นทางตรงตัว** (ห้องเดียวอาจมีหลาย tie-start — แม่นกว่าที่แผ่นเพลงเดาจาก segment)
- `:2621-2650` trigger: `setTimeout` debounce (**ไม่ใช้ rAF** — บทเรียน B111/B114) + `document.fonts.ready` + ResizeObserver + `window.resize`
- `:2657` `watch(() => props.active)` — Studio ใช้ `v-show` เก็บ editor ไว้ mounted ตอนซ่อน (`Studio.vue:401`) `onMounted` จึงรันตอนกว้าง 0 · ต้อง re-measure ตอนกดเข้า "แก้ไข" **(จุดนี้พังจริงตอนทดสอบ ไม่ใช่แค่ env — แก้แล้ว)**
- template `:3178-3195` overlay SVG (`.ed-arc-layer`, `no-print`) + `:data-li`
- CSS `:4846-4855` `position:absolute` (ไม่เป็น flex item) · `pointer-events:none` · `z-index:2` · `fill: var(--note-blue)` เท่าแผ่นเพลง

## 5. ข้อที่ env ของ session นี้ขับไม่ได้ — ต้องให้คนตรวจต่อ

| ข้อ | สถานะ | เหตุผล |
|---|---|---|
| **A6 มือถือ 360/412** | ⚠️ **ยังไม่ได้ทดสอบจริง** | `resize_window` รายงาน success แต่ `innerWidth` ค้างที่ 3200 (outerWidth ขยับ แต่ viewport ไม่) — **emulate ความกว้างมือถือไม่ได้ในเครื่องนี้** |
| **A5 พิมพ์ (Ctrl+P)** | ⚠️ **ตรวจจาก DOM เท่านั้น** | overlay หน้าแก้ไขเป็น `.no-print` (`styles.css:796` → `display:none !important`) · overlay แผ่นเพลง (`.tie-overlay`) ไม่มี `no-print` จึงยังพิมพ์เหมือนเดิม · **แต่ตาม [[feedback_verify_print_from_pdf]] ต้องสั่งพิมพ์ PDF จริงถึงจะนับ** |

**สิ่งที่พอยืนยันได้แทน A6 (invariant ไม่ขึ้นกับความกว้าง)** — วัดทั้ง 4 บรรทัดของ #12:
- `overlayW === stripW` เป๊ะ (797.5 = 797.5) ทุกบรรทัด
- overlay อยู่ในกรอบ strip ทุกบรรทัด (`overlayFitsStrip: true`)
- ทุก path อยู่ในกรอบ overlay (`pathsInsideOverlay: true`)
- `pointer-events: none` (แตะโน้ต/คำร้องข้างใต้ได้ปกติ) · `document` ไม่มี horizontal overflow

overlay ถูกกำหนดขนาดจาก `.ed-strip` เอง จึง **กว้างเกิน strip ไม่ได้ตามโครงสร้าง** และที่มือถือทุกห้องจะตกคนละแถว = เข้าทาง "แบ่งครึ่ง" ที่พิสูจน์แล้ว — แต่ยังต้องมีคนเปิดมือถือจริงยืนยัน

**หมายเหตุวิธีทดสอบ:** เคส A1 สร้างโดย set `input.value` + dispatch `input` event (ทางเดียวกับที่ v-model รับคีย์จริง) แล้ววัด layout จริงที่ได้ออกมา — ไม่ได้แตะ pointer/hover จึงไม่กระทบข้อควรระวังเรื่อง synthetic pointer · **ไม่ได้บันทึกลงฐานข้อมูล** (แก้ใน state อย่างเดียว)
