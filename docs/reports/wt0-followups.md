  (รูปแบบหน้าพิมพ์จริง A4/footer = งาน WT-B/US-B02 · story นี้แค่ปุ่ม trigger)

## ผลทดสอบ
- **unit:** ✅ ผ่านทั้งหมด **16/16** (`npx vitest run`) — รวมชุดใหม่ 4 เคส:
  - "เปิดเพลง" เห็นครบทั้ง 3 โหมด
  - เลือกเพลง → นำทาง `/song/:id` เฉพาะตอนกดปุ่ม "เปิดเพลง" (ไม่ใช่ตอนแค่เลือก)
  - เปิดเพลงจาก "ดู" → คงอยู่ที่ "ดู" ไม่เด้งเข้า "แก้"
  - โหมดแผ่นมีปุ่มพิมพ์ + คลิกแล้วเรียก `window.print()`
- **build:** ✅ `npm run build` ผ่าน
- **ตรวจจริงบนเบราว์เซอร์ (port 5301):** ✅
  - ปุ่ม "เปิดเพลง" โผล่ครบทุกโหมด · ค้นหา "พระเจ้า" → เจอ "1. พระเจ้าเป็นความรัก"
  - อยู่โหมด "ดู" → เปิดเพลง → hash เป็น `#/song/…` · โหมดยังเป็น "ดู" · ชื่อเพลงขึ้นบนแถบ
  - โหมดแผ่น → ปุ่ม "พิมพ์" เรียก `window.print()` ได้ (verify ด้วยการ stub `window.print`)
- **วิธี tester ลอง:** เปิด <http://localhost:5301> → ที่แถบบนมีปุ่ม "เปิดเพลง"
  1. อยู่โหมด **ดู** กด "เปิดเพลง" → พิมพ์ค้นหา → เลือกเพลง → กด "เปิดเพลง" → เพลงเปลี่ยนโดย**ไม่ออกจากโหมดดู**
  2. สลับไป **แผ่น** → เห็นปุ่ม **พิมพ์** มุมขวา → กดแล้วขึ้นกล่องพิมพ์ของเบราว์เซอร์
  3. ลองซ้ำจากโหมด **แก้** ก็ยังกด "เปิดเพลง" ได้เหมือนกัน

## ข้อสังเกต / คำถามถึง SA
- **ซ้ำซ้อนเล็กน้อยในโหมดแก้:** โหมดแก้มีเมนู "เพลง → เลือกเพลงเพื่อแก้…" ของ `EditorMode`
  อยู่แล้ว ตอนนี้จะมี "เปิดเพลง" ของเชลล์ซ้อนอีกอันในโหมดแก้. ผม**ไม่ลบของ `EditorMode`**
  เพราะ `EditorMode.vue` เป็น collision file ของ **WT-D** (DS-D01/D02/D03/D04) — DS-05 จึงจำกัด
  ให้แตะแค่ `Studio.vue`. เสนอ: ให้ WT-D (หรือ follow-up ถัดไป) ตัด "เลือกเพลงเพื่อแก้…"
  ออกจากเมนู "เพลง" ของ editor ทีหลัง เพื่อไม่ให้ซ้ำ. ไม่กระทบการใช้งาน (สองทางโหลดเพลงได้ทั้งคู่).
- watcher เดิมบังคับ `mode='view'` ตอน id เปลี่ยน — ผมเอาออกเพื่อทำ AC "โหลดในโหมดปัจจุบัน".
  ผลพลอยได้: กด back/forward ระหว่างเพลงก็คงโหมดเดิมด้วย (พฤติกรรมนี้สมเหตุผลกับ US-05)
- ไม่มี bug แบบ "ได้ยิน" ในงานนี้ (ไม่แตะเสียง/playback)
