## 2. พิสูจน์: PDF ตรงกับโมเดลเป๊ะทุกเส้น

รันโมเดลจริง (`resolveContent` + `beamGroups` ด้วย `syllables` จริงของเพลง 101) นับว่า**ควรมี**คานกี่แท่ง
แล้วเทียบกับ**ที่มีจริงในไฟล์ PDF**:

| | โมเดลบอกว่าควรมี | มีจริงในไฟล์ PDF |
|---|---|---|
| คาน (runs) | **8** | **8** |
| แท่งคานที่วาด (bars) | **16** | **16** |

> ตัวเลข 16 ตรงกับที่ PM วัดบนจอ ("beam element 24 · แสดงจริง 16") พอดี
> (24 คือทั้งหน้ารวม **ดู/ฝึกร้อง** ที่ `v-show` ค้างไว้ด้วย · ที่เป็นของ **แผ่นเพลง** จริงๆ = 16)

### ตรวจตำแหน่งทีละเส้น — ตรงกับตัวเลขข้างล่างทุกเส้น

| หน้า | คาน (x0–x1, y, กว้าง) | ตัวเลขที่อยู่ใต้ |
|---|---|---|
| 1 | 207.00–217.12 · y209.25 · 10.12pt | `1` `7` |
| 1 | 207.00–217.12 · y210.00 · 10.12pt | `1` `7` |
| 1 | 271.12–283.50 · y209.25 · 12.38pt | `3` `•` `4` |
| 1 | **277.50–283.50 · y210.00 · 6.00pt** | `•` `4` |

- คู่แรก (`1 7` = 2 ชั้นทั้งคู่) → **L1 กับ L2 ซ้อนกันเป๊ะ** ยาวเท่ากัน = คานคู่ยาวคลุมทั้งคู่ ✅
- คู่ที่สอง (`3.` 1 ชั้น + `4` 2 ชั้น) → **L1 ยาว 12.38 คลุมทั้งสองตัว · L2 ยาวแค่ 6.00 ชิดขวา (จบที่ 283.50 เท่ากัน)** = **ขีดสั้น (partial beam) ของ B111 พิมพ์ออกมาถูกต้อง** ✅
- ผลบนกระดาษจริง: ตัว `3•` ได้ **1 เส้น** · ตัว `4` ได้ **2 เส้น** — **ตรงตามที่พี่เปาขอเป๊ะ**

รูปแบบนี้ซ้ำเหมือนกันทั้ง 4 ชุด (หน้า 1 สามชุด · หน้า 2 หนึ่งชุด)

### `beforeprint` ทำงานจริงไหม → **ทำงาน**
ความกว้างคานบนกระดาษ = **10.12 / 12.38pt** (≈13.5 / 16.5px) แต่ตอนวัดบนจอที่ scale 1× ได้ **33.58px**
ถ้า `applyAllBeams` **ไม่ได้** วัดใหม่ตอนพิมพ์ คานจะยาว 33.58px แล้วเลยตัวเลขไปไกล
แต่ทุกเส้นวางตรงกับขีดใต้ตัวเลขคลาดไม่เกิน ~0.1pt → **`beforeprint` วัดใหม่ตาม layout ของกระดาษจริง สำเร็จ**

### ความหนาเส้นไม่ใช่ปัญหา
| ชนิด | จำนวน | ความหนา |
|---|---|---|
| ขีดใต้ตัวเลข ราย-ตัว | 316 | 0.38pt ทุกเส้น |
| คานคลุม | 16 | 0.38pt (8) · 0.75pt (8) |

คาน **หนาเท่าหรือหนากว่า** ขีดราย-ตัว → ไม่ได้จางกว่ากัน

## 3. แล้ว "97 คู่ที่ไม่มีคานเชื่อม" คืออะไร

มันคือ **คู่ที่โมเดลตั้งใจไม่มัดตั้งแต่แรก** ไม่ใช่คานที่หายไป

รันโมเดลนับคู่ตัวเลขที่มีขีดใต้ติดกัน แล้วดูว่าอยู่คานเดียวกันไหม (เพลง 101 ทุกบรรทัด):

```
คู่ที่ติดกันทั้งหมด : 172
อยู่คานเดียวกัน    : 8
ไม่อยู่คานเดียวกัน : 164     ← ตรงกับที่ PM นับจากกระดาษ (133) ในสัดส่วนเดียวกัน
```

**ต้นเหตุ = `issue8` (มัดคานตามพยางค์)** ใน `src/lib/notation.js` บรรทัด 220-240:

> *คานจะขาดที่ขอบจังหวะ · ตัวหยุด · `-` · triplet **และ (issue8) ก่อนโน้ตทุกตัวที่ขึ้นคำร้องใหม่***

เพลงไทยส่วนใหญ่ **1 โน้ต = 1 พยางค์** → แทบทุกตัวขึ้นคำใหม่ → **คานขาดแทบทุกจุด**
ตัวอย่างจากเพลง 101 ที่ไม่ถูกมัด: `4_ 3_ 2_ 1_` · `.7_ 1_ 2_` · `3. 3_ 4_ 3_ 2_ 1_` — ล้วนเป็นชุดเขบ็ตที่หนังสือเพลงทั่วไป**มัดติดกันหมด**

### ผลกระทบทั้งคลัง (109 เพลง)

| กฎที่ใช้ | คาน (runs) | คู่ที่ถูกมัด | % |
|---|---|---|---|
| **ตอนนี้ — มัดตามพยางค์ (issue8)** | 516 | 518 / 4,763 | **10.9%** |
| ถ้ามัดตามจังหวะล้วน | 3,238 | 3,272 / 4,763 | **68.7%** |

→ ถ้าเปลี่ยนเป็นมัดตามจังหวะ **คานจะเพิ่มขึ้น ~6.3 เท่า** ทั้งคลัง

## 4. ของเดิมหรือของใหม่ → **ของเดิม 100%**

รันโมเดลเดียวกันบน **`c7f60a3` = โค้ดที่ live อยู่ตอนนี้**:

```
c7f60a3 (LIVE) : runs 8 · barElements 8 · pairs 172 · beamed 8 · notBeamed 164
B111 (ของเรา)  : runs 8 · barElements 16 · pairs 172 · beamed 8 · notBeamed 164
```

- **จำนวนคานเท่ากันเป๊ะ · คู่ที่ไม่ถูกมัดเท่ากันเป๊ะ**
- B111 ทำแค่เปลี่ยน 8 แท่ง → 16 แท่ง (เพิ่มเส้นชั้นที่ 2 ให้ถูก) — **ไม่ได้เอาคานออกแม้แต่เส้นเดียว**

> 🔴 **แปลว่า live ตอนนี้พิมพ์ออกมาก็คานน้อยแบบนี้อยู่แล้ว** ตั้งแต่ issue8 ขึ้นระบบ
> เรื่องนี้**ใหญ่กว่างาน B111** และไม่เกี่ยวกับ B111 เลย

## 5. ความเห็นตรงๆ: overlay ที่วัดพิกัด เหมาะกับการพิมพ์ไหม

**เหมาะ · และมันทำงานถูกอยู่แล้ว — ไม่ต้องรื้อ**
หลักฐาน: โมเดลบอก 16 แท่ง → กระดาษได้ 16 แท่ง วางตรงทุกเส้น คลาด ≤0.1pt รวมขีดสั้นของ B111
ถ้าจะรื้อวิธีวาด ต้องมีเหตุผลอื่น (เช่น performance) **ไม่ใช่เพราะพิมพ์ผิด เพราะมันไม่ได้ผิด**

ข้อสังเกตเล็กน้อย (ไม่ใช่บั๊ก แต่ควรรู้): ระยะห่างระหว่างเส้นชั้น 1 กับชั้น 2 บนกระดาษวัดได้ **0.74-1.12pt**
ออกแบบไว้ 2.5px (≈1.9pt) → บนกระดาษเส้นสองชั้นชิดกันกว่าที่ออกแบบเล็กน้อย
ยังแยกออกได้ (มีช่องว่างจริง) แต่ถ้าอยากให้ชัดขึ้นตอนพิมพ์ ปรับ `BEAM_GAP` แยกสำหรับ print ได้ — **งานเล็กมาก**

## 6. ทางเลือก + ขนาดงาน (ให้พี่เอมตัดสิน)

**คำถามจริงคือเรื่องกฎการมัดคาน ไม่ใช่เรื่องพิมพ์** — และ `issue8` เป็นกฎที่ **พี่เปาเคยขอเอง**
(มี golden test 10 ตัวใน `notation.beam.test.js` อ้าง "รูป1/รูป2" จากหนังสืออ้างอิง)
**ต้องถามพี่เปาก่อนว่าเปลี่ยนใจหรือเปล่า — ห้ามเดา**

| ทาง | ทำอะไร | ขนาด | ข้อดี | ข้อเสีย |
|---|---|---|---|---|
| **A. เล็ก** | ให้ **แผ่นเพลง** ส่ง `syllables=null` → มัดตามจังหวะเฉพาะตอนพิมพ์ | **~1-2 ชม.** (แก้ 1 บรรทัดใน `SongSheet.vue` + เทสต์) | ได้คานกลับมา 6.3 เท่าทันที · ไม่แตะกฎเดิม | **จอกับกระดาษไม่ตรงกัน** — ผมไม่แนะนำ |
| **B. เล็ก-กลาง** | เปลี่ยนกฎเป็น "มัดตามจังหวะ" ทั้งระบบ | **~ครึ่งวัน** (แก้ `beamGroups` + เขียน golden test issue8 ใหม่ 10 ตัว + กวาดคลังดูของเสีย) | ตรงกับหนังสือเพลงสากล · จอ=กระดาษ | **ล้มการตัดสินใจเดิมของพี่เปา** ต้องได้ไฟเขียวก่อน |
| **C. กลาง** | ทำเป็นตัวเลือก "มัดตามจังหวะ / มัดตามคำร้อง" ให้ผู้ใช้สลับ | **~1-2 วัน** (กฎ + UI + persist + คู่มือ + เทสต์) | ไม่ต้องเลือกข้าง · รองรับทั้งสองความเห็น | เพิ่มตัวเลือกให้ผู้ใช้คิด (ขัด KISS) |
| **D. รื้อวิธีวาดคาน** | overlay → CSS/SVG ใน flow | **~3-5 วัน** | ไม่ต้องวัดพิกัด | **ไม่จำเป็น — ข้อมูลพิสูจน์แล้วว่าพิมพ์ถูก** ผมไม่แนะนำ |

### ผมเสนอ
1. **ก่อนอื่น: บอกพี่เอมว่าไม่ใช่บั๊กการพิมพ์ และ live วันนี้ก็เป็นแบบนี้** — B111 ยังปล่อยได้ตามปกติ
2. เอาตัวเลข 10.9% vs 68.7% ไปให้ **พี่เปา** ดู แล้วถามคำถามเดียว: *"ชุดเขบ็ตอย่าง `4 3 2 1` ที่แต่ละตัวมีคำของตัวเอง อยากให้มัดติดกันไหม"*
3. ได้คำตอบแล้วค่อยเลือก **B** หรือ **C** (ไม่เอา A เพราะจอ≠กระดาษ · ไม่เอา D เพราะไม่จำเป็น)
