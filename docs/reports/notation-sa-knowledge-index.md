## แหล่งที่อ่าน

### 1. `work/ปรับ pl edit ui/แปลงโน้ตเพลงเป็นอัลกอริทึม.md` (บทสนทนา P'Aim + พี่เปา + Gemini)
บันทึกการซักถามจากแผ่นเพลงจริง (เพลง 773, 105, 68 ฯลฯ) จนตกผลึกเป็นข้อสรุปเชิงมาตรฐาน:

- **เครื่องหมายควบคุมการไหล = สากล 100%** — repeat `‖: :‖`, volta กล่อง `1.` `2.`
  (First/Second Endings) · เจียนพู่ยืมมาจากโน้ต 5 เส้นทั้งชุด ใช้กฎเดียวกัน
- **การจัดหน้าแบบหนังสือเพลงโบสถ์ = ไม่สากล** — คำว่า `(รับ)` · เนื้อข้อ 2-4 แยกไปอีกหน้า ·
  เนื้อรับพิเศษของข้อ 4 ห้อยท้าย = การประหยัดกระดาษ ไม่ใช่มาตรฐาน
- **`:‖` ไม่ได้แปลว่า "รับ"** (พี่เปาชี้) — ต้องมีคำว่า "รับ" กำกับถึงจะเป็นท่อนรับร่วมของทุกข้อ ·
  ถ้าไม่มี = ดนตรีวนจริงแต่เนื้อเป็นของใครของมัน (เพลง 105)
- **สถาปัตยกรรมที่เคาะ:** input ได้ 2 แบบ (สากล / แบบโบสถ์) → core เก็บมาตรฐานเดียว
  (คลี่เป็นเส้นตรง 1:1 · MusicXML-compatible) → display ได้ 2 แบบ
- **4 use case ที่ระบบต้องรองรับ** (เอกสารแนบท้าย · จากคลังเพลงจริง)
  1. วนท่อนหลักเก็บเนื้อทีละข้อ แล้วปล่อยผ่านเข้าท่อนรับร่วม
  2. เปลี่ยน time signature กลางเพลง (เพลง 515, 433) → ต้องเก็บรายห้อง ไม่ใช่ global
  3. ท่อนรับสลับแถวเนื้อตามเลขข้อ (เพลง 669, 432 · ข้อ 1-4 แถวบน / ข้อ 5 แถวล่าง)
  4. multi-voice ลูกคู่ร้องเหลื่อมเวลาในห้องเดียวกัน (เพลง 6, 17, 19)
- **Pain point ฝั่งพี่เปา** (Gemini สรุปจากบทสนทนา — ยังไม่ verify กับของจริง):
  พิมพ์ในกล่องข้อความยาวแล้วเบี้ยว · ไม่ WYSIWYG · แก้/ลบรายโน้ต-รายห้องแล้วเนื้อหลุดตำแหน่ง

### 4. `OneDrive/.../pleng.phrakham.life/jianpu/` (คลังความรู้ที่ P'Aim รวบรวม)
- **Wikipedia "Numbered musical notation"** = มาตรฐานอ้างอิงตัวจริง: จุดบน/ล่าง = อ็อกเทฟ
  (ซ้อนกันได้หลายจุด · **จุดล่างวางใต้เส้นใต้จังหวะ**) · เส้นใต้ 1 เส้น = ½ · `-` ต่อท้าย = +1 จังหวะ ·
  **จุดหลังโน้ต = ประจุด (×1.5) · 2 จุด = ×1.75** · `0` = หยุด (ยาวขึ้นใช้ `0` ซ้ำ ไม่ใช่ `-`) ·
  accidental วาง**หน้า**ตัวเลข (`#4`) · movable-do (`1=C`) · time signature เขียนเป็นเศษส่วน ·
  ⚠️ **จุด staccato หน้าตาชนกับจุดอ็อกเทฟ** — มาตรฐานแก้ด้วยจุดหนากว่า/ห่างกว่า หรือใช้สามเหลี่ยมคว่ำ ·
  bar line / repeat / 1st-2nd ending "look very similar to" ของโน้ต 5 เส้น (เลขกล่องตัวเล็กกว่าเลขโน้ต) ·
  มี ASCII input convention ที่ใช้กันบนเว็บ (`'` = อ็อกเทฟบน · `( )` = slur) — ตรงกับ syntax ของ pleng
- **Domisol (โปรแกรมเจียนพู่จริง · 5 ภาพ)** = reference UI ที่ใกล้โจทย์เราที่สุด:
  แถบล่างแบ่ง 3 แท็บ `123` / `tr` / `f` — `123` = แป้นตัวเลข 3 แถว (อ็อกเทฟบน / กลาง / ล่าง)
  + `—` + `.` + แถวค่าตัวโน้ต + `♯ ♭ ♮` · `tr` = NOTE (tie/staccato/…) · GRACE · PHRASE (slur/tuplet/
  chord/gliss) · FINGER · TECHNIQUE · **BAR (`‖:` `:‖` `1.` `2.` `+Bar` `−Bar`)** · TIME SIGNATURE ·
  `f` = DYNAMICS · HAIRPIN · **NAV SYMBOLS (Segno, Coda)** · **JUMPS (D.C. Fine · D.S. Coda ·
  To Coda · Fine)** · โมเดล interaction = "คลิกโน้ตแล้วพิมพ์ทับ · Enter แทรกตัวใหม่ · Esc ต่อท้าย"
  → **สำคัญ: P'Aim ปฏิเสธ "แถบเครื่องมือด้านล่าง" แบบนี้ · เอาคีย์บอร์ด + ป๊อปอัปใกล้เคอร์เซอร์แทน ·
  แต่ *รายการสัญลักษณ์* ของ Domisol ใช้เป็น checklist ความครบได้**
- MuseScore jianpu plugin (`.qml` 5 เวอร์ชัน + EasyJianpu) = วิธี render เจียนพู่ทับ MuseScore
  (ต้องใช้ฟอนต์ `01SMN.ttf`) — เป็น render reference ไม่ใช่ input reference

### 5. `src/lib/notation.js` + `src/lib/songModel.js` (โค้ดจริง สาขา `studio-shell-redesign`)
- **`parseNotes()`** — lexer ระดับตัวอักษร เว้นวรรคได้/ไม่ได้ · syntax ที่รับจริงวันนี้:
  `~`(tie) `#|b|n` `.`นำหน้า(อ็อกเทฟล่าง ซ้อนได้) เลข0-7 `'`(อ็อกเทฟบน · รับ `‘ ’ ′` ของ iOS ด้วย)
  `_`/`__`(เส้นใต้ 1-2) `.`ท้าย(ประจุด 1-2 จุด) `~`(tie) `^`(fermata) · `-`/`–` = ต่อเสียง ·
  `0` = หยุด · `( )` = slur · `{ }` = triplet · `|` ถูกข้าม (bar เป็น item แยก ไม่ใช่ตัวอักษร)
  → **จุดสำคัญ: `.` มี 3 ความหมายจริงในโค้ด และตัด ambiguity ด้วยกฎตำแหน่ง**
  (`.` ก่อนเลข = อ็อกเทฟล่าง · `.` ท้าย token = ประจุด · `5..5` = ประจุด+อ็อกเทฟล่าง)
- **`noteBoxKinds()` / `syllableSlots()` / `attackSlots()`** — จำแนกกล่องโน้ตเป็น attack / held / struct
  = ฐานของการจับคู่พยางค์
- **`beamGroups()`** — เส้นใต้เชื่อมตาม **พยางค์** ไม่ใช่ตามจังหวะ (ตั้งใจต่างจาก Gould/Behind Bars
  เพราะ target = เพลงร้องไทย)
- **`slurSpans()` / `melismaSpans()` / `arcPlan()`** — slur ข้าม segment/บรรทัด + เอื้อนที่ derive
  จากช่องพยางค์ว่าง (ไม่ต้องมี `( )` ในข้อมูล)
- **`songModel.js`** — `migrateToV2()` · `resolveContent()` (กาง arrangement เป็นบรรทัด render +
  ติด `_stanza` `_entryIndex` `_stanzaLine` สำหรับ click-to-edit) · `melodyLineSignature()`
  (ตัด pickup bar ก่อนเทียบ) · **`resolvePlayOrder()` = seam ที่เขียนคอมเมนต์ไว้ว่าเป็นจุดเสียบ
  D.C./D.S./Coda เฟส 2** · ตอนนี้มีแค่ `resolveStrophicOrder()` (`afterEachVerse` = "ร้องรับทุกข้อ")
- `expandRepeats()` (ใน `midi.js`) = กาง repeat/volta ตอนเล่น · **จำกัด 2 รอบ · 3+ endings ยังไม่รองรับ**

## ขัดแย้ง / ต้องระวัง

1. **Gemini เสนอ "Measure-Based Grid + แถบ palette แบบ MuseScore" — P'Aim เคาะสวนทาง**
   (คีย์บอร์ด + ป๊อปอัปใกล้เคอร์เซอร์) · และเคาะ "หน้ากากอนุชนก่อน · ไม่ทำ 5 เส้น · ไม่ทำแนวเบส"
   ซึ่งตัด use case 4 (multi-voice) ออกจากสโคปรอบนี้ → **ยึดคำเคาะของ P'Aim**
2. **Gemini เสนอ "คลี่เป็นเส้นตรง ไม่มีการย้อนในชั้นข้อมูล"** แต่ **v2 เก็บ repeat/volta ไว้ในข้อมูล
   แล้วคลี่ตอน runtime** (`expandRepeats`) — ต่างกันจริง แต่ v2 ถูกกว่าสำหรับงานพิมพ์
   (เก็บสัญลักษณ์ = พิมพ์แบบหนังสือได้ · คลี่ตอนเล่น = เล่นถูก) · **ไม่ต้องแก้ตามคำแนะนำ Gemini**
3. **Wikipedia: รอบซ้ำ 3+ ครั้ง เขียน volta ได้ (`1. (repeat), 2, 3`)** — แต่โค้ดจำกัด 2 รอบ
   → use case "ข้อ 2-3 ไม่ย้อน" (เพลง 105) ยัง encode ไม่ได้ตรงๆ
4. **`docs/song-model-v2.md` มีทั้งดีไซน์และ handoff log ปนกัน** — บางบรรทัดคือแผน บางบรรทัดคือของจริง
   ต้องอ่านหัวข้อ "Hand-off status" ประกอบเสมอ
