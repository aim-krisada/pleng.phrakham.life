ตัวแปลง **ไม่ได้อยู่บน branch ที่ใช้งานปัจจุบัน** — ถูกลบไปแล้ว ต้องขุดจากประวัติ git:

```

tools/encode_ties.py      อยู่บน branch  da-ties   (คอมมิต 4185d3a)
tools/detect_arcs.py      อยู่บน branch  da-ties
```

หลักฐานที่ใช้ เรียงจากน้ำหนักมากไปน้อย:

1. **รันตัวแปลงจริงวันนี้** บนไฟล์ต้นฉบับ (PDF+DOCX) → output ลงโฟลเดอร์ชั่วคราวของผมเท่านั้น ✅ ไม่แตะ DB
2. **ผลดิบที่ commit ไว้ตอนนำเข้า** — `tools/import-all-120.sql` บน `da-import` = output ของตัวแปลงล้วนๆ 120 เพลง ที่ยังไม่มีมนุษย์แตะ = ตัวควบคุมที่สมบูรณ์แบบ
3. **ผลของตัวใส่เส้นโค้ง** — `tools/import-ties.sql` (อยู่บน branch ปัจจุบันแล้ว) 48 เพลง
4. **อ่านโค้ดตัวแปลงบรรทัดต่อบรรทัด** เทียบกับ spec ของ parser (`src/lib/notation.js`)
5. **รัน parser ทั้ง 2 เวอร์ชัน** (base กับ G1) ทับสตริงอาการทั้ง 7 จุด

สคริปต์ตรวจทั้งหมดอยู่ในโฟลเดอร์ชั่วคราวของ session นี้ (ไม่ commit — เป็นเครื่องมือใช้ครั้งเดียว)

## คำตอบ 6 ข้อ

### 1. ตัวแปลงสร้างลำดับ modifier ยังไง — มีจุดที่ประกอบสตริงผิดไหม

**มี 1 จุด และเป็นจุดเดียว** — `tools/parse_song.py:138` (branch `da-import`):

```python
tok = ('.' if oc == 'low' else '') + (pending) + t + ("'" if oc == 'high' else '') + '_' * u
#      ^^^^^^^^^^^^^^^^^^^^^^^^^     ^^^^^^^^^
#      จุดเสียงต่ำ มาก่อน            ชาร์ป/แฟลต มาทีหลัง   ← สลับที่กัน
```

เทียบกับ spec ที่ parser ต้องการ (`src/lib/notation.js:35-37`) — อ่าน accidental **ก่อน** แล้วค่อยอ่านจุดเสียงต่ำ:

```js
if (s[j] === '#' || s[j] === 'b' || s[j] === 'n') { accidental = s[j]; j++ }   // บรรทัด 35
let low = 0
while (s[j] === '.') { low++; j++ }                                            // บรรทัด 37
```

**ผลคือ:** โน้ตที่มีทั้งจุดเสียงต่ำ **และ** ชาร์ป/แฟลต จะออกมาเป็น `.#4` แทนที่จะเป็น `#.4` เสมอ — **ไม่ใช่บางครั้ง แต่ทุกครั้ง 100%** เพราะเป็นการต่อสตริงแบบตายตัว ไม่มีเงื่อนไข

ส่วนลำดับอื่นในบรรทัดเดียวกัน (`'` จุดบน แล้ว `_` ขีดเขบ็ต) **ถูกต้องตาม spec** และจุดเพิ่มค่า `.` ที่ต่อท้ายทีหลัง (บรรทัด 144) ก็ต่อท้ายสุด **ถูกต้อง** — จึงมีที่ผิดที่เดียวจริงๆ

**สำคัญ: ตัวแปลงไม่มีทางสร้างอาการ 1 และ 3 ได้เลย**

- **อาการ 1 (`^`)** — ค้นทั้งไฟล์ 579 บรรทัด **ไม่มีตัวอักษร `^` โผล่ที่ไหนเลย** และสแกนผลดิบ 120 เพลง เจอ `^` **0 ครั้ง** · `^` (เฟอร์มาตา) เป็นฟีเจอร์ที่ผู้ใช้พิมพ์เองในหน้าแก้ (`src/lib/notation.js:65` · `src/components/NoteRow.vue:276`)
- **อาการ 3 (วงเล็บ)** — ลูปอ่าน PDF (บรรทัด 133-149) รับเฉพาะ เลข `-` `.` เส้นกั้นห้อง และ `:` · **ตัวอักษร `(` `)` ถูกทิ้งทั้งหมด ไม่เคยถูกเขียนลง token** และสแกนผลดิบ 120 เพลง เจอวงเล็บ **0 ครั้ง**

### 2. ทำซ้ำได้ไหม — 🔑 หลักฐานชี้ขาด

**ได้ · ทำซ้ำแล้ววันนี้ ตรงกันทุกตัวอักษร**

รันตัวแปลงเวอร์ชันปัจจุบันบนไฟล์ต้นฉบับของเพลง #109 (output ลงโฟลเดอร์ชั่วคราว ไม่แตะ DB):

```
py -3.14 parse_song.py "109 เวลาก่อสร้างพระวิหาร ล232G397.pdf" \
                       "109 เวลาก่อสร้างพระวิหาร ล232G397.docx" -o rerun/109.json
```

**ผลรันดิบ วันนี้:**

```
RE-RUN TODAY, song 109, stanza A line 4/4
   bar 1 : .3_
   bar 2 : .2 .3_ .#4 .5_          ← ตรงกับใบแจ้งเป๊ะ
   bar 3 : .6_ 2 2 1_
   bar 4 : .7. .6.
   bar 5 : .5. .5
  bad-order tokens in this re-run: ['.#4']
```

เทียบกับใบแจ้งพี่เปา ข้อ 2.1: *"ห้องนั้นทั้งห้องตอนนี้ `.2 .3_ .#4 .5_`"* — **ตรงกันทุกตัวอักษร ทั้งห้อง**

และสแกนผลดิบทั้ง 120 เพลง — **โน้ตที่มีทั้งจุดเสียงต่ำและชาร์ป/แฟลตมี 15 ตัว และผิดลำดับทั้ง 15 ตัว (100%)**:

```
at-risk population (low-octave + accidental) = 15  in songs [49, 64, 73, 92, 109, 111]

  (49)  '.b7_'   line 2/5    | (73)  '.#4_'  line 5/5
  (64)  '.#4_' '.#4'  lines 1,3,7 /8   (6 ตัว)
  (92)  '.#5_'  lines 1,2,3,4,7 /13   (5 ตัว)
  (109) '.#4'   line 4/4     | (111) '.#5_'  line 2/3
```

**สรุปข้อ 2: อาการ 2 = ตัวแปลง 100% ไม่มีข้อสงสัย** — ทำซ้ำได้ทั้งจากผลดิบที่ commit ไว้ และจากการรันสดวันนี้

**⚠️ ข้อที่ใบแจ้งพี่เปายังไม่ได้บอก:** ใบแจ้งเจออาการนี้แค่ **1 จุด (เพลง #109)** แต่ตัวแปลงสร้างไว้ **15 จุด ใน 6 เพลง** — เพลง **49, 64, 73, 92, 111** อาจยังพังอยู่และยังไม่มีใครแจ้ง **หรืออาจถูกแก้ไปแล้ว** ผมไม่ได้เปิดฐานข้อมูลตรวจตามกติกา → **นี่คือรายการที่ควรให้ Tester ไล่เช็คบนเว็บจริง**

### 3. ทำไมกระจุกที่บรรทัดสุดท้ายของท่อน

**หักล้าง — ไม่มีตรรกะตัดท่อนที่ทำงานต่างออกไปตรงขอบ และข้อมูลก็ไม่ได้กระจุกจริง**

ดูตำแหน่งบรรทัดของทั้ง 15 จุดที่ตัวแปลงสร้าง:

| เพลง | บรรทัดที่เจอ | อยู่บรรทัดสุดท้ายไหม |
|---|---|---|
| 49 | 2 จาก 5 | ไม่ |
| 64 | 1, 3, 7 จาก 8 | ไม่ |
| 73 | 5 จาก 5 | ✅ ใช่ |
| 92 | 1, 2, 3, 4, 7 จาก 13 | ไม่ |
| 109 | 4 จาก 4 | ✅ ใช่ |
| 111 | 2 จาก 3 | ไม่ |

**2 จาก 6 เพลง** เท่านั้นที่อยู่บรรทัดสุดท้าย — ไม่ใช่รูปแบบ

ในโค้ดก็ยืนยันตรงกัน: การประกอบ token (บรรทัด 138) ทำงานที **ต่อ 1 ตัวอักษร** โดย**ไม่รู้เลย**ว่าตัวเองอยู่บรรทัดไหน ห้องไหน ใกล้จบท่อนหรือยัง — ไม่มีตัวแปรตำแหน่งเข้ามาเกี่ยวเลย จึงเป็นไปไม่ได้ทางโครงสร้างที่จะพังเฉพาะตรงขอบ

**คำอธิบายที่น่าจะจริงกว่า:** ที่ 3 ใน 5 เพลงในใบแจ้งบังเอิญอยู่บรรทัดสุดท้าย เป็นเรื่อง**ดนตรี ไม่ใช่โค้ด** — บรรทัดสุดท้ายของท่อนคือจุดจบวรรค ทำนองมักไหลลงไปโน้ตเสียงต่ำ (`.2 .3 .#4 .5`) ซึ่งเป็นบริเวณเดียวที่จุดเสียงต่ำจะไปเจอกับชาร์ปได้ · ที่สังเกตกันไว้ไม่ผิด แต่กลุ่มตัวอย่างแค่ 5 เพลงเล็กเกินกว่าจะสรุปเป็นรูปแบบ

### 4. วงเล็บไม่ครบคู่ เกิดจากการตัดท่อนแล้วเส้นโค้งถูกหั่นครึ่งหรือเปล่า

**หักล้าง — และเจอต้นตอจริงของเพลง #90 แล้ว**

ตัวใส่เส้นโค้ง (`encode_ties.py`) ออกแบบมาให้**ป้องกันเรื่องนี้ไว้แล้ว** — `apply_slur` (บรรทัด 135-145) จะใส่ `(` และ `)` **ในห้องเดียวกันเท่านั้น** ถ้าเส้นโค้งคร่อมข้ามห้อง มันจะ `return False` แล้วรายงานเป็น `cross-seg` ให้คนตรวจ **ไม่ใส่ครึ่งเดียว**

สแกนผลของมันทั้งหมดเพื่อยืนยัน:

```
songs in import-ties.sql: 48
lines checked: 309   unbalanced: 0   caret(^) tokens: 0
segments unbalanced: 0
```

**วงเล็บครบคู่ 100% ทั้ง 309 บรรทัด** — ทั้งระดับบรรทัดและระดับห้อง

**แล้ว `)` ของเพลง #90 มาจากไหน?** เทียบ 3 เวอร์ชันของบรรทัดเดียวกัน:

```
ตัวแปลง (import-all-120.sql):   1_ 1_ .6_ 1_ .7 .7 ...        ← ไม่มีวงเล็บเลย
ตัวใส่เส้นโค้ง (import-ties):   1_ ( 1_ .6_ 1_ ) .7 .7 ...    ← ครบคู่ ถูกต้อง
ที่พบในใบแจ้ง:                  1_ {1_ .6_ 1_})               ← `(`→`{` , เพิ่ม `}` , ลืมลบ `)`
```

**เห็นชัดว่าเกิดอะไรขึ้น:** มีคนแก้ทีหลัง เปลี่ยนเส้นโค้งเอื้อน `( )` เป็นโน้ตสามพยางค์ `{ }` — เปลี่ยน `(` เป็น `{` แล้วใส่ `}` เข้าไป **แต่ลืมลบ `)` ตัวเดิมทิ้ง** เหลือค้างอยู่

นี่คือ**การแก้ด้วยมือทีหลัง ไม่ใช่ตัวแปลง** — และคำแนะนำในใบแจ้งพี่เปา (ทาง ก: ลบ `)` ทิ้ง) **ถูกต้องแล้ว** หลักฐานชุดนี้ยืนยันให้ ไม่ต้องเปิดหนังสือแล้วสำหรับจุดนี้

ส่วนเพลง **"หากท่านรู้จักพระเยซู"** (ไม่มีเลขเพลง) — **ไม่ได้อยู่ในชุดนำเข้า 120 เพลงเลย** (ชุดนั้นเป็นเลข 1-120 ทั้งหมด และเพลงนี้ไม่มีเลข) จึงไม่เคยผ่านตัวแปลงหรือตัวใส่เส้นโค้งตัวนี้ · เช่นเดียวกับ **เพลง #760** — เลขเกินช่วง 1-120 ไปมาก ไม่ใช่ของชุดนี้

> **นี่คือเหตุผลที่อาการ 1 (`^`) ทั้ง 4 จุดกระจุกอยู่ที่ #103 กับ #760:** #760 ไม่ได้มาจากตัวแปลง ส่วน #103 ถึงจะมาจากตัวแปลง แต่ทำนองที่พบในใบแจ้ง (`1_ 1_ 1_ / 5^.` 4 บรรทัด) **ไม่ตรงกับที่ตัวแปลงเคยสร้าง** (`7 7 7 / 7 4` 6 บรรทัด) เลย = ถูกคีย์ใหม่ด้วยมือไปแล้วทั้งท่อน

### 6. มีอาการอื่นที่ยังไม่มีใครเจอไหม

**ตรวจแล้ว 2 เรื่อง — เรื่องแรกเคลียร์ เรื่องที่สองพบสัญญาณแต่ยังไม่ฟันธง**

#### 6.1 โน้ตที่ระบบอ่านไม่ออก — ✅ เคลียร์ ไม่มีอย่างอื่น

เอา token **ทุกตัว**ที่ตัวแปลงสร้างใน 120 เพลง ยัดเข้า parser จริงทั้ง 2 เวอร์ชัน:

```
=== base parser ===
  unreadable "raw" fragments:  ".#" x14   ".b" x1        ← มีแค่อาการ 2 อย่างเดียว
=== g1 parser ===
  unreadable "raw" fragments:  NONE                      ← G1 เก็บครบ
```

**ยืนยันว่าอาการ 2 คือความผิดพลาดชนิดเดียวที่ตัวแปลงสร้างซึ่งทำให้ระบบอ่านไม่ออก** ไม่มีชนิดอื่นแฝงอยู่

#### 6.2 ⚠️ ข้อสงสัยใหม่ — จังหวะในห้องเกินมา 0.25 อย่างเป็นระบบ

วัดจำนวนจังหวะทุกห้องเทียบกับเครื่องหมายประจำจังหวะ (ตัดห้องหัว-ท้ายบรรทัดออก เพราะห้องจริงต่อข้ามบรรทัดได้ = ไม่ใช่ของพัง):

```
ห้องทั้งหมดที่ไม่ตรง        : 1113/3163 (35.2%)   ← ตัวเลขนี้หลอก ส่วนใหญ่คือห้องต่อข้ามบรรทัด
ห้องกลางบรรทัดที่ไม่ตรง     :  243/1799 (13.5%)   ← ตัวเลขที่ควรดู
```

ในจำนวนนี้มี **14 เพลง** ที่ห้องส่วนใหญ่ยาวไม่ตรงกับเครื่องหมายประจำจังหวะ และบางเพลง**ผิดแบบเดียวกันทุกห้อง**:

```
  #68   ts=4/4 (ควรได้ 4)  แต่ห้องส่วนใหญ่ = 4.25   [ตรงกัน 100% ของห้อง]
  #105  ts=3/4 (ควรได้ 3)  แต่ห้องส่วนใหญ่ = 3.25   [ตรงกัน 100% ของห้อง]
  #33   ts=4/4 (ควรได้ 4)  แต่ห้องส่วนใหญ่ = 3      [ตรงกัน 100% ของห้อง]
  #108 #80 #39 #37 = 4.25 · #7 #54 = 4.5 · #28 #120 = 5 · #38 = 3.5 · #20 = 3
```

**ทำไมน่าสงสัย:** ความสม่ำเสมอ 100% พร้อมส่วนเกินคงที่ (+0.25 = โน้ตเขบ็ตสองชั้นหนึ่งตัว) เป็นลายเซ็นของ**ความผิดพลาดเชิงระบบ ไม่ใช่ดนตรี** — ถ้าเป็นเพลงที่จังหวะแปลกจริง มันจะไม่เกินเท่ากันเป๊ะทุกห้อง

**⛔ ผมยังไม่สรุปว่านี่คือของพัง** เพราะเป็นไปได้ 2 ทางและผมแยกไม่ออกโดยไม่เปิดหนังสือ:
- **(ก) อ่านเครื่องหมายประจำจังหวะผิด** — ตัวแปลง (บรรทัด 408-410) หาจาก regex ในหัวเรื่อง ถ้าไม่เจอจะ**เดาเป็น 4/4** พร้อมคำเตือน · ถ้าเพลงจริงเป็น 9/8 (=4.5 จังหวะ) แล้วถูกเดาเป็น 4/4 ตัวเลขจะออกมาแบบนี้เป๊ะ → **แปลว่าโน้ตถูก แต่ metadata ผิด**
- **(ข) นับขีดเขบ็ต/จุดเพิ่มค่าผิด** — `underline_count` (บรรทัด 92-99) ตัดเพดานที่ 2 ขีด และการต่อจุดเพิ่มค่า (บรรทัด 143-145) ใช้ระยะห่างเป็นตัวตัดสิน → **แปลว่าโน้ตผิดจริง**

**วิธีตรวจให้จบ:** เปิดหนังสือต้นฉบับของเพลง **#68 หรือ #105** (2 เพลงที่ผิดสม่ำเสมอ 100% = ตัวอย่างที่สะอาดที่สุด) ดูแค่ 2 อย่าง — (1) เครื่องหมายประจำจังหวะที่พิมพ์ไว้คืออะไร (2) นับจังหวะในห้องแรกด้วยตา · ถ้าเครื่องหมายในหนังสือไม่ตรงกับที่ระบบเก็บ = ทาง (ก) แก้ง่าย · ถ้าตรงแต่จำนวนจังหวะไม่ตรง = ทาง (ข) ต้องแก้ตรรกะอ่าน PDF

**หมายเหตุสำคัญ:** ตัวเลขชุดนี้มาจาก**ผลดิบตอนนำเข้า** เพลงเหล่านี้อาจถูกแก้ไปแล้วบนเว็บจริง — ต้องให้ Tester เช็คสถานะปัจจุบันก่อนลงมือทำอะไร

## G1 คุ้มครองอะไรแล้ว · อะไรยังต้องแก้ที่ตัวแปลง

รัน parser ทั้ง 2 เวอร์ชันทับสตริงอาการทั้ง 7 จุด (`raw` = ชิ้นที่ระบบอ่านไม่ออกแล้วพิมพ์เป็นตัวอักษรดิบ):

```
=== base (parser ปัจจุบันบน studio-shell-redesign) ===
  S1 #103        "5^."             raw="."    beats=1
  S1 #760-1      "2^."             raw="."    beats=1
  S1 #760-2      "#4^_"            raw="_"    beats=1
  S1 #760-3      "1^_"             raw="_"    beats=1
  S2 #109        ".#4"             raw=".#"   beats=1
  S3 unnumbered  "~6)"             raw=none   beats=1
  S3 #90         "1_ {1_ .6_ 1_})" raw=none   beats=1.5

=== g1 (feat-order-free-modifiers) ===
  S1 #103        "5^."             raw=none   beats=1.5   ✅
  S1 #760-1      "2^."             raw=none   beats=1.5   ✅
  S1 #760-2      "#4^_"            raw=none   beats=0.5   ✅
  S1 #760-3      "1^_"             raw=none   beats=0.5   ✅
  S2 #109        ".#4"             raw=none   beats=1     ✅
  S3 unnumbered  "~6)"             raw=none   beats=1     ← ไม่เปลี่ยน
  S3 #90         "1_ {1_ .6_ 1_})" raw=none   beats=1.5   ← ไม่เปลี่ยน
```

และยืนยันว่า G1 ให้ความหมายเหมือนกันเป๊ะ ไม่ใช่แค่ "ไม่พัง":

```
.#4  →  {accidental:"#", low:1, pitch:"4", ...}
#.4  →  {accidental:"#", low:1, pitch:"4", ...}     เหมือนกันทุกฟิลด์ ✅
5^.  →  {pitch:"5", dots:1, dotted:true, fermata:true}
5.^  →  {pitch:"5", dots:1, dotted:true, fermata:true}   เหมือนกันทุกฟิลด์ ✅
```

**สรุปการคุ้มครอง:**

| | อาการ 1 (`^`) | อาการ 2 (`.#`) | อาการ 3 (วงเล็บ) |
|---|---|---|---|
| **G1 กลบให้ไหม** | ✅ ครบ | ✅ ครบ | ❌ ไม่ช่วย |
| **ต้นเหตุ** | คนพิมพ์ | **ตัวแปลง** | การแก้ทีหลัง |
| **ต้องแก้ที่ตัวแปลงไหม** | ไม่ต้อง | ✅ **ต้อง** (`parse_song.py:138`) | ไม่ได้ (ไม่ได้มาจากตัวแปลง) |

**เรื่องอาการ 3 ที่ต้องเข้าใจให้ตรงกัน:** วงเล็บไม่ครบคู่ **ไม่ได้ทำให้ระบบอ่านโน้ตไม่ออก** (`raw=none` ทั้ง base และ G1 · จังหวะก็ถูก) — ผลคือ**เส้นโค้งเอื้อนวาดผิดที่หรือไม่วาด** เท่านั้น จึงเป็นเรื่องของ **lint ในหน้าแก้** ไม่ใช่เรื่อง parser · ปัจจุบัน `src/lib/notationLint.js` ตรวจ `‖:` `:‖` ที่ไม่ครบคู่อยู่แล้ว (บรรทัด 169-180) **แต่ไม่ได้ตรวจ `( )`** → นี่คือช่องว่างที่ควรอุด ไม่งั้นความผิดพลาดตอนแก้ด้วยมือแบบเพลง #90 จะเกิดซ้ำได้เรื่อยๆ โดยไม่มีอะไรเตือน

## ข้อสรุป

**ต้นเหตุ = ทั้งคู่ แต่แยกกันชัดเจน ไม่ปนกัน**

- **อาการ 2 (`.#4`) = ตัวแปลง 100%** — ฟันธงได้เต็มปาก · ทำซ้ำได้ทั้งจากผลดิบที่ commit ไว้ และจากการรันสดวันนี้ที่ได้สตริงเดิมทุกตัวอักษร · เป็นจุดเดียวในโค้ด (`parse_song.py:138`) · **ยังพังอยู่ ณ วันนี้**
- **อาการ 1 (`^`) = คนพิมพ์** — ฟันธงได้เต็มปาก · ตัวแปลงไม่มีตัวอักษร `^` อยู่ในโค้ดเลย และผลดิบ 120 เพลงมี `^` ศูนย์ครั้ง · เพลง #760 ไม่ได้อยู่ในชุดนำเข้าด้วยซ้ำ
- **อาการ 3 (วงเล็บ) = การแก้ด้วยมือทีหลัง** — ฟันธงได้ · ผลตัวแปลงไม่มีวงเล็บเลย ผลตัวใส่เส้นโค้งวงเล็บครบคู่ 309/309 บรรทัด · ร่องรอยของเพลง #90 บอกชัดว่าเกิดจากการเปลี่ยน `( )` เป็น `{ }` แล้วลืมลบ `)`

**สมมุติฐานตั้งต้นของพี่เอมถูกส่วนเดียว** — ตัวแปลงเป็นต้นเหตุจริง แต่ของ **1 ใน 7 จุด** ไม่ใช่ทั้งหมด · และข้อสังเกตเรื่อง "กระจุกที่บรรทัดสุดท้าย" **หักล้างแล้ว** เป็นเรื่องบังเอิญจากกลุ่มตัวอย่างเล็ก

**สิ่งที่ยังสรุปไม่ได้ (ขาดหลักฐานอะไร):**

1. **จังหวะเกิน +0.25 ใน 14 เพลง (ข้อ 6.2)** — ขาด **หนังสือต้นฉบับของเพลง #68 กับ #105** เพื่อแยกว่าเป็น "อ่านเครื่องหมายประจำจังหวะผิด" (metadata ผิด โน้ตถูก) หรือ "นับขีดเขบ็ตผิด" (โน้ตผิดจริง)
2. **สถานะปัจจุบันของ 15 จุดอาการ 2** — ขาดการเช็คฐานข้อมูลจริง (ผมไม่แตะตามกติกา) · เพลง **49, 64, 73, 92, 111** อาจยังพังอยู่ ต้องให้ Tester ไล่ดูบนเว็บ
