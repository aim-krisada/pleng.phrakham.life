# แก้รูที่ทำให้ "ธีม/หมวด" ของเพลงหายตอนกดอนุมัติ (ประตูที่สองของ B108)

ไฟล์ที่เกี่ยวข้อง: `db/011-approve-keep-theme.sql` (ตัวแก้) · `db/011-rollback-approve-keep-theme.sql` (ตัวถอย) ·
`db/011-approve-keep-theme.test.js` (หลักฐาน)

---

## 1. เกิดอะไรขึ้น (ภาษาคน)

ในฐานข้อมูลมีคำสั่งชื่อ `approve_and_publish` — คือปุ่ม **"อนุมัติและเผยแพร่"** ที่ผู้ตรวจกดเวลาอนุมัติร่างของสมาชิก

คำสั่งนี้เขียนไว้ประมาณว่า *"เอาธีมจากที่แอปส่งมา ไปใส่ให้เพลง"*

ปัญหาคือ ฐานข้อมูลแยกไม่ออกระหว่าง 2 อย่างนี้:

| แอปส่งมาว่า | ความหมายที่ควรจะเป็น | สิ่งที่ฐานข้อมูลทำจริง |
|---|---|---|
| "ธีม = ความรอด" | ใส่ธีมนี้ | ✅ ใส่ธีมนี้ |
| "ธีม = ว่าง" (คนลบทิ้งเอง) | ลบธีมทิ้ง | ✅ ลบธีมทิ้ง |
| **ไม่พูดถึงธีมเลย** (แอปอ่านของเดิมไม่ได้ เลยไม่กล้าส่ง) | **อย่าไปยุ่งกับธีมเดิม** | ❌ **ลบธีมทิ้ง** |

บรรทัดสุดท้ายคือบั๊ก — **"ไม่รู้" กลายเป็น "ลบ"** และหน้าจอขึ้นว่า "✅ อนุมัติและเผยแพร่แล้ว" ตามปกติ ไม่มีใครรู้ตัว

**ขนาดของความเสี่ยง** (นับจากข้อมูลจริงบนเซิร์ฟเวอร์วันนี้ 23 ก.ค. อ่านอย่างเดียว ไม่ได้แตะอะไร):
เพลงที่เผยแพร่แล้ว **162 เพลง** · มีธีม **80 เพลง** · อยู่นอกเล่มอนุชน **80 เพลง** (เล่มใหญ่ 54 · เด็กเล็ก 26)
ทุกเพลงในกลุ่มนี้ห่างจากการเสียธีม/ถูกย้ายเล่มแค่ "การกดอนุมัติหนึ่งครั้ง"

ช่องเดียวกันนี้ยังเปิดอยู่กับ **ชื่อภาษาอังกฤษ** และ **เลขเพลง** ด้วย (เขียนแบบเดียวกันเป๊ะ) — 011 ปิดทั้งสามช่องพร้อมกัน

> งาน B108 ที่เพิ่งรวมเข้ามาแล้วแก้ **ฝั่งแอป** ไปถูกทางแล้ว (แอปจะไม่ส่งค่าที่มันเดาเอง) —
> แต่รูนี้อยู่ **ฝั่งฐานข้อมูล** จึงยังเปิดอยู่ ต้องปิดด้วย SQL งานนี้ไม่ได้แก้ทับของเดิม ไม่ได้แตะโค้ดแอปเลยสักบรรทัด

## 2. แก้ยังไง

เปลี่ยนกฎเป็น **"พูดถึง = เขียน · ไม่พูดถึง = ไม่แตะ"**

- แอปส่งธีมมา (จะเป็นค่าจริงหรือค่าว่างก็ตาม) → ทำตามที่ส่ง **เหมือนเดิมทุกอย่าง** (ลบธีมเองยังลบได้ปกติ)
- แอปไม่พูดถึงธีม → **ปล่อยของเดิมไว้** (เมื่อก่อน = ลบ)

พูดอีกแบบ: **ตรงไหนที่เรารู้ ทำเหมือนเดิม · ตรงไหนที่เราไม่รู้ เปลี่ยนจาก "ลบ" เป็น "ไม่แตะ"** —
ไม่มีเคสไหนที่กลายเป็นอันตรายกว่าเดิม

และกฎนี้ตรงกับที่แอปทำอยู่แล้วพอดี: แอป (ทั้งตัว live บน `main` และตัวบน base) จะแนบหมวด/ธีมมา
**เฉพาะตอนที่มันรู้จริงเท่านั้น** ตามที่ B108 แก้ไว้ → **ไม่ต้องแก้โค้ดแอปเพิ่มเลย**

## 3. ⚠️ เรื่องที่ต้องบอกก่อน — ผมอ่านของจริงบนเซิร์ฟเวอร์ไม่ได้

ผมพยายามเปิดดูว่า คำสั่งตัวจริงบน Supabase ตรงกับไฟล์ `db/006` ในรีโปไหม **แต่อ่านไม่ได้** เพราะกุญแจที่อยู่ในโค้ด
เป็นกุญแจสาธารณะ (publishable) ซึ่งดูได้แค่ข้อมูลเพลง ดู "หน้าตาคำสั่ง" ไม่ได้ (เซิร์ฟเวอร์ตอบว่าต้องใช้กุญแจลับ)
และผมไม่รันอะไรบนฐานข้อมูลเองตามกติกา **ผมจึงไม่ยืนยันว่าตรง และไม่เดาว่าตรง**

**ทางออก: ให้ตัว SQL ตรวจเองแทน** — 011 จะเปิดดูคำสั่งตัวจริงก่อนเป็นอย่างแรก แล้ว

- ตรงกับ `db/006` → ลงมือแก้ แล้วขึ้นข้อความ `011 applied.`
- **ไม่ตรง** (มีคนไปแก้บนเซิร์ฟเวอร์เอง / ยังไม่เคยรัน 006 / ไม่มีคำสั่งนี้เลย) → **หยุดทันที ไม่แก้อะไรสักอย่าง**
  แล้วพิมพ์คำสั่งตัวจริงออกมาให้ทั้งดุ้น → **ก๊อปข้อความนั้นส่งกลับมาให้ทีมดูก่อน อย่าฝืนรันอะไรต่อ**
- รันซ้ำอีกรอบ → ขึ้นว่า `011 already applied` แล้วจบ ไม่พังไม่ซ้ำซ้อน

ที่ต้องทำแบบนี้เพราะ ถ้าเขียนทับดื้อๆ แล้วบังเอิญมีคนเคยแก้คำสั่งนั้นบนเซิร์ฟเวอร์ไว้ งานของเขาจะหายเงียบ —
ซึ่งก็คือบั๊กแบบเดียวกับที่เรากำลังแก้อยู่พอดี

## 4. ระหว่างรัน คนที่ใช้งานอยู่จะเป็นยังไง

**ใช้ได้ตามปกติ ไม่มีช่วงที่ระบบล่ม** เพราะ:

- **ไม่แตะข้อมูลเพลงเลยแม้แต่แถวเดียว** — ไม่อ่านเพื่อแก้ ไม่เขียน ไม่ลบ ไม่ย้าย ไม่ backfill
  011 เปลี่ยนแค่ "วิธีทำงาน" ของคำสั่งเดียว
- **ไม่มีการลบคำสั่งทิ้งแล้วสร้างใหม่** ใช้ `create or replace` = สลับไส้ในทีเดียวจบ
  จึงไม่มีเสี้ยววินาทีไหนที่ใครกดปุ่มแล้วเจอ "ไม่พบคำสั่ง" · สิทธิ์การเรียกใช้ก็ติดอยู่กับที่เดิม ไม่หลุด
- **คนที่กดอนุมัติอยู่พอดีตอนนั้น** ครั้งนั้นจะทำงานจบด้วยกฎเก่า ครั้งถัดไปใช้กฎใหม่ — ไม่มีการค้างกลางคัน
- **แอปตัวเก่าที่ live อยู่ (`main`) ไม่ต้องแก้ ไม่ต้อง deploy ใหม่** เพราะมันส่งข้อมูลมาแบบเดิมทุกประการ
  และกฎใหม่ก็ตีความข้อมูลชุดเดิมนั้นเหมือนเดิมทุกกรณี — เว้นกรณีเดียวคือกรณีที่เมื่อก่อนมันลบธีมทิ้ง

---

## 5. ต้องรัน SQL ก่อน หรือขึ้นโค้ดก่อน

### 👉 รัน SQL ได้เลย และ**ไม่ต้องขึ้นโค้ดตามเลย** — งานนี้ไม่มีการแก้โค้ดแอป

รอบนี้ลำดับไม่มีทางผิด เพราะกฎใหม่ครอบกฎเก่าทั้งหมด:
โค้ดเก่า + SQL ใหม่ = ทำงานถูกต้อง (นี่แหละคือตัวแก้) · โค้ดใหม่ + SQL เก่า = ก็แค่เป็นสภาพวันนี้ที่บั๊กยังอยู่

(ต่างจากรอบ B108 ที่ต้องรัน `db/010` ก่อนขึ้นโค้ด เพราะรอบนั้นโค้ดจะไปเขียนคอลัมน์ที่ยังไม่มี — รอบนี้ไม่มีคอลัมน์ใหม่)

## 6. พี่เอมต้องกดอะไรบ้าง

1. เปิด Supabase → **SQL Editor** → New query
2. ก๊อปทั้งไฟล์ `db/011-approve-keep-theme.sql` วางลงไป → กด **Run**
3. ดูข้อความที่ขึ้นในช่อง output:
   - `Live function matches db/006. Replacing with the 011 version…` + `011 applied.` → ✅ **เรียบร้อย จบ**
   - ขึ้นคำว่า **`STOP:`** → ❌ **ยังไม่มีอะไรถูกแก้** (ปลอดภัย) → ก๊อปข้อความทั้งหมดส่งกลับมาให้ทีม
   - `011 already applied` → เคยรันไปแล้ว ไม่ต้องทำอะไรต่อ
4. อยากเช็กซ้ำว่าติดจริง (ไม่บังคับ · แค่อ่าน ไม่แก้อะไร) — รันท้ายไฟล์ที่คอมเมนต์ `--- Verify ---` ไว้
   บรรทัดธีมต้องอ่านได้ว่า `theme = case when p_song ? 'theme' then … else theme end`
5. ทดสอบของจริง: เอาเพลงที่มีธีมและอยู่เล่มใหญ่มา 1 เพลง → ให้สมาชิกส่งร่าง → กดอนุมัติ → **ธีมกับเล่มต้องอยู่ครบ**

## 7. ถ้าพัง จะถอยยังไง

ก๊อปทั้งไฟล์ `db/011-rollback-approve-keep-theme.sql` วางใน SQL Editor → Run → กลับไปเป็น `db/006` เป๊ะ ทันที

- ถอยได้ทุกเมื่อ ไม่ต้องรอ ไม่ต้องปิดระบบ ไม่กระทบข้อมูล (เหตุผลเดียวกับข้อ 4)
- ไม่ต้อง deploy อะไรตาม เพราะไม่เคยแก้โค้ดแอป
- **แต่ระวัง:** ถอยกลับ = กลับไปใช้ตัวที่ลบธีมทิ้ง ถ้าเจอปัญหา บอกทีมก่อนดีกว่าถอยเลย
  (ของเดิมไม่เคยมีไฟล์ถอย — รอบนี้ทำไว้ให้แล้ว)
