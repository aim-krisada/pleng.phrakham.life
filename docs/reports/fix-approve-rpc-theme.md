## 1. เกิดอะไรขึ้น (ภาษาคน)

ในฐานข้อมูลมีคำสั่งชื่อ `approve_and_publish` = ปุ่ม **"อนุมัติและเผยแพร่"** ที่ผู้ตรวจกด

คำสั่งนี้แยกไม่ออกระหว่าง 2 อย่าง:

| แอปส่งมาว่า | ที่ควรจะเป็น | ที่เกิดขึ้นจริง |
|---|---|---|
| "ธีม = ความรอด" | ใส่ธีมนี้ | ✅ ใส่ธีมนี้ |
| "ธีม = ว่าง" (คนตั้งใจลบ) | ลบธีม | ✅ ลบธีม |
| **ไม่พูดถึงธีมเลย** (แอปอ่านของเดิมไม่ได้ เลยไม่กล้าส่ง) | **อย่าไปยุ่ง** | ❌ **ลบทิ้ง** |

**"ไม่รู้" กลายเป็น "ลบ"** — และจอขึ้น "✅ อนุมัติและเผยแพร่แล้ว" ตามปกติ ไม่มีใครรู้ตัว

**ช่องนี้เปิดอยู่ 3 ช่อง** (ทั้งเพลงเก่าและเพลงใหม่):

| ช่อง | ผลถ้าแอปไม่ส่งค่ามา |
|---|---|
| **เลขเพลง** | ถูกล้างเป็นค่าว่าง 🔴 |
| **ชื่อภาษาอังกฤษ** | ถูกล้าง |
| **ธีม** | ถูกล้าง |

ส่วน **ชื่อไทย · เนื้อเพลง · หมวด · ป้ายตรวจ** มีตัวกันอยู่แล้ว — **011 ไม่ไปแตะเลย**

**ขนาดความเสี่ยง** (นับจากของจริง อ่านอย่างเดียว 23 ก.ค.): เพลงเผยแพร่แล้ว **162 เพลง** ·
มีธีม **80 เพลง** · อยู่นอกเล่มอนุชน **80 เพลง** (เล่มใหญ่ 54 · เด็กเล็ก 26) · และ **ทุกเพลงมีเลขเพลง**

> งาน B108 ที่ merge ไปแล้วแก้ **ฝั่งแอป** ให้ "ไม่ส่งค่าที่เดาเอง" ซึ่งถูกสำหรับ **หมวด** (ฝั่ง SQL มีตัวกัน)
> แต่ **ไม่ช่วยเรื่องธีม** — เพราะฝั่ง SQL ไม่มีตัวกัน "ไม่ส่ง" จึงเท่ากับ "ลบ" **รูนี้ยังเปิดอยู่วันนี้**
> งานนี้ไม่ได้แก้ทับ B108 และไม่ได้แตะโค้ดแอปเลยสักบรรทัด

## 2. แก้ยังไง

เปลี่ยนคำถามจาก *"ค่าว่างไหม"* เป็น ***"เขาส่งช่องนี้มาหรือเปล่า"***

- **ส่งมา** → ทำเหมือนเดิมทุกประการ (จะใส่ค่าใหม่ หรือจะตั้งใจลบ ก็ยังทำได้ปกติ)
- **ไม่ส่งมา** → **ไม่แตะของเดิม** (เมื่อก่อน = ลบทิ้ง)

จุดสำคัญ: ฝั่งที่ทำงานตอน "ส่งมา" คือ**สูตรเดิมทุกตัวอักษร** — ไม่ได้เขียนใหม่ ไม่ได้ปรับ
เพราะฉะนั้นทุก payload ที่แอปส่งอยู่ทุกวันนี้ ได้ผลลัพธ์เท่าเดิมเป๊ะ **เปลี่ยนเฉพาะกรณีที่เมื่อก่อนมันลบทิ้ง**

**ไม่ใช้ `coalesce`** เพราะจะทำให้ "ตั้งใจลบธีม" ทำไม่ได้อีกเลย — การเอา "ไม่มีค่า" ไปปนกับ "ค่าคือว่าง"
คือรากของบั๊ก B108 พอดี ถ้าแก้ด้วยวิธีนั้นก็เท่ากับสร้างบั๊กฝาแฝดขึ้นมาใหม่

**เพลงใหม่ (เส้น insert)** มีรูเดียวกัน → ให้ไปหยิบค่าจาก**ร่างของเพลงนั้นเอง**แทนการเขียนค่าว่าง
(คอลัมน์หมวด/ธีมของร่างมีอยู่แล้วตั้งแต่ `db/010`) และหมวดจะลอง "ค่าจากร่าง" ก่อนตกไปที่ค่าสำรอง `อนุชน`
— ค่าสำรองตัวนี้แหละคือรากของ B108

**สิ่งที่จงใจไม่แก้:** เส้น insert ยังบันทึกคนอนุมัติเป็นเจ้าของเพลงเหมือนเดิม (`auth.uid()`)
ไฟล์ `db/006` ในรีโปตั้งใจแก้เรื่องนี้แต่ไม่เคยถูกรัน — **ผมไม่แอบรันให้พร้อมกัน** เพราะเป็นการเปลี่ยนพฤติกรรม
คนละเรื่องที่ไม่มีใครสั่ง ให้เป็นการตัดสินใจแยกของ PM

## 4. ระหว่างรัน คนที่ใช้งานอยู่จะเป็นยังไง

**ใช้ได้ตามปกติ ไม่มีช่วงที่ระบบล่ม และไม่มีอะไรต้องย้อนภายหลัง**

- **ไม่แตะข้อมูลเพลงสักแถว** (ข้อ ก) — เปลี่ยนแค่ "วิธีทำงาน" ของคำสั่งเดียว ไม่ใช่ "ข้อมูล"
  → **ไม่มีวันต้องย้อนฐานข้อมูล** งานที่พี่เปาทำระหว่างนั้นจึงไม่มีทางหาย
- **ไม่มีการลบคำสั่งทิ้งแล้วสร้างใหม่** (ใช้ `create or replace` = สลับไส้ในทีเดียว) → ไม่มีเสี้ยววินาทีที่ใครกดปุ่มแล้วเจอ
  "ไม่พบคำสั่ง" · สิทธิ์เรียกใช้ก็อยู่ที่เดิม
- **คนที่กำลังกดอนุมัติพอดี** ครั้งนั้นจบด้วยกฎเก่า ครั้งถัดไปใช้กฎใหม่ ไม่ค้างกลางคัน
- **แอปตัวเก่าไม่ต้อง deploy ใหม่** (ข้อ ค)

## 6. พี่เอมต้องกดอะไร

1. Supabase → **SQL Editor** → New query
2. ก๊อปทั้งไฟล์ `db/011-approve-keep-theme.sql` วาง → **Run**
3. อ่านข้อความที่ขึ้น:
   - `Live function matches…` + `011 applied.` → ✅ **เรียบร้อย จบ**
   - ขึ้นคำว่า **`STOP:`** → ❌ **ยังไม่มีอะไรถูกแก้เลย** (ปลอดภัย) → ก๊อปข้อความทั้งหมดส่งกลับให้ทีม
   - `011 already applied` → เคยรันแล้ว ไม่ต้องทำอะไร
4. (ไม่บังคับ) เช็กซ้ำแบบอ่านอย่างเดียว — รันบรรทัดท้ายไฟล์ที่คอมเมนต์ `--- Verify ---` ไว้
5. ทดสอบจริง: เอาเพลงที่มีธีม + อยู่เล่มใหญ่ 1 เพลง → ให้สมาชิกส่งร่าง → กดอนุมัติ →
   **ธีม เล่ม เลขเพลง ชื่ออังกฤษ ต้องอยู่ครบ**

**ตัว SQL ตรวจของจริงให้เองก่อนลงมือทุกครั้ง** — ถ้าคำสั่งบนเซิร์ฟเวอร์ไม่ใช่ตัวที่พี่เอมอ่านออกมาให้ (เช่นมีคนไปแก้มือ
หรือมีคนรัน `db/006` แทรกเข้ามา) มันจะ **หยุดทันที ไม่แก้อะไรเลย** แล้วพิมพ์ของจริงออกมาให้ส่งกลับมาที่ทีม
