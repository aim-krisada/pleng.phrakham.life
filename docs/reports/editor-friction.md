## ข้อ 1 · issues7 — "ดูผลทั้งเพลง ตัวเล็กมาก ยืดขยายไม่ได้อีก"

### ต้นเหตุจริง (มี 3 ชั้นซ้อนกัน)

**ชั้นที่ 1 — บั๊กคำนวณ ทำให้ตัวเล็กกว่าที่ออกแบบไว้เอง**
โค้ดวัดความกว้างคอลัมน์อ่านด้วย `parseFloat(cs.maxWidth)` แต่เบราว์เซอร์คืนค่ามาเป็นสูตรดิบ
`"min(1160px, 100%)"` → `parseFloat` = **NaN** → ตกไปใช้ `|| Infinity` → วัดคอลัมน์เป็น **1455px**
ทั้งที่ของจริงกว้าง **1160px เป๊ะ** (วัด `getBoundingClientRect` แล้ว)

ผลข้างเคียงที่ย้อนศร: **จอยิ่งกว้าง ตัวยิ่งเล็ก** — พิสูจน์สดแล้ว จอ 1280 → 9.06px · จอ 1536 → **7.47px**

**ชั้นที่ 2 — ตัวลากขยายเป็นทางตัน (นี่คือคำว่า "ไม่ได้อีก")**
หน้าต่างอยู่มุมขวาบน (`right:16px`) และตัวลากขยายได้แค่ไปทางขวา/ลงเท่านั้น
วัดจริงที่จอ 1536: หน้าต่างกว้าง 720px อยู่ที่ x=785 → **ขยายได้อีกแค่ 27px แล้วชนขอบจอ**
พี่เปาพูดถูกทุกตัวอักษร — ปุ่มที่ดูเหมือนจะขยายได้ แทบไม่ขยับเลย

**ชั้นที่ 3 — หน้าต่างเล็กเกินกว่าจะอ่านออกตั้งแต่ต้น**
แผ่นเพลง**ย่อลงให้พอดีหน้าต่าง** ดังนั้นหน้าต่าง 720px = ตัวหนังสือ ~9px (แผ่นจริง 16px)
นี่คือผลของ parity ที่พี่เปาขอเองใน issues3 (ให้ตกบรรทัดเหมือนหน้าแผ่นเพลง):
จะให้ตกบรรทัดเหมือนคอลัมน์ 1160px ในหน้าต่าง 720px ได้ **ต้องย่อ** — เลี่ยงไม่ได้
**หน้าต่างยิ่งกว้าง = ตัวยิ่งใหญ่** จึงเป็นกลไกของมันอยู่แล้ว แค่ก่อนหน้านี้ทำให้กว้างไม่ได้

### แก้ยังไง
1. ดึงตัวเลข px ออกจากสูตรแทน `parseFloat` ตรงๆ → วัดคอลัมน์ได้ 1160px ตามที่ออกแบบไว้จริง
2. **ตัวลากขยาย: พอชนขอบขวา ให้หน้าต่างเลื่อนไปทางซ้ายแทนการหยุด** → ลากได้จนเต็มจอ
3. ค่าเริ่มต้นหน้าต่าง 720px → **980px** (อ่านออกตั้งแต่เปิด แต่ยังเห็นโต๊ะแก้ไว้ "แก้ไปดูไป")

### ผลที่วัดได้จริง (จอ 1536 · Aa 100%)
| | ก่อน | หลัง |
|---|---|---|
| เปิดมาเฉยๆ | 7.47px | **13.73px** (+84%) |
| ลากตัวขยายสุด | 720→747px (โต 27px) | 980→**1528px** · ตัวหนังสือ **21.75px** |

## 🔴 issues7 รอบ 2 — P'Aim ลองจริงแล้วยังไม่ผ่าน (commit `31f8720`)

> *"ควรใช้พื้นที่เต็ม · เหลือที่ทางขวาเยอะ · ขยายขนาดตัวอักษรไม่ได้ (ต้องขยายยังไง) · ควรมีปุ่มขยาย font ในหน้านั้นเอง — **ดูตรงไหนแก้ตรงนั้น**"*

### 🎯 ต้นเหตุจริงของ **ทั้งสองอาการ = ข้อเดียวกัน** (วัดจากหน้าแผ่นเพลงจริง ไม่ใช่เดา)
เพลงจริง (#66) บนหน้าแผ่นเพลง คอลัมน์ **1160px** · ตัวอักษร 16px:
| | |
|---|---|
| บรรทัดที่แสดงจริง | 10 บรรทัด · กว้าง **291–397px** |
| **บรรทัดกว้างสุด** | **397px = 34% ของคอลัมน์** |
| **ที่ว่างด้านขวา** | **66%** |

**เนื้อเพลงต้องการแค่ ~24.8em แต่ preview ย่อให้พอดี "คอลัมน์" 68.4em**
→ **มันย่อตัวหนังสือลงเพื่อให้พอดีกับความว่างเปล่า** → **ตัวเล็ก + เหลือที่ทางขวา = อาการเดียวกัน ต้นเหตุเดียวกัน**
_(ระวังกับดัก: ตอนแรกผมวัดได้ 34 บรรทัด แต่ **24 บรรทัดมี width=0 (ซ่อนอยู่)** → ตัวเลขแรกเชื่อไม่ได้ · กรองเฉพาะที่แสดงจริงถึงได้ตัวเลขข้างบน · **เกือบเอาตัวเลขผิดไปออกแบบ**)_

### แก้ยังไง
1. **ปุ่มขยาย font อยู่ในหัวหน้าต่างนั้นเอง** (− / % / +) · **จำค่าไว้** · กดที่ % = คืน 100%
   **ตอบคำถาม PM ตรง ๆ: แยก scale — ไม่ใช้ Aa ตัวเดียวกัน** · เหตุผล: "ดูตรงไหนแก้ตรงนั้น" ไม่ควรแปลว่า **ยื่นมือไปจัดหน้าแก้ไขข้างหลัง** · **verify สดแล้วว่าหน้าแก้ไขไม่ขยับ** (คง 16px · `readingFontScale` ยังเป็น 1) · **"Aa" เดิมยังอยู่ครบ ไม่ได้ย้าย**
   **กลไก:** หาร em-width ไม่ใช่คูณ font — เพราะหน้ากว้าง 100cqw เสมอ ถ้าคูณ font แผ่นจะล้นหน้าต่าง = **scroll แนวนอน (ผิด ui-standards §2)** · หาร em = ตัวใหญ่ขึ้น ไม่มี h-scroll · **ที่ zoom 1 = parity เดิมไม่ถูกแตะ**
2. **หน้าต่างจำขนาด/ตำแหน่งที่ลากไว้** — เดิม **ลืมทุกครั้งที่ปิด** = ครึ่งหนึ่งของอาการ "ไม่ใช้พื้นที่เต็ม" · clamp ตอนเปิดกันหลุดจอ
3. **เลิกใช้เลข 980 ตายตัว** ตามกฎ P'Aim → **กว้างตามจอ (65vw)** มีพื้น/เพดานกันจอเล็ก-ล้น

### ผลที่วัดได้จริง
| จอ | ก่อน (รอบ 1) | หลัง |
|---|---|---|
| 1536 เปิดมาเฉย ๆ | 998px · 14px | **14px · เหลือขอบขวา 31px** |
| 1536 กด + | **ทำไม่ได้ (ไม่มีปุ่ม)** | **200% → 28px** |
| 1536 หลัง reload | **ลืมหมด กลับไป default** | **จำได้: 1528px · 150% · 32.62px · เหลือขอบขวา 4px** |

### 🔴 ผมทำบั๊กเองอีกตัว — เจอเพราะทดสอบ 412px จริง
`Math.max(20, em / zoom)` — เพดาน 20em เดิมมีไว้**กันค่าวัดพัง** แต่ผมเอาไปใส่**หลัง**หารด้วย zoom
→ บนมือถือคอลัมน์แค่ ~21em → **ทุก zoom ชนเพดาน → ปุ่ม + ตายสนิท** (วัดได้ 19.4px ก่อน/หลังเท่ากัน)
**แยกให้ขาด:** 20em กันค่าวัดพัง (ใช้ก่อนหาร) · 8em กัน h-scroll (ใช้หลังหาร)

### 🪤 test ของผม fake-pass 2 รอบก่อนจะจับบั๊กได้จริง (บทเรียน Tier-B)
1. รอบแรก: jsdom ไม่มี layout → คอลัมน์ตกไปใช้ fallback **42em (กว้าง)** = **เคสที่ไม่เคยพัง** → ย้อนโค้ดเป็นบั๊กแล้วยังเขียว
2. รอบสอง: จำลองคอลัมน์มือถือแล้ว แต่ **assertion อ่อนไป** — กด + ครั้งเดียว 21.2→20 "ก็ยังลดลง" แม้ติดเพดาน → ยังเขียว
3. รอบสาม: **กด + สองครั้งติด ต้องลดลงทุกครั้ง** → ย้อนโค้ด = **แดงจริง** (`expected 20 to be less than 20`)
**เกณฑ์เดียวที่เชื่อได้ = ย้อนโค้ดกลับแล้วต้องแดง** ไม่ใช่ "เขียวแล้วแปลว่าดี"

## ข้อ 2 · issues9 — "ทำไมต้องกดปักหมุดก่อนถึงจะเซฟร่างได้"

### ต้นเหตุจริง — **แรงกว่าที่ PM คาด**
PM ตั้งโจทย์ว่าเป็นเรื่อง "ค่าเริ่มต้นควรอยู่แถบหลักไหม" ของจริงหนักกว่านั้น:

ในหน้า ⚙ แต่ละแถวจะ render ตัวควบคุมตามชนิด — `menu` ได้ dropdown · `toggle` ได้สวิตช์ · `slider` ได้แถบเลื่อน
แต่ **`btn` ไม่มีสาขาเลย** → "บันทึกร่าง" (เป็น `btn`) แสดงแค่ ไอคอน + ชื่อ + **ช่องว่าง** + หมุด

**คือมันกดใช้จากหน้า ⚙ ไม่ได้จริงๆ — ปักหมุดคือทางเดียวที่จะได้ปุ่มที่กดแล้วทำงาน**
พี่เปาไม่ได้เข้าใจผิด เขาอธิบายพฤติกรรมจริงของระบบเป๊ะๆ

**ภาพพี่เปายืนยันตรงกับโค้ด:** ในภาพ `issues9/20260717-141753.png` แถว "บันทึกร่าง" มีแต่หมุด (ไม่มีสวิตช์)
ส่วนแถว "ดูผลทั้งเพลง" (เป็น `toggle`) มีสวิตช์ — ตรงกับที่โค้ดบอกทุกประการ

### แก้ยังไง (ไม่รื้อระบบปักหมุด ตามที่ PM กำชับ)
1. **"บันทึกร่าง" ได้ที่นั่งถาวรบนแถบหลัก** (row 2) — คำสั่งที่ใช้บ่อยที่สุดของคนพิมพ์ 124 เพลง ต้องกดถึงทันที
   ไม่ต้องตั้งค่าก่อนใช้ · ระบบปักหมุดยังอยู่ครบสำหรับตัวอื่น
2. **ปิดคลาสบั๊ก:** `btn` ในหน้า ⚙ ตอนนี้มีปุ่มกดใช้ได้จริง — ทุกแถวในหน้านั้นทำสิ่งที่มันเขียนไว้
3. **กันปุ่มโผล่ 2 ที่:** พี่เปา**ปักหมุด 'draft' ไว้แล้ว**ในเครื่องเขา (เห็นหมุดแดงในภาพ) ถ้าไม่กัน
   ปุ่มจะขึ้นทั้งแถบหลักและแถวหมุด = ซ้ำ · ตอนนี้ของที่มีที่นั่งบนแถบแล้วจะไม่ถูกวาดซ้ำจากหมุด
   (ล้างค่าเก่าให้เอง ไม่ต้องให้พี่เปาทำอะไร · ตรงตาม ui-standards §2 single source of action)

## ข้อ 3 · issues11 (เฉพาะการเข้าถึง) — "ต้องเลื่อนมาบนสุดถึงจะกดใช้ได้"

### ต้นเหตุจริง
`.edhead` (แถบหัวที่ถือเครื่องมือของบรรทัดที่กำลังแก้ — ⋯ · ฮุก · ซ้ำ · ย้ายขึ้น/ลง · ลบบรรทัด)
เป็น `position: static` = **เลื่อนหนีไปกับหน้า** → แก้บรรทัดที่ 20 ต้องเลื่อนกลับขึ้นบนสุดทุกครั้งที่จะใช้เครื่องมือ

### แก้ยังไง
ทำให้แถบ**ติดอยู่ใต้หัวเว็บ** · ค่า `top` **วัดจากความสูงหัวเว็บจริง ไม่ hardcode**
(หัวเว็บสูงไม่เท่ากันแต่ละขนาดจอ — 64px ที่เดสก์ท็อป · 114px ที่ 360px)

> **เจอบั๊กข้างเคียง (ไม่ได้แตะ · นอกขอบเขต):** `NAV.rail` ใช้ `top: 58px` แบบ hardcode
> ทั้งที่หัวเว็บสูง 64–114px → rail มุดใต้หัวเว็บ · ผมเลี่ยงกับดักนี้ด้วยการวัดเอา

### 🔴 ผมทำพังเอง แล้วแก้แล้ว — SA เตือนถูก (และแย่กว่าที่เตือน)
SA ชี้ว่า **`.cshead` (แถบเครื่องมือท่อน) sticky อยู่แล้ว** ที่ `top: 58px` แบบ hardcode (B085 · ไฟล์เดียวกัน)
**ผมไม่ได้ตรวจจุดนี้ในรอบแรก** · พอวัดจริงที่จอ 1536 · scroll 1600:

| | ก่อนแก้ (ที่ผมเกือบส่ง) | หลังแก้ |
|---|---|---|
| edhead | 64→158 | 64→158 |
| cshead | **58→126 (มุดใต้ edhead)** | **158→226** |
| ซ้อนกัน | **62px** | **0px** |
| ปุ่ม cshead กดได้ | **❌ ไม่ได้** (โดน `ed-chip` ของผมบัง) | **✅ ได้** |

# 🛑 งานต่อ: ปุ่มลบเพลง — **หยุดตามกติกา ยังไม่ลงมือ**

**กฎ P'Aim:** Admin ลบได้ทุกเพลง · **Editor ลบได้เฉพาะเพลงของตัวเอง** (ทั้งคู่ไม่ว่าตรวจแล้วหรือยัง)
PM สั่ง: *"verify RLS ก่อน · ถ้าไม่ตรง = ต้องแตะ DB → หยุด รายงาน PM"* → **verify แล้ว: ไม่ตรง · หยุด**

## ครึ่งแรก (Admin) — ✅ ใช้ได้อยู่แล้ว ไม่ต้องแก้
`EditorMode.vue:2357` = `v-if="isApprover && loggedIn && editingId && !reviewingDraft"`
**ไม่มีเงื่อนไข `verified` เลย** → Admin ลบได้ทุกเพลงไม่ว่าตรวจแล้วหรือยัง = **ตรงกฎ P'Aim อยู่แล้ว**
(`isApprover` = `tier === 'approver'` ← `profile.role === 'approver'` · `profiles.role` มีแค่ 2 ค่า: `editor` / `approver` → **approver = Admin** · ยืนยันจาก `db/002:9`)

## ครึ่งหลัง (Editor) — 🔴 ทำไม่ได้ตอนนี้ · ติด 3 ชั้น

**ชั้น 1 · RLS ปฏิเสธตรง ๆ** (`db/002:75`)
```sql
create policy "Approvers can delete songs" on public.songs
  for delete using (public.app_role() = 'approver');
```
→ Editor ลบเพลงที่เผยแพร่แล้ว **ไม่ได้เลย ไม่ว่าเป็นของใคร** · **ต้องแก้ policy = แตะ DB จริง**

**ชั้น 2 · 🔴 ข้อมูล "เจ้าของ" ไม่มีอยู่จริง — ผมยิง API จริงนับมาแล้ว (`songs` เปิดอ่าน public)**
| สถานะ | จำนวน |
|---|---|
| ยังไม่ตรวจ · `author_id = NULL` | **104** |
| ตรวจแล้ว · `author_id = NULL` | **17** |
| ตรวจแล้ว · `author_id` มีค่า | **3** |
| **รวม** | **124** |

→ **121 จาก 124 เพลง (97.6%) ไม่มีเจ้าของ** · **ต่อให้แก้ policy วันนี้ พี่เปาก็เป็นเจ้าของ 0 เพลง → ลบอะไรไม่ได้อยู่ดี ปุ่มจะโผล่มาแล้วกดไม่ได้**

**ชั้น 3 · 🔴 `author_id` ของเพลง = "คนกดอนุมัติ" ไม่ใช่ "คนเขียน"** (`db/004:177`)
```sql
insert into public.songs (..., author_id) values (..., auth.uid())
```
บรรทัดนี้อยู่ใน `publish_draft` ซึ่ง **กันไว้ว่าต้องเป็น approver เท่านั้นถึงเรียกได้** (`:168`) และเป็น `security definer`
(`auth.uid()` อ่านจาก JWT ของ**คนเรียก** → definer ไม่เปลี่ยนค่านี้) → **ค่าที่ได้คือ uid ของผู้อนุมัติเสมอ**
ทั้งที่ตัวร่าง `d` **มี `d.author_id` = คนเขียนจริงอยู่ในมือแล้ว แต่โค้ดไม่ได้ใช้**
→ **แปลว่าถึงจะแก้ policy เพลงที่พี่เปาเขียนใหม่ก็จะกลายเป็นของ P'Aim ทุกเพลง → กฎ "ลบของตัวเอง" ไม่มีวันทำงานให้พี่เปาเลย**
(สอดคล้องกับข้อมูล: 3 เพลงที่มี `author_id` = `verified` ทั้งหมด = ผ่าน `publish_draft` มา)

## ⚠️ สิ่งที่ผม **ไม่** ได้พิสูจน์ (ไม่เคลมเกินจริง)
ผมลองยิง DELETE แบบไม่ล็อกอินด้วย **id ปลอม** ได้ HTTP 204 — **แต่ตีความไม่ได้** เพราะ RLS ปฏิเสธ กับ "ไม่มีแถวตรง" คืน 204 เหมือนกัน
**ผมไม่ทดสอบด้วย id เพลงจริง** เพราะนั่นคือการลบงานพี่เปาของจริง → **ข้อนี้ยังไม่รู้ ต้องตรวจ policy ในแดชบอร์ด**

## ทางเลือกให้ PM/P'Aim เคาะ (ผมไม่ตัดสินเอง)
| | ทำอะไร | ราคา |
|---|---|---|
| **A** | แก้ `publish_draft` ให้ใช้ `d.author_id` (คนเขียน) + เพิ่ม delete policy `author_id = auth.uid()` | **แตะ DB 2 จุด** · ได้ผลเฉพาะ**เพลงใหม่** · **121 เพลงเดิมยังไม่มีเจ้าของ → พี่เปายังลบไม่ได้** |
