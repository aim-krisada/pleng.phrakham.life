## 1. ตัดสิน source: **DOCX (เนื้อไทย) + PDF geometry (ทำนอง) — ใช้คู่กัน**

เทียบ 5 เพลง (1, 10, 100, 103, 109) + สแกนทั้ง 120 ไฟล์ ได้ข้อสรุปชัด:

| ข้อมูลที่ต้องการ | PDF (pdftotext ข้อความ) | PDF (pdfplumber geometry) | DOCX (Word) |
|---|---|---|---|
| เนื้อร้องไทย | ❌ กลายเป็น `,` `.` หมด | ❌ (ตัวอักษรไทยเพี้ยน) | ✅ **ครบ** |
| ตัวเลขโน้ต 1–7 | ✅ (แต่ระยะเพี้ยน) | ✅ พร้อมพิกัด x/y | ✅ ครบ |
| จังหวะ (ขีดใต้ = เขบ็ต) | ❌ หาย | ✅ เป็น `rects` เส้นบาง | ✅ เป็น underline ของ run |
| จุดออกเทฟ สูง/ต่ำ | ❌ หาย | ✅ เป็น `curves` บน/ล่างตัวโน้ต | ⚠️ **114/120 เป็นรูปวงกลมลอย** (จับคู่ยาก) |
| เครื่องหมาย ♯ ♭ ♮ | ❌ **หาย** (`F♯m`→`Fm` ผิดคอร์ด!) | ✅ | ✅ inline `♯2 ♮2` |
| คอร์ด | ⚠️ เพี้ยน (เสีย ♯) | ✅ ตัวอักษร + พิกัด | ✅ (คั่นด้วย tab) |
| ย้อน/จบ (`‖: :‖` volta) | ⚠️ | ✅ (สัญลักษณ์) | ✅ `║ :` inline |

**สรุป:** PDF ข้อความล้วน **ใช้ไม่ได้** (เสียเนื้อไทย + ♯ + จังหวะ). DOCX ได้เกือบครบ **ยกเว้นออกเทฟ**
(114/120 เพลงฝังจุดออกเทฟเป็น "รูปวงกลมลอย" — พิกัด absolute เช่น `396240 340995` = ที่ PM เคยเห็นเป็น noise —
ผูกกับตัวโน้ตไม่ได้ถ้าไม่ render). **ทางออก = อ่านทำนองจากเรขาคณิตของ PDF (pdfplumber) + เนื้อไทยจาก DOCX**
เพราะ PDF เป็นที่เดียวที่ "วาดจุดออกเทฟไว้ตรงตำแหน่ง" อ่านได้แม่น

## 2. เครื่องมือ + วิธี (parser)

- ไฟล์: `tools/parse_song.py` (เข้า repo แล้ว) · run ด้วย **Python 3.14-64** (`...\pythoncore-3.14-64\python.exe`)
- dependency: `pdfplumber` (เรขาคณิต PDF) · `python-docx` (เนื้อ Word) · `pythainlp` engine `dict` (ตัดพยางค์ไทย) — ครบบนเครื่องนี้แล้ว
- **ทำนองจาก PDF:** ตัวเลขโน้ต = ตัวอักษร มีพิกัด · จุดออกเทฟ = `curve` เหนือ/ใต้ตัวโน้ต · ขีดเขบ็ต = `rect` เส้นบาง ·
  คอร์ด = แถวตัวอักษรด้านบน จับคู่ตามพิกัด x · **แบ่งห้องด้วยการนับจังหวะ** (barline เป็น glyph ฟอนต์พิเศษ เลยนับ beat ให้ครบ 4 แทน)
- **เนื้อจาก DOCX:** ตัดพยางค์ด้วย pythainlp `dict` (แม่นมาก) · เคารพช่องว่าง/ขีด `-` ที่ผู้พิมพ์ใส่เอง (เช่น `ส-ถิต`) · ตัดลูกน้ำทิ้ง
- **โมเดล v2:** ทำนองแต่ละบรรทัด = stanza · **ทำนองซ้ำ → รวมเป็น stanza เดียว** (เพลง 1: บรรทัด 1/3/7 = stanza A ใช้ซ้ำ 3 รอบ) ·
  arrangement = ลำดับเล่นจริง แต่ละแถวผูก stanza + พยางค์ของข้อนั้น + label (จับ `(รับ)` / `1.` อัตโนมัติ)
- **ปรัชญา = seed:** ทุกจุดที่ไม่ชัวร์ → ใส่ warning ไม่เดามั่ว · คนเกลาต่อใน Studio ("Claude หว่าน · พี่เปาเกลา")

## 4. ความเสี่ยง (เรียงตามผลกระทบ) — เตรียมชนใน Step 2

1. **ออกเทฟจากเรขาคณิต (สูงสุด · 114/120 เพลง)** — ต้องจับคู่ "จุดวงกลม" กับตัวโน้ตด้วยพิกัด x + บน/ล่าง.
   นำร่องแม่นในเพลง 1 แต่ต้อง validate ทั้งเล่ม (เพลงตัวโน้ตแน่น/2 จุด = เสี่ยงพลาด) → มี beat-check จับผิดให้
2. **ตัดพยางค์ไทย + เอื้อน** — pythainlp แม่น ~ตรง แต่ที่ **จำนวนพยางค์ ≠ จำนวนโน้ต** (เพลง 1: 14 vs 15) = จุดเอื้อน/ตัดพลาด → ใส่ warning ทุกจุด คนเกลา slur ใน Studio
3. **ย้อนกลับ/จบ 2 แบบ (repeat + volta)** — เพลง 100 มี `‖: … :‖` + จบ 2 แบบ (1./2.) + **3 ข้อซ้อนบนทำนองเดียว**. parser flag ไว้ ยังไม่แปลง repeat อัตโนมัติ (163 `║`, 139 `:` ทั้งเล่ม)
4. **หลายข้อบนทำนองเดียว → arrangement หลายแถว** — เพลง 100 = 2-3 ข้อใต้ทำนองเดียว (นี่คือเหตุผลที่เลือก v2). parser รับได้ แต่ต้องตรวจว่าข้อ 2/3 พยางค์เท่าทำนองไหม
5. **จับคู่คอร์ด → โน้ต** — ใช้พิกัด x ค่อนข้างแม่น แต่คอร์ดกลางห้องเสี่ยงเลื่อน → ตรวจสายตา
6. **movable-do (คีย์)** — บางเพลง header เขียน `(C Major)` แต่คอร์ดเป็น B major (เพลง 10). ตัวเลข = do อ้างอิง C, คอร์ด = คีย์จริง B → **ต้องเคาะว่าเก็บ `key` เป็นอันไหน** (parser flag ให้แล้ว)
7. **ห้องหัว/ท้ายไม่เต็ม (pickup)** — ห้องจบวลี 3 หรือ 1 จังหวะ (ทำนองลากข้ามบรรทัด) = ปกติของ source ไม่ใช่บั๊ก → flag เฉย ๆ

**แผน backup (Step 2 จะทำ):** ก่อนทับของเดิม — DA เขียน SQL `select ... into backup_songs` / export JSON เพลงเดิมทั้งหมดใน Supabase → P'Aim run เก็บก่อน upsert (DA เขียน DB ไม่ได้ · publishable key = read)

## 7. อัปเกรด parser (Step 2)
- **แบ่งห้องจาก barline จริง** (glyph `U+F0BD`) ไม่ใช่การนับจังหวะ → จับ pickup/ห้องไม่ปกติได้ถูก (fallback นับจังหวะเฉพาะบรรทัดที่ไม่พิมพ์ barline · flag)
- **key = รากคอร์ดตัวสุดท้าย** (เพลงลงจบที่ tonic) · header key ต่างเมื่อไหร่ → เก็บ `content.headerKey` + warning
- กรอง **footer หน้า/หน้าเนื้อล้วน/คำแปล**: บังคับว่าแถวโน้ตต้องมีแถวคอร์ดอยู่บน (กัน "100.1" ถูกอ่านเป็นโน้ต)
- **สร้าง SQL upsert อัตโนมัติ** ต่อเพลง (มี `-- REVIEW` ยกทุก warning ไว้หัวไฟล์ให้พี่เปาเกลา)

## 8. Risk log (เรียงตามผลกระทบ)
1. **repeat + volta + หลายข้อ (เพลง 100)** — parser flag แต่ยังไม่แปลง `‖: :‖`/จบ2แบบ อัตโนมัติ · หลายข้อบนทำนองเดียวทำ arrangement row บวม (flag "48 พยางค์ vs 12 โน้ต") → **ต้องเกลามือใน Studio** · เพลงพวกนี้ ~15-20% ของเล่ม (163 `║`, 139 `:`)
2. **เอื้อน/ตัดพยางค์** — pythainlp แม่น แต่จุดที่พยางค์≠โน้ต (off-by-1/2) = เอื้อน → flag ทุกจุด (sylMis ในแอป=0 เพราะ pad ช่องว่าง แต่ warning ชี้จุดให้เกลา slur)
3. **octave จาก geometry** — นำร่องแม่น แต่โน้ตแน่น/2 จุด เสี่ยง → มี beat-check + สายตาช่วย
4. **movable-do key** — เก็บคีย์คอร์ด (ถูกสำหรับ transpose) แต่ถ้าเพลงลงจบไม่ใช่ tonic จะเดาคีย์พลาด → flag เมื่อ header ต่าง
5. **บาง system ไม่มี barline พิมพ์** (เพลง 100 บางบรรทัด) → fallback นับจังหวะ + flag
6. **pairing PDF system ↔ DOCX lyric** ต้องจำนวนตรงกัน → flag เมื่อไม่ตรง
