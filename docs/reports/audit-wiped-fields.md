## 1. ทำไมผมตอบเองไม่ได้ (ไม่ใช่ไม่ได้ลอง)

ตาราง `song_revisions` (ประวัติการแก้) **อ่านไม่ได้ด้วยกุญแจสาธารณะที่อยู่ในโค้ดแอป** —
ลองแล้ว เซิร์ฟเวอร์ตอบ 200 แต่คืนศูนย์แถว: `Content-Range: */0` (เทียบกับตาราง `songs` ที่คืน `0-0/164`)
แปลว่า RLS ปิดตารางนี้ไว้จากคนที่ไม่ได้ล็อกอิน ซึ่ง**ถูกต้องแล้ว** (ประวัติใครแก้อะไรไม่ควรเปิดสาธารณะ)

→ ผมจึงเตรียม **คำสั่งอ่านอย่างเดียว** ให้ P'Aim รันแทน แล้วส่งผลกลับมา

⚠️ **อีกข้อจำกัดที่ต้องบอก:** ตัวเลข 164 เพลงที่ผมนับได้ อาจไม่ใช่ทั้งหมด — `db/005` ตั้ง RLS ให้คนทั่วไป
เห็นเฉพาะเพลงที่ `verified = true` และผมตรวจแล้วพบว่าที่มองเห็นได้ **164 แถวเป็น verified ทั้งหมด**
ถ้ามีเพลงที่ยังไม่ verified อยู่ ผมจะมองไม่เห็นเลย → ตัวเลขจากคำสั่งของ P'Aim (ซึ่งเห็นทุกแถว) คือตัวจริง

## 2. ตัวแยก "บั๊กล้าง" ออกจาก "คนแก้เอง"

พี่เปาแก้ข้อมูลเองทุกวัน บางเพลงเขาอาจลบเลขเพลงเองตั้งใจ — **ห้ามเหมารวม** ตัวแยกคือ:

| กอง | สัญญาณในประวัติ | แปลว่า |
|---|---|---|
| **A. บั๊ก** | แถวที่ค่าหาย **มีคู่ฝั่งร่างผูก `op_group` เดียวกัน** (ปุ่มอนุมัติเขียนประวัติ 2 แถวพร้อมกันเสมอ) | มาจากปุ่ม "อนุมัติและเผยแพร่" = บั๊กตัวนี้ |
| **B. คนแก้เอง** | ไม่มีคู่ แต่**มีชื่อผู้กด** | แก้เพลงตรงๆ ในหน้าแก้ — อาจตั้งใจลบ ต้องให้คนตัดสิน |
| **C. บอกไม่ได้** | ไม่มีคู่ และ**ไม่มีชื่อผู้กด** | แถวเก่าก่อน `db/004` ตอนนั้นยังไม่บันทึกชื่อ → **แยกไม่ออก ต้องไม่นับรวม 2 กองบน** |

> **จุดที่ผมเกือบพลาด:** ตอนแรกผมใช้ "`op_group` ว่าง = บอกไม่ได้" ซึ่งผิด —
> การแก้เพลงตรงๆ ก็ไม่มี `op_group` เหมือนกัน ทำให้ "คนแก้เอง" ถูกเหมาไปกอง C หมด
> เทสต์จับได้ ตัวแยกจริงคือ **"มีคู่ฝั่งร่างไหม"** ไม่ใช่ "`op_group` ว่างไหม"

**ข้อจำกัดที่เหลืออยู่ ต้องพูดตรงๆ:** ถ้าผู้ตรวจ **ตั้งใจลบธีมแล้วกดอนุมัติ** จะตกอยู่ในกอง A เหมือนกัน
เพราะประวัติมองไม่เห็นเจตนา — คำสั่งที่ 2 จึงแสดง **ชื่อคนกด + วันเวลา** ไว้ให้คนตัดสินเอง ไม่ตัดสินแทน

## 3. 4 คำถามที่ต้องตอบ กับคำสั่งที่ตอบให้

| คำถามของ PM | ตอบด้วย |
|---|---|
| 1. โดนล้างจริงกี่เพลง / คนแก้กี่เพลง / บอกไม่ได้กี่เพลง | **คำสั่งที่ 1** — แยก 3 กอง พร้อมนับแยกรายฟิลด์ |
| 2. เกิดก่อนหรือหลัง 20 ก.ค. (วัน B108 ฝั่งแอปขึ้น live) | **คำสั่งที่ 1** คอลัมน์ `ก่อน_20ก_ค` / `ตั้งแต่_20ก_ค` — ถ้าหยุดหลัง 20 ก.ค. = ประตูที่ 1 ได้ผลจริง เหลือแต่ประตู RPC |
| 3. ยังเกิดอยู่ไหมวันนี้ | **คำสั่งที่ 1** คอลัมน์ `ใน_24ชม_ล่าสุด` |
| 4. ซ่อมได้ไหม | **คำสั่งที่ 2** คอลัมน์ `ค่าเดิม_*` — ประวัติเก็บค่าก่อนแก้ไว้เต็ม ถ้าไม่ว่าง = **ซ่อมได้โดยไม่ต้องเดา ไม่ต้องเทียบหนังสือ** · และกรองเฉพาะเพลงที่ **วันนี้ยังว่างอยู่** (ถ้ามีคนกรอกกลับไปแล้วจะไม่ขึ้น) |

**คำสั่งที่ 3** ตอบอีกมุมที่คำสั่งที่ 1 มองไม่เห็น: เพลงใหม่ที่เผยแพร่ครั้งแรกแล้วค่าจากร่างไม่ติดไปด้วย
(ไม่ใช่ "ถูกลบ" แต่ "ไม่เคยถูกพาไป" — เส้น INSERT)
