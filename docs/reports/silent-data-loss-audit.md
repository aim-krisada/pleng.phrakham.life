ขอบเขต: ทุกเส้นทางที่ข้อมูลผู้ใช้ถูก **แปลง / บันทึก / โหลดกลับ** บนสาขา `studio-shell-redesign`
วิธี: อ่านโค้ดทุกเส้นทาง + **รันของจริง** — ดึงเพลงที่เผยแพร่แล้วทั้ง 160 เพลงจาก Supabase (อ่านอย่างเดียว

## รากของปัญหาเดียวกัน — "ประกอบวัตถุใหม่ทีละฟิลด์"

ไล่ระบุฟิลด์ที่ "รู้จัก" ทีละตัว อะไรที่ไม่ได้ระบุไว้จะหลุดหายโดยไม่มีใครรู้:

### 🔴-2 อนุมัติร่าง = ลบ "ธีม" ของเพลงทิ้ง + ย้ายเพลงไปเล่มอนุชน (B108 ประตูที่สอง)

- **ที่:** [db/006-author-id-fix.sql](db/006-author-id-fix.sql) (และ `db/004` เดิม) ท่อน `update public.songs set … theme = p_song->>'theme'` · [EditorMode.vue:1562](src/components/EditorMode.vue#L1562) `p_song`
- **สถานการณ์จริง:** สมาชิกส่งร่างมาให้ตรวจ → ผู้อนุมัติเปิดร่างนั้น (`loadDraft` → `applyRow`) → กด **อนุมัติและเผยแพร่**

- **ข้อมูลที่หาย:** ธีมของเพลงหายถาวร + หมวดถูกย้ายเป็นอนุชน

- **ผู้ใช้รู้ตัวไหม:** **ไม่รู้** — ข้อความที่ขึ้นคือ "✅ อนุมัติและเผยแพร่แล้ว"
- **สำคัญสำหรับการวางแผนแก้:** ถ้าแก้ B108 ด้วยการ "เพิ่มคอลัมน์ลง `song_drafts`" อย่างเดียว **เส้นทางนี้ยังพังอยู่**
  เพราะ `p_song` ประกอบทีละฟิลด์ และ SQL ยังเขียนทับ `theme` แบบไม่มี `coalesce`
- **สถานะหลักฐาน:** พิสูจน์แล้วจากการอ่าน SQL + โค้ด + นับข้อมูลจริง (ไม่ได้รัน RPC — ห้ามเขียน DB)

### 🟡-5 แถวข้อร้องที่ชี้ทำนองไม่เจอ = เนื้อทั้งข้อหายจากแผ่นเพลงเงียบๆ

- **สถานการณ์:** JSON ที่เขียนมือ/นำเข้า มีข้อร้องชี้ไปยัง stanza id ที่สะกดผิดหรือถูกลบ
- **ข้อมูลที่หาย:** เนื้อทั้งข้อไม่ขึ้นบนแผ่นเพลง ไม่ขึ้นตอนพิมพ์ ไม่มีข้อความเตือน (ข้อมูลยังอยู่ใน JSON — กู้ได้)
- **ความเสี่ยงต่อเนื่อง:** ผู้ใช้เห็นว่าข้อหาย จึงพิมพ์ใหม่ → ได้ข้อซ้ำซ้อนในไฟล์
- **ทำไมยังไม่พังวันนี้:** `removeStanza:590` ลบแถวที่อ้างถึงทิ้งพร้อมกัน จึงไม่เกิดจากการใช้งานปกติ

### 🟡-6 แปลง v1 → v2 ตัดคำที่เกินทิ้ง

- **ที่:** [songModel.js:56](src/lib/songModel.js#L56) `syls[w++] ?? ''` — ถ้าคำมากกว่าช่องโน้ต คำที่เกินหายไป
- **บรรเทาแล้วบางส่วน:** `migrateToV2` คืน `warnings` และหน้าแก้แสดงกล่องเตือน ([EditorMode.vue:2917](src/components/EditorMode.vue#L2917))
  → ผู้ใช้ **เห็น** ว่ามีกี่จุดที่ไม่พอดี
- **ทำไมยังไม่พังวันนี้:** 160/160 เพลงในระบบเป็น v2 หมดแล้ว ไม่มี v1 เหลือ
- **จะระเบิดเมื่อ:** ต่อทาง "อัปโหลด JSON" (ไฟล์ v1 เก่าของผู้ใช้จะวิ่งผ่านทางนี้) หรือย้ายข้อมูลชุดเก่าเข้ามา

### 🟡-7 "✓ ตรวจแล้ว" แสดงค่าเดา ไม่ใช่ค่าจริง ตอนเปิดร่าง

- **ที่:** [EditorMode.vue:1300](src/components/EditorMode.vue#L1300) `verified.value = data.verified ?? false`
- **สถานการณ์:** เปิดร่างมาตรวจ — `song_drafts` ไม่มีคอลัมน์ `verified` → ปุ่มแสดงว่า "ยังไม่ตรวจ" เสมอ

- **ผลกระทบ:** ยังไม่ทำข้อมูลหาย (กดแล้วเขียน `true` ทับ `true`) แต่เป็นรากเดียวกับ B108 —

## ยังไม่พิสูจน์ (ห้ามอ้างว่าตรวจแล้ว)

1. **ไม่ได้ทดสอบในเบราว์เซอร์จริงเลยแม้แต่ข้อเดียว** — ทุกข้อพิสูจน์ด้วยการอ่านโค้ด + รันฟังก์ชันแปลงข้อมูลใน node
   เท่านั้น 🔴-1 (การแก้ในหน้าฝึกร้องหาย) ควรให้ Tester ยืนยันด้วยมือก่อนจ่ายงานแก้
2. **ไม่ได้รัน `approve_and_publish` จริง** — ห้ามเขียน DB จึงยืนยันเฉพาะจาก SQL ต้นฉบับใน `db/006`
   **ยังไม่ได้ตรวจว่า SQL ที่รันอยู่บน Supabase จริงตรงกับไฟล์ในรีโปหรือไม่** (006 มีหมายเหตุว่า "รันหลัง 005 + PM เคาะ")
3. **ไม่ได้อ่านตาราง `song_drafts` ของจริง** (RLS ต้องล็อกอิน) — สรุปคอลัมน์จาก `db/002` เท่านั้น
4. **ไม่ได้จำลอง quota เต็ม / โหมดส่วนตัว** สำหรับ 🟠-3
5. **ไม่ได้ตรวจเส้นทาง `EditorMode` ที่เหลืออีก ~3,500 บรรทัด** (คัดลอก/วาง · undo/redo · แก้เป็นชุด) —
   ตรวจครบเฉพาะเส้นทาง โหลด → แปลง → เซฟ ตามที่ได้รับมอบหมาย
6. `holds` — ยืนยันว่าไม่มีโค้ดใน repo อ่านมัน แต่**ไม่ทราบว่าใครใส่มาและตั้งใจให้ทำอะไร** (ควรถามฝั่งนำเข้าข้อมูล)
