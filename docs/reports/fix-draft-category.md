# B108 — หมวด/เล่ม หาย แล้วถูกเปลี่ยนเป็น "อนุชน" เอง

Branch `fix-draft-category` (base = live round 33 `23f2e19`).
**Phase 2** (กันเขียนทับ) + **Phase 1** (ร่างเก็บหมวดเองได้) อยู่ในรอบเดียวกัน — P'Aim รัน
`db/010-draft-category.sql` แล้ว PM ยืนยันคอลัมน์มีจริงบน DB → บั๊กปิดครบในการ deploy ครั้งเดียว

## อาการที่รายงาน

ผู้เขียนใส่เพลงไว้ที่ **เล่มใหญ่** · พอพี่เปาเปิดขึ้นมาแก้ กลายเป็น **อนุชน** · ถ้ากดเผยแพร่
ต่อ เพลงนั้นก็ถูกย้ายเล่มจริงในฐานข้อมูล (ข้อมูลเสียหายเงียบๆ)

## Root cause — ยืนยันแล้วจากโค้ดจริง

`song_drafts` **ไม่มี** คอลัมน์ `category` และ `theme` เลย (`db/002-draft-review-system.sql`
บรรทัด 40-52) — มีแต่ตาราง `songs`

1. `draftRow()` เก็บแค่ `song_id, number, title_th, title_en, content` → หมวดที่เลือกไว้หายทันที
   ตอนบันทึกร่าง
2. เปิดร่างกลับมา → `loadDraft()` → `applyRow(d)` → `meta.category = data.category ?? 'anuchon'`
   → แถวร่างไม่มีคีย์นี้ → **ตกไปที่ 'อนุชน'**
3. ตอนเผยแพร่ `saveDirect()` เขียน `row.category = meta.category || 'anuchon'` ทับลง `songs`
   → เพลงถูกย้ายเล่มเงียบๆ
4. เส้นทาง **ผู้อนุมัติ** (`approve()`) พังแบบเดียวกัน — ส่ง `category: meta.category || 'anuchon'`
   เข้า RPC `approve_and_publish` · และ RPC ตัวนี้ `coalesce(p_song->>'category', category)`
   (ปลอดภัยถ้าไม่ส่ง) แต่ `theme = p_song->>'theme'` **ไม่ coalesce** → ถ้าส่งค่าเดาไป ธีมหายด้วย

ตัวเลขในฐานข้อมูลวันนี้: anuchon 46 / lem-yai 29 / dek-lek 21 — ใน 46 นั้นบางเพลงน่าจะโดนย้าย
มาด้วยบั๊กนี้ (การไล่ย้อนหลัง = งานแยก ไม่อยู่ในรอบนี้)

## Phase 2 — สิ่งที่แก้ไปแล้ว (ไม่แตะฐานข้อมูล)

ทั้งหมดอยู่ใน `src/components/EditorMode.vue` แกนคือธง **แยกรายช่อง** `categoryKnown` +
`themeKnown`:
**true = ค่าในช่องนั้นมาจากของจริง (แถวเพลงที่เผยแพร่แล้ว / ดึงมาจาก song_id ของร่าง /
คนกดเลือกช่องนั้นเอง) → เขียนลงฐานข้อมูลได้ · false = เป็นแค่ค่าตั้งต้น → ห้ามเขียนทับ**

### กติกา "รู้จริง" (knownness)

`known = มีคีย์ **และ** ค่าไม่เป็น null` — ห้ามใช้แค่ "มีคีย์"
เพราะ `db/010` เพิ่มคอลัมน์ให้ **ทุกแถวร่างที่มีอยู่เดิม** โดยค่าเป็น NULL ทั้งหมด ถ้านับแค่มีคีย์
ร่างเก่าทุกใบจะกลายเป็น "รู้จริง" ทันที แล้วเราก็จะเอา null/ค่าเดา เขียนกลับทับของจริง = บั๊กเดิมกลับมา
ทางประตูหลัง · ร่างเก่าที่ค่าเป็น null → ถือว่ายังไม่รู้ → ไปดึงจากเพลงที่เผยแพร่แล้ว
(`resolveBook()`) · ดึงไม่ได้ → ยังไม่รู้ → **ไม่ส่งคีย์นั้นเลย**

## ช่องที่เคยเหลือ — ปิดแล้ว

เดิม Phase 2 อย่างเดียวยังปิดไม่ได้: เพลงใหม่ที่ยังไม่เคยเผยแพร่ เก็บหมวดข้ามการบันทึกร่างไม่ได้
เพราะ `song_drafts` ไม่มีที่เก็บ · ตอนนี้ `db/010` รันแล้ว + `draftRow()` เขียนจริงแล้ว →
**เขียนเพลงใหม่ → เลือก เล่มใหญ่ → บันทึกร่าง → ปิด → เปิดร่างกลับมา = ยังเป็น เล่มใหญ่**

ที่ปิดครบตอนนี้:
1. เพลงที่เผยแพร่แล้วไม่ถูกย้ายเล่มเอง — ทั้งทางผู้เขียนเผยแพร่เอง และทางผู้อนุมัติ
2. ธีมไม่ถูกล้างเป็น null ตอนอนุมัติ
3. ร่างของเพลงใหม่เก็บหมวดของตัวเองได้
4. ร่างเก่า (ก่อน migration, ค่าเป็น null) ไม่ถูกตีความว่าเป็นการเลือกจริง
5. เพลงที่ตั้งใจไม่จัดเล่ม (category null) ไม่ถูกยัดเข้า อนุชน ตอน re-publish

**ข้อสังเกตเล็กๆ (ไม่ทำข้อมูลเสียหาย):** ถ้าคนตั้งใจเลือก "— ไม่ระบุธีม —" แล้วบันทึกร่าง ค่าที่เก็บ
คือ null ซึ่งอ่านกลับมาเป็น "ยังไม่รู้" → ตอนเปิดร่างของเพลงที่เผยแพร่แล้ว ระบบจะไปดึงธีมเดิมของเพลง
มาแสดงแทน (การล้างธีมยังทำได้ปกติถ้าเผยแพร่ในรอบเดียวกันโดยไม่ปิดร่าง)
