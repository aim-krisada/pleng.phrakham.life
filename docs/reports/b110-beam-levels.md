## 1. สิ่งที่แก้

### `src/lib/notation.js`
- **เพิ่ม `beamLevels(run)`** (ก่อน `beamGroups`) — แปลง 1 คาน (run) เป็น "โครงเส้นรายชั้น":
  ไล่ทีละชั้น `level = 1..max(underlines)` แล้วหาช่วงที่ติดกันของโน้ตที่มี `underlines >= level`
  - ช่วงยาว ≥ 2 ตัว → เส้นเต็มช่วงนั้น
  - ช่วงยาว 1 ตัว → **ขีดสั้น (partial)** · ทิศ `'left'` ถ้ามีตัวก่อนหน้าใน run · ไม่งั้น `'right'`
  - ทำงานได้ทุกความลึก (3 ขีด/เขบ็ต 3 ชั้น ก็เข้าสูตรเดียวกัน)
- **`beamGroups()` → `flush()`** (เดิมบรรทัด 276) เพิ่ม `levels: beamLevels(run)`
  `u2` **ยังอยู่เหมือนเดิม** (ไม่พัง caller/เทสต์เก่า) แต่ **เลิกใช้วาด**

### `src/components/NoteRow.vue`
- `beams` computed → `flatMap` โครง `levels` ออกมาเป็น "1 แท่งต่อ 1 ชั้น" (มี fallback ถ้าไม่มี `levels`)
- `applyBeam()` — geometry ใหม่:
  - หา **baseline** = ขอบล่างของตัวเลข **หลังหักความหนา border-bottom ของตัวมันเอง**
    (`.num.u2` สูงกว่า `.num.u1` 2.5px — เดิมวัดจาก `rect.bottom` ดิบ ทำให้ y ขึ้นกับว่าตัวแรกเป็น u1 หรือ u2)
  - `top = baseline + (level-1) * (1.5 + 1)` · `height = 1.5` เสมอ
    → ชั้น 1 = baseline..+1.5 · ชั้น 2 = +2.5..+4 · ชั้น 3 = +5..+6.5
    **ตรงกับที่ `border-bottom: 4px double` เดิมวาดเป๊ะ** → กลุ่มที่ 1 ชั้นล้วน / 2 ชั้นล้วน หน้าตาไม่เปลี่ยน
  - ขีดสั้น: กว้าง = ความกว้างตัวเลข + `max(4, ครึ่งความกว้างตัวเลข)` ยื่นไปทางที่ระบุ
- template: `<i class="beam">` มี `data-beam-level / -start / -end / -partial` (ใช้ตรวจ/เทสต์ได้)
- CSS: ลบ `.beam.beam-u2` (แท่งหนา 4px แบบ double) ทิ้ง — ทุกแท่งเป็นเส้น 1.5px เท่ากันหมด

**ไม่ได้แตะ:** `midi.js` · `parseNotes` · duration · กฎการตัดกลุ่ม (จังหวะ/พยางค์/triplet/ตัวหยุด/`-`) → เสียงเหมือนเดิม 100%

### ก่อนแก้ (stash โค้ดออกแล้ววัดซ้ำบนหน้าเดียวกัน)
```
ตัวเลข:  3•[u1] l=2.4 r=13.4   |   4[u2] l=24.9 r=36.0   |   5[u0] l=40.7 r=51.7
คาน:     1 แท่ง  class="beam beam-u2"  l=2.4  r=36.0  top=19.70  h=4.00
```
→ แท่งคู่หนา 4px **คลุมตั้งแต่ 2.4 ถึง 36** = คลุมตัว `3.` (1 ชั้น) ด้วย → **ตัว 1 ชั้นโชว์ 2 เส้น = บั๊กที่พี่เปาเจอ**

### หลังแก้
```
ตัวเลข:  3•[u1] l=2.4 r=13.4   |   4[u2] l=24.9 r=36.0   |   5[u0] l=40.7 r=51.7
คาน L1 (เต็ม run)      l=2.4   r=36.0  top=22.70  h=1.50
คาน L2 (partial,left)  l=19.4  r=36.0  top=25.20  h=1.50
```
**นับเส้นที่พาดกึ่งกลางแต่ละตัว:**

| ตัวโน้ต | ขีดที่พิมพ์ | เส้นที่วาดจริง | ผล |
|---|---|---|---|
| `3•` | u1 (1 ชั้น) | **L1 อย่างเดียว = 1 เส้น** | ✅ |
| `4`  | u2 (2 ชั้น) | **L1 + L2 = 2 เส้น** | ✅ |
| `5`  | u0 (ไม่มีขีด) | 0 เส้น | ✅ |

ตรวจเพิ่ม:
- **baseline ตรงกันทั้งสองตัว** — `3•` bottom 23.7 − border 1px = 22.7 · `4` bottom 26.7 − border 4px = 22.7 → L1 นอนที่ 22.7 พอดี (ไม่เหลื่อม)
- **ระยะชั้น 2** = 25.2 − 22.7 = 2.5px ตรงกับที่ `4px double` เดิมวาด
- **ขีดสั้นยื่นซ้าย** — ตัว `4` left = 24.9 · L2 left = 19.4 → ยื่นเข้าหาตัว `3•` 5.55px (= ครึ่งความกว้างตัวเลข) ✅
- **ทดสอบซ้ำที่ font scale 2.2×** (ปุ่ม Aa) — ยังได้ `3•=1 เส้น · 4=2 เส้น` เหมือนเดิม
- คานโผล่เองตามปกติหลังโหลด 2.5 วิ โดยไม่ต้องยิง resize (ไม่ได้ทำ timing พัง)

## 5. ความเสี่ยง / จุดที่อาจ regress

1. **จำนวน DOM element เพิ่ม** — คานที่มี 2 ชั้นเดิมใช้ `<i>` 1 ตัว ตอนนี้ใช้ 2 ตัว
   → ทุกตัวมี `ResizeObserver` ของตัวเอง แผ่นเพลงยาวๆ มี overhead เพิ่มนิดหน่อย (516 คาน → ~530 แท่ง) ไม่น่ามีผลจริง แต่ยังไม่ได้วัด
2. **ตำแหน่ง y ขยับ 2.5px สำหรับคาน 2 ชั้น** — เดิมคานคู่ถูกวาดที่ `rect.bottom − 4` โดย `rect` มาจาก**ตัวแรก**ของ run
   ถ้าตัวแรกเป็น u1 (บั๊ก) คานจะลอยสูงกว่าที่ควร 2.5px · ตอนนี้ยึด baseline จริงเสมอ
   → **กลุ่ม 2 ชั้นล้วนไม่ขยับ** (ตัวแรกเป็น u2 อยู่แล้ว) แต่ถ้าเจอกลุ่มผสมที่เคยเห็นจนชิน หน้าตาจะ "ต่ำลง" เล็กน้อย — **เป็นการแก้ให้ถูก ไม่ใช่ regression**
3. **`getComputedStyle` ต่อคาน 1 ครั้งใน `applyBeam`** — เรียกทุกครั้งที่ resize/print
   ถ้าพบว่าช้าบนเครื่องอ่อน ให้ส่ง `underlines` ของตัวแรกมาทาง binding แทน (แก้ง่าย ไม่กระทบตรรกะ)
4. **fallback jsdom/ไม่มี layout ยังเหมือนเดิม** — `!a.width` → `display:none` แล้วขีดราย-ตัว (`.num.u1/.u2`) โผล่แทน ซึ่ง**ถูกอยู่แล้ว**ตั้งแต่ต้น
5. **ทิศขีดสั้นของตัว 2 ชั้นที่อยู่กลาง run** (เช่น `5_ 6__ 7_`) ตอนนี้เลือก `'left'` เสมอ
   ตามมาตรฐาน engraving เคสนี้กำกวม — **ถ้า P'Aim/พี่เปาอยากได้อีกทิศ แก้จุดเดียวใน `beamLevels()`**
