# รายงาน — B064 · แก้ speed (BPM) ไม่สะท้อนหน้าฝึกร้องหลัง publish

**Branch:** `wt-b064` · **worktree:** `../pleng-b064` · **port:** 5395 (`--host`)
**LAN (มือถือ):** `http://192.168.1.124:5395/` → เปิดเพลง → แก้ไข → เปลี่ยน BPM → publish → ฝึกร้อง

## บั๊ก (พี่เปาเจอสด)
แก้ speed (BPM) ในหน้าแก้ไข → publish (เซฟลง DB สำเร็จ) → ไปหน้าฝึกร้อง → **ยังเป็น speed เดิม ไม่ใช่ที่แก้**

## Root cause (ยืนยันแล้ว)
ทั้งหน้าแก้ไขและหน้าฝึกร้องอยู่ใน `Studio.vue` shell เดียวกัน (สลับ mode ไม่ใช่ navigate ใหม่). การแก้ BPM ในตัวแก้ไข emit `change` ขึ้นไปที่ `Studio` → `liveSong.content.bpm` อัปเดต → `viewerSong.content.bpm` (prop ของ `SongViewer`) reactive ตามทันที **แล้ว**.

แต่ `SongViewer.vue` เก็บ speed/คีย์ไว้ใน **local ref สแนปช็อต** ที่ init ตอน mount เท่านั้น:
- `tempo = ref(props.song.content.bpm || 92)` (line 59)
- `displayKey = ref(props.song.content.key || 'C')` (line 56)

ทั้งคู่ re-sync **เฉพาะตอน `props.song.number` เปลี่ยน** (watch line ~188 — จงใจตาม DS-A04: คงค่าที่ผู้ฟังเลือกไว้ข้ามการแก้เพลงเดิม). พอแก้ BPM ของเพลงเดิม (number ไม่เปลี่ยน) watch ไม่ยิง → `tempo` ค้างค่าเดิม → หน้าฝึกร้องเล่นความเร็วเก่า. คีย์ (`displayKey`) เป็นบั๊กคลาสเดียวกันเป๊ะ.

`timeSignature` + เนื้อ + โน้ต ไหลผ่าน computed `resolved` (spread `content`) → reactive อยู่แล้ว → refresh ครบ ไม่ใช่บั๊ก.

## สิ่งที่แก้
`src/components/SongViewer.vue` — เพิ่ม 2 watch ที่ยิงเมื่อ **ค่าที่เก็บใน content เปลี่ยนเอง**:
```js
watch(() => props.song?.content?.bpm, (b) => { if (b) tempo.value = b })
watch(() => props.song?.content?.key, (k) => { if (k) displayKey.value = k })
```
- แก้ BPM/คีย์ = เจตนาชัดว่าอยากเปลี่ยน → sync เข้าหน้าฝึกร้อง
- **DS-A04 ยังอยู่ครบ:** แก้เนื้อล้วน (ไม่แตะ bpm/key) → watch ไม่ยิง → ค่าที่ผู้ฟังเลือก (ทรานสโพส/ความเร็ว) ไม่โดนล้าง
- ไม่แตะ watch เดิม (reset เต็มตอนเปลี่ยนเพลง) · ไม่แตะ EditorMode / SongList / SongSheet / songModel
