**ผูกงาน 2 อัน:** `work/ปรับ pl edit ui` (โมเดล/อัลกอริทึม) + `work/editor-ux` (UX หน้าแก้ไขที่คนใช้ทุกวัน)

## 0. สรุปหัวใจ (อ่านอันเดียวพอ)

**สถาปัตยกรรม "สมองสากล + หน้ากาก 2 แบบ" ที่ G กับพี่เอมตกผลึก — ถูกต้อง และ pleng สร้างไปแล้ว ~70% รันจริงบนเว็บ**

G แนะให้ "ยื่น schema ให้โปรแกรมเมอร์ร่าง" — แต่ความจริงคือ **schema มีอยู่แล้ว ไม่ใช่หน้ากระดาษเปล่า** อยู่ที่ `docs/song-model-v2.md` สร้างเสร็จ verify แล้ว live แล้ว
→ **ความเสี่ยงที่ใหญ่ที่สุดคือ ทำเหมือนเริ่มใหม่จากศูนย์** ทั้งที่ควร "ต่อยอดของที่มี" · งานจริงเหลือแค่ 4 ชิ้นที่ยังไม่ได้ทำ (ดู §3)

## 1. G เสนออะไร (จากไฟล์สนทนา)

**"สมองสากล + หน้ากากแปลงกาย" = 3 ส่วน:**
1. **Input (หน้าจอ):** พิมพ์ได้ 2 แบบ — สากล (`:‖` กล่อง `1.` `2.`) หรือแบบโบสถ์คุ้นเคย (`(รับ)`)
2. **Core (หลังบ้าน):** แปลงเป็น**โครงเส้นตรงมาตรฐาน** (MusicXML-like · play-order กางออกเป็น 1:1) ทันที
3. **Display:** แสดงได้ 2 แบบ — สากล หรือ แบบโบสถ์

**"Algorithm Flow" (จุดที่ G เรียกว่าทองคำ):** ตัดเพลงเป็น**บล็อก** (ข้อ1-4 · ท่อนรับ-A-รอบ1/รอบ2/จบ · ท่อนรับ-B...) แล้ว**ดึงมาต่อกันเป็นเส้นตรงตามรอบ** — โปรแกรมไม่ต้องเข้าใจกล่อง `1.`/`2.` ก็แสดงถูก 100%

**G ตอบเรื่องมาตรฐานไว้ถูกแล้ว** (ผมยืนยัน): `:‖` + กล่อง `1.`/`2.` = **มาตรฐานสากลจริง** · การเขียน `(รับ)` แล้วแยกเนื้อไปหน้าอื่น = **ไม่สากล เป็นการจัดหน้าแบบไทย** → เก็บ core เป็นสากล แสดง/พิมพ์แบบโบสถ์ได้ = ถูกทาง

## 2. ของจริงในโค้ด pleng ตอนนี้ (ยืนยันจาก `src/lib/songModel.js`)

**โมเดล v2 = แนวคิดเดียวกับ G เป๊ะ แต่สร้างไปแล้ว:**

| G เรียกมันว่า | ในโค้ด pleng คือ | สถานะ |
|---|---|---|
| บล็อกทำนอง (เขียนครั้งเดียว) | **`stanzas[]`** (ทำนอง A, B… ไม่มีเนื้อ) | ✅ live |
| บล็อกเนื้อ + ลำดับการร้อง | **`arrangement[]`** (แต่ละแถว link stanza + ใส่เนื้อ 1 พยางค์/โน้ต) | ✅ live |
| "Algorithm Flow" ดึงบล็อกต่อเป็นเส้นตรง | **`resolvePlayOrder(content)`** + `resolveContent()` | ✅ live |
| เครื่องหมายย้อนสากล `:‖` กล่อง `1.`/`2.` | **`repeat-start` / `repeat-end` / `volta`** + `expandRepeats()` | ✅ live (เล่นเสียงกระโดดถูก) |
| แปลงเป็นเส้นตรง 1:1 | `expandRepeats()` คืน play order เป็น index จริง (verify แล้ว: `A‖:B C(v1):‖ D(v2)` → `0,1,2,0,1,3,4`) | ✅ live |
| Core มาตรฐานสากล (MusicXML) | — | ❌ **ยังไม่มี** |

**แปลว่า:** pleng ไม่ได้เริ่มจากศูนย์ · "สมองสากล" (core ที่กางลำดับการร้องเป็นเส้นตรง) **ทำงานอยู่แล้ว** ทั้งการแสดงผลและการเล่นเสียง

## 3. gap จริง — งานที่ยังไม่ได้ทำ (นี่คือสิ่งที่ต้องคุยกับ G ว่าจะทำอันไหนก่อน)

1. **D.C. / D.S. / Coda / Fine (กระโดดข้ามท่อน)** — repeat/volta ทำแล้ว แต่การกระโดดแบบ D.C./Segno **ยังไม่ทำ** · มี comment ในโค้ดชี้จุดไว้แล้ว (`songModel.js:186` "they add more cases here; the display pass never changes") = ออกแบบเผื่อไว้ เติม case เพิ่มได้ ไม่ต้องรื้อ
2. **Display แบบ compact สากล** — ตอนนี้ render 2 แบบ: กางเต็ม (follow-along) + พิมพ์ซ้อนข้อ (ประหยัดกระดาษ) · **แบบ compact ที่มี `:‖` กล่อง `1.`/`2.` แบบสากล ยังไม่มี** (นี่คือ "หน้ากากสากล" ที่ G พูดถึง — ฝั่ง display)
3. **`(รับ)` เป็น input shortcut** — B102 มี resolver สัญลักษณ์ play-order แล้ว (`(รับ)`=`***`) แต่ "ปุ่มลัดพิมพ์ท่อนรับแบบโบสถ์" ที่ G เสนอ ยังไม่เป็นชั้น input เต็มตัว
4. **MusicXML export** — "คุยกับโลกได้" (เปิดใน MuseScore) · โมเดล v2 ใกล้พอจะ serialize เป็น MusicXML ได้ แต่ **ตัว exporter ยังไม่มี** = งานเขียน serializer ตรงๆ ไม่ใช่งานรื้อโมเดล

**ข้อควรระวัง (จาก memory `pleng-render-data-gaps`):** ก่อนเคลมว่า v2 รองรับเคสไหน ต้องเช็คว่า **data encode สิ่งที่จะ render จริง** — บางเพลงในคลังอาจยังเป็น v1 (เนื้อแบนราบ) ไม่ใช่ v2 → ต้องดูว่าจะ migrate หรือรองรับทั้งคู่

## 4. ทำไม editor-ux ผูกกับเรื่องนี้ (P'Aim ชี้ถูก)

โมเดล v2 ข้างบน = **"สมอง"** · หน้าแก้ไข (editor) = **"มือ" ที่คนใช้ป้อนสมองนั้นทุกวัน** · บั๊ก editor-ux เกือบทั้งหมดคือความเจ็บปวดของการใช้ editor v2 ที่ซับซ้อน:

| editor-ux bug | ผูกกับโมเดลยังไง |
|---|---|
| **issues4** โครงเพลง collapse=ทำนอง | = panel `arrangement` vs `stanzas` (2 พาเนลที่ v2 สร้าง) UX ยังไม่ลงตัว |
| **issues6** คลิกโน้ต preview→เด้งไปแก้ | = การเดินระหว่าง "ดูผลรวม" (resolved) กับ "แก้ stanza" (source) — คนละ index กัน (โค้ดหมายเหตุไว้เอง) |
| **issue21** พิมพ์ 2 คอร์ดในห้อง | = chord-at-note-box (v2 step 4 ทำแล้วบางส่วน · อาจแค่ต่อให้ครบ) |
| **issues10** ลบทั้งเพลง | = editor จัดการเพลง/ร่าง |
| **issues17** อันไหนต้องตรวจ | = flow ส่งตรวจ (ไม่ผูกโมเดลตรง) |

**นัยเชิงกลยุทธ์:** ถ้ากำลังจะขยับโมเดล (เพิ่ม D.C./display สากล) → **อย่าเพิ่งรื้อ editor-ux ทีละจุดแบบสุ่ม** เพราะ panel เดียวกันจะโดนแก้ซ้ำ · แต่บั๊กที่**ไม่แตะโมเดล** (issues10 ลบเพลง · issue20 แถบติด · issues15.2 แถบดำ) แก้ได้เลยไม่ต้องรอ

## 5. คำตอบตรงๆ ต่อคำถาม architecture ของ G

**Q: บรรทัด 5 เส้น / เจียนพู่ มีมาตรฐานสากลบอกให้ทำแบบนี้ไหม?**
→ มี · `:‖` + volta กล่อง `1.`/`2.` = สากลจริง (G ตอบถูก) · และ **pleng ก็เก็บแบบมาตรฐานนี้เป็น SSOT อยู่แล้ว** (`repeat/volta` + `arrangement`) ส่วน `(รับ)` เป็นแค่ชั้นสะดวก — ตรงกับที่ G เสนอเป๊ะ

**Q: ควรร่าง Database Schema แบบไหน?**
→ **ไม่ต้องร่างใหม่ — มีแล้วที่ `docs/song-model-v2.md`** (stanzas + arrangement + repeat/volta) · ควรให้ G รีวิว/ต่อยอด schema นี้ ไม่ใช่ออกแบบจากศูนย์ · สิ่งที่ต้องเติม schema จริงๆ = field สำหรับ D.C./Segno + flag สำหรับโหมด display สากล

**Q: ตัดเพลงเป็นบล็อกแล้วต่อเป็นเส้นตรง (Algorithm Flow) — ทำได้ไหม?**
→ **ทำอยู่แล้ว** · `resolvePlayOrder()` + `expandRepeats()` คืนลำดับ index จริง verify แล้ว · เคสของ G (ข้อ1-3 ใช้รับ A · ข้อ4 ใช้รับ B แต่จบร่วมกัน) = แค่เรียง `arrangement` ตามนั้น ซึ่งโมเดลรองรับ

**Q: ควรบอกโปรแกรมเมอร์ให้ศึกษาโครงข้อมูลของ MuseScore/MusicXML ไหม?**
→ ควร — **แต่เพื่อเขียน exporter (gap #4)** ไม่ใช่เพื่อออกแบบ core ใหม่ · core ของ pleng ตัดสินใจโครงไปแล้วและใกล้ MusicXML พอจะ map ได้

## ภาคผนวก — สรุป `song-model-v2.md` ให้ G อ่าน (ไม่ต้องเปิดโค้ด)

- **1 เพลง = `stanzas[]` (ทำนอง เขียนครั้งเดียว ไม่มีเนื้อ) + `arrangement[]` (ลำดับการร้อง แต่ละแถว = link stanza + เนื้อ 1 พยางค์/โน้ต attack)**
- แก้ทำนองที่เดียว → ทุกข้อที่ใช้ทำนองนั้นอัปเดตตาม
- 1 พยางค์ = 1 โน้ตที่ตี · เอื้อน = slur (โน้ตที่ถูกโยงไม่กินพยางค์) · `-` = ลากต่อ
- `repeat-start`/`repeat-end`/`volta` = `:‖` และกล่อง `1.`/`2.` มาตรฐาน · `expandRepeats()` กางเป็น play-order จริงตอนเล่น
- รองรับทั้ง v1 (เนื้อแบน เพลงเก่า) และ v2 พร้อมกัน — เพลงเก่าไม่พัง
- **ยังไม่มี:** D.C./D.S./Coda jump · display compact สากล · MusicXML export
