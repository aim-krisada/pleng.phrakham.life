```text
ขอบคุณครับ ผมเอาคำถามชุดก่อนไปคุยกับเจ้าของงานแล้ว เขาตอบกลับมาแบบนี้
รอบนี้ผมอยากได้ "ทางออกที่ดีที่สุดระดับโลก" ไม่ใช่แค่ทางที่พอไปได้
และขอให้คุณค้านผมทุกจุดที่ค้านได้

=== คำตอบ/ข้อสั่งการจากเจ้าของงาน (ยกมาตรงตัว) ===

(1) เรื่องจุด "." ที่ทับซ้อน 4 ความหมาย — ห้ามยอมแพ้
    "เราต้องคิดนอกกรอบ user centric มันต้องทำได้ง่ายและดีได้ อย่าเพิ่งยอมแพ้
     ผมคิดว่า keyboard ง่ายกว่า เวลาทำเพลง ไม่ต้องขยับ mouse ↔ keyboard ไปมา"
    → ทางแก้แบบ Domisol (ทำอ็อกเทฟเป็นปุ่มกด ห้ามพิมพ์จุด) = ไม่เอา
    → ต้องเป็นทางที่ "อยู่บนคีย์บอร์ดล้วน" แต่ยังแยก 4 ความหมายได้ชัด
      (อ็อกเทฟบน / อ็อกเทฟล่าง / จุดประ / staccato)

(2) ต้องรองรับหลายอัตราจังหวะในเพลงเดียว
    ตอนนี้ระบบเก็บเลขจังหวะได้ค่าเดียวต่อเพลง → เพลงที่เปลี่ยนจังหวะกลางเพลง
    ระบบขึ้นเตือน "ห้องนี้จำนวนจังหวะไม่ครบ" ผิดทุกห้องตั้งแต่จุดที่เปลี่ยน

(3) ระบบเตือน (validation) ต้องแม่น
    "แย่กว่าไม่มีคำเตือนเลย เห็นด้วย ระบบต้องดีกว่านี้ ไม่งั้นคนใช้ ignore = useless"

(4) property ของเพลง (คีย์ / เลขจังหวะ / tempo / ฯลฯ) ต้องแก้จากที่เดียวแบบรวมศูนย์
    ของเดิมมีอยู่แล้วแต่อนุญาตให้ออกแบบใหม่ได้เลย

(5) เรื่อง "คลี่การวนร้องเป็นเส้นตรง" เจ้าของงานให้มุมใหม่:
    ในโหมดฝึกร้อง การคลี่ท่อนที่ต้องร้องซ้ำออกมาตรงๆ น่าจะดีกับผู้ใช้มากกว่า
    เพราะผู้ใช้ไม่ต้องเลื่อนจอขึ้นลงข้ามหลายย่อหน้าตามเลขข้อ = ปวดหัว
    → กรอบที่เราจะยึด: "เก็บสัญลักษณ์ซ้ำไว้ในข้อมูล (เพื่อพิมพ์หนังสือให้เหมือนต้นฉบับ)
      + คลี่เป็นเส้นตรงตอนแสดงในโหมดฝึกร้อง" = คนละชั้น ไม่ขัดกัน

(6) ผู้ใช้จริง (คนคีย์เพลงหลัก) ยังเข้าใจว่าระบบ "ไม่มีเครื่องหมายซ้ำเลย"
    ทั้งที่ระบบมีมาสักพักแล้ว → เป็นปัญหาการมองเห็นฟีเจอร์ ไม่ใช่ฟีเจอร์ขาด

=== คำถามรอบ 2 ===

A. (ต่อจากข้อ 1) ในโลกนี้มีรูปแบบการป้อนข้อมูลบนคีย์บอร์ดล้วนแบบไหนบ้าง
   ที่แก้ปัญหา "อักขระตัวเดียวหลายความหมาย" ได้จริง
   เช่น modal input (โหมดเหมือน vim), กดปุ่มเดิมซ้ำเพื่อวนค่า (multi-tap),
   dead key, prefix/postfix ที่ต่างกัน, การกดสองปุ่มพร้อมกัน (chord key),
   หรือการตีความจากบริบทแล้วให้ผู้ใช้แก้ทีหลัง
   → ขอให้ **ฟันธงว่าอันไหนดีที่สุด** สำหรับคนที่พิมพ์เพลงรัวๆ ทั้งวัน
     ไม่ใช่ลิสต์ตัวเลือกเฉยๆ · ให้เหตุผลด้วยหลักการออกแบบ input ที่อ้างอิงได้
     และบอกด้วยว่าทางที่คุณเลือกจะพังตอนไหน

B. ถ้าเราให้พิมพ์บนคีย์บอร์ดล้วน ผู้ใช้จะ "เห็น" ไหมว่าสิ่งที่พิมพ์ไปแปลว่าอะไร
   ควรมีการแสดงผลย้อนกลับ (feedback) แบบไหนถึงจะรู้ทันทีว่าจุดที่เพิ่งพิมพ์
   กลายเป็นอ็อกเทฟหรือจุดประ โดยไม่ต้องละสายตาจากที่พิมพ์

C. (ข้อ 2+4) การเก็บ "คุณสมบัติของเพลง" ที่เปลี่ยนกลางเพลงได้ (เลขจังหวะ คีย์ tempo)
   ควรออกแบบยังไงให้ (ก) แก้จากที่เดียวแบบรวมศูนย์ตามที่เจ้าของงานต้องการ
   และ (ข) ยังเปลี่ยนค่ากลางเพลงได้ — สองข้อนี้ดูขัดกันเอง
   มาตรฐาน MusicXML/MuseScore แก้ปมนี้ยังไง และเราควรลอกโครงไหนมา

D. (ข้อ 3) หลักการออกแบบระบบเตือนที่ไม่ทำให้คน "ปิดหูปิดตา" (alert fatigue)
   ควรเตือนตอนไหน (ทันทีที่พิมพ์ / ตอนออกจากห้อง / ตอนกดบันทึก)
   ควรแยกระดับ "ผิดแน่" กับ "น่าสงสัย" ยังไง
   และกฎข้อไหนที่ไม่ควรเตือนเลยเพราะเดาใจผู้ใช้ไม่ได้

E. (ข้อ 5) ยืนยันกรอบ "เก็บสัญลักษณ์ซ้ำในข้อมูล + คลี่เป็นเส้นตรงตอนแสดงผล" ให้หน่อย
   ถ้าเห็นด้วย ชั้นการแสดงผลควรออกแบบยังไงให้ผู้ใช้ไม่หลง
   (เช่น ต้องบอกไหมว่า "นี่คือรอบที่ 2 ของท่อนรับ", ควรให้สลับดูแบบต้นฉบับได้ไหม,
    ตอนพิมพ์ลงกระดาษควรใช้แบบไหนเป็นค่าเริ่มต้น)
   ถ้าไม่เห็นด้วย บอกว่าพังตรงไหน

F. (ข้อ 6) ฟีเจอร์ที่มีอยู่แต่ผู้ใช้ไม่รู้ว่ามี — หลักการแก้ที่ได้ผลจริงคืออะไร
   และอะไรคือวิธีที่ "ดูเหมือนจะช่วยแต่จริงๆ ไม่ช่วย" (เช่น ทัวร์แนะนำตอนเปิดครั้งแรก)

G. รวมทุกอย่างข้างบน — ถ้าคุณเป็นคนออกแบบระบบนี้ใหม่วันนี้
   โดยยังอยู่ในกรอบที่เจ้าของงานเคาะแล้ว (คีย์บอร์ดเป็นหลัก + ป๊อปอัปติดเคอร์เซอร์)
   คุณจะออกแบบชุดสัญลักษณ์และวิธีป้อนออกมาหน้าตายังไง
   ขอเป็นข้อสรุปที่ฟันธง ไม่ใช่ทางเลือกหลายทาง

=== ข้อกำหนดคำตอบ ===
- ค้านผม อย่าเออออ · ถ้าคุณคิดว่ากรอบที่เจ้าของงานเคาะไว้ (คีย์บอร์ด+ป๊อปอัป) ผิด
  ให้บอกพร้อมเหตุผลเป็นข้อๆ — ผมจะบันทึกไว้ให้เขาตัดสินเอง แต่ผมจะไม่เปลี่ยนกรอบเอง
- ถ้าอ้างระบบจริงได้ ให้ระบุชื่อ · ถ้าไม่แน่ใจว่าระบบนั้นทำจริงไหม ให้บอกว่าไม่แน่ใจ
- ตอบเป็นข้อ สั้น ตรง ไม่ต้องสุภาพเกินจำเป็น
```
