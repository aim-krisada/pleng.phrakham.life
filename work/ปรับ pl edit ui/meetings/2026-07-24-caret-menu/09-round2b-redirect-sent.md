ขอบคุณครับ แต่ผมไม่ได้ถามว่าจะเลือกระบบโน้ตไหน — **ระบบล็อกแล้ว** เป็น jianpu มาตรฐาน: จุดใต้เลข = ต่ำ (พิมพ์เป็น `.5`) · จุดบนเลข = สูง (พิมพ์เป็น `5'`) · parser รองรับอยู่แล้ว ผมเปลี่ยนไม่ได้

คำถามผมคือ **ปัญหา UX ของการรับคีย์** จุดเดียว:

ในโปรแกรม `.` มี 2 ความหมายแยกด้วยตำแหน่ง — **จุดก่อนเลข = ต่ำ** (`.5`) · **จุดหลังเลข = โน้ตประจุด** (`5.`) แต่ตอนรับคีย์ พอผู้ใช้กด `.` โปรแกรมยัง**ไม่รู้ว่าจะมีเลขตามมาไหม**

จุดยืนผม: ทำ `.` เป็น **dead-key prefix** — กด `.` แล้วรอ ถ้าคีย์ถัดไปเป็นเลข = ต่ำ · ถ้าไม่ใช่เลข = ประจุดบนโน้ตปัจจุบัน

**ตอบ 3 ข้อนี้พอครับ (สั้น ๆ):**
1. **จุดพังของ dead-key นี้** คืออะไร โดยเฉพาะเคส: ผู้ใช้คลุมโน้ต `3` อยู่ ตั้งใจกด `.` เพื่อทำ `3.` (ประจุด) แล้วเผลอพิมพ์ `5` ต่อ (ตั้งใจจะไปโน้ตใหม่) → โปรแกรมดันได้ `.5` (ต่ำ) แทน — จะ**ตัดสินความกำกวมนี้ยังไงให้ผิดน้อยสุด**?
2. ควร **โชว์ visual** ตอน `.` ค้าง (pending ต่ำ) ไหม หรือเงียบ?
3. **ไม่มี timeout** (one-shot จนกดคีย์ถัดไป) — เห็นด้วยไหม หรือควรมี escape?
