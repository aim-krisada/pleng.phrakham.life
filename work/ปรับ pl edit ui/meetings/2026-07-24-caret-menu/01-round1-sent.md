คุยภาษาไทยนะครับ ผมเป็นสถาปนิกระบบ (SA) ของเว็บโน้ตเพลงคริสตจักรไทย pleng.phrakham.life — หน้าแก้เพลงเป็น **inline WYSIWYG editor** (แก้บนแผ่นเพลงจริง เหมือน Google Docs โหมดแก้) โน้ตเป็นตัวเลข jianpu (0-7)

**ผมมีจุดยืนมาแล้ว อยากให้คุณค้านเฉพาะจุดที่ค้านได้จริง** · ถ้าอ้างมาตรฐาน/พฤติกรรมโปรแกรมอื่น **ขอ URL + ชื่อ element/ปุ่มที่ผมเปิดตรวจเองได้** (รอบก่อน ๆ เคยอ้าง MuseScore ผิดเวอร์ชันแล้วต้องถอน ผมจึงตรวจทุกข้อ · การติดป้าย "ไม่ชัวร์" ไม่ได้ทำให้ข้อความผิดกลายเป็นถูก) · ถ้าเป็นความเห็นจากประสบการณ์ ให้เขียนกำกับว่าเป็นความเห็น

## บริบทโมเดลเคอร์เซอร์ปัจจุบัน (ต้องเข้าใจก่อน)
- เคอร์เซอร์วันนี้ = **"เลือกโน้ต" (block cursor)** — โน้ต 1 ตัวถูกไฮไลต์ · ลูกศร ←→ เดินทีละโน้ต · โน้ตแต่ละตัวมี "หน่วยเนื้อร้อง" ใต้มัน (↑↓ สลับโน้ต↔เนื้อ)
- **พิมพ์เลข = ทับ (overwrite) ที่โน้ตที่เลือก เป็นค่า default** (ทีมล็อกไว้: "กด 1 = ใส่ 1 ตรงที่อยู่" เพราะงานที่ทำบ่อยสุด = แก้พิตช์โน้ตที่ผิด) · มีปุ่ม Insert สลับเป็นโหมด "แทรก" (ริปเปิลดันขวา)
- คำสั่งที่ทำ **กับโน้ตตัวหนึ่ง**: ชาร์ป/แฟลต (# b n) · เขบ็ต/จุด/โยง/เฟอร์มาตา (_ . ~ ^) · ขึ้นอ็อกเทฟ (') · ใส่คอร์ด · ทำเป็นตัวหยุด (พิมพ์ 0) — ทั้งหมดต้องมี "โน้ตเป้าหมาย" ชัด
- ปัจจุบัน **Delete = ทำโน้ตที่เลือกเป็นตัวหยุด (rest) อยู่กับที่** · **Backspace = ลบทั้ง cell (ดึงโน้ตหลังมาชิด)**
- engine ข้อมูลรองรับ "แทรกก่อน/หลังโน้ต" อยู่แล้ว (มี flag before=true/false)

## ปัญหาที่ผู้ใช้จริง (พี่เปา · คนคีย์เพลงมืออาชีพ เดสก์ท็อปทุกวัน) เจอ
1. **แทรกโน้ตก่อนตัวแรกไม่ได้** — ในห้องที่มี "2 5 2" เขาอยากแทรก "5" ไว้**หน้าตัว 2 ตัวแรก** แต่โมเดล block ไม่มีตำแหน่ง "หน้าโน้ตตัวแรก" (ตำแหน่งซ้ายสุดที่เลือกได้คือตัว 2 เอง)
2. **Del/Backspace รู้สึกกลับด้าน** — เขาขอให้สลับ Del กับ Backspace

## จุดยืนผม (PM วินิจฉัยว่าต้นตอ = โมเดล block ควรเป็น "caret ระหว่างโน้ต" แบบ text) — ขอให้ค้าน
**Design ที่ผมจะเขียนลงสเปก:**
- เปลี่ยนเป็น **caret ระหว่างโน้ต (between-notes caret)** — โน้ต N ตัว มีช่องหยุด N+1 ช่อง (g0 = หน้าตัวแรก … gN = หลังตัวสุดท้าย) · วาดเป็นแท่งกะพริบในช่องว่าง → "หน้าโน้ตตัวแรก" กลายเป็นตำแหน่งจริงที่เห็นได้ (แก้ปัญหา 1)
- **แต่ยังต้องมี "โน้ตที่ active" ให้คำสั่งกับโน้ต** (ชาร์ป/มาร์ค/คอร์ด/rest): active = **โน้ตทางซ้ายของ caret** (ถ้า caret อยู่ g0 = โน้ตทางขวา) · โน้ต active ไฮไลต์ให้เห็นเป้า → เท่ากับ **selection + caret อยู่ด้วยกัน** (โน้ตไฮไลต์ + แท่ง caret ที่ขอบ)
- **พิมพ์เลข = ยังทับ (overwrite) โน้ต active เป็น default** (รักษาการล็อกของทีม — งานแก้พิตช์ที่บ่อยสุดยังกด 1 ทีเดียว) · **โหมด Insert = แทรกที่ช่อง caret** (ที่ g0 = แทรกหน้าตัวแรก → แก้ปัญหา 1)
- **Backspace = ลบโน้ต "ทางซ้าย" ของ caret + caret ถอยซ้าย 1 ช่อง · Delete = ลบโน้ต "ทางขวา" ของ caret + caret อยู่กับที่** (ตรงตามธรรมเนียม text editor สากล — Backspace ลบตัวก่อน caret, Delete ลบตัวที่/หลัง caret) → ได้สิ่งที่พี่เปาอยากได้ **โดยไม่ต้องสลับปุ่มมั่ว ๆ**
- "ทำเป็นตัวหยุด (rest)" ที่เคยผูกกับ Delete → ย้ายไป **พิมพ์เลข 0** (ทับ active) ซึ่งทำได้อยู่แล้ว จึงไม่เสียความสามารถ

## คำถามที่อยากได้คำตอบเป๊ะ
1. **โมเดลไฮบริด "caret ระหว่างโน้ต + active = โน้ตซ้าย + พิมพ์=ทับ active / Insert=แทรกที่ช่อง"** — มันcoherentไหม หรือมี **โมเดลมาตรฐานที่สะอาดกว่า**? โปรแกรมโน้ตจริง (MuseScore/Finale/Dorico) ตอน note-input จัดการ **"แทรกก่อนโน้ตตัวแรก"** และ **ทิศทางการลบ (ก่อน/หลัง caret)** ยังไง — **ขอ URL handbook ที่ระบุ element/ปุ่มจริง** (อย่าเดาเวอร์ชัน)
2. **overwrite-เป็น-default (แก้พิตช์เร็ว) ปะทะ text-caret ที่ปกติ "พิมพ์=แทรก"** — การให้ "พิมพ์=ทับโน้ตซ้าย" ทั้งที่ caret เป็นช่องว่าง จะ**สับสน**ไหม? มี editor จริงที่ทำ **selection+caret อยู่ด้วยกัน** (ทับ selection แต่ caret แสดงตำแหน่งแทรก) ให้อ้างอิงไหม? หรือคุณคิดว่าควร**พลิกเป็น insert-default** ไปเลย (แล้วยอมให้ "แก้พิตช์" แพงขึ้น)? ฟันธงมา + เหตุผล
3. **Backspace=ลบซ้าย / Delete=ลบขวา** — ผมยึด Wikipedia "Backspace deletes the character before the cursor · Delete deletes text at or following the cursor" · มี **โปรแกรมโน้ต** ที่จงใจทำต่างจากนี้ (แล้วมีเหตุผลดี) ไหม?
4. **มือถือ:** จะให้ **แตะเพื่อวาง caret ในช่องระหว่างโน้ต** ได้ยังไงให้โดน (ช่องว่างมันบาง) · แตะโดนโน้ต = active โน้ตนั้น แต่จะวาง caret หน้า/หลังยังไง · pitfall เรื่องแป้นพิมพ์จอบัง caret/ป๊อปอัป (ผมวัด visualViewport อยู่แล้ว) มีอะไรอีก?
5. **จุดตายตอนใช้ทั้งวัน** (ไม่ใช่ตอนลองครั้งแรก) ที่ผมมองไม่เห็น — โดยเฉพาะพฤติกรรมที่ขอบห้อง (bar line) และปลายบรรทัด: caret ที่ท้ายห้อง/ท้ายบรรทัด กด Insert หรือ Del ควรทำอะไร?

## เรื่องรอง (Task A — เมนู) ขอความเห็นสั้น ๆ พอ
ย้าย "เปิดเพลงอื่น" (ค้นหา+เลือกเพลง) จากปุ่มเด่นบนแถบ ไปซ่อนในเมนู **⋮ เพิ่มเติม (overflow)** · แล้วเปลี่ยน search ให้ใช้ engine จริง (จับโน้ต "5561"↔"5 5 6 1" · fuzzy พิมพ์ผิด · เลขในเล่มหนังสือ) แทน substring ธรรมดา
- ซ่อน "ค้นหาเพลง" ไว้ใน overflow menu จะ **discoverability แย่ลง**ไหม สำหรับคนที่ต้องสลับเพลงบ่อย? มี pattern มาตรฐาน (Material/HIG) ว่าอะไรควร/ไม่ควรอยู่ใน overflow?
