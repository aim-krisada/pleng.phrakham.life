## บริบทหน้าแก้เพลง (ที่ต้องรู้)

เป็น **inline WYSIWYG editor** — แก้บนแผ่นเพลงจริง ไม่มีกล่องแยก (เหมือน Google Docs โหมดแก้)
โน้ตเป็นตัวเลข (jianpu) · เคอร์เซอร์ = โน้ต/พยางค์ที่เลือก ( index ภายใน · บนจอคือ cell ที่ไฮไลต์) ·
เดินเคอร์เซอร์ด้วยลูกศร: `←→` = โน้ตก่อน/ถัดไป · `↑↓` = สลับโน้ต↔เนื้อ (หรือข้ามบรรทัดด้วย Ctrl) ·
`space` = หน่วยถัดไป · desktop พิมพ์คีย์บอร์ดล้วน · มือถือมีแถบปุ่มลอยเหนือแป้น (visualViewport)
มีสถานะ "บันทึกแล้ว/ยังไม่บันทึก" + ปุ่ม "เสร็จ" ออกจากโหมดแก้ (ยืนยันถ้ายังไม่บันทึก)

**ที่ล็อกไว้:** พื้นผิวเดียว · ไม่มีโหมดซ้อน · touch ≠ pointer · WCAG 2.2 AA (เป้ากด ≥24px) ·
⛔ เครื่องผู้ใช้เป็นโน้ตบุ๊กจอสัมผัสที่รายงาน `hover:none` ทั้งที่เสียบเมาส์ → ห้ามผูกอะไรกับ hover/tooltip

## 3 เรื่องที่อยากให้ค้าน (ผู้ใช้เจอจริงตอนใช้พรีวิว)

### 1. Auto-scroll เคอร์เซอร์เข้าจอ
**ปัญหา:** กด `↑↓` เดินเคอร์เซอร์ในเพลงยาว → cell ที่กำลังแก้หลุดออกนอกจอ ผู้ใช้ต้องเลื่อนมือเอง
**จุดยืนผม:** ทุกครั้งที่เคอร์เซอร์ย้าย (จาก keyboard nav **และ** ตอนแก้/แทรก) → `scrollIntoView({block:'nearest'})`
ที่ cell ที่เลือก · เผื่อ `scroll-margin` กันโดนบัง 2 อย่าง: (บน) แถบเครื่องมือ/สถานะบันทึก · (ล่างมือถือ) แป้นพิมพ์จอ
**คำถาม:**
- `block:'nearest'` เทียบกับ `'center'` — อันไหนดีกว่าสำหรับ**พิมพ์รัว** (center เด้งเยอะเกินจนเวียนหัวไหม)?
- **smooth vs instant** ตอนพิมพ์รัว — smooth มีคิว animation ค้างถ้าเคอร์เซอร์ขยับถี่ ๆ ไหม · มาตรฐาน/ข้อควรระวังว่าไง?
- มี edge case อะไรที่ `scrollIntoView` **เงียบ/ไม่ทำงาน**บนเบราว์เซอร์จริง (ไม่ใช่ iframe/pane) ที่ผมต้องกันไว้?

### 2. ซ่อนแท็บโหมดบนสุด (ฝึกร้อง/แผ่นเพลง/แก้ไข) ตอนอยู่ในโหมดแก้ inline
**ปัญหา:** ระหว่างแก้ แท็บ 3 อันบนสุดไม่จำเป็น = รกจอ (พี่เอมอยากให้หายไปเลยตอนแก้)
**จุดยืนผม:** ตอนเข้าโหมดแก้ inline → **ซ่อนแท็บทั้งแถบ** · ออกจากโหมดแก้ด้วยปุ่ม **"เสร็จ"** ที่มีอยู่แล้ว
(บนแถบสถานะบันทึก) · พอออก → แท็บกลับมา
**คำถาม:**
- การซ่อน navigation ทั้งแถบตอนเข้าโหมด focus (เหลือทางออกทางเดียวคือ "เสร็จ") — มีหลักหรือ pitfall อะไร?
  (ผมนึกถึง "ทางออกต้องชัดเสมอ" ของ Nielsen #3 User control and freedom — ปุ่มเสร็จพอไหม หรือต้องมี Esc ด้วย?)
- ควร **animate** การซ่อน/แสดงแท็บ (slide/fade) หรือ **หายทันที**? อันไหนทำให้ผู้ใช้ไม่งงว่าแท็บหายไปไหน?
- ระวังอะไรเรื่อง **layout shift** ตอนแท็บหาย (เนื้อเลื่อนขึ้นกระตุกไหม · เคอร์เซอร์ที่กำลังแก้เด้งไหม)?

### 3. 🎯 คอร์ด = พิมพ์/แก้ตรงจุดที่กด เหมือนโน้ต (ไม่ใช่ไปกดในแถบไกล ๆ)
**ปัญหา:** ตอนนี้ใส่คอร์ดต้องไปกดปุ่ม "คอร์ด" ในแถบเครื่องมือ แล้วเปิดกล่อง — พี่เอมบอกว่า "ขึ้น ๆ ลง ๆ หลายที"
เสียจังหวะ (แถบอยู่ไกลจากตำแหน่งที่กำลังทำงาน)
**จุดยืนผม (ทีมเคาะแล้วว่าเห็นด้วย):** ทำ **chord-at-cursor เป็นทางหลัก** — กดที่โน้ต → ป๊อปอัปเล็กเด้งตรงนั้น →
พิมพ์ชื่อคอร์ด (autocomplete + ลิสต์คอร์ดในคีย์) → ยืนยัน · แนว MuseScore (กด `Ctrl+K`/เลือกโน้ตแล้วพิมพ์คอร์ดตรงตำแหน่ง)
**คำถามที่ต้องได้คำตอบเป๊ะ (นี่คือส่วนที่ยากสุด):**
- **เลื่อนไปใส่คอร์ดตัวถัดไปยังไง** — MuseScore ใช้ **Space = ไป beat/โน้ตถัดไปแล้วพิมพ์คอร์ดต่อได้เลย** (ไม่ต้องปิดกล่อง)
  · ผมอยากได้แบบนั้น แต่ในตัวแก้ผม `space` = "หน่วยถัดไป" อยู่แล้วตอนแก้โน้ต — **จะชนกันไหม ควรใช้ space หรือ tab?**
  ขอมาตรฐาน/ตัวอย่างจริงว่าโปรแกรมโน้ตเขาผูกปุ่มไหนกับ "คอร์ดถัดไป" (MuseScore/Sibelius/Dorico ทำยังไง · ขอ URL)
- **แก้คอร์ดเดิม vs เพิ่มใหม่** — กดโน้ตที่มีคอร์ดอยู่แล้ว = ป๊อปอัปเด้งพร้อมคอร์ดเดิมให้แก้ · โน้ตที่ยังไม่มี = ช่องว่าง · ถูกไหม?
- **ลบคอร์ด** — พิมพ์ค่าว่างแล้ว Enter? หรือปุ่มลบในป๊อปอัป? อันไหน error-proof กว่า?
- **มือถือ:** ป๊อปอัปคอร์ด **ต้องไม่ถูกแป้นพิมพ์บัง** — ผมจะวัด `visualViewport` แล้ววางป๊อปอัปเหนือแป้นเสมอ · มี pitfall อะไรอีก?
- **autocomplete:** ควรกรองจากคอร์ดในคีย์ก่อน แล้วยอมพิมพ์คอร์ดนอกคีย์ได้ด้วย (maj7, sus, /bass) · จุดที่มักพังตอนทำ chord autocomplete คืออะไร?
