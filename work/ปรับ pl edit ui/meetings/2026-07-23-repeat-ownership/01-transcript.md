# คุย G — เจ้าของสัญลักษณ์ 2 ระดับ + แท็บป๊อปอัป (23 ก.ค. 2026)

## คำถามที่ส่ง

ผมเป็นสถาปนิกระบบของแอปโน้ตเพลงเจียนพู่ (โน้ตตัวเลข) สำหรับคริสตจักรไทย ขอคำตอบอิงมาตรฐานสากลจริง ถ้าไม่มีมาตรฐานกำหนดไว้ให้บอกตรงๆ ว่าไม่มี อย่าเดา

บริบท: โมเดลข้อมูลเราแยก "ทำนอง" (stanza) กับ "การร้องแต่ละรอบ" (arrangement entry ที่ลิงก์ทำนอง + ใส่เนื้อร้องของข้อนั้น) ข้อ 1/2/3 ที่ทำนองเดียวกันจะลิงก์ทำนองเดียวกัน แก้ทำนองที่เดียวได้ผลทุกข้อ

คำถามที่ 1 (สำคัญที่สุด): เรามีเพลงจริงที่ "ทำนองเดียวกันทุกข้อ แต่ข้อ 2 กับ 3 ไม่ต้องวนซ้ำเหมือนข้อ 1" คือความต่างอยู่ที่ "รอบการร้องของข้อนั้น" ไม่ใช่ที่ตัวทำนอง
- ในมาตรฐานสากล (MusicXML, MEI, หรือ engraving practice ตาม Gould/Behind Bars) เขาเก็บ/เขียนความต่างระดับ "ข้อ" แบบนี้อย่างไร มีชื่อเรียกไหม
- เราคิดจะเก็บ override ไว้ที่ arrangement entry (เช่นฟิลด์ชื่อ endingOverride) แต่ชื่อนี้สื่อว่า "เปลี่ยนห้องท้าย" ทั้งที่เคสจริงคือ "ข้อนี้ไม่วนเลย" ซึ่งคนละเรื่อง
- ถามตรงๆ: รูปแบบการวนร้อง/การกระโดดทั้งหมดที่ดนตรีสากลรองรับมีอะไรบ้าง (repeat, 1st/2nd ending, D.C., D.S., Coda, Fine, ฯลฯ) และถ้าจะออกแบบที่เก็บ override ระดับข้อให้ครอบ "ทุกรูปแบบ" ต้องเก็บอะไรบ้าง มีอะไรที่ override ระดับข้อทำไม่ได้โดยธรรมชาติไหม

คำถามที่ 2: การมีสัญลักษณ์ 2 ระดับความเป็นเจ้าของ (ระดับทำนอง = ทุกข้อได้เหมือนกัน กับ ระดับข้อ = ทับเฉพาะข้อนั้น) มาตรฐานหรือซอฟต์แวร์ระดับโลกเรียกแนวคิดนี้ว่าอะไร มีกับดักอะไรที่คนทำมักพลาด

คำถามที่ 3: หน้าจอทำเพลงของเราใช้บนเดสก์ท็อป/แท็บเล็ตเป็นหลัก (มือถือใช้ "ฟัง" เป็นหลัก) ตอนนี้มีป๊อปอัปแป้นสัญลักษณ์ 21 ปุ่ม และจะมีสัญลักษณ์เพิ่มอีกมาก (คำสั่งวนซ้ำ กระโดด เปลี่ยนจังหวะกลางเพลง ฯลฯ) ผู้ใช้จริงบอกว่า "เครื่องมือเยอะ หาไม่เจอ" และเสนอให้แบ่งเป็นแท็บ
- การแบ่งแท็บเป็นคำตอบที่ถูกไหมสำหรับของที่จะโตขึ้นเรื่อยๆ บนจอกว้าง หรือมีรูปแบบที่ดีกว่า
- ข้อจำกัดสำคัญ: ผู้ใช้พิมพ์คีย์บอร์ดเป็นหลัก ไม่อยากยกมือไปเมาส์ และเครื่องเขารายงาน hover:none ทั้งที่มีเมาส์ จึงใช้ tooltip เป็นทางเดียวไม่ได้

## คำตอบเต็มจาก G

คำถามที่ 1: การจัดการความต่างระดับข้อและการวนซ้ำในมาตรฐานสากล
1. มาตรฐานสากลเก็บความต่างระดับ "ข้อ" อย่างไร และมีชื่อเรียกไหม

คำตอบตามมาตรฐานสากล: ใน MusicXML, MEI หรือการพิมพ์โน้ตตามคู่มือของ Elaine Gould (Behind Bars) ไม่มีการแยกเก็บโครงสร้างทำนองของแต่ละข้อออกจากกัน ในระดับข้อมูลแบบซอฟต์แวร์สมัยใหม่ เพราะมาตรฐานเหล่านี้มองโน้ตเป็นลำดับเวลาเชิงเส้น (Linear Score)

วิธีที่สากลจัดการ:

Text/Lyrics Alignment: ตัวโน้ตชุดเดิมถูกเขียนขึ้นมา 1 บรรทัด (หรือ 1 ชุดทำนอง) แล้วนำ เนื้อร้องของแต่ละข้อ (Stanza/Verse) มาวางใต้โน้ตบรรทัดเดียวกัน โดยใช้หมายเลขข้อกำกับ (เช่น 1., 2., 3.)

ความต่างเรื่องการวนซ้ำ (Repeat/Ending): หากข้อ 1 ต้องวนซ้ำแต่ข้อ 2 และ 3 ไม่ต้องวน มาตรฐานสากลถือว่า โครงสร้างจังหวะและการวนซ้ำเป็นส่วนหนึ่งของทำนองหลัก หากมีข้อใดข้อหนึ่งร้องไม่เหมือนโครงสร้างหลัก (เช่น ข้ามทวนซ้ำ) โน้ตสากลจะไม่ใช้การ "override" ที่ข้อมูล แต่จะใช้วิธี เขียนโน้ตบรรทัดแยก (Alternative Staff / Variant) สำหรับข้อนั้นโดยเฉพาะ หรือใช้เครื่องหมายกำกับเนื้อร้องเช่น (ข้อ 2-3 ข้ามการวนซ้ำ) ให้ผู้ร้องสังเกตเอง

2. ชื่อเรียกที่เหมาะสมสำหรับการเก็บ Override

ฟิลด์ endingOverride ไม่ถูกต้องตามความหมายทางดนตรี เพราะ Ending (Voltası / 1st-2nd Ending) หมายถึงห้องเพลงท้ายรอบที่ต่างกัน แต่เคสของคุณคือ "การเปลี่ยนเส้นทางการเดินเพลง (Navigation Flow)"

ชื่อแนะนำในระดับสถาปัตยกรรมข้อมูล:

navigationOverride หรือ flowOverride (สื่อถึงการเปลี่ยนลำดับการวิ่งของเพลง)

repeatStructureOverride (สื่อถึงโครงสร้างการวนซ้ำที่เปลี่ยนไป)

3. รูปแบบการวนซ้ำ/กระโดดทั้งหมดในดนตรีสากล และการออกแบบข้อมูลเพื่อรองรับ "ทุกรูปแบบ"

รูปแบบการกระโดดและวนซ้ำมาตรฐาน (Navigation Commands) มีดังนี้:

Repeat Sign (เครื่องหมายย้อนซ้ำ): จุดคู่พร้อมเส้นกั้นห้อง (|:, :|) วนกลับไปจุดเริ่มต้นหรือเครื่องหมายย้อนกลับเปิด

Volta Brackets (1st/2nd Ending - โวลตา): ช่องทางเลือกที่ 1 และ 2 สำหรับรอบที่วนกลับมาไม่เหมือนกัน

Dal Segno (D.S.): กระโดดกลับไปที่สัญลักษณ์ Segno ($$)

Da Capo (D.C.): กระโดดกลับไปที่จุดเริ่มต้นเพลง (Capo)

Fine: จุดสิ้นสุดเพลงเมื่อวิ่งกลับมาจากการเจอ D.C. หรือ D.S.

Coda / To Coda: การกระโดดข้ามห้องเพลงไปยังท่อนท้าย (Coda section) เพื่อจบเพลง

วิธีออกแบบข้อมูลเพื่อรองรับ Override ระดับข้อให้ครอบคลุม:

การทำ override ระดับข้อ (Stanza/Arrangement Entry) ไม่ควรให้ผู้ใช้ไปแก้โครงสร้างโน้ตทีละตัว แต่ควรเป็นการแก้ "ค่าพารามิเตอร์การควบคุมการเดินเพลง (Control Flow Parameters)" ดังนี้ครับ:

JSON
{
  "stanzaNumber": 2,
  "flowControlOverride": {
    "skipRepeats": true,          // ข้ามเครื่องหมายย้อนซ้ำทั้งหมดในข้อนี้
    "activeVolta": 2,             // บังคับให้วิ่งเข้าช่องทางเลือกที่ 2 ทันที (ไม่ต้องผ่านช่อง 1)
    "customJumpMap": [            // สำหรับเคสซับซ้อน (Optional)
      { "fromMeasure": 12, "action": "skip_to", "targetMeasure": 16 }
    ]
  }
}

ข้อจำกัด: อะไรที่ Override ระดับข้อทำไม่ได้โดยธรรมชาติ?

เปลี่ยนความยาวห้องเพลงหรือเปลี่ยนตัวโน้ตไม่ 1 รูปแบบ: ถ้ารอบที่ 2 ทำนองโน้ตเปลี่ยนไปเลย (ไม่ใช่แค่เรื่องการวนซ้ำ) การ Override จะล้มเหลวโดยสิ้นเชิง เพราะโครงสร้างพิกัดห้องเพลง (Measure Index) จะไม่ตรงกัน เคสดังกล่าว ต้องแตกเป็นทำนองใหม่ (New Stanza/Melody Entity) เท่านั้น ไม่สามารถใช้ระบบลิงก์ทำนองเดิมได้

คำถามที่ 2: แนวคิดข้อมูล 2 ระดับ และหลุมพรางที่มักพลาด

ชื่อเรียกสากล: ในทางซอฟต์แวร์และระบบจัดการเนื้อหา (CMS / Design Systems) เรียกว่า Inheritance & Override Pattern (การสืบทอดคุณสมบัติและการทับค่าเฉพาะจุด) ส่วนในบริบทเครื่องมือทำเพลง/DAW จะใกล้เคียงกับแนวคิด Master/Instance หรือ Template with Local Overrides

กับดักที่คนทำมักพลาด:

Broken Reference (ลิงก์ขาด): เมื่อผู้ใช้แก้ทำนองหลัก (ระดับทำนอง) แล้วลืมคิดผลกระทบกับข้อที่มีการ Override ไว้ ทำให้จังหวะหรือห้องเพลงที่ถูกสั่งกระโดด/ข้าม (เช่น ข้ามไปห้อง 8) กลายเป็นห้องว่างหรือผิดตำแหน่งเพราะห้องเพลงหลักถูกลบหรือเลื่อน

Data Bloat / State Complexity: การเก็บบันทึกสถานะทับซ้อนหลายชั้น (เช่น ทำนองแก้ -> ข้อแก้ -> เนื้อร้องแก้) ทำให้ระบบ Debug ยากมากเวลาโน้ตแสดงผลเพี้ยนในหน้าจอแสดงผลจริง

คำถามที่ 3: การจัดการ UI แป้นสัญลักษณ์สำหรับเดสก์ท็อป/แท็บเล็ต (Keyboard-First + Hover:none)
1. การแบ่งแท็บเป็นคำตอบที่ถูกไหม?

คำตอบ: ถูกครึ่งเดียว และอาจสร้างปัญหาใหม่ การกองปุ่ม 21+ ปุ่มไว้ที่เดียวคือความผิดพลาดเมื่อเครื่องมือโตขึ้น แต่การใช้ "แท็บ" อย่างเดียวบนเดสก์ท็อปจะบังคับให้ผู้ใช้ต้องคลิกสลับแท็บไปมา ซึ่งขัดกับพฤติกรรมของสถาปนิก/นักดนตรีที่ต้องการความเร็วในการทำงาน

2. รูปแบบที่ดีกว่าสำหรับ Desktop/Tablet ที่เน้นคีย์บอร์ดและมีข้อจำกัดเรื่อง Hover

เมื่อผู้ใช้พิมพ์คีย์บอร์ดเป็นหลักและหน้าจอมีข้อจำกัดเรื่อง hover:none (เช่น บนแท็บเล็ต หรือระบบสัมผัสที่ต่อคีย์บอร์ด) รูปแบบ UI ที่ดีที่สุดคือ:

Command Palette (Modal / Floating Search Bar - คล้าย Ctrl+K หรือ VS Code):

ให้ผู้ใช้กดปุ่มลัด (เช่น / หรือ Ctrl+K) แล้วพิมพ์ชื่อสัญลักษณ์ที่ต้องการได้ทันที (เช่น พิมพ์คำว่า "ย้อนซ้ำ" หรือ "Coda" แล้วกด Enter) โดยไม่ต้องละมือจากคีย์บอร์ดไปหาเมาส์คลิกเปิดแท็บ

Contextual Toolbar / Popover ตามตำแหน่งเคอร์เซอร์ (Inline Trigger):

เมื่อพิมพ์โน้ตหรือเคอร์เซอร์อยู่ที่ห้องเพลงไหน ให้แสดงปุ่มลัดเฉพาะกลุ่มที่จำเป็นขึ้นมาใกล้จุดพิมพ์ (Popup ขนาดเล็กที่กดเลือกได้ด้วยปุ่มลูกศร + Enter บนคีย์บอร์ด)

Categorized Keyboard Shortcuts (Shortcuts Mapping):

กำหนดตัวอักษรย่อบนคีย์บอร์ดให้ชัดเจน (เช่น กด R สำหรับ Repeat, กด F สำหรับ Fine) พร้อมแสดงป้ายคีย์บอร์ดกำกับ (Keyboard Badge) ตัวเล็กๆ ไว้ที่ปุ่มแทนการใช้ Tooltip เพราะระบบ hover:none ทำให้ Tooltip แบบปกติแสดงผลลำบากบนจอสัมผัส
