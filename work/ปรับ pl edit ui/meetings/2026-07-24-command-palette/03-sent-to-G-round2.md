# รอบ 2 — ข้อความที่ส่งให้ G (ทั้งดุ้น)

ขอบคุณครับ 3 ข้อที่ผมรับทันทีเพราะเป็นของจริง: (1) `[` ต้องไม่แย่งโฟกัสและต้องปล่อยคีย์ไหลผ่าน
(2) `scrollIntoView({block:'nearest'})` ตอนเลื่อน ↑↓ (3) การเปิด/ปิด palette ห้ามไปโผล่ในประวัติ Undo
— ข้อ 3 นี่ผมมองข้ามจริง ๆ และมันจะพังแบบเงียบ ๆ ด้วย

ตอนนี้ขอไล่ 4 ข้อที่ผมยัง**ไม่รับ**หรือยังไม่ปิด ขอคำตอบตรง ๆ ทีละข้อครับ

## 1. ⚠️ ข้ออ้าง ARIA ของคุณ ผมขอตรวจก่อน

คุณเขียนว่า *"ต้องมี `aria-haspopup="listbox"`"* — ผมเปิด W3C ARIA APG หน้า Combobox Pattern แล้ว
รายการคุณสมบัติที่**บังคับ**คือ `role=combobox` · `aria-expanded` · `aria-controls` · `aria-activedescendant`
ส่วน `aria-haspopup` **ค่าปริยายของ combobox คือ `listbox` อยู่แล้ว**

**คำถามตรง ๆ: มีประโยคไหนใน APG หรือใน ARIA spec ที่บอกว่า *ต้อง* ใส่ `aria-haspopup="listbox"`
สำหรับ popup ที่เป็น listbox? ถ้าไม่มี ขอให้ตอบว่า "ไม่มี ผมพูดเกิน"** ตรง ๆ ได้เลยครับ
ผมต้องเขียนลงสเปกที่ dev จะทำตามจริง จึงรับข้ออ้างมาตรฐานที่เปิดตรวจไม่ได้ไม่ได้
(รอบก่อน ๆ คุณเคยยืนยันเรื่องมาตรฐานผิดแล้วถอน 2 ครั้ง ผมจึงถามซ้ำทุกครั้ง ไม่ได้กวนครับ)

เช่นเดียวกับที่คุณอ้าง *"NN/g Help and Documentation Guidelines"* — ขอชื่อบทความ/URL ที่เปิดได้จริง
ถ้าเป็นความเห็นจากประสบการณ์ บอกว่าเป็นความเห็นก็พอ ผมจะเขียนกำกับให้ถูก

## 2. กติกา Enter ตอน `[` เปิด autocomplete แบบไม่แย่งโฟกัส

ปมคือ **Enter ในตัวแก้ของเราแปลว่า "ไปโน้ตตัวถัดไป" อยู่แล้ว** ถ้า autocomplete ยึด Enter ไปเมื่อไหร่
คนพิมพ์รัวจะเจอ "กด Enter แล้วได้ของที่ไม่ได้ตั้งใจ" ซึ่งแย่กว่าการไม่มี autocomplete

**กติกาที่ผมจะเขียน:** เปิดรายการมาแล้ว **ยังไม่ไฮไลต์แถวไหนเลย** →
Enter/space/อักขระอื่น **ไหลผ่านลงบรรทัดตามปกติทุกตัว** · ต่อเมื่อผู้ใช้กด `↓` (หรือแตะแถว) จึงมีแถวถูกเลือก
และ Enter จึงแปลว่า "เอาแถวนี้" · Esc = ปิดรายการเฉย ๆ ตัวอักษรที่พิมพ์ไว้ยังอยู่

**คำถาม: กติกานี้ปลอดภัยพอสำหรับคนพิมพ์รัวไหม หรือมันทำให้ autocomplete ช้าลงจนไร้ประโยชน์
เทียบกับแบบ VS Code ที่ preselect แถวแรกให้เลย? ถ้าคุณเคยเห็นระบบที่เจอปัญหานี้จริง เล่าเคสให้ฟังหน่อย**

## 3. ค่าที่ต้องกรอก — ผมยังไม่ถอยจาก snippet placeholder ด้วยเหตุผลนี้

ข้อกังวลเรื่อง IME ไทยของคุณ **ใช้ไม่ได้กับเคสของเรา** เพราะค่าที่ต้องกรอกทุกตัวเป็น
**ตัวเลข/ASCII ล้วน** (`[ซ้ำ 2 รอบ]` → "2" · `[3/4]` · `[T:120]` · `[Key:C]`) ซึ่งไม่ผ่าน composition ของ IME ไทย
ส่วนปัญหา Esc 2 รอบ ผมปิดด้วยการนิยามให้ชัด: ตอนนั้น **palette ปิดไปแล้ว** เหลือแค่ข้อความในบรรทัด
ที่มีตัวเลขถูกคลุมอยู่ → Esc = ยกเลิกการคลุม (ข้อความยังอยู่) ไม่มีสถานะซ้อน

**คำถาม: ยังเหลือจุดพังอะไรที่ผมมองไม่เห็นในทางนี้ไหม (ที่ไม่ใช่เรื่อง IME ไทยและไม่ใช่ Esc ซ้อน)?**
ถ้าไม่มี ผมจะเขียนว่า "ถ้าเจอ IME glitch จริงตอนทดสอบ ให้ถอยไปใช้ inline input step ตามที่ G เสนอ"

## 4. คำสั่งที่ "ไม่ลงบรรทัด" — ผมไม่แยกเป็น palette 2 ตัว แต่แยกด้วยโครงในตัวเดียว

คุณเสนอให้แยกขาดตั้งแต่รุ่นแรก ผมเห็นด้วยกับ**เหตุผล** (mental model ต่างกัน · กดผิดแล้วเสียสมาธิ)
แต่ไม่เห็นด้วยกับ**วิธี** เพราะ palette 2 ตัว = ผู้ใช้ต้องจำว่าของที่ต้องการอยู่ประตูไหน = ระบบ 2 ชุด
ซึ่งเป็นสิ่งที่ทีมสั่งห้ามไว้ตรง ๆ

**ทางที่ผมจะเขียน:** ประตูเดียว ลิสต์เดียว แต่
(ก) คำสั่งที่ไม่ลงบรรทัดอยู่**ท้ายสุดเสมอ** ใต้หัวข้อ "คำสั่ง (ไม่ลงบรรทัด)"
(ข) **ไม่มีทางถูกไฮไลต์เอง** — ต้องกด `↓` ลงไปถึงหรือแตะเท่านั้น
(ค) หน้าตาแถวต่างกันชัด (ไม่มีช่องข้อความ mono ท้ายแถว เพราะไม่มีอะไรลงบรรทัด)

**คำถาม: (ก)+(ข)+(ค) ปิดความเสี่ยง "กดผิดแล้วระบบทำงานที่ไม่ได้ตั้งใจ" ที่คุณห่วงได้ครบไหม
ถ้ายังไม่ครบ ขาดอะไร?**

ตอบเป็นข้อ ๆ สั้น ๆ ได้เลยครับ ไม่ต้องเกริ่น
