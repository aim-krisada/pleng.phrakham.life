### G ตอบ (หลังส่วนที่ 1/1) — OK

ยินดีครับพี่ เป็นบรีฟที่ชัดเจนและมีกรอบคิดทาง UX/Architecture ที่แน่นมากครับ รอบนี้ผมจะสแกนหา "จุดตาย" จากประสบการณ์จริงที่ระบบมักจะพังเมื่อเจอกับ User สาย Power User (พิมพ์รัวๆ) และระบุข้ออ้างอิงให้พี่ไปเปิดเช็กต่อได้ทันทีครับ

1. ข้อค้านและจุดตายรายข้อ (พร้อมอ้างอิง)
🔴 ข้อ A: การเปิดด้วย [ อาจสร้าง "การติดขัด (Friction)" ในการพิมพ์รัว

มุมมองค้าน: การเปิด Palette ทันทีเมื่อพิมพ์ [ จะมีปัญหาเมื่อ พี่เปาพิมพ์เร็วแบบไม่มองจอ หากเขากำลังพิมพ์คำสั่งที่เขาจำได้ขึ้นใจอยู่แล้ว เช่น [ซ้ำ 2 รอบ] การที่ Popover เด้งขึ้นมาแบบ Auto-complete อาจเกิดอาการ UI Lag หรือ Enter/Spacebar โดนแย่ง Action (Interrupted Flow)

ข้อผิดพลาดมาตรฐาน ARIA: ตามมาตรฐาน W3C ARIA APG (Combobox Pattern) เมื่อใช้ Pattern นี้ ระบบต้องมั่นใจว่าการกดพิมพ์แบบต่อเนื่องจะไม่ส่งผลกระทบให้ Text Input เสียโฟกัส หรือกิน Keystroke ถัดไป

ทางแก้/ปรับแต่ง: หากต้องการใช้ [ ควรเป็น Trigger แบบ Silent Autocomplete (อยู่ใต้เคอร์เซอร์โดยไม่แย่ง Focus) และหากผู้ใช้กดพิมพ์อักขระถัดไปโดยไม่กด ↑↓/Enter ระบบต้องยอมให้ข้อความไหลลง Document ต่อไปได้ทันทีโดยไม่ Delay

🔴 ข้อ F: Snippet Placeholder (Highlight ตัวเลข) มีจุดตายเมื่อใช้กับ IME ไทย

มุมมองค้าน: การแทรกข้อความแล้วสั่ง Selection (Highlight ตัวเลข) เพื่อให้พิมพ์ทับ มีปัญหากับ Keyboard Input Method Editor (IME) และการคุม Focus บน Browser

จุดพังจริง:

ถ้าผู้ใช้พิมพ์ภาษาไทยทับลงไปบนตัวเลขที่กำลัง Highlight บาง Browser บน OS ไทย (โดยเฉพาะ Windows + Thai Kedmanee) จะเกิดปัญหา Composition String ทำให้ตัวอักขระเพี้ยนหรือ Highlight หลุดก่อนจะพิมพ์จบ

หากผู้ใช้กด Esc เพื่อยกเลิกขณะอยู่ใน state "Highlight ตัวเลข" เคอร์เซอร์จะหลุด หรือต้องกด Esc 2 รอบ (รอบแรกปิด highlight / รอบสองปิด palette)

ข้อแนะนำ: ใช้การ "ถามค่าในแถวของ Palette (Inline Input Step)" ดีกว่าในแง่ Architecture เพราะไม่ต้องวุ่นวายกับการจัดการ DOM/Selection ใน Editor หลัก พอผู้ใช้กด Enter ใน Palette → Palette เปลี่ยนเป็นช่องใส่ตัวเลข → กด Enter อีกที ข้อความสมบูรณ์ค่อยถูก Insert ลง Editor ทีเดียว (Atomic Transaction)

🔴 ข้อ G: จุดที่มักจะตกหล่นใน W3C ARIA APG (Combobox)

พี่ระบุว่าจะใช้ aria-activedescendant ชี้แถวที่เลือก ซึ่งถูกต้องตามมาตรฐาน แต่มี 2 จุดที่ dev มักจะตกหล่นและพังเมื่อใช้งานจริง:

aria-expanded State Management:

อ้างอิง: W3C ARIA Authoring Practices Guide (APG) - Combobox Pattern

ข้อที่ต้องตรวจ: ตัว Element ตัวแม่ (ที่เป็นช่องค้นหา) ต้องมี aria-haspopup="listbox" และสั่ง Toggle aria-expanded="true|false" ให้ถูกต้องทุกครั้งที่ Palette เปิด/ปิด ไม่เช่นนั้น Screen Reader และ Accessibility Tools บน OS จะหลง State

Virtual Scroll / DOM Recycled Issue:

ถ้าผลการค้นหามีรายการเยอะ แล้วใช้ aria-activedescendant ชี้ไปยัง id ของ Element ใน Listbox... ตัว Element นั้น "ต้องโชว์อยู่ใน Viewport จริง" หาก Scroll ลงไปแล้ว Element ที่เลือกโดน Overflow ซ่อนอยู่ Scrollbar ของ Listbox จะไม่ Auto-scroll ตาม เว้นแต่จะเขียน JS สั่ง element.scrollIntoView({ block: 'nearest' }) ควบคู่ไปด้วยเสมอ

2. จุดตายที่มองไม่เห็น (เมื่อคนใช้ทั้งวัน)
1. ปัญหา "เคอร์เซอร์กระโดด" หลังปิด Palette (Focus Restorations)

เมื่อกด Esc เพื่อปิด Palette เคอร์เซอร์ต้องกลับไปที่บรรทัดเดิม พร้อมด้วย State ของ Undo/Redo (History Stack) ที่ไม่เสีย

จุดพัง: การเปิด Palette แล้วยกเลิก (กด Esc โดยไม่ได้เลือกอะไร) บางระบบแอบไป Mutation DOM หรือสร้าง Temporary Range ไว้ ทำให้เมื่อกด Ctrl+Z กลายเป็น Undo การเปิด Palette แทนที่จะเป็น Undo โน้ตเพลงที่พิมพ์ก่อนหน้านี้

2. การแยกประเภทใน ข้อ B (แยกขาด vs ปนกัน)

คำแนะนำ: ควรแยกให้ขาดตั้งแต่รุ่นแรก (ข้อ B)

เหตุผล: คำสั่งประเภท "แทรกข้อความ" (Insert Text) กับคำสั่ง "สั่งการระบบ" (Action e.g. พิมพ์/ฟัง/บันทึก) มี Mental Model ต่างกัน

การแทรกข้อความ = ผลลัพธ์อยู่ที่ "ตำแหน่งเคอร์เซอร์"

การสั่งการระบบ = ผลลัพธ์อยู่ที่ "ทั้งหน้าต่าง/ระบบ"

หากนำมารวมกัน เมื่อพี่เปาพิมพ์ [ซ้ำ] เขาคาดหวังให้มันลงไปที่เคอร์เซอร์ แต่ถ้าบังเอิญกดผิดไปโดนคำสั่งสั่งการ (เช่น "เล่นเสียงซ้ำทั้งเพลง") มันจะ interrupt สมาธิการพิมพ์โน้ตทันที

3. ข้อ H (ทางค้นพบ): ไอเดียที่ 4 ที่ได้ผลจริง

ข้อ H ที่พี่เตรียมไว้ (ปุ่มบน Toolbar + เปิดด้วย [ + ข้อความสะกิด) ถือว่าครอบคลุมแล้วครับ แต่ถ้าต้องการ "กลไกที่ 4" ที่มีหลักฐานทาง UX รองรับและเหมาะกับสายพิมพ์มืออาชีพ:

💡 วิธีที่ 4: "Empty-State Search Hint / Pattern Teaching" (คำใบ้ในช่องค้นหาแบบปรับเปลี่ยนตามบริบท)

หลักการ (Ref: NNNG - Neilsen Norman Group: Help and Documentation Guidelines): การให้ความช่วยเหลือที่ดีที่สุดคือการให้ใน "จังหวะที่ผู้ใช้กำลังจะใช้งาน" (In-context guidance)

การทำงานจริงในระบบ:

เมื่อพี่เปากดปุ่มบน Toolbar (แถบปุ่มเดิมที่เขาคุ้นเคย) ให้ระบบทำการ Highlight หรือกะพริบเน้น Keyboard Shortcut Badge ที่อยู่ข้างๆ ปุ่มนั้นแป๊บหนึ่ง

เมื่อผู้ใช้เปิด Palette ขึ้นมา แต่อันยังไม่ได้พิมพ์อะไรเลย (Empty State) แทนที่จะโชว์แค่รายการคำสั่งเรียบๆ ให้มีส่วน Header เล็กๆ เขียนว่า:

💡 เกร็ดมือไว: พิมพ์ [ ในบรรทัดโน้ตได้เลยโดยไม่ต้องกด Ctrl+K

Inline Keyboard Hint ข้างชื่อปุ่มบน Toolbar: บนแถบปุ่ม 21 ปุ่มเดิม ให้ใส่ Badge คำสั้นๆ ไว้ท้าย tooltip หรือบนตัวปุ่มเลยว่า [Ctrl+K] เพื่อเชื่อมโยงว่าปุ่มที่เขาคลิกทุกวัน สามารถเรียกผ่าน Ctrl+K ได้เช่นกัน

สรุปข้อเสนอแนะเพื่อนำไปปรับใช้

เรื่อง [ Autocomplete (ข้อ A): ทำได้ แต่ต้องทบทวนเรื่อง Keystroke Passthrough ให้ดี เพื่อไม่ให้ขัดจังหวะคนพิมพ์เร็ว

เรื่อง Snippet Placeholder (ข้อ F): แนะนำให้ลองเทสการพิมพ์ภาษาไทยทับ Highlight บน OS จริง หากพบ IME Glitch ให้ถอยมาใช้ Inline Palette Input แทน

เรื่อง Accessibility (ข้อ G): อย่าลืม scrollIntoView() เมื่อกด ↑↓ เลื่อนรายการใน Listbox

เรื่องหมวดหมู่ (ข้อ D): การใช้ editorCommands.js เป็น Single Source of Truth ถือเป็น Architecture ที่ถูกต้องและแน่นหนาที่สุดแล้วครับ
