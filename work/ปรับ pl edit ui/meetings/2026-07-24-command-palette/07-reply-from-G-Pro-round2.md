### G ตอบ (หลังส่วนที่ 1/1) — OK

ยินดีครับ ในฐานะ SA คุยกับ SA เรามาเจาะสเปกกันทีละข้อ ผมจะแยกชัดเจนระหว่าง "มาตรฐาน/ทฤษฎีที่ตรวจสอบได้" กับ "ความเห็นจากประสบการณ์ออกแบบระบบ (Opinion/Experience)" เพื่อให้คุณนำไปตัดสินใจต่อได้ทันที

1. ทางค้นพบ (Discoverability) ที่มีหลักฐานรองรับ

คำตอบ: วิธีที่ 4 คือ Ghost Text (ข้อความจางบนพื้นที่ว่าง)

หลักฐานอ้างอิงที่เปิดตรวจได้: กฎ 10 Usability Heuristics for User Interface Design ของ Nielsen Norman Group (NN/g) ข้อที่ 6. Recognition rather than recall (การมองเห็นแล้วจำได้ ดีกว่าต้องนึกเอาเอง) และบทความ "In-Context Help" ของ NN/g

ทำไมถึงได้ผล: ผู้ใช้ไม่ต้องละสายตาจาก Caret (จุดที่กะพริบรอรับข้อความ) ทันทีที่ขึ้นบรรทัดใหม่ หรือพื้นที่ว่าง ให้แสดงข้อความจางๆ ว่า พิมพ์ [ เพื่อแทรกคำสั่ง... พอพิมพ์อักขระแรก ข้อความนี้จะหายไป

(ความเห็นเสริม): นี่คือ UX Pattern ที่ Block Editor ทุกตัว (Notion, Slack, Confluence) ใช้เพื่อเปลี่ยนผ่านผู้ใช้จากยุค Toolbar มาสู่ยุค Command Palette จนกลายเป็นมาตรฐานอุตสาหกรรม (De facto standard) สำหรับเครื่องมือประเภท Text-heavy

2. 2 ประตู ([ และ Ctrl+K) จะสร้าง 2 Mental Model ไหม?

คำตอบ: สร้างแน่นอนครับ และจะเป็นความสับสนที่อันตรายมากด้วย

จุดตายที่ต้องค้าน: คุณบอกว่า [ ไม่แย่งโฟกัส และ ไม่เลือกแถวแรกให้ (Enter ไหลผ่าน) ส่วน Ctrl+K เลือกแถวแรกให้

(ความเห็นจากประสบการณ์): นี่คือการทำลาย Muscle Memory หากผู้ใช้จำว่าต้องการพิมพ์คำสั่ง "ซ้ำ" เขาเปิดได้ 2 ทาง แต่ปุ่ม Enter ซึ่งเป็นปุ่มยืนยันสุดท้าย ดันทำงานไม่เหมือนกัน (ฝั่งนึงขึ้นบรรทัดใหม่ ฝั่งนึงแทรกคำสั่ง) นี่ไม่ใช่ "2 ประตูสู่ห้องเดียวกัน" แต่เป็น "2 เครื่องมือที่หน้าตาเหมือนกันเป๊ะแต่กลไกไกปืนต่างกัน"

เคสที่พัง: ลองดู Auto-complete ของ IDE ยุคเก่าที่บางทีพ่น Snippet บางทีแค่แนะนำคำ ถ้า Enter หวังผลไม่ได้ 100% คนคีย์เร็วจะหยุดใช้

ข้อเสนอ: ต้องทำให้พฤติกรรมเหมือนกันเป๊ะ หากพิมพ์ [ แล้ว List โผล่ ต้องมีแถวที่ถูก Focus เสมอ (อาจจะเป็น Exact match หรือแถวแรก) เพื่อให้กด Enter แล้วจบงานได้เหมือน Ctrl+K

3. การเรียงตอนยังไม่พิมพ์: ตายตัว (Fixed) vs ใช้บ่อย (MRU)

คำตอบ: ตายตัว (Fixed) จะแพ้เมื่อรายการยาวเกิน Viewport ของ Palette

หลักฐาน/แนวปฏิบัติที่ค้าน: Hick's Law (Hick-Hyman Law) เวลาในการตัดสินใจแปรผันตามจำนวนความซับซ้อนของตัวเลือก และ Fitts's Law

อธิบาย: คุณใช้ตรรกะของ "การจำพิกัด" (Spatial Memory - Fitts's Law) ซึ่งถูกต้องสำหรับ 2D Toolbar ที่ปุ่มอยู่กับที่ แต่ Palette คือ 1D List ที่ต้อง Scroll หากมี 30 รายการ แล้วคำสั่งที่พี่เปาใช้บ่อยที่สุด (เช่น ย้อนต้นเพลง) อยู่หมวดสุดท้ายในลำดับที่ 28 พี่เปาจะต้องกดลูกศร ↓ 28 ครั้ง หรือต้องจำใจพิมพ์ค้นหาทุกครั้ง

(ความเห็น): เส้นแบ่งอยู่ตรงที่ "มันบังคับให้ผู้ใช้ต้อง Scroll หรือไม่" หากรายการเกิน 10-15 บรรทัด (หลุดจอ) การเรียงแบบตายตัวจะกลายเป็นภาระทันที ข้อแนะนำคือ 3-5 ลำดับแรกควรเป็น MRU (เพื่อ O(1) Access) แล้วเส้นคั่น ตามด้วยหมวดตายตัว

4. เผื่อโต — ลิสต์แบนราบแตกหักที่ตรงไหน?

คำตอบ: เริ่มพังที่ประมาณ 50-60 รายการ หรือเมื่อเริ่มเกิด Search Collision (คำค้นหาชนกันมากเกินไป)

เกณฑ์ตัดสินใจ: เมื่อพิมพ์คำค้นหา 2 อักขระ (เช่น "ซ้") แล้วผลลัพธ์ที่โผล่ขึ้นมามีมากกว่า 7-9 รายการ (เกิน Miller's Law: 7±2) ผู้ใช้จะต้องหยุดพิมพ์เพื่อมากวาดสายตาหาอีกรอบ

สิ่งที่ควรเพิ่ม (มาตรฐานอ้างอิง): ระบบ Namespaces/Prefix routing (เช่น > หรือ @) อ้างอิงตามสถาปัตยกรรมของ VS Code Command Palette (อ้างอิง W3C ARIA Combobox pattern สำหรับ Multi-mode)

พิมพ์เฉยๆ = ค้นหาคำสั่งโครงสร้าง

พิมพ์ @ (หรืออักขระอื่น) = ค้นหาสัญลักษณ์ (Symbols) หรือคอร์ด

วิธีนี้ลด Scope ของ Search Pool ลง ทำให้ผลลัพธ์แม่นยำขึ้นโดยไม่ต้องใช้ UI ตัวกรอง (Filter dropdown) ที่ต้องใช้เมาส์คลิก

5. ทะเบียนคำสั่งไฟล์เดียว (SSOT) มักพังตรงไหน?

คำตอบ: (ความเห็นจากประสบการณ์การวางสถาปัตยกรรม) มักจะพังที่ 3 จุดนี้เมื่อระบบโตขึ้น:

Contextual Availability (สิทธิการแสดงผลตามบริบท): คำสั่ง A อาจกดจาก Toolbar ได้เสมอ แต่ใน Palette ห้ามแสดงถ้าไม่ได้ Highlight ข้อความ (หรือกลับกัน) ทะเบียนไฟล์เดียวมักขาด evaluateContext(currentState)

Display vs Search Data: ชื่อบนปุ่ม Toolbar ต้องสั้นกุด (เช่น "ซ้ำ") แต่ใน Palette ต้องอธิบายชัดเจน (เช่น "ซ้ำท่อนนี้ N รอบ") และต้องมี Search Aliases ซ่อนไว้ ทะเบียนต้องรองรับโครงสร้างแบบ:
{ id: 'repeat', toolbarLabel: 'ซ้ำ', paletteLabel: 'ซ้ำท่อน (Repeat)', aliases: ['วน', 'volta'] }

Action Payload: ปุ่มบน Toolbar กดแล้วจบ แต่ใน Palette อาจต้องการ Argument เสริม ถ้าทะเบียนรับแค่ String ธรรมดา อนาคตจะทำ Action ที่ซับซ้อน (เช่น เรียก Modal หรือ Call API) ไม่ได้

6. ความเสี่ยงเดียวที่ใหญ่ที่สุด ที่ทำให้มืออาชีพเลิกใช้?

คำตอบ: "Latency / UI Jitter (ความหน่วงและอาการกระตุกของการวาด UI) ที่ขัดจังหวะกระแสการพิมพ์ (Typing Flow)"

คนพิมพ์เร็วระดับสัมผัส (Touch typist) คิดแล้วนิ้วขยับทันที ถ้า Ctrl+K หรือ [ ใช้เวลาเรนเดอร์ DOM แม้เพียง 100-200ms หรือมีอาการ Layout ขยับ (Shift) นิ้วเขาจะพิมพ์อักขระต่อไปหรือกด Enter ไปแล้ว ผลคือคำสั่งไม่ติด หรือพิมพ์ขยะลงไปใน Editor

วิธีตรวจ (Metrics): วัดค่า Cancellation Rate (เปิด Palette แล้วกด Esc หรือลบทิ้งทันที) และวัด Time-to-Execute (เวลาตั้งแต่กด Ctrl+K จนถึงกดยืนยัน)

ถ้า Time-to-Execute ของพี่เปาไม่ลดลงเลยใน 1 สัปดาห์ หรือ Cancellation Rate สูงทะลุ 20% แปลว่า Palette ของคุณช้าเกินไปสำหรับเขา หรือเขาพิมพ์เร็วกว่า UI ที่ตอบสนอง
