# ส่งให้ G Pro (รอบ mid-bar jumps) — 2026-07-24

**บริบท:** ต่อจากรอบ D.C./D.S./Segno/Coda (แชต "การจัดการ Jump Marker ใน MusicXML" / "MusicXML Edge Cases").
รอบนี้เจาะ **mid-bar** อย่างเดียว. verify chip = **Pro**.

---

ผมเป็นสถาปนิกระบบ (SA) ของแอปโน้ตเพลงโบสถ์ (jianpu/โน้ตตัวเลข) กำลังปิดช่องว่าง **"การกระโดดกลางห้อง" (mid-bar jump)** ต่อจากรอบก่อนที่เราตกลง D.C./D.S./Segno/Coda/Fine ระดับบรรทัดไว้แล้ว. ผู้ใช้ยืนยันว่าหนังสือเพลงต้นฉบับมี Segno/Coda/จุดย้อน **วางกลางห้อง** (ไม่ตรงเส้นห้องพอดี) จริง จึงต้องรองรับใน v1.

**บริบทเทคนิคสำคัญของระบบผม (ต่างจากซอฟต์แวร์โน้ตทั่วไป — โปรดใช้ประกอบคำตอบ):**
- ระบบผม **ไม่มี time-signature / ไม่บังคับความยาวห้อง** เลย. เส้นห้อง `|` เป็นแค่เส้นแบ่งสายตา. โน้ตแต่ละตัวถือค่า "จำนวนบีต" ของตัวเอง. playback = บวกบีตไปเรื่อย ๆ ตามลำดับโน้ต. ไม่มีการ validate ว่าห้องครบ 4 บีตไหม.
- โน้ตทุกตัวมีพิกัด (บรรทัด, ลำดับในบรรทัด) แม่นอยู่แล้ว → ผมเลื่อนจุดเริ่ม/จบของ "ช่วงเล่นซ้ำ/รอบย้อน" ไปลงกลางห้องได้ทางเทคนิค.

**สมมุติฐานของผม (อยากให้ช่วย verify/ค้าน แต่ละข้อขอ URL มาตรฐานที่ผมเปิดตรวจเองได้ — G เคยพลาดข้อเท็จจริงโน้ต ผมจะตรวจทุกข้อ):**

**Q1 — timing ของ mid-bar jump target ตามมาตรฐาน.** เมื่อ **จุดหมายของ D.S./D.C. al Coda (Segno หรือ Coda) อยู่กลางห้อง** (partway ในห้อง) มาตรฐาน (MusicXML 4.0 `<sound segno/coda>` + นิยาม `<measure>`) บอกว่า playback ควรเล่น "เศษห้อง" จาก marker ถึงเส้นห้องยังไง? เล่นตาม duration ที่เขียนของแต่ละโน้ต **ตรง ๆ** (ไม่มีการเติม/completion เมตริก) ใช่ไหม? ขอ URL.

**Q2 — split-bar / anacrusis = notation หรือ playback responsibility?** ในการ engraving การกระโดดกลางห้องมักจับคู่กับห้อง pickup (anacrusis) โดย "เศษห้องหลังกระโดด + ห้อง pickup รวมกัน = 1 ห้องเมตริก". คำถามคือ กฎ "รวมกันได้ 1 ห้อง" นี้เป็นกฎของ **playback** (player ต้องบังคับ) หรือเป็นแค่ **convention การเขียนโน้ต** (คนเขียนต้องใส่ duration ให้ถูกเอง · player แค่เล่นตามที่เขียน)? ผมเชื่อว่าเป็นอย่างหลัง (ผมจึงให้ engine เล่นตาม duration ที่ encode ไม่ต้องคำนวณเมตริก) — ยืนยันหรือค้าน + URL/แหล่งอ้าง.

**Q3 — pitfall ตอนกระโดดเข้าห้องกลางคัน ที่ engine ที่ถูกต้องต้องจัดการ.** ผมนึกออก 2 อย่าง: (i) **accidental ที่ตั้งไว้ก่อนหน้าในห้องเดียวกัน** (เช่น `#` บนโน้ตต้นห้อง แล้วกระโดดเข้ากลางห้อง — โน้ตหลังจุดเข้ายังต้องติด `#` ของห้องนั้นไหม?) (ii) **tie/slur ที่คร่อมจุดเข้า** (marker ตกบนโน้ตที่กำลังลากเสียงค้าง). **นอกจาก 2 ข้อนี้ ยังมี pitfall อะไรอีกที่ผมตกหล่น?** (เช่น chord symbol ค้าง, tuplet, tempo/rubato ที่ตั้งกลางห้อง ฯลฯ) — enumerate ให้ครบเท่าที่มาตรฐาน/ประสบการณ์ระบุ + URL ที่เกี่ยวข้อง.

**Q4 — marker บนโน้ตที่กำลังลากเสียงค้าง (mid-sustain) valid ไหม?** จุดหมายกระโดด (Segno/Coda/Fine) วางบน "โน้ตต่อเสียง (tie continuation)" กลางเสียงลาก — มาตรฐาน/practice อนุญาตไหม หรือ marker ต้อง anchor ที่ **note onset (attack)** เสมอ? (ผมวางแผน snap ไป attack ใกล้สุด — ถูกทางไหม?)

**Q5 — UX precedent การวาง+เห็น marker กลางห้อง.** สำหรับผู้ใช้ไม่ใช่นักดนตรีอาชีพ (คนทำเพลงโบสถ์) วิธีที่ชัดสุดในการให้ **วาง marker กลางห้อง** และ **เห็นว่ามันกลางห้อง** (ต่างจากซอฟต์แวร์ที่ simplify ให้ marker ตกเส้นห้องเท่านั้น) คืออะไร? มี precedent ใน iReal Pro / MuseScore / Dorico เรื่อง mid-measure marker ไหม? ถ้าเป็นความเห็นให้เขียนกำกับ.

ตอบเป็นไทย. ทุกข้อที่อ้างมาตรฐาน/พฤติกรรมโปรแกรมอื่น ขอ **URL + ชื่อ element/attribute/ปุ่ม** ที่ผมเปิดตรวจเองได้. ข้อที่เป็นความเห็นให้เขียน "(ความเห็น)" กำกับ.
